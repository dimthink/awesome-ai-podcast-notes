# 开篇引入

AI 行业正在讲述一个极为单一的故事。模型基准测试每三个月翻倍，自主任务时长突破 17 小时，CEO 们开始公开预测"牛马时代"的终结。如果你只看新闻标题，你几乎已经可以看到那个没有人类的未来。

但 Dan Shipper 看到的，是一个完全相反的世界。

作为媒体与工具公司 Every 的创始人兼 CEO，Dan 和他的 30 人团队可能是这个星球上最"AI 浸入"的一群人——全员使用 Claude Code、Codex、OpenClaw，从工程师到编辑到销售，每个人都在用 AI 深度重构自己的工作方式。按照常理，他们应该是最不需要人的公司。

事实恰恰相反：**他们在过去一年，团队规模翻了一倍**。

这期对话的核心命题，就藏在这样一组令人不安的矛盾之中：为什么自动化程度越高，人类参与的深度和广度反而在增加？为什么拥有最多 AI 工具的人，反而在更疯狂地工作？为什么一个如此"AI 优先"的人，会斩钉截铁地说"自动化是个谎言"？

Dan 的回答不是一句鸡血口号，而是一套完整的逻辑框架。他逐层拆解了三个关键维度：**我们工作的底层架构将如何分化**（超级 Agent vs. 个人 Agent vs. 工作表面）、**SaaS 和工具层的经济逻辑将如何逆转**（用户自带 AI 的范式）、以及**人类的价值将在何处重新锚定**（品味、判断、审美——而非单纯的编码或写作能力）。

这篇文章不是对一次谈话的摘要，而是对 Dan 整个预测体系的一次深度重构。我们将跟随他的逻辑线，穿过"Agent 架构双轨化"、"CLI 已死"、"SaaS 末日不复存在"、"基准测试陷阱"等层层谜题，最终抵达那个核心反直觉点：**AI 的真正作用不是取代人，而是把所有人都变成了"经理"——而经理的工作，从来不会减少，它只会变得更大、更难、也更有价值。**

准备好了吗？让我们走进这个 AI 的悖论深处。

# 详细内容

# 一、Agent 架构的双轨制：当每个人都变成"经理"

## 1.1 超级 Agent：不是你驯化它，而是它需要你

**核心观点**

AI 代理的部署正在从"每个人都拥有自己的个人 Agent"模式，转向"一个公司级的超级 Agent"模式——因为当下的 AI 代理仍然极端依赖人类持续的关注、修正和维护。

**深度阐述**

Dan 最初对 Agent 的想象，充满了诗意。他引用《黄金罗盘》——每个角色肩膀上都有一个灵魂的化身——认为每个人都将拥有一个自己的数字恶魔，与之对话、分工、共同成长。当 OpenAI 的 OpenClaw 问世时，Every 内部全员第一时间上马，每个员工都在搭建属于自己的个人 Agent。

然后现实打了所有人的脸。

"这个东西总是在坏。你得去 SSH 进你的服务器，你得去修各种奇怪的问题……大多数人不愿意花这个时间，也花不了这个时间。"

Dan 用了一个精准的词来形容当下的 AI 代理：**"fiddly"**（需要你不断摆弄、折腾）。它不是一个开箱即用、完美自运行的解决方案。它更像是一个需要不断被照看、被"浇灌"的盆栽。

这个洞察是 Dan 整个预测体系中最根本的一块砖：**每一个 Agent 都需要一个在乎它的人类。** 一旦你丢下它不管，它就立刻退化为一个不那么好用的玩具。

于是事情走向了一个反直觉的方向。他原本预测的是人人拥有个人 Agent，但现在他看到的趋势是：**一个公司级的超级 Agent，配一个专门照看它的"前置部署工程师"（Forward Deployed Engineer）**。

**关键原话**

> "It really needs a human who cares about it... The minute you sever that connection, the agent is not really that useful anymore."
> ——[章节 1.2]
这意味着什么？他举了两个例子：Shopify 的"River"（内部超级 Agent），以及 Ramp 的那个同名神奇的 Agent。它们不是一个"所有任务都做得完美"的全能神，而是一个**被精心维护、实时调整的公共设施**。

这个预测的第二层暗线是：**Agent 的部署不是一条"从个人到公司"的扩张路径，而是一条"从公司到个人"的下沉路径。** 模型越好，Agent 越独立，"fiddly"的部分越少，个人 Agent 才会真正回归。但在此之前，中控式超级 Agent + 专属维护者，才是主流范式。

**个人感受**

Dan 说这段话时有一个非常微妙的情绪转变。从提到"每个人都有自己的恶魔"时那种眼睛发亮的孩子气，到描述"这个东西一直在坏"时那种苦笑——他亲身经历了这个理想化落地的过程，而这恰恰给了他的话以真实分量。

**延伸思考**

如果将这个框架延伸，会发现一个更有趣的结构性问题：**AI 代理的维护本身，正在变成一种新的全职工作。** 这意味着"自动化"并不是一条消灭工作的直线，而是一条生出一条分支的曲线——一边是 AI 执行任务，另一边是人类管理 AI。这条分支本身正在成长为一个全新的职业品类。

## 1.2 CLI 已死：为什么我们速通命令行时代却在加速撤退

**核心观点**

CLI（命令行界面）是 AI 代理的"美妙但短暂"的初恋期。它本质上是人类被迫妥协的产物。当 GUI 工作表面（如 Codex、Claude Co-work）成熟后，CLI 将被迅速抛弃——因为人类需要的是**可视化协作**，而非纯文字窗口。

**深度阐述**

这个预测可能是整场对话中情感冲击力最强的一个。

Dan 坦诚地回顾了一个被很多人忽略的认知过程：当 Claude Code 刚刚发布并爆火时，所有人（包括他自己）都认为"CLI 才是 AI 代理的正确形态"。因为它直接、高效，可以绕过复杂的图形界面，直接调用底层系统。

但 Dan 给出了一个极其犀利的反思：

> "We speed ran the CLI era. It was nice while it lasted, but I think CLIs are over."
这不是一次理性的"预测"，更像是一个"清醒的告别"。

为什么？因为 CLI 有一个致命的缺陷：**它是为一个人/一台机器设计的对话窗口，而不是为两个智能体（人类 + AI）协作设计的。**

**"不是 CLI 让人爱上 Claude Code，而是 Claude Code 的核心能力恰好承载在 CLI 上。"** 但这个载体并不是唯一的可能。

现在，Codex 的桌面应用出现了一个关键性功能：它把浏览器放进了 Agent 里面。Dan 描述他的日常：

> "当我写文档时，我在 Codex 里打开内嵌浏览器，进入 Proof（他的在线 Markdown 编辑器）。Codex 在看着我，我在看着 Codex，所有事情都在一个地方完成。它不仅能回应用、写文档，还能去帮我做研究、操作我的电脑——做任何我能做的事。"
这个范式转变背后有一个根本性逻辑：**人类需要的不是"A 完成工作、B 接收结果"的异步过程，而是"A 和 B 共同可视化地沉浸在同一个工作流中"。**

**关键数据/案例**

Dan 提到，他现在已经连续 10 天保持了收件箱为零的状态。"你知道我是什么样的人，这在以前是不可能的。"

他的工作流变成了：让 Codex 通过他们的邮件 Agent Cora 收集所有邮件，Codex 渲染出一个页面，他看着页面开始"独白"式地回复——"去查这个人的背景"、"去收集过去四年的相关文档"、"把回复发给律师确认"。

这个体验的质变在于：**他再也不用离开自己的工作环境去切换到另一个 SaaS 工具。** 所有工具、所有数据、所有操作，都在 Codex 的单一表面内发生。

这种模式一旦稳固，就不太可能回到 CLI。因为 GUI 的**可视性**、**可并发性**、**可干涉性**是无法被替代的。

**个人感受**

Dan 说"CLI 已死"的语气里没有傲慢，更像是一种见证者的陈述。他经历过那个狂热期，但当他亲眼看到 Codex 在一个更友好的表面复制并超越 CLI 的能力时，他果断更新了自己的预测。这种"诚实面对现实变化而不断更新预测"的姿态，本身就是一种值得学习的认知习惯。

## 1.3 双 Agent > 单 Agent：为什么你的 Agent 需要和其他 Agent 对话

**核心观点**

"两个 Agent 协同 > 一个 Agent 独立" 正在成为一个底层原则。AI 代理之间可以通过更丰富、更准确的上下文进行沟通，其效率远超人类直接与 Agent 交互。

**深度阐述**

这个点可能是整场对话中相对容易被忽略、但长期来看意义深远的一个。

Dan 提出一个场景：他的 Codex 正在和另一个公司的 Agent 交互。Codex 在交互过程中，可以传递大量关于 Dan 个人的上下文——"我们正在做什么项目"、"Dan 喜欢什么风格"、"过往的沟通记录"。这种上下文量级，是 Dan 自己手动输入永远不可能达到的。

> "Two agents are better than one."
这句话的意思不是"更多的 Agent 等于更多的效率"——而是说：**当两方的 Agent 以"智能代理-智能代理"的方式进行对话时，人类变成了一个更高层级的监督者，而非传话筒。**

举个例子：如果一个 SaaS 工具只服务直接使用人类界面的用户，那么它的 onboarding 流程需要引导用户完成大量配置。但如果它假设用户的访问方式是"用户的 Agent 来敲门"，那么用户只需要粘贴一段 Prompt 给 Codex，Codex 就会访问该工具的 Agent 界面，自动完成全部配置，并带回一个高度个性化的结果。

更有趣的是，当出现问题时，"Codex，去修一下"变成了一个真实的闭合回路。Codex 会直接以"Agent 对 Agent"的方式访问工具的排查接口，Debug 并修复。

**延伸思考**

这其实指向了一个更深层次的架构变化：**SaaS 产品不再是无生命的工具集，它们自身也需要变成"Agent-Habitable"（Agent 可居）的。** 这意味着 CLI、清晰的 API、允许批量操作、支持回滚和日志的界面。如果 SaaS 不提供对 Agent 友好的接口，它的用户——不是人类用户，而是 Agent 用户——就会消失。而这种"Agent 用户"的增长，可能是未来的主要增长引擎。

# 二、SaaS 的末日是个谎言：当用户变成了 AI 代理

## 2.1 反向思考：为什么 Agent 越多，SaaS 越值钱

**核心观点**

"AI 终结 SaaS"是一个流行的头条观点，但 Dan 认为它完全说反了：Agent 不是 SaaS 的终结者，而是 SaaS 真正的**超级用户**。SaaS 公司的收入不仅不会下降，反而会因为 Agent 的涌入而暴涨。

**深度阐述**

这一部分足以让很多正在焦虑的 SaaS 创始人松一口气。Dan 给出了一个极其朴实、却极容易被忽略的逻辑链条：

**如果每个人（包括 Agents）都在用 SaaS，并且 Agent 可以同时执行 100 个操作（比人类快且多），SaaS 的请求量会爆炸式增长。"它是个巨大的基础设施挑战和定价挑战，但对我来说，这让我极度看好 SaaS。"**

"Agent 化"导致了两个结果：

1. **用户基数扩张**：每个 SaaS 的"用户"数量会从"公司人数"翻到"公司人数 × 每个人拥有的 Agent 数量"。Dan 团队才 30 人，但他们的 Agent 数量远远不止 30——每个人可能有多个 Agent 在并行处理不同任务。
1. **计算成本转移**：当用户在 Codex 里使用 Proof 写文档，背后是 Codex 在烧用户的 Token，而不是 Proof 在烧自己的 Token。Dan 的解释是：
> "When I run the agent on that website, I'm using my tokens. I'm not using the vendor's tokens. So it puts SaaS back in its place."
这句话的意义在于，SaaS 公司原本面临一个巨大焦虑："如果用户都用 AI 写内容、浏览我的工具，我的推理成本谁来支付？" 如果 AI 成本由模型公司（OpenAI/Anthropic/Google）承担，SaaS 公司就不需要再为每一此 AI 调用支付巨额推理费。它的角色退回到提供**内容结构、数据权限和工作流**，而 AI 算力变成了用户自带的基础设施。

**关键数据/案例**

> "We still pay for a ton of SaaS and our SaaS spend is up year over year."
Every 是一个全员 AI 化、全员使用 Codex 的公司。但它的 SaaS 账单不降反升。这个案例本身，就是对"AI 终结 SaaS"最直接的反驳。

**个人感受**

Dan 说"我会在现在买 SaaS 股票"的时候，是真的在笑。这个态度是另一种反主流。当市场被"Agent 会取代一切"的恐惧笼罩，他看到的却是"Agent 会成为一切的超大甲方"。

## 2.2 全新的产品规范：双用户（人类 + Agent）的界面设计

**核心观点**

如果一个 SaaS 工具只考虑"人类使用"这一种场景，它将错过未来 18 个月最大的结构性机会。真正的差异化在于：**能不能让人类和 Agent 在同一工作空间中同时操作，互不干扰，并能展示对方做了什么。**

**深度阐述**

Dan 提出了一个前瞻但具体的产品设计挑战。如果 Agent 会成为主流用户，SaaS 的产品界面需要做出根本性调整。

- **审批台（Approval Inbox）**：Agent 可以一次完成 100 项操作（批量修改文档、批量发送邮件）。但它不能擅自执行所有操作——特别是事关重大业务决定。它需要把操作日志、变更结果汇总成一个"审批队列"，让人类快速确认。
- **回滚机制（Rollback）**：Agent 的特点是"快而大量"——但速度快并不等于正确率高。当 Agent 一口气改了 50 个文件格式，结果改坏了一个关键部分时，一个"一键回滚 10 分钟前"的能力就变成了刚需。
- **并发可见性**：当 Agent 和人类同时操作同一个文档时，Agent 做的事情必须在界面上实时对用户可见（而不只是日志文件）。这种"双光标"、"双操作流"的体验，是旧式文档工具完全无法处理的。
**关键原话**

> "The classic piece of productivity software is really mostly meant for a human to use... Now we are moving into a new paradigm where the human and the agent are on the same piece of work together. And you need to have visibility into what the agent is doing."
**案例细节**

Every 内部有一个 Agent 叫 "Claudie"（管理咨询业务的 Agent），以及很多个类似的专门的 Agent。当用户遇到了 Bug，他们不是给客服发邮件，而是："我的 Agent 给你们的 Agent 写了一封 Bug Report。" 这个 Bug Report 精确到什么程度？"这就是我操作的过程，这里是完整的复现步骤，这是代码中可能出错的代码段。" 然后收到报告的公司超级 Agent，可以直接将其转化成 GitHub Issue，派一个 Agent 去修复。这是 Dan 最兴奋的"闭合循环"之一。

# 三、基准测试的陷阱与人类价值的重新锚定

## 3.1 "资深工程师基准测试"：一个反思性思想的起源

**核心观点**

Dan 构建了一个名为"资深工程师基准测试"的内部评估工具，来测量 AI 模型在实际项目中的修复能力。这个测试的结果，暴露了当前模型的一个根本局限——不是它不能完成任务，而是它不会判断"什么才是真正重要的事"。

**深度阐述**

这个故事来自 Dan 的个人经历，也是整场对话中最"出人意料"的高潮段落之一。

Dan 用他自己的工具 Proof（一个文档/代码写作应用）独自"vibe code"（凭感觉盲写）了一个侧边项目。结果项目一上线服务器就不停崩溃。他对着 Codex 喊"去修！"Codex 尝试打了好几个补丁，结果补丁又引起了更多的问题——最后他几乎一夜没睡，还因为疯狂敲代码得了"Vibe 程序员肘"（Vibe Coder Elbow）。

最终，他请了两名资深工程师，分别独立对同一代码库进行重写。结果两个人都从头重写了整个代码库——而不是去修那几个 Bug。

这个经历让他意识到一个根本差异：**当一个标准的人类资深工程师看到一堆"vibe code slop"时，他的本能是："这根本就是一团糟啊。" 他会把系统性地推倒重来作为最优方案。而一个模型，如果你给它说"这里有 Bug，请修复"，它会真的只去修 Bug。**

于是 Dan 构建了一个基准测试：向每个模型提供完全相同的 Prompt——"这是一个 vibe code slop，如果你要从第一原则重写，你怎么做？"

**结果：**

- 所有编码模型（在 GPT 5.5 之前）： **30 / 100 分**
- 人类资深工程师：  **88-92 / 100 分**
- GPT 5.5（使用 Opus 4.7 计划）： **62 / 100 分**
62 分是巨大的进步。但 Dan 指出两个问题：

1. **当所有模型达到 90 分，这个基准测试本身就会被改掉。/当 AI 完全胜任现有任务时，评判标准会自然抬升。**
1. **模型目前达到 62 分的前提是，提供给它一个关键的 Prompt：它无法自发地决定"我应该从头重写"这种行动。它只是在执行指令。**
**关键原话**

> "What a human senior engineer does is they go look at the code base and they're like, 'This is a piece of [**]. This guy doesn't know what he's doing.' And then they say, 'We're going to have to rewrite a lot of this.'"**
**个人感受**

Dan 讲这个故事时的语气，是一个经历过"模型让我震惊"到"模型令我困惑"再到"模型给我制造灾难"全过程的人的分享。他从失败中为自己构建了一个方法论——他称之为"翻石头"——每当新模型发布，他就拿那个旧的不能解决的问题去试。AI 不能解决，不意味着永远不能解决；今天的 62 分，明天可能变成 90 分。

但这个测试的核心洞察是：**真正能区分人类与模型的，不是执行能力——而是定义问题、重新框定问题的能力。**

## 3.2 "昨天的人类能力变廉价"：这才是 AI 的真实经济效应

**核心观点**

"AI 会让所有工作消失"是一个错误的经济模型。真实的经济效应是：**模型会把"昨天已被证明过的人类能力"变廉价，从而释放人类的创造力，去探索"从来没被做过的事情"。**

**深度阐述**

这段话是全篇最核心的"金句"推导。

Dan 的表述是：

> "Models make yesterday's human competence cheap."
这是一个非常聪明的模型视角。它意味着：

- 当模型收敛所有已有的人类知识（作为训练数据）并通过廉价推理分发出去时，昨天的技能——写一封标准化的邮件、做一个基础的 Landing Page、写一段标准 CRUD 代码——原本只有少数人能做，现在所有人都能做。
- 当所有人能做同一件事时，这件事的"溢价"就消失了。它成为基础建设（commoditized）。
- **而此时，唯一具备溢价能力的是"去做别人还没做过的事"**——而这部分恰恰是模型无法从已有训练数据中获取的，因为它从未被完成过，没有数据。
这就是 Dan 解释为什么 Every 在招人、为什么他认为"AI 工作末日"不存在的原因：

> "The AI job apocalypse is not really a thing."
**延伸思考**

Dan 提出的这种"昨天能力变廉价"的经济模型，与经济学家 W. Brian Arthur 的"技术的本质"有深刻的呼应：新技术不是把旧技术变成废物，而是把旧技术"吸收为底层的模块"，从而让人站在更高层级开始创新。Dan 用更具体的工作场景验证了它。

## 3.3 "一切都在变，什么都没有变"：人类在新时代的真实角色

**核心观点**

AI 时代最特殊的认知状态，是同时感受到"一切都在变"和"什么都没有变"。基础架构（邮件、Slack、文档）还在，但每一项工作的内容、技能组合和分配方式都已天翻地覆。

**深度阐述**

Lenny 和 Dan 在对话快结束时，有一段非常珍贵的共同反思——他们尝试同时拥抱这两种矛盾感受。

Lenny 说："一方面看起来很多没有变——SaaS 还在、邮件还在，我们还是用 Slack 沟通。但另一方面，每个岗位都已经完全改变了。工程师不写代码了，PM 不写 PRD 了，设计师的设计流程被压缩到几乎不存在了。"

Dan 回应说，这就是他真实感受到的状态。

> "I'm simultaneously extremely AI pilled and very bullish on humans... And I feel like there are these things where they're going to be pretty similar to how they are, and that's probably good."
他举了一个例子来解释这种"新旧并存"：他的多数邮件现在是由 GPT-5.5 加上 Codex 写的。他对此完全坦白。

> "There's a difference between an AI generated document that's slop and not... the slop one is that it took them less time to make it than it takes me to read it, and they don't stand behind every line."
人类不是需要不再使用 AI 写东西——而是需要对自己产出的东西负责。如果一个人发了一封 AI 生成的邮件被问到具体内容却说不出来，那就是问题。但如果一个产品经理借助 AI 生成了一份优秀的策略文档，自己看完后认可，并愿意为它负责——这在 Dan 看来是完全正面的。

**关键原话**

> "If you send me an AI generated document, I think that's great. And if we talk about it and it's clear you have no idea what's in it, big no-no."

# 四、谁会赢：PM 与全栈设计师的黄金时代

## 4.1 "全栈 PM"的崛起

**核心观点**

**产品经理（PM）可能是 AI 时代最具职业优势的人群之一。** 当 AI 把"执行"的成本降至几乎为零，能决定"做什么"和"什么值得做"的人，获得了系统性的权力。

**深度阐述**

Dan 用了他内部团队的一个例子——Marcus。这是一个在 Axios 做 PM 出身的人，有非常好的产品嗅觉。他离职一年，全身心浸入 AI 工具，特别是 Cursor。他用"lightly technical"来形容 Marcus——有点技术底子，懂数据库迁移是什么，但一年前绝不可能把他当成一个全职工程师来用。

但现在的 Marcus 成了一个危险的角色：

> "He ships faster than almost anyone on the team."
原因很简单：他理解产品的本质需求，AI 帮他干掉了所有"不需要思考的执行"。他不需要再组织一个完整工程团队、写一份 30 页的 PRD、等两周的排期。他打开 Cursor，描述他要什么，Agent 实现它。他做代码审查，对产品行为做决策，然后明天就可以发布一个小功能。

PM 变强了，不是因为他们学会了写 1000 行代码，而是因为他们**学会了提出正确的 10 行**。

**个人感受**

Lenny 和 Dan 关于 PM 的对话明显有一种兴奋共鸣。Lenny 表态说"这可能是我的播客历史评分最高的一期"，Dan 回应说"这是我最反主流的一次发言"。这种轻松互动掩盖了背后的严肃结论：**当一个行业告诉你"工程师会被取代"、"PM 会被取代"、"设计师会被取代"时，真实情况可能是——拥有真实判断力的个体，会驾驭 Agent 而变得不可取代。**

## 4.2 全栈设计师：对抗 "Slop 化" 的最后堡垒

**核心观点**

**设计师是 AI 时代最大的"隐藏赢家"。** 当 AI 让所有人可以大规模生产"平庸且同质化"的内容时，真正具备视觉品味和交互直觉的设计师，拥有终极稀缺价值——"与众不同"。

**深度阐述**

Dan 指出，AI 编码的普遍问题是"所有 AI 写出来的东西看起来都差不多"。因为所有模型基于同样的数据训练，默认模式趋于平均化。而这个趋势在经济层面，意味着选择成本没有变，但分辨成本变高了。

"Slop"是 Dan 反复使用的词，意思是那种"功能上有、但完全缺少美学、没有灵魂"的产物。在 slop 泛滥的环境中，能做出"不一样"的东西的人，其价值不是线性提升，而是指数级提升。

设计师正好是这个稀缺能力的拥有者。在以往，设计师无法直接付诸实施完整的想法——他们做完设计稿，交给工程师，工程师压缩实现、剪裁细节——最后成品永远偏离原设计。现在设计师可以直接用 Codex 把交互推成 Pull Request。"他们不需要交接了。" Dan 的团队里，设计师现在直接往仓库里推 PR，不再需要工程团队出手。设计师自己把代码写出来，测试通过。

**关键原话**

> "They now feel so empowered to go build stuff. They have all these ideas to make things look amazing and these interesting interactions. And that's the exact thing that's really hard to do with vibe coding because it just all looks the same."
**延伸思考**

从经济学角度看，设计师的崛起意味着"差异化溢价"的回归。AI 降低了基础生产的边际成本，但它**不能降低"识别和创造差异化"的边际成本**——这恰恰是人类的独特空间。设计师（以及 PM）拥抱 Agent，不是"工具的使用"，而是一种能力套现。

## 4.3 "前置部署工程师"：AI 时代的新工种

**核心观点**

Dan 预测了一个正在快速成型的新角色：**前置部署工程师（Forward Deployed Engineer）**。这不是 AI 的替代者，而是 AI 的"监护人"——不是去"写 AI"的代码，而是去"维护和配置"公司的 Agent。

**深度阐述**

这里回到了我们章节一的核心："每个 Agent 需要一个在乎它的人类。"

一个前置部署工程师的工作方式和传统的软件工程师完全不同。Dan 内部有个叫 Nitesh 的员工，他就是这个角色。他每天的大量时间是——在 Slack 上和一个叫 Claudie 的 AI 代理对话。他一早打开 Slack，给 Claudie 发消息："你这里为什么做了这么笨的事？改一下配置。" Claudie 回复："好的，我改好了，你看下日志？"

这意味着什么？**"写代码"不再是这个岗位的核心技能，"配置 Agent、训练 Agent、审查 Agent 的行为、校正 Agent 的错误"才是。**

Dan明确指出，即使AI模型越来越强，Agent 独立能力越来越高，这个角色也不会消失。因为越多的Agent，意味着越多的维护需求。这是一个螺旋上升：模型进步 → Agent 数量增加 → 管理 Agent 的团队规模增加。

> "Even the big model companies have teams of people that run these agents... I don't think those teams are going away."
## 4.4 "骑乘模型"：普通人能做的唯一正确选择

**核心观点**

面对这个快速变化的时代，Dan 给出的行动指南只有四个字：**骑乘模型（Ride the Models）。**

**深度阐述**

这个建议听起来简单，但 Dan 解释的深度填补了它真正的内涵。他说：

> "The edge of AI is not in San Francisco. The edge of AI is wherever AI meets a real human doing something."
旧金山的顶尖模型团队在做模型，但他们不知道该怎么用这个模型做你每天面对的具体工作。每个普通人，在一个新模型发布时，拥有一次独一无二的机会——成为全地球第一批发现"这个模型能解决我工作中哪个特定痛苦"的人。

Dan 提倡的行为模式是：**每次新模型发布，把你之前解决不了的那个问题拿到新模型面前，重新"翻那块石头"（turn the rock over again）。**

AI 无法替代的工作，不是某类特定的职业，而是"系统性地尝试、判断、再尝试"的过程。只要保持这个循环——不停地用新模型解决自己的具体问题——你就在骑乘模型。你不会过时，因为新能力每次都会先到你的手上。

**关键原话**

> "The way to ride the models is to be curious and playful, to apply the new model to whatever it is that you care about, and to keep turning over rocks."

# 精华收获

**1. 自动化不是一个省力的封闭循环，而是一个制造新需求的开放循环。**

Dan 反复强调的核心洞察是：每当你自动化了一件事，你产生的结果需要被审查、被维护、被整合。你需要一个人来照看这个 Agent。**自动化从来不会减少对人类注意力的绝对需求——它只是转移了它的位置。**

**2. "昨天的人类能力变廉价"是理解 AI 经济学的唯一正确框架。**

如果你还在担忧"我的技能是否会被取代"，请把这个框架套用在自己身上：你的技能是否属于"过去十年中被证实且被模型数据覆盖的范围"——如果是，你需要升级到"新的人类能力"。不是放弃旧技能，而是在它之上建立起新的判断力。

**3. 产品经理和设计师是最具潜力的 AI 赢家。**

当"从无到有"的构建成本被压低到接近零时，**"做什么"和"做得不一样"** 的价值变得从未如此重要。PM 和设计师恰恰是这两个能力的拥有者。如果你属于这两个领域，不要等待被"裁撤"，而应该立刻成为公司里第一个用 Codex 写代码的人。

**4. CLI 是一个过渡态，GUI+Agent 才是终极界面。**

人类需要可视化地参与并监控 Agent 的工作流，而不仅仅是给一个 CLI 指令然后等结果。如果你正在做一个面向未来 18 个月的产品，你需要设计的是"人类与 Agent 共享"的双用户界面。

**5. SaaS 没有末日，你需要做的只是——让你的工具支持 Agent 访问。**

所有的 SaaS 公司应该闭眼买入自己股票的期权，而非套现。Agent 带来的用户数量增长会远超过人类的增长。那些无法让 Agent 易于接入的产品，才会被时代抛弃。那些门槛开放、支持日志、支持回滚、让 Agent 可以自动化大部分低阶操作的产品，会统治下一阶段的 SaaS。

**6. 基准测试=欺骗。真正的工作是"拎出对的事情"。**

你可以看到 AI 在明确任务下获得了 62 分、72 分、92 分——但它不会自发地说："现在我们已经到达天花板，让我重定义规则吧。"**这种元认知，即决定"此题已无需再答，我们该出一套新试卷"的能力，是 AI 和人类之间目前最坚实的鸿沟。**

**7. 人人都是"AI 经理"。**

如果你拥抱这个新角色，你将永远有工作。如果你的策略是躲避和等待，你将被甩在后面。**骑乘模型（Ride the Models）**——成为最早用新模型解决你具体问题的那个人。
