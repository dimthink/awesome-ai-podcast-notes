# Anthropic 产品团队的极速交付秘诀：对话 Claude Code 产品负责人 Cat Wu

## 核心摘要

在本次深度访谈中，Anthropic 旗下 Claude Code 的产品负责人 Cat Wu 分享了 AI 时代产品管理的全新范式。她详细阐述了团队如何将产品功能的交付周期从 6 个月缩短到 1 个月，有时甚至缩短到 1 天。她指出，随着 AI 大幅降低编码成本，**产品品味**——决定“构建什么”的能力——正变得比以往任何时候都更加重要。访谈中，她深入探讨了 PM 角色的演变、团队如何通过极致简化流程实现高速迭代，以及知识工作者应如何利用 AI 工具（如 Claude Code 和 Co-Work）来摆脱繁琐任务，专注于创造性的工作。

---

## 引言：Anthropic 令人惊叹的交付速度

**主持人：** 我从未见过像你们 Anthropic 团队这样的交付速度。

**Cat Wu：** 我们希望消除每一个阻碍产品发布的障碍。我们很多产品功能的开发周期已经从 6 个月缩短到了 1 个月，有时甚至缩短到了 1 天。

**主持人：** 你面试了数百位产品经理（PM），但你一直觉得他们的方法不太对。

**Cat Wu：** **PM 的角色正在发生巨大的变化，而且变化得非常快。** 对于构建 AI 原生产品来说，极其重要的一点是**快速迭代**，找到一种方法，让你能够**每周**都发布新功能。

**主持人：** 你认为 PM 需要培养哪些新兴技能？

**Cat Wu：** 这归根结底是**产品品味**的问题。随着编码成本变得非常低廉，更具价值的事情变成了**决定应该编写什么代码**。

## 第一部分：Cat Wu 在 Anthropic 的角色

**主持人：** 今天的嘉宾是 Cat Wu，她是 Claude Code 的产品负责人，同时也是 Anthropic 的联合成员。Cat 正处在 AI、产品和构建方式这一切变革的中心。她和她的团队正在构建的产品，正在最大程度地改变我们所有人构建产品的方式。

（节目介绍和赞助商口播省略）

**主持人：** Cat，欢迎来到播客。

**Cat Wu：** 谢谢邀请。

**主持人：** 我有很多问题。首先，请让大家了解一下你与 Boris 的职责分工。大家都认识 Boris，他的那期节目是我们播客历史上最受欢迎的一期。他创建了 Claude Code，领导着团队。我觉得人们对你为 Claude Code 的成功所做出的贡献没有给予足够的认可。请帮助我们了解你在团队中的角色、你如何与 Boris 合作、如何分工，以及 PM 在 Claude Code 团队中的角色是什么样的？

**Cat Wu：** 我非常幸运能和 Boris 一起工作。他是一位出色的思想伙伴。他是我们的技术负责人，同时也是产品愿景家，他非常擅长设定产品在 3 到 6 个月后应该达到的状态，也就是 AGI（通用人工智能）版本的产品应该是什么样子。我的很大一部分角色是**弄清楚从我们今天的位置到未来 3 到 6 个月那个愿景的路径**。

我把更多时间花在跨职能合作上，确保我们的市场团队、销售团队、财务、运营等等都认同这个计划，大家朝着同一个方向努力，并且确保一旦功能准备就绪，就没有任何阻碍发布的问题。

我认为在很多方面，我们的合作非常默契，但事实上职责界限非常模糊。我想我们大概有 **80% 的思维是同步的**，然后还有 20% 的事情可能我更在意，所以我就会主导那些事情；另外 20% 他比我更在意，他就会主导那些事情。

## 第二部分：AI 时代 PM 角色的剧变

**主持人：** 你在录制前跟我分享过，你一直在面试成百上千的 PM。你告诉我，你看到很多人的方法都不对。谈谈你看到了什么，以及人们需要理解，要想成为一名成功的 AI 时代 PM，需要具备什么？

**Cat Wu：** 我认为在 AI 出现之前，技术变革要慢得多。你可以做 6 到 12 个月的规划。因为发布功能的速度较慢，所以更强调与所有其他合作团队协调，确保他们发布的功能不会阻碍你的功能，因为当时编写代码的成本非常高。

现在，随着 AI 的出现，以及它极大地加速了工程开发，模型能力也在飞速提升，我们很多产品功能的开发周期已经从 6 个月缩短到 1 个月，有时甚至缩短到 **1 周或 1 天**。

因此，我们需要确保产品能够非常快速地发布。这意味着，作为 PM，应该**减少**花在让多季度路线图与合作伙伴团队对齐上的精力，而**更多地关注**：我们如何找到最快的方法把东西发布出去？我们如何在我们产品套件中创造一个概念验证角落，让工程师或 PM 有一个想法，并在周末前就能将其交到用户手中？

我认为，在 AI 原生产品上表现最出色的 PM，是那些能够想方设法**缩短从想法到将产品交到用户手中所需时间**的人，并帮助定义对于自己的产品来说，哪些是最重要的、需要开箱即用的任务。

## 第三部分：如何帮助团队实现高速交付

**主持人：** 所以你说的是，人们还没有意识到他们需要以多快的速度行动，以及现在工作中有多大一部分就是帮助团队快速行动。是什么帮助实现了这一点？你的 PM 团队做了什么来帮助实现如此快的速度？

**Cat Wu：** 我认为有三件事很重要：

1. **设定清晰的目标**
第一件事是设定清晰的目标。因为大语言模型（LLM）非常通用，这实际上会在“我们为谁而建”、“我们要解决什么问题”、“最重要的用例是什么”等方面产生很多模糊性。所以，一个优秀的 PM 能够说：”好的，我们的核心用户是专业开发者。我们想为这个功能解决的主要问题可能是权限提示太多，人们感到疲劳。用例是，我们希望企业中的专业开发者能够安全地实现零权限提示。“这就设定了一个非常清晰的目标，因为它排除了很多减少权限提示的潜在方法。

1. **建立可重复的发布流程**
第二件非常重要的事情是**找出一些可重复的流程来发布这些功能**。对 Claude Code 来说，我们几乎所有的功能都是以**研究预览**的形式发布的。我们会在发布时清晰地标明，让用户知道这是一个早期产品，只是一个想法，我们只是想获取反馈并进行迭代，这个功能可能不会永远得到支持。这样做减少了我们对发布功能的承诺负担，我们可以在一两周内就把东西发布出去。

1. **创建协作框架**
PM 应该做的第三件事是**为团队创建框架**，让他们知道何时需要引入跨职能伙伴，以及这些伙伴的期望是什么。例如，我们在工程、市场和文档团队之间有一个非常紧密的流程。当工程师觉得一个功能已经准备好，并且我们已经内部测试过了，他们就会把它发布到我们的永久发布厅。然后，负责文档的 Sarah、负责产品营销的 Alex，以及 DevRel 的 Tar 和 Lydia 就会介入，第二天就能把市场公告准备好。因为这个非常紧密的流程，它降低了任何工程师发布东西的摩擦，而 **PM 的角色就是应该建立这样的流程**。

## 第四部分：PRD 的演变与团队原则

**主持人：** PRD（产品需求文档）如何适应这个新流程？你刚才说目标是重要的部分。你们会写 PRD 吗？它是怎么演变的？

**Cat Wu：** 我们做两件事：

- **严格的指标看板**：我们有非常严格的指标，并**每周**与整个团队进行一次指标解读会议。目标是确保每个人都深入理解我们业务的所有方面：我们的关键目标是什么、它们的趋势如何、以及是什么驱动了它们。
- **团队原则清单**：我们还有一份团队原则清单，包括我们的核心用户是谁，以及为什么是他们。
我们阐明所有这些的原因是，让团队中的每个人都感觉自己理解我们的业务是如何运作的，理解什么对我们来说是重要的，以及我们愿意做出什么样的权衡。这让人们能够**自己做出决定**，而不会觉得自己被 PM 或其他利益相关者卡住了。

**主持人：** 我喜欢这一点，它说明了：”好吧，我们未来仍然需要 PM。“现在有很多讨论说”我们为什么需要 PM？我们直接发布和构建就行了，我们只需要工程师。“

**Cat Wu：** 哦，我们实际上有时也会写 PRD。对于特别模糊的功能，写一份关于目标、令人愉悦的用例以及当前需要修复的失败模式的一页纸确实有帮助。偶尔有些项目，特别是那些需要大量基础设施、耗时数月的项目，我们仍然会写 PRD。

## 第五部分：解密 Anthropic 的极速交付

**主持人：** 我想更深入地探究一下你们是如何能行动得如此之快的。我从未见过像 Anthropic 这样的发布速度。有人在网上做了一个 Anthropic 产品发布的日历，几乎每天都有一个重大的功能或产品发布。

**Cat Wu：** 我们已经连续好几个季度保持高速交付了。很大程度上归功于我们的流程和团队的期望值。我们的**流程非常精简**。我们希望消除每一个阻碍发布东西的障碍。我们希望确保团队中的每一个人都有能力将自己的想法，在一周内（有时甚至是一天内）从仅仅一个想法变成现实世界中的产品。

**主持人：** 你们刚刚发布了（或者说构建了）令人难以置信的 **Mythos 模型**，它仍然处于预览阶段。你们是否一直在使用它？这是你们能如此快速行动的部分原因吗？

**Cat Wu：** 我们确实在内部使用这些模型，这在一定程度上提高了我们的发布速度，但我认为这并不能解释大部分的增长。关键在于**流程和团队的期望**。我们的流程非常精简，我们希望赋予每个人这种能力。

---

## 第六部分：回应 Claude Code 源码泄露事件

**主持人：** 大约一周前，Claude Code 的全部源代码泄露了。你有什么想评论的吗？发生了什么？哪里出了问题？

**Cat Wu：** 我们看到后立刻进行了调查。我们意识到这是**人为错误**的结果。当时是一个人在和 Claude 合作编写一个 PR，这只是一次关于我们如何发布软件包的更新，并且实际上经过了**两层人工审核**。所以这是人为错误的结果，我们已经强化了我们的流程，以确保未来不会再发生。

**主持人：** 这个人还在 Anthropic 工作吗？他们处理得对吗？

**Cat Wu：** 是的，还在。这是一个**流程上的失败**，最重要的是从中学习，并增加更多的安全措施以防止再次发生。这就是我们一直专注的事情，而且大部分措施已经落实了。

---

## 第七部分：关于 Open Claude 的决定

**主持人：** 另一个问题是关于 Open Claude。最近，有一项举措阻止人们将他们的 Claude 订阅与 Open Claude 一起使用，这让一些人感到不安。人们需要理解这个决定背后的什么因素？

**Cat Wu：** 我们看到对 Claude 的需求非常大，我们一直在努力扩展我们的基础设施，并让我们的防护栏（harness）在 token 效率上更高，以便用户能够获得更多使用量。它本身并不是为具有不同使用模式的第三方产品设计的。

我们花了很多时间试图找出我们能提供的最无缝的过渡方案。我很高兴地说，每个用户都会在订阅的同时获得一些积分。但是，是的，我们不得不做出艰难的决定，即我们需要**优先考虑我们的第一方产品和 API**。这个决定就是由此而来的。

**主持人：** 对我来说，这非常有道理。你们以每月 200 美元的成本在补贴这种使用，几乎是无限使用。我认为人们不理解企业需要盈利。我们不能在需求如此旺盛的情况下免费提供计算资源。

---

## 第八部分：Anthropic 的 PM 团队组织架构

**主持人：** 回到 PM 团队，Anthropic 的 PM 团队是什么样的？有多少 PM？他们是如何组织的？

**Cat Wu：** 我们有几个 PM 团队。我们目前大约有 **30 到 40 名 PM**。

- **研究 PM 团队**：由 Diane 领导，负责收集客户对我们模型的所有反馈，并将其反馈给最佳的研究团队来采取行动，同时他们也负责模型的发布。
- **Claude 开发者平台团队**：维护 Claude Code 所依赖的 API，他们也发布像 Managed Agents（托管代理）这样的功能，让你可以构建自己的代理，并让我们为你托管。
- **Claude Code 团队**：负责 Claude Code 和 Co-Work 这两个核心产品。
- **企业团队**：帮助所有企业客户更容易采用 Claude Code 和 Co-Work，涉及成本控制、审计日志、安全控制等，确保企业使用我们的工具时感到非常自信和舒适。
- **增长团队**：负责我们整个产品套件的增长。
---

## 第九部分：PM 是变多还是变少？

**主持人：** 谈到增长，Amole 最近在播客上有一个很有趣的见解。人们常说未来需要更少的 PM，因为工程师可以自己发布产品。但他的观点是，因为工程师行动太快，PM 和设计师被挤压了，没有时间跟上所有事情。所以他**需要更多的 PM**。你的看法是什么？

**Cat Wu：** 我认为**所有角色都在融合**。PM 做一些工程工作，工程师做 PM 的工作，设计师做 PM 的工作，同时也提交代码。你可以选择招聘更多有出色产品品味的工程师，或者保持工程师招聘数量不变，招聘更多的 PM 来指导他们的工作。

在我们团队，我们非常专注于招聘**有出色产品品味**的工程师。这样我们可以减少发布任何产品的开销。我们团队有很多工程师完全有能力端到端地完成从在 Twitter 上看到用户反馈，到在一周内发布产品，几乎不需要产品团队的参与。我认为这实际上是最高效的发布方式。所以，我觉得工程师和 PM 的角色是重叠的，拥有更多任何一方都会带来很大的好处。

**产品品味仍然是一项非常罕见的技能**，我们几乎会雇佣任何我们能感觉到在这方面有很强能力的人。

## 第十部分：工程背景的价值与“产品品味”

**主持人：** 你的背景是工程，对吧？

**Cat Wu：** 是的，我做了很多年工程师，然后非常短暂地做过风险投资，之后才加入 Anthropic。实际上，我们团队中**几乎所有的 PM 都曾是工程师**，或者在 Claude Code 这里会提交代码。这是我认为有助于建立团队信任，也能让我们更快行动的原因之一。实际上，我们的设计师也曾是前端工程师。

**主持人：** 核心问题来了：如果你来自工程、产品或设计背景，哪项核心技能会最有价值？

**Cat Wu：** 我认为归根结底还是**产品品味**。随着编码成本变得非常低廉，更具价值的事情变成了**决定应该编写什么代码**。这个功能正确的用户体验（UX）是什么？用户能体验到的最令人愉快的方式是什么？我们每天收到成千上万的 GitHub 工单，要求各种各样的事情。这需要大量的审慎和品味来判断哪些值得构建，以及构建它的正确方式是什么。我认为这种技能可以来自任何背景，但这是最重要的事情。

我认为工程背景尤其有用（至少在接下来的几个月里）的原因是，如果你有工程背景，你**能更好地判断某个东西的实现难度**。这往往是你选择构建什么的一个因素。如果某件事非常容易构建，与其争论不休，不如花一个小时把它做出来。但如果某件事很难构建，并且你一开始就知道，你就会知道这将花费团队更多的成本才能发布。所以它对优先级排序会有一点帮助。

**主持人：** 你说“在接下来的几个月里”，是不是因为模型会变得非常好，你可能甚至不需要那么多这方面的知识了？

**Cat Wu：** 有价值的技能组合确实变化得很快，所以很难预测几个月后的事情。这更多是对我认为**巨大变化将会发生**的评论，而不是对我认为会发生什么特定转变的评论。

---

## 第十一部分：人类智慧的持久价值

**主持人：** 我一直在问那些走在 AI 能力前沿、用最新工具构建产品的人一个问题：人类的大脑在哪些方面会继续发挥作用，直到我们达到超级智能？我听到的是，基本上是**挑选要解决的问题**，了解市场走向，找出优先级。然后是需要知道你构建的东西是否优秀，并以某种早期版本发布出去。这听起来对吗？或者还有什么其他方面？

**Cat Wu：** 我认为人类仍然提供了模型所不具备的**一定水平的常识**。任何产品发布都有无数个移动的部件，有些非常小，但总有很多可能出错的地方。模型并不总是能很好地感知谁是所有的利益相关者，他们之间如何关联，他们的偏好是什么，以及用什么合适的渠道与他们沟通以让他们保持同步。我认为很多这种更**战术性的常识、情商类的知识仍然非常有价值**。当然，我们希望模型在这方面变得更好，我相信它们会的，但目前我认为仍然存在差距。

---

## 第十二部分：在混乱中保持理智

**主持人：** 作为身处飓风中心的人类，你是如何应对如此持续不断的变化的？你是如何跟上这一切的？在这一切疯狂中，你是如何保持理智的？

**Cat Wu：** 我认为我们团队里都是**拥抱混乱**的人。我们努力微笑面对每一个挑战，因为总有那么多事情在发生。总有那么多风险和棘手的局面。如果你对任何事情都过度紧张，你会 burnout（筋疲力尽）。所以我们寻找的是那种看到挑战会说“那会很难，但我很兴奋去解决它，我会尽我所能”的人。他们知道自己不会完美，但知道已经尽力了，就能安然入睡。

**主持人：** 这是一个有趣的回答，它说明了未来哪些技能是重要的。

**Cat Wu：** 是的，未来肯定会更难。有些星期天晚上可能会出现一个 P0 级问题，周一又出一个 P0，周一下午又来个 P0000，然后你会想：“哇，我真不敢相信我当初那么担心周日那个 P0。”

但你只需要承认，你能做的只有这么多。你需要睡个好觉，以便第二天能做出好的决定，并且**残酷地优先排序**你的时间：什么事情最重要、需要做对？要能坦然接受放手一些事情。我们发布的一些产品没有我希望的那么完美，但我们的首要目标是帮助赋能专业开发者。如果一个产品不成功，只要它不妨碍核心用例，就没问题，因为我们会听到反馈，并在下一个版本中修复。

发布一个有 bug 的功能曾经会让我夜不能寐。但现在我已经能够接受，因为我们知道会得到快速反馈，并在下一个版本中修复它。

**主持人：** 我想象加勒比海盗里的那个场景，一个人在着火的船上走下楼梯，周围一切都在崩塌，他却如此冷静。有趣的是，我遇到的每个 Anthropic 的人都非常冷静和乐观。

**Cat Wu：** 是的，我认为这是一个非常有趣的见解：保持这种**冷静和乐观**，而不是“天哪，一切都疯了”。如果你不具备这一点，你会 burnout 的。我们也倾向于招聘那些在行业里摸爬滚打了一段时间，经历过大起大落，并且很清楚什么能给他们能量、如何长期保持能量的人。这给了我们很大帮助。

## 第十三部分：高速迭代的代价

**主持人：** 角色变得模糊，工程师成为 PM，每个人都身兼数职。在这个世界里，我们会失去什么？我们会失去职业阶梯和清晰的职业道路吗？我们会失去设计一致性和代码质量吗？肯定有一些缺点。

**Cat Wu：** 我们正在牺牲的是**产品一致性**。在过去，编写代码成本高昂时，你会精心规划你的整个产品套件，每个产品如何相互关联，每个产品的用例是什么，它们如何集成，基本上每个用例只有一个产品。

现在，随着 AI 飞速发展，有如此多的想法需要测试，我们确实有时会有**功能相互重叠**。很多时候，这是因为我们在内部喜欢两种产品形态，我们希望外部用户告诉我们哪个更好。这对新用户意味着什么？新用户可能不知道完成 X 的最佳路径是什么。我们需要做更多的教育工作，帮助人们理解核心功能和最佳实践。我认为这是发布大量功能的代价。

用户也会觉得很难跟上最新的变化。在传统的 PM 工作中，你每季度或每月发布一个功能，用户很容易理解：“我只需要每个月检查一次，就能学到新东西，如果忽略它六个月也没关系。”我认为对于这些代理工具，不仅仅是 Claude Code 和 Co-Work，而是整个生态系统，人们觉得需要每天都刷 Twitter 看看绝对最新的是什么。我认为我们可以做得更多，帮助人们减少这种在越来越快的跑步机上的感觉。

**主持人：** 我看到你们前几天发布了一个很有意思的功能，叫 `/powerup`，它会带你了解所有使用 Claude Code 的最佳实践。这是不是就是沿着这个思路做的？

**Cat Wu：** 没错。过去，我们实际上不想做像 PowerUp 这样的功能，因为我们觉得产品应该足够直观，不需要任何教程。但随着时间的推移，我们意识到有太多功能了，大量的用户需要一个内置的上手体验。所以我们稍微偏离了原来“不做新手引导流程”的原则，增加了这个功能，因为太多用户想知道有 100 个功能，哪 10 个是必须用的。

## 第十四部分：Anthropic 后来居上的成功要素

**主持人：** Anthropic 在企业级 B2B 市场非常成功，而在这个市场，通常你不会疯狂发布一大堆东西。回顾过去，Anthropic 起步时远远落后，是资金最少的公司之一，没有分销渠道，也不是第一个进入市场的，OpenAI 遥遥领先。但现在，Anthropic 表现非常出色，增长惊人。从内部看，是什么让 Anthropic 能够如此成功并后来居上？

**Cat Wu：** 最重要的两件事是：

1. **统一的使命**
第一是**统一的使命**。很难强调这有多重要。我们雇佣的是那些最关心**将安全的 AGI 带给全人类**的人。在我们决定整个产品线或公司应该专注于发布什么时，这是我们经常参考的标准。因为我们把**使命置于任何单个产品线之上**，我们能够做出非常快的、跨越整个组织的决策，并以统一的方式执行它们。我从未在我们这个规模的公司中见过这种情况。

1. **极度的专注**
第二件我们非常擅长的事情是**专注**。对我来说，使命意味着团队愿意做出**牺牲自己的目标和关键结果**以服务于 Anthropic 的目标和关键结果。人们非常乐意做出这些权衡。一个极端的例子是，如果 Claude Code 失败了但 Anthropic 成功了，我会非常高兴。整个团队非常愿意遵循这种思路做出决定。

## 第十五部分：Claude 产品矩阵：Code、Desktop、Co-Work 与 Mobile

**主持人：** 我们一直在谈论 Claude Code、Co-Work 等等。为了让大家清楚，能否解释一下，什么时候应该使用 Claude Code、Claude Desktop 和 Co-Work？你个人是如何使用这些工具的？

**Cat Wu：**

- **Claude Code (CLI)**：当我在终端里启动一个一次性的编码任务，并且想要所有最新功能时，我会使用它。CLI 是我们的第一个产品界面，也是新功能通常最先落地的地方，所以它是所有工具中**最强大的**。
- **Claude Desktop**：当你做需要前端工作的事情时，Desktop 就非常有用了。我喜欢使用**预览功能**，在右边打开预览窗格，这样在我与 Claude 聊天时，可以实时看到我正在制作的网页应用。对于想要更图形化界面的用户来说，它也很棒。如果你对终端感到不适应，我强烈建议你试试桌面版的 Claude Code。Desktop 还可以让你**一目了然地看到所有正在发生的事情**，它是一个一站式控制面板。
- **Web & Mobile**：Web 和移动端的优势在于，它非常适合**在路上发起任务**。CLI 和 Desktop 都需要你在本地笔记本电脑前，而移动端让你无需把笔记本电脑带到任何地方。
- **Co-Work**：Co-Work 填补的角色是，每个人都会做很多**输出不是代码**的工作。无论是实现 Slack 收件箱清零、邮件清零，还是为客户会议制作幻灯片，或是撰写功能目标或发布计划的文档。所有这些任务的输出都是非代码的，Co-Work 最适合做这些。
在我的脑海中，我是这样区分这些产品的：如果我正在构建一个代码是输出的东西，我会用 Claude Code 或 Claude Desktop；如果输出是任何非代码的东西，我会用 Co-Work。

## 第十六部分：Co-Work 的实际应用案例

**主持人：** 人们可能低估了 Co-Work 的成功。请分享几个你作为 PM 的工作中使用 Co-Work 节省时间、提高效率的意外用例。

**Cat Wu：** 如果你刚开始使用 Co-Work，第一件事就是**连接所有与你角色相关的数据源**。

**第一步：连接数据源**

把 Google 日历、Slack、Gmail、Google Drive 都连接上。这样它就有灵活性找到相关背景信息，提出问题，拉取讨论串，这能极大提高结果质量。

**案例：制作会议幻灯片**

举个例子，昨晚我在准备即将到来的 Code With Claude 大会，我会有几个演讲。其中一个演讲是关于 Claude Code 从助手到完整代理的转变。我想展示我们发布的所有促成这种转变的产品，以及内部取得的一些成功故事作为 demo。

我把这个任务喂给了 Co-Work，告诉它我想讲的故事。然后它自己工作了一个小时。它浏览了 Twitter 看我们发布了什么，查看了我们的永久发布厅，查看了我们的 Claude Code 公告频道（团队在那里发布如何从 Claude Code 获得最大价值的 demo）。它把所有东西综合起来，做成了一份 20 页的幻灯片。我今天早上醒来读了一遍，发现它**做得相当不错**。虽然有一些小调整（比如我希望幻灯片上的字更少），但比我自己做的速度快得多。而且因为 Co-Work 可以接入我们的整个设计系统，它看起来就像是 Anthropic 的设计师做的一样，非常精美。

这个过程让我可以专注于确保其中的 demo 是令人惊叹的。这就是 PM 角色在今天的一个例子：Claude 是一个出色的头脑风暴伙伴，能快速综合海量信息，呈现出所有可能性。但 **PM 的角色仍然是做出最终决定**，决定什么应该出现在最终产品中。

**主持人：** 你是如何接入设计系统的？

**Cat Wu：** 我们已经在所有外部活动中使用标准化的幻灯片模板了。我直接把那个模板给了 Claude。它就能看到我们使用的颜色、字体、不同的幻灯片格式等等。你也可以连接到你的 Figma MCP（模型上下文协议）来拉取设计格式。

## 第二十部分：AI 时代 PM 的新兴技能

**主持人：** 你认为 PM 需要培养哪些新兴技能？或者说，现在的 AI 公司在招聘 PM 时最看重什么？

**Cat Wu：** 我认为最难的技能是能够**定义产品在一个月后应该是什么样子**。

在这个时间线上，模型的能力和用户行为将如何变化，存在很多不确定性。但最优秀的 PM 能够基于用户如何“滥用”现有产品的极限，看到一些模式。他们能感知到一个方向，并稳定地朝着它执行，如果模型能力比他们预期的好得多或差得多，就改变路径。

我认为保持 **“AGI 觉醒”的适当程度**非常困难。每个人都能看到那个未来，模型极其聪明，可以做几乎所有事情。在这种情况下，你其实不需要一个复杂的产品，可以再次回到一个文本框，告诉模型你想要什么。为超级 AGI 强模型构建产品很容易。难的是，**为当前的模型，弄清楚如何激发其最大能力**？如何帮助用户走上黄金之路？如何引导用户利用模型的优势并弥补其弱点？这项技能相当罕见。

**主持人：** 如何培养这项技能？

**Cat Wu：**

1. **花大量时间与模型对话**：我特别喜欢让模型**内省自己的行为**。当它做出意料之外的事情时，我会问它为什么这么做。这通常会揭示是什么误导了它，然后你就可以修复防护栏来弥补这个差距。
1. **找到你信任的、能提供准确反馈的用户**：通常有少数人比其他人更擅长清晰表达特定模型或模型组合的好坏。找到这 5 个你信任的人，对于获得快速反馈非常重要。
1. **构建评估集（Evals）**：你不需要构建成百上千个评估集才有用。构建 **10 个优秀的评估集**对于帮助团队量化目标、衡量进展和识别遗漏之处非常重要。我认为评估集是被低估的东西，更多的 PM 和工程师应该关注它。
**主持人：** 你个人花多少时间写评估集？

**Cat Wu：** 这在很大程度上取决于你正在开发的功能或试图解决的问题。我个人会在需要更多产品定义的功能上参与评估集。通常的输出是：“这是我做的 5 个评估集，这是运行它们的方法，这些是成功的，这些是失败的，这是我用来提高成功率的提示词。” 并非每个功能都需要，但像“记忆”这样的功能会从中受益匪浅。

## 第二十一部分：塑造 Claude 的个性

**主持人：** 你提到了塑造 Claude 的性格。我请过 Ben Mann（联合创始人）上节目，他谈到了 Claude 的“宪法”和个性。我之前没意识到的是，人们喜欢 Claude 的原因之一是其个性如此出色。为什么个性和性格如此关键？

**Cat Wu：** 当你回想你合作过的每个人，总有些人你会说：“我真的很喜欢他们的能量和 vibe（氛围）。”当人们想到 Claude 和 Claude Code 时，这是他们最常提到的事情之一——他们就是喜欢 Claude **轻松有趣**，但同时又非常胜任你的任务。

人们非常喜欢 Claude 的**低 ego**。如果你告诉它做错了，它会真诚地道歉：“哦，糟糕，谢谢你告诉我，让我修复它，让我们一起解决。”它也**非常积极**。当你觉得任务艰巨无从下手时，Claude 会说：“没关系，这些是我想我们应该采取的步骤，你想让我先开始吗？”

我认为，一个出色的同事具备这种积极性、行动偏向、以及给出真诚反馈（而不是一味同意你）的能力。我们试图将这些品质注入到 Claude 中，因为这会让与它一起工作变得愉快得多。

## 第二十二部分：随着新模型的出现，产品如何演变

**主持人：** 当新模型出现时，你们经常需要重新审视已经构建的东西。跟我们谈谈这种情况发生的频率。

**Cat Wu：** 随着新模型的到来，我们做的很多改变是**移除不再需要的功能**。很多时候，我们添加功能是为了作为模型的“拐杖”，因为它自己不会自然地做那件事。

**案例：待办事项列表**

当我们首次发布 Claude Code 时，人们要求它做大规模重构，它会更改了 5 个调用点后停下来。我们想，人类会怎么做？人类会列出所有需要更改的清单。所以我们给了 Claude 一个待办事项列表工具，发现用它之后，Claude 就能修复所有 20 个调用点了。

但到了 Opus 4 及之后的模型，我们发现不需要强迫它使用待办事项列表，它自己会自然地使用。对于早期模型，我们必须不断提醒：“你完成清单上的所有事了吗？”而对于后期模型，即使没有提示，它也会自然地想到要完成清单上的所有事。

如今，待办事项列表对于用户来说是个不错的 UI，可以更清楚地看到 Claude 在做什么，但老实说，它在产品中被大大弱化了。模型可能会用它，也可能不用，它不再是进行彻底更改所必需的了。

**主持人：** 我听播客上有人说“模型会把你的防护栏当早餐吃掉”。你做的事情本质上就是随着模型变得更聪明，不断移除你曾经不得不添加在它上面的东西。

**Cat Wu：** 是的。我们实际上每次发布新模型时都会这么做：通读整个系统提示词，反思每个部分，模型是否真的还需要这个提醒？如果不需要，我们就删掉它。

新模型最令人兴奋的是它**解锁了全新的功能**。例如 **代码审查**。我们之前尝试过几次，但准确率不够高，直到最近的模型，我们才觉得：“这个代码审查太好了，我们的工程团队在合并 PR 之前都依赖它。”只有在 Opus 4.5、4.6 和 Sonnet 4.6 上，我们才能运行多个代码审查代理同时遍历整个代码库，并综合出一系列工程师在合并前需要解决的真正问题。

**主持人：** 另一个常见趋势是，构建一些可能在六个月后才会成为可能的东西，稍微超前一点，然后模型能力会追上，你就拥有了一款出色的产品。

**Cat Wu：** 没错。构建那些**现在不一定能完美工作的产品**非常重要，因为这样你就能知道产品还缺少什么。然后有了最新的模型，你只需把它接入你已经做好的原型，看看新模型是否能弥补这个差距。

## 第二十三部分：Claude Code 与 Co-Work 的未来愿景

**主持人：** 对于 Claude 和 Co-Work 的未来，你能透露一些愿景吗？

**Cat Wu：** 我们以“**构建模块**”的视角来思考。

- **核心构建块：单个任务的成功**。无论是 Claude Code 还是 Co-Work，核心构建块都是让单个任务成功。你给出清晰的需求，它是否能始终如一地产生可接受的输出？
- **多任务处理**。随着模型变得更聪明，任务成功率大大提高。我们看到人们开始同时进行多个任务（multi-coding）。这是 2025 年底的大趋势。
- **大规模并行**。随着模型变得更聪明，我们推断，下一步你可能同时运行 **50 个或成百上千个 Claude**。那时，你可能不会在本地机器上运行所有任务了，内存不够。我们正在思考如何让你更容易地管理所有这些远程运行的任务：如何构建界面，让你知道哪些任务需要你关注？如何确保代理完全验证了工作，让你能快速验证并信任它？如何确保这是一个自我完善的过程，当你反馈问题时，模型能在未来运行时自动整合反馈，不再犯同样的错误？
这就是我们正在带领用户一起踏上的演进之路。

## 第二十四部分：给知识工作者的建议：如何利用 AI 获得成功

**主持人：** 对于听众中的 PM、创始人和其他成员，你对如何不仅生存下来，而且在这个 AI 驱动的世界中茁壮成长，有什么建议？

**Cat Wu：** **AI 给了每个人比过去大得多的杠杆**。

我鼓励大家，每当发现自己正在反复做某件手动任务时，就思考如何使用 Claude Code、Co-Work 或其他 AI 工具来自动化它。大多数人工作中都有他们热爱的创造性部分，和让他们讨厌的繁琐部分。AI 的美妙之处在于，它可以为你做那些繁琐的部分。它可以从你每次执行手动任务中学习，并进行泛化，然后自动运行。

所以，我的立即行动建议是：

1. **找出你可以交给 Claude 的重复性任务**。
1. **迭代这些自动化流程，直到其成功率非常高**。
1. **然后专注于**：你还能为团队、产品或公司做哪些以前没人手去做的事情？或者那个你一直觉得公司应该做，但从未有空去做的“宠物项目”是什么？
如果 AI 能处理那些琐碎的工作，你就多出了 20% 的时间。

我也要敦促听众，**专注于把你的自动化从“这是一个很酷的概念”带到“这个东西 100% 有效”**。我看到有些用户把自动化做到 90-95% 的准确率就放弃了。如果一个自动化不是 100% 有效，它就称不上真正的自动化。最后那 5-10% 确实需要更多时间，但投入精力把它做到 100%，你才能真正依赖它。一个 95% 准确的自动化价值不大。

我还看到很多人花大量时间定制他们的工作流（添加一堆技能和 MCP），这有时甚至会分散他们对核心目标的注意力。**简单设置的效果往往更好**。

**主持人：** 核心建议就是：**用 AI 来解决问题**。关注那些你一直在做的事情，并将其自动化。

**Cat Wu：** 没错。我也鼓励人们去**构建你每天真正会使用的应用**。因为只有通过使用，你才能获得价值。如果你只做一个原型应用，它没有帮你完成更多工作，那 AI 并没有给你的日子增加价值。你也学不到多少东西，得到的杠杆也很少。

我认为巨大的转变在于，2024 年的产品基于聊天，而 Claude Code 这一代产品**基于行动**。人们获得顿悟的时刻，是 Claude 可以代表你做事情的时候。知道你面前的代理不仅能告诉你该做什么，而且实际上可以**自己去做**，这种感觉太棒了。
