**Lenny**：为什么你认为 AI 和前沿模型在目前还不太擅长设计？

**Andrew**：设计更难评价，因为人类的品味部分是反馈机制的一部分。这在当前技术下仍有点遥不可及。

**Lenny**：现在的产品团队形态与几年前相比有何不同？

**Andrew**：OpenAI 的每个人都很有主动性，有很多好主意，所以每个人都在构建一切。并不是人们在做根本不同的角色或关注不同的事情，而是流程颠倒过来了。实施不再是最昂贵的部分，我敢说是品味（taste）。

**Lenny**：你觉得这种一切全能的崩溃即将到来，这就是未来吗？还是我们会继续保持角色分化？

**Andrew**：有一些事情我害怕。我听说很多公司在说我们要取消产品角色，每个人都成为构建者，然后会发生的是……

## 产品工作的颠覆

**Andrew**：作为领导者，最难的事情之一就是流程的颠覆。我现在通常认为，任何人都可以构建任何东西。如果你和模型交谈，你可以实现任何你想要的功能，这不再是软件的难点。这创造了一个环境，人们可以制造各种各样的东西。你给人们无限的 token，OpenAI 的每个人都非常主动，有很多好主意，所以每个人都在构建一切。而回顾我们长期以来使用的产品流程，情况正好相反：先是研究和构思，然后可能是原型设计，但实施是昂贵的，所以你希望通过文档、研究、原型来降低所有实施的风险。现在这完全改变了。现在我确信，对于某个急需的功能，可能有 90 个不协调的团队在并行实施和尝试。所以简短的回答是：它颠倒了。实施不再是最昂贵的部分，我敢说是品味（taste）。更重要的是筛选过程：在这 90 个尝试中，哪些是好的？我们应该如何整合？它应该是其他功能的一部分吗？切换开关应该分几段？所有这些都是。

**Lenny**：“品味”真是一个热词。我想再回到这一点。你提到的 90 个原型很有趣。为了确保我理解正确：在 OpenAI 有一个想法，过去人们写文档，现在人们直接创建原型，导致有很多原型，然后团队从中选择方向。是这样吗？

**Andrew**：这种情况很多。很多产品领导者说 PRD（产品需求文档）已死，原型为王。我其实并不完全相信。因为实施变得廉价，很容易直接跳到原型，尤其是非工程师。但我也注意到，对于工程师来说，他们很容易写很多不值得读的文档。如果实施很丰富，那么选择正确的媒介很重要。如果目的是产品清晰度，可能需要文档；如果目的是让用户试用以测试交互模式，则是原型。但关键是，现在选择媒介很重要。

**Lenny**：我想到一个词“原始标记”（primal mark），艺术家创作时第一笔决定了后续。所以有时候原型可能是错误的第一步，因为你会被这个原型所限制。你似乎在说，在某些情况下文档仍然有用。

**Andrew**：是的。过去，媒介隐含了流程的阶段。如果你看到的东西看起来像生产应用，那意味着它处于流程后期，假设已经被验证。现在这些东西分离了。所以我们需要清楚这个原型或文档是在做什么，因为你不希望过度锚定在一个探索性的原型上，但它看起来太像生产版了。

**Lenny**：当谈到好的品味时，具体是什么？是决定投资哪个方向，还是判断这个是否可以发布？人们听到这个词，不知道自己有没有好的品味。它具体表现为什么？

**Andrew**：这是一个很好的问题。有来自 Linear 的产品负责人的推特说人们过度强调品味的审美方面。他用 Paul Graham 举例，说 Paul Graham 显然有很好的品味，但他穿工装短裤。所以品味有很多层面。有审美部分，也有系统思维部分：如何适应整体，未来方向是什么，如何呈现。有很多是更广泛的上下文。当然也有关于交互动画是否合适等。我可能太注重这些了。但真正的品味问题是：如果我们能构建任何东西，那么目标是什么？我们如何达到？

## 设计过程：已死还是重生？

**Andrew**：我们可能有很多一致。设计过程本身我已经不支持了，在 AI 之前就不赞成。我在创业时不喜欢那种案例研讨式的流程。过去的流程假设实施昂贵，所以必须穷尽问题空间和解决方案空间再实施。后来有了 Figma 和 Origami，我们可以快速模拟，但问题是我们现在可以把所有实施都拉进来。所以我们要么使用原型，要么使用简化版产品（如 baby Codex）来探索。所以，说设计过程已死，既对也不对。如果拘泥于具体工具和具体步骤，那确实死了，但抛弃整体流程和阶段划分则是错误的。

**Lenny**：你的背景涵盖了所有职能：工程师、设计师、产品经理、创始人，现在你监督桌面应用，设计不归你直接管吗？

**Andrew**：这取决于星期几。我们紧密合作，坐在一起，但汇报线不固定。

**Lenny**：那么 Codex 的设计过程是怎样的？

**Andrew**：有很多关于角色崩溃的说法，我们并没有看到完全的角色崩溃。但在 Codex 团队中，角色重叠更多。设计师写代码，产品经理写代码等。我们描述为：团队之间的重叠更多，每个人的范围不再由边界定义，而是由工作重心定义。如果平均来看，设计师也有不少代码编写。Codex 的设计过程深受“dog fooding”（吃自家狗粮，即使用自己的产品）循环的影响。我们经常为了改进产品而不改进自己的流程，以便在实际使用中发现问题。这种状态不舒服，但每周都在变化。

**Lenny**：我很喜欢这个观点：你的角色是你花时间做的事情的平均值。如果是 PM 工作就是 PM，如果是工程就是工程师。OpenAI 是第一个称员工为“技术成员”（Member of Technical Staff, MTS）的公司吗？

**Andrew**：不是，可能最早是 Xerox。但在研究型公司中很常见。

**Lenny**：它似乎预示着角色的融合。你认为长此以往，我们会走向所有人都是多面手，还是继续角色划分？

**Andrew**：我担心断然取消角色的概念，因为它可能抹杀专业领域和最佳实践。很多公司说取消产品角色，大家都成构建者，但结果是产品管理的整个知识体系被抛弃。我认为“这不是你的领地”的边界可以消失，但需要平衡，不是每个人都能做所有事。每个专业都有技能成分。工程师往往认为其他角色只是随便混混，但事实并非如此。现在更重要的是，切换角色更容易了，更容易学习最佳实践，不再将角色效果与使用特定工具绑在一起。但人们经常夸大这些变化。

**Lenny**：Codex 团队有多少人？工程师、设计师、PM 的构成。

**Andrew**：每次有人问，我都说在10到几千之间。因为 Codex 是公司所有工作的结晶。但我们有两位数的工程师，设计团队大约一半，产品人员几个，但产品很分散。团队有很多前创始人和有品味的人。

**Lenny**：你提到“区域联防”（zone defense）这个概念来比喻产品工作，能展开说一下吗？

**Andrew**：我和 Alexander 多次讨论这个比喻。如果两个产品人员工作太近，通常不是好信号。你想作为产品人员找到空白区域，特别是在新世界中，策展和引导很重要。因此需要品味制造者从构思到成品全程引导。所以你需要覆盖整个公司范围。你分散开来，填补空缺，并雇佣有产品意识的工程师，这样不需要大家都集中评审。

**Lenny**：我注意到现在最有价值的人是能将想法从构思到实现，且有品味判断好坏的人。这是你招聘和判断人才的标准吗？

**Andrew**：是的。当前核心就是这样。同时 IC（独立贡献者）和管理也融合了，IC不再是逐字符敲代码，而是管理代理和协作。管理者也在不同粒度管理。我寻找有领域掌握和品味的人，能在无限内容中识别信号。

## 规划与时机

**Lenny**：你提到规划，随着发展节奏，现在很难做路线图规划。你们如何规划？多久看多远？产出是什么？

**Andrew**：我们并不革新。基本思路是：越是短期的需要越详细。我们不计划9个月，因为任何精确度都是虚假的。在应用端，我们只需要知道模型的能力时间线。在我的上一家公司，我们开始根据模型能力驱动功能，产品流程变成了：列出所有感兴趣的功能，原型化所有，决定哪些现在可行，其余等待模型进步。因为功能的成败取决于模型是否足够聪明，而不是功能形态。

**Lenny**：Codex 应用的故事很有趣。如果你在11月发布2月的版本，可能会失败，唯一区别是模型。所以策略是构建现在不工作但将来会工作的东西？

**Andrew**：是的。我们有很多这种探索。但要明确，这是设计过程的早期阶段。人们容易有肌肉记忆，以为写了代码就应该发布。我们之前有在应用内浏览器的尝试，从 Atlas 到 Operator 再到 Codex，同一个功能随着模型智能的提升而效果不同。所以不要固执地认为“这不工作所以是坏功能”。另外，有时野心太大，形式与智能不匹配。比如最初的 Codex web 版本太宏大，但模型不够强；而 Claude Code 完全是本地、不是代理型，反而更匹配当时模型的能力。所以这可能要尝试多次。

**Lenny**：这确实是这个播客的一个主题：构建那些尚未奏效但会随着模型变好而奏效的东西。还有另一种雄心的主题：对你所承担的事情更加雄心勃勃。这是你的方法吗？就是构建一堆可能还不工作的事情，留着等模型赶上来？

**Andrew**：是的，我们有很多这样的做法。但挑战在于要清楚处于设计过程的哪个阶段。人们有肌肉记忆——我写了代码，所以应该发布。不，这意味着你有一个可以测试未来模型的工件。我们之前尝试的应用内浏览器就是一个例子。从 Agent 工作到现在的版本，同样的功能因智能不同而结果完全不同。所以不要固执。另外，也要注意不要过于雄心勃勃，形式与智能不匹配。最初 Codex web 版本野心太大，而 Claude Code 更适用。所以产品可能需要发布六次才能奏效，形状可能根本没变。

**Lenny**：所以有很多变量：模型时间线、用户接受度、团队能力。是否鼓励团队更大胆？

**Andrew**：这是核心挑战。产品一旦存在，人们很容易聚焦于小毛病并优化。但我们也有自下而上的探索文化，因为可能未来的努力会颠覆现有产品。所以需要设计一个能同时处理颠覆和改进的流程。

**Lenny**：展望一下，AI 如何改变构建产品的方式？从手写代码到 AI 写代码，再到编码成为指导 AI。现在有代理和循环，最前沿的团队怎么操作？

**Andrew**：循环已经是上周的事了。现在问题是代码是有监督还是无监督编写。我们在探索自动开发，比如夜间垃圾收集。但模型通常增加复杂性，需要删代码。要使开发完全自动化还不行，还需要教会模型构建正确的抽象。我们还没有到“睡觉吧，让应用自己成长”的地步，但我们在努力。

**Lenny**：你认为会走到那一步吗？

**Andrew**：我不知道，我不说不。

**Lenny**：你现在作为产品/工程负责人，如何在工作中使用 AI？有什么人们可能不知道的用法？

**Andrew**：我觉得我有世界上最好的工作。当初开发 Codex 时，目标是让我自己能用它写代码。现在我的角色转变了，需要做更多产品发现和协调，所以我开始用 Codex 来做这些。例如，我每天早上会收到一份简报，汇总所有 Slack 渠道的关注事项。我还可以回复让它给我五个问题等等。我会设置自动任务：遍历 Slack，按类别关注重要事情。第一次可能需要调整，但可以用对话微调。

**Lenny**：怎么设置？流程是什么？

**Andrew**：我做一个自动化任务或定时任务，告诉它哪些渠道重要。然后运行，如果需要调整，我可以直接说“下次运行时注意这个”或“减少某个工作流的权重”。它会更新通知方式。但这是我自己设置的，因为这是产品发现的一部分。对于普通用户，我们还需要简化。

**Lenny**：人们没有意识到你们的应用可以像 Open Claw 那样工作？比如设置每天早上检查某件事并汇报。

**Andrew**：是的。如果你不知道怎么设置，可以直接问应用，它会帮你设置。可能还需要连接器。最不济的是，如果连连接器都没有，它可以用计算机控制（computer use）来操作。比如我要建一个过滤垃圾邮件的应用，需要去 Google Cloud Console 设置 API，我用计算机控制让它去做。

**Lenny**：这个设计连接器与计算机使用的决策很微妙。

**Andrew**：是的，我们观察用户的个人工作流，从中提炼共性，然后让这些共性成为产品的一等功能，比如记忆（memory）功能。

**Lenny**：关于浏览器使用，有人预言我们会开始在 Codex 内运行 SaaS 应用，而不是开 Chrome。你觉得这会是方向吗？

**Andrew**：我们尝试过多种浏览器形状：Operator、Tragic、Atlas、Chrome 扩展，现在有应用内浏览器。我们学到很多。浏览器应该只给代理用还是让用户当作日常浏览器？这有很多权衡。键盘快捷键等一致性问题很难。这个应用需要同时满足从初学者到极客的各种用户，挑战很大。

**Lenny**：那么 Codex 的愿景是什么？一年、两年、十年后什么样？

**Andrew**：我们最初有 CLI，然后决定做应用。我们不确定，但相信这是一个工具。在内部 dog food 时，发现不仅工程师，连市场、财务、法务都在用，他们认为 Codex 努力很有前途，甚至建议将 Codex 能力加到 ChatGPT 和 Atlas 里。但问题是，即使其他应用更适合他们，他们也不肯离开 Codex 应用。所以开发者工具和通用知识工作工具不是二选一。我们相信我们构建的形状是正确的，可以深入垂直领域，如金融、法律。通过建立正确的扩展原语和通用模型，你可以做任何事。这就是我们思考的方式。

**Lenny**：所以是“超级应用”的概念？

**Andrew**：有人这么说，现在我得整天听这个词。但想法是作为一个基地，可以在应用内完成一切，也可以打开其他应用（如 Excel）。应用可以通过连接器或计算机使用与它们交互。比如我们有一个故事，摄影师 Brent 用 Codex 编辑视频，Codex 自己构建了一个 Premiere Pro 扩展来操作。所以有两个模型：一是让 Codex 与已有工具无缝交互，二是让 Codex 打开 Web 应用并增强它。我们同时在做。

**Lenny**：我很喜欢这个 Premiere 的故事，它说明可以更大胆地尝试 AI 能做什么。

## 失败角落

**Lenny**：我们现在进入一个固定环节——“失败角落”。很多人看到你似乎一切都顺利，但人们可能看不到你失败的时期。你能否分享一个职业生涯中失败但学到了重要一课的故事？

**Andrew**：你这么说让我觉得这是我第一次感觉自己没有在失败。我曾经是一个创业公司的创始人，最终几乎是将公司零散出售。那几年非常挣扎，是一个高度监管的领域。后来又去了另一家初创公司，尝试 AI 工具，同样是在受限制的行业，总是屡试屡败。所以对我来说，失败是常态。有时候只是时间点、技能、激情和市场的吻合。现在我们用 Codex 项目与 ChatGPT 结合，也有过无数微失败——我们认为形状是对的，但在 Slack 里有 2000 条消息说我们多愚蠢。这正是我热爱 OpenAI 的地方：人们会直接告诉我们。产品之所以好，是因为它经历了内部这样的循环。我大概失败了 10 到 15 年才走到这一步。我仍然对现在的一切感到惊讶。
