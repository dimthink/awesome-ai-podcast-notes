## 开场：回顾与预言

**主持人 Lenny：** 上次你上这个播客时，你有一个热门的观点，说人们低估了 **Claude Code（Anthropic 的编程代理工具）**。你真是对得不可思议。这一集我们要聊聊你还有哪些其他预测。

**Dan Shipper：** AI 导致的失业大潮并不会真的发生。我非常非常看好产品经理和全栈设计师。

**Lenny：** 你们在过去一年里员工数量翻倍了，这对于一家如此拥抱 AI 的公司来说，并不是人们预期会看到的。

**Dan：** 我同时极度相信 AI，又极度看好人类。**自动化是一个谎言。每个 AI 代理都需要一个人类。** 我们现在有这么多自动化、这么多 AI，可我的工作反而更多了。

**Lenny：** 创造力，感觉会变得越来越有价值，这样才能从人们不断推出的各种“垃圾”中脱颖而出。

**Dan：** 模型做的事情，本质上是让昨天的人类能力变得廉价，因此它就被商品化了，不再有价值。人类要做的是：“看，我们有这么多昨天冻结的人类能力，我要怎么用它们来创造一些新的、有趣的东西？”

**Lenny：** 你对工作方式会如何改变有怎样的预测？

**Dan：** 工作会在两个主要方向上分化。**第一，每个人至少会有一个可以委托工作的 AI 代理；第二，你大部分工作实际上都会在你的电脑上完成，环境就像 Codex（OpenAI 的编程助手）或 Claude Co-work（Anthropic 的协作工作区）那样。**

**Lenny：** 你预测的是，SaaS 工具将运行在 Codex 或 Claude Code 内部？

**Dan：** 我觉得“SaaS 末日”的说法很蠢。我现在就会买 SaaS 股票。AI 代理的作用是增加 SaaS 的用户数，而不是消灭它。

**Lenny：** 很多人开始转向命令行（CLI），试图在终端里工作。

**Dan：** 我们已经快速经历了 CLI 时代。它很好，但我觉得 CLI 已经结束了。

## 第一部分：预测的未来——如何预见？

**Lenny：** Dan，感谢你的到来，欢迎回到播客。

**Dan：** 谢谢邀请。总是很愉快。

**Lenny：** 上次你来时，你有一个几乎是随口说出的热门观点，说人们低估了 Claude Code，尤其是将 Claude Code 用于非工程工作——比如修复文件、整理硬盘等那些大家没想过的用途。一年前没人讨论这个，你完全正确。之后他们开发了 Cohere，正是基于这个具体想法：将 Claude Code 用于非技术工作。Codex 也在进入这个领域。我觉得这已经是 Anthropic 过去一年成功的重要原因——让非技术人员使用这些东西。所以你在这一点上非常超前。我甚至写了篇 newsletter 文章来扩展这个想法，问大家如何将 Claude Code 用于非工程工作，收到了很多例子，那成了我第二受欢迎的文章。很明显，你对未来方向有独特的洞察。所以这一集的主题是继续聊聊你预测的未来，以及产品开发者会面临哪些变化。首先请你简单介绍一下你和你的团队是如何运作的，这给了你这种独特的视角。

**Dan：** 谢谢你的介绍。关于预测未来，我们在 Every 的做法是：不去凭空猜测，而是共同活在未来的场景里。Every 的每个人都是 AI 早期采用者。我们现在大约有 30 人，一年前是 15 人，规模翻了一倍。我们有工程师、设计师、作家、编辑、销售、客服。每个人都有些“喜欢探索、喜欢实验、充满好奇、全身心投入 AI”的特质。我认为这创造了一个未来的小口袋，我们都活在里面，比别人走得更远。因为在其他公司，人员构成不一，有早期采用者，也有中间派和反对者。另外，由于我们的角色——评测模型、担任一些 AI 品味引领者——我们能在模型发布前就接触到它们。所以我们能进行 alpha 和 beta 测试，在一定程度上帮助引导方向，这非常酷。所以我觉得，与其说预测未来，不如说创造这样一个环境，然后注意观察正在发生的事。核心还有一点：把观察到的东西写下来。把未来说出来，某种程度上会让它成为现实，对自己、团队和所有读者都是如此。

关于 Claude Code 的例子就很自然。Claude Code 发布时我们就试用了——那是我们的工作，我们尝试所有新模型公司的新东西。当时它还有点早。但大概在 Sonnet 3.5 或 Sonnet 3.7 时，我们在做内部评估，然后发现：“天啊，这太疯狂了。他们去掉了代码编辑器。”从那时起，我们就开始转向一个所有人在终端用英语和电脑对话的时代。所以我能看到：“哦，这个趋势开始了。”因为我的工作有一部分就是摆弄新东西，于是我又想：“我能不能把它用在我的写作上？怎么用？”然后它就这么展开了。你会觉得：“嗯，还没完全成熟，但对我来说已经很有用了。”我们内部有一个“伸手测试”：你早上醒来会不会自然地想去用它？

**Lenny：** 这种组合很棒：使用最新工具，并且能敏锐地意识到什么才是新奇、不同、有趣的。因为你必须写出来，所以这形成了完美的配方。

## 第二部分：工作方式如何改变

**Lenny：** 我会把这次对话分成三个预测类别：第一，工作方式在未来几年将如何改变；第二，我们从事的工作的形态会是什么样子，如何变化；第三，谁会在这种未来中最成功 / 你现在应该做什么才能成功。

**Dan：** Lenny，我只有一个要求：一年后我们再来，你打分。

**Lenny：** 我们评分。好，现在我们开始。你对未来一年工作方式的改变有什么预测？

**Dan：** 我有个最喜欢的视角：如果看一些基准测试，比如衡量模型能自主执行任务多长时间的 benchmark，最新模型可以达到 17 小时 50% 的准确率。这很疯狂。模型进步是指数级的。但我的感受是：一年后回看，我们会发现人类实际上还有更多工作要做。这是一个有趣的悖论。我的主要预测是：工作将在两个主要方向分化，就是 AI 代理的使用方式。

**第一，每个人（至少每个公司）会拥有至少一个可以与之对话、可以委托工作的 AI 代理。** 这就像 OpenAI 的 OpenClaw 那样的东西。

**第二，你大部分工作实际上会在你的电脑上通过 Codex 或 Claude Co-work 这样的环境来完成。** 这种环境会成为你完成所有工作的“操作系统”——无论是邮件、文档等等。

我想按顺序聊聊这两点。

### 2.1 可委托的 AI 代理：从个人代理到超级代理

**Dan：** 第一点是你会拥有可以委托工作的代理，很可能在 Slack 里，但也可以在任何地方。有趣的是，这个架构会是什么样子？每个人一个代理？每个团队一个？还是只有一个专门化的代理？最初 OpenClaw 发布时，我们内部每个人都用了，我坚信会是“每人一个代理”的世界，代理会变成你的一部分——就像《黄金罗盘》里的灵神一样。但后来我完全改变了看法。我认为现在的模式会是“超级代理”——整个公司一个代理。Shopify、Ramp 都已经有了。我发现个人代理的问题在于：设置和维护太费劲了。会频繁出问题，需要 SSH 登录等等，大多数人不想或不能花这个时间。**一个 AI 代理要想有用，它需要一个关心它的人类。** 一旦你切断了这种联系，代理就没用了。所以模型开始转向公司一个代理，由一位“向前部署的工程师”负责确保代理为全公司工作。随着模型变得更独立，个人代理会重新出现，但现在是从上到下，先有一个顶层代理，然后逐步细分。

**Lenny：** 所以你的预测是公司会有一个超级代理，每个人都能和它对话。

**Dan：** 对。

### 2.2 全新工作界面：Codex / Claude Co-work 成为操作系统

**Lenny：** 那另一个工作界面呢？Codex 和 Co-work。

**Dan：** 这个太酷了。Anthropic 意识到，如果把 AI 代理放在你的电脑上，它就能访问你拥有的一切。它用终端，因此有超级权限。它们真的理解如何用终端，因为网上有大量训练数据。这创建了一个强大的编码范式。当人们还在把编码模型当结对编程伙伴时，它们已经成功实现了“代理在你的电脑上”。后来大家发现，一旦有了能在电脑上构建任何东西的编码代理，它实际上能用于任何类型的工作。

然后 Open AI 也推出了 Codex 桌面应用。它吸取了 Anthropic 的经验，直接从 Claude Code 跳到 Co-work，又进一步。我觉得现在这就是一场赛马。Codex 是我目前的日常工具。我用它做几乎所有事：写作时，我在 Codex 里打开内置浏览器，登录在线文档编辑器，让 Codex 运行并看着我。它不仅能回复和写作，还能去研究、能用我的电脑做任何我能做的事。我已经连续 10 天收件箱清零了——这太疯狂了。因为我用 Codex 配合邮件代理 Cora，它会收集所有邮件，渲染一个页面，我只需对着它说话。

**所以，更好的模式不是把 AI 放进浏览器，而是把 AI 代理放在你的电脑上，再把浏览器放进 AI 里。** 这种组合目前还很少见，但一年内会变得非常普遍。

**Lenny：** 这比听起来更深刻。你预测的是，不是 AI 被嵌入 SaaS，而是 SaaS 工具会运行在 Codex 或 Claude Code 内部。

**Dan：** 对，这是很重要的二阶效应。当我在代理里用 Proof（一个在线 Markdown 编辑器）或其他网站时，代理能访问该网站和我电脑的一切。我用自己的 token，而不是 SaaS 提供商的 token。这就把 SaaS 放回了它应有的位置：你要为代理提供一个友好的界面。一旦做到这点，你就不需要再操心在自己的产品里内建 AI 了。例如，Proof 的用户不需要我为 token 付费，他们自带 AI。这会改变 SaaS 公司的商业模式。

**Lenny：** 那么 Cursor 呢？

**Dan：** Cursor 也看到了同样的趋势，但我觉得他们目前更明确地选择了为程序员服务的路线。这可能限制他们在更广泛知识工作领域的发展。但每个模型公司都意识到，拥有一个 harness（框架）来最大化模型潜力非常重要。像 Cursor 被 SpaceX 接近收购，说明大家都在意识到不能只做模型部分。最终这个 harness 能做任何类型的知识工作。

**Lenny：** 对于那些非 OpenAI 或 Anthropic 的产品公司，如果这个预测成真，他们的产品会慢慢被用到这些代理内部。你建议他们现在做什么准备？

**Dan：** 我会建议为这种模式做准备。传统的生产力软件（如 Slack、Word、PPT）大多只为人设计。现在人们在用 CLI，让代理独立使用。但我们正在进入一个新范式：人类和代理在同一个工作上协同。两者都需要相互可见，来回交流。这样的软件会非常不同。比如，Proof 不需要很多格式化功能，因为代理可以做；同时需要新的交互方式，比如代理能同时做很多事情，如何展示给用户？你需要批准功能、收件箱摘要、日志和快速回退。基础设施也要不同，因为代理可以在几秒内发送数十亿请求。这正是 GitHub 现在遇到问题的原因——用户数暴涨其实是代理在提交代码。

这种新世界的例子：当用户有问题时，他们的代理会发送 bug 报告，比人类写的详细得多，包括复现步骤、甚至推测代码位置。然后我们收到后变成 GitHub issue，再派代理去修复。这种闭环非常酷。

### 2.3 CLI 结束了吗？

**Lenny：** 很多人正转向 CLI 和终端工作。你的预测是不是意味着人们会离开 CLI，回到代理并行的方式？

**Dan：** **CLI 结束了。** 我们快速跑了一遍 CLI 时代，它很不错，但 GUI 的出现是有原因的。在 GUI 里你能得到同样的好处，而且体验更好。对于非程序员尤其如此。甚至我们的技术人员现在大多数也不再主要用 CLI。他们用 Codex、Claude Code、Cursor 这类工具。

### 2.4 两种工作模式总结

**Lenny：** 回到大局，两种模式：一是通过 Slack 和公司超级代理聊天执行任务；二是在你自己的电脑上运行 Codex 或 Claude Code，所有工作都生活在那里面。这会是一个巨大变化。

**Dan：** 还有一点：我绝对不是“代理至上主义者”。我们会有很多不同的代理。两个代理比一个好。当我的 Codex 与另一个代理交互时，它能提供比我输入更丰富的背景信息。这种加速效应很有价值。

**Lenny：** 关于 SaaS：你提到用户在用代理访问 SaaS 时用自己的 token，这对 SaaS 商业模式影响很大。

**Dan：** 我认为这实际上可能拯救他们的利润率。现在大家都在急着加 AI 功能，觉得代理是主要交互方式，但这需要耗费 token。而我认为一旦我把 Codex 或 Co-work 作为主要工作界面，我仍然需要 SaaS。所以我现在就会买 SaaS 股票。**SaaS 的末日是假的。** 代理实际上增加了 SaaS 的用户数。我们内部所有人在用代理的同时，SaaS 支出依然年年增长。代理用的 SaaS 产品量会激增。因此我非常看好 SaaS。

## 第三部分：工作的形态会如何变化

**Lenny：** 第二部分关于工作形态的预测。

**Dan：** 一个明显变化是：拉取请求（PR）的数量激增。顾问、运营、编辑等非技术角色也开始创建 PR。之前只有技术人员能做的事，现在更广泛的人能做了。这给另一端带来了压力——如何处理所有这些新代码。例如，OpenClaw 每天收到数千个 PR，然后他们启动 5 万个 Codex 实例去筛选和合并。这会越来越普遍。难点变成：合并哪个 PR？如何保持整体一致性？怎么决定删什么？

**Lenny：** 我听到很多人说现在每个人什么都能做，导致困惑：“我到底该做什么？”你觉得呢？

**Dan：** 这确实是个问题。但在 Every，大家普遍是多面手，喜欢涉猎各种事情。可能会逐渐稳定下来：营销人员仍然做营销，即使他们会碰网站——那已经是营销的一部分了。另外，一些新职位正在出现，比如“向前部署的工程师”。每个代理需要一个人。模型公司内部也有团队专门跑这些代理。随着模型能力增强，代理数量增长，但人类依然在管理它们。这看起来像是一个特定类型的人：负责确保代理正常运行、做正确的事。

**Lenny：** 所以“工作末日”暂时不会来。

**Dan：** 对，我同时极度 AI 上瘾，又非常看好人类。

### 3.1 更多审核，更多系统构建

**Lenny：** 你描述的还包括：更多工作变成审核其他人的输出（AI 或人类），以及 babysitting AI 代理。

**Dan：** 我更想说是“构建系统”而非“babysitting”。向前部署的团队是在建设一个整个系统，让知识不足的人也能安全使用。这对数据科学团队也一样：如果有个数据科学代理处理所有基础查询，数据科学家就能专注更难的问题。

### 3.2 哪些角色变化最少？

**Lenny：** 我一直在想：哪个产品技术角色目前变化最少？工程师全在用 AI；产品经理很多工作被替代；设计流程已死；数据科学完全不同了。你觉得呢？

**Dan：** 有趣的是，CEO 和投资者似乎仍可以很“可选”地用或不用这些工具。但实际上你的公司能走多远取决于 CEO 对 AI 的直觉。不过目前很多 CEO 觉得“那是下面人做的事”。短期内他们可以保持工作模式不变。销售可能变化也小，虽然顶端的 BDR 可以用 AI 做线索开发，但核心工作类似。客户支持则彻底变了。

### 3.3 AI 生成的内容会变得更受欢迎

**Dan：** 另一个趋势：我们会在文档和邮件中读到更多 AI 生成的内容，并且会喜欢它。在代码中我们已经接受了 AI 写的计划文档。在 Every，我们 2025 年底的季度规划全用了 Notion 代理。每个团队与代理对话，代理询问过去一年的情况、目标、指标，然后生成质量极高的战略报告。我发现，AI 生成的东西有区别：好的 AI 文档是关键，但如果用户对内容一无所知，那就是大忌。我认为对于策略文档，GPT-5.5 在良好指引下写出的内容，比大多数人自己敲出来的好得多。大多数人写策略文档很烂。同样，我的大部分邮件现在是 GPT-5.5 写的。我几乎想标注它来自 AI，因为它写的正是我想说的。这对我来说，作为作家，也很重要。人类写作依然重要，但我们只有标签使用 AI 的情况。

## 第四部分：谁会在未来成功？现在该做什么？

**Lenny：** 第三个类别：谁会在 AI 未来中成功？大家现在该做什么？

**Dan：** **我非常非常看好产品经理（PM）。** 内部有个例子：Marcus，我们写作应用 Spiral 的负责人。他原本是 PM，在 Axios 带过团队。休假一年后深度学习了 AI，学会了用 Cursor。他的技术能力现在算“轻度”，但编码模型已经能让他把技术知识和尖锐的产品感结合起来。他比几乎所有人发货都快，且有极佳的用户直觉。他不再需要组织整个团队，可以自己完成。所以我非常看好那些拥抱 AI 的产品经理。

**其次，全栈设计师。** 设计师通常有很多美好的交互想法，但过去工程师跟不上。现在他们可以直接构建了。他们能做出与众不同的东西，从所有“雷同”的输出中脱颖而出。这很强大。

**Lenny：** 所以“AI 工作末日”不是真的。

**Dan：** 对。我认为公司重组很多可以用“过度招聘”来解释，AI 是个方便的借口。但大规模失业不会发生。模式是：每个新模型让昨天的人类能力变得廉价；这些东西被迅速普及，变得千篇一律、商品化。人类要做的是：利用这些冻结的能力，创造新的、有趣的东西。模型总是滞后于那些真正利用它们创造新领域的人。这会在很长时间内持续。

### 4.1 如何“驾驭模型”（Ride the Models）

**Lenny：** 那具体做法呢？在 Salesforce 的 PM 应该怎么驾驭模型？

**Dan：** 很多公司限制员工使用最新模型……但在业余时间也要尝试。关键是保持好奇心和玩乐心态。当新模型发布时，试试你之前知道它做不了的事，看看现在能不能做了。比如我的“高级工程师 benchmark”，之前所有模型都 30 分，现在 GPT-5.5 到了 62 分。**你要在在意的任务上不断“翻石头”（turn the rock over）**，因为现在不行，以后可能就行。而且，AI 的前沿不在旧金山，而在于 AI 与真实人类、真实任务结合的地方。因为在旧金山制造模型的人并不完全知道如何使用它。所以你每次能用上最新模型，你就有机会成为第一批发现它新用途的人。例如我们在布鲁克林，但我们比很多旧金山的人走得更远，因为我们把它用在一切事情上。如果你坚持这么做，你很难被淘汰。

**Lenny：** 现在有一个很棒的特点：无论你有多少钱，都能立即使用最先进的模型。

**Dan：** 是的。如果 IBM 发明了 AI，它一定贵得只有大公司能用。但硅谷文化是“让智能廉价到可以忽略不计”，这让我们每个人都有了这个工具。
