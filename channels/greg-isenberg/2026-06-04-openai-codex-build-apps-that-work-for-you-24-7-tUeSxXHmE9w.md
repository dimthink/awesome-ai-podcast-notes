### [第一部分：挑战预设——为什么 Codex Sites 不是 Replit 的竞品] `[章节 0:00 - 5:00]`

**核心观点**

Codex Sites 的目标不是和别人竞争“一键建站”，而是在生态级层面创造一个**可以自主运行和持续演化的产品**。

**深度阐述**

很多人在看到 Codex Sites 时的第一反应其实和 Greg 一样：这听起来太像 Replit Agent、Bolt 或者 Lovable 了。在那些平台上，你输入一句 prompt（“给我做一个 SaaS 落地页”），十几秒后它就帮你配好服务器、数据库、域名，然后发布上线。这对非技术人员来说简直是神迹。

Greg 没有绕弯子。他直接点头承认对手的优势：**“Replit 和 Lovable 在你只想要一个 prompt 解决所有问题的时候，它们太棒了——它们有编辑器、有数据库、有服务器、有托管，还连了域名。你甚至可以直接在里面注册域名。”** 这段话很重要，因为它建立了信任。他不是来吹牛或卖课的。他是一个在产品前沿实战的创业者，他的诚实让后续的论述有了重量。

然后他话锋一转，说出了真正的核心差异：**Codex Sites 不是让你做一个“瞬间部署的静态快照”，它是让你做一个“能在 Codex 生态里持续被代理操作的活系统”。**

这里有一个微妙但极其重要的区别。Bolt/Replit 给你的是一个在瞬间完成的“成品”，你后续更新它，依然需要回到编辑器，手动改代码，重新部署。但 Greg 想要的是不同的东西。他讲了他的个人痛点：“我的电子报有 158,000 人订阅。当它变成 160,000 的时候，我还得手动去登陆后台、找到那个数字、改成新的、再保存。就为了改一个数字。” 这个场景平淡无奇，甚至有点可笑。但在传统互联网、甚至传统低代码平台，这就是每天发生的现实。网站必须手动维护。

Codex Sites 要解决的，正是这个根本性的时间成本不平等：**不再是人类去服务产品，而是产品在 AI 的调度下自动响应变化。**

但 Greg 没有回避它的现状。他非常坦诚地列出了它目前缺失的部分：

- 缺少内置数据库（需要 Prompt 去加）
- 缺少支付系统
- 缺少邮件发送
- 缺少分析工具
- 缺少密码管理 Vault
- **目前无法绑定自定义域名**（只能生成那种又长又丑的 Codex 内部链接）
这个列表读下来，如果一个人只想快速做个落地页，那他的确应该继续用 Replit。**但如果你的目标不仅仅是做一个落地页，而是创建一个能自我进化的内核，那 Codex Sites 的思维层级就是另一个维度了。**

Greg 给出的定位非常精准：**“如果你对自主应用这个理念感兴趣，如果你住在 Codex 生态里，那么 Codex Sites 非常、非常、非常酷。”** 三个 “very”，是极度克制的兴奋。

**[个人感受]**

Greg 说这句话时有一丝微妙的迟疑，他似乎在小心地平衡“一个正在狂热的信徒”和“一个负责任的评论者”这两个角色。他反复强调自己跟 OpenAI 没有利益关联：“I have no affiliation with OpenAI.” 他是一个单纯被产品打动的人。

他提到一个月的契机——Riley Brown 上了他的播客，当时他还是 “I’m in the Anthropic ecosystem. Convince me to use Codex.” 短短一个月，从抗拒到 daily driver，这种转变本身就值得解读。它不是被收购式的皈依，而是一点一点被产品的独特价值击穿的必然结果。

**[延伸思考]**

这里浮现出一个更大的框架：**“在哪个生态里建造产品”正成为比“用什么工具写代码”更重要的战略决策。** 这就像是移动互联网早期选择 iOS 还是 Android，或者更早选择 AWS 还是自建机房。生态决定了你的产品能接上什么能力（Plugins、模型、记忆、操作），也决定了它能以多快的速度进化。Greg 敏锐地看到了这一点：选择哪个人工智能平台，其实是在选择你的产品未来的 “operating system”。

### [第二部分：建造“创业想法操作系统”——六个 Prompt 的解剖] `[章节 5:00 - 10:00]`

**核心观点**

一个好的应用不是从设计稿开始的，而是从对 AI 的精确策略性约束开始的。Greg 的六步法是一个完整的产品管理思维框架。

**深度阐述**

口舌说完，我们来真的。

Greg 选择了一个极其实用的微型产品来演示：**Startup Ideas OS**。这是一个内部看板，有五个竖列——Inbox, Researching, Validating, Building, Killed。每张卡片包含：Idea Name, Buyer, Pain, Proof, Next Step, Score。

他在六步里呈现了一个完整的产品迭代过程。让我们逐帧还原。

**第一段 prompt：场景构建**

Prompt: “Build a startup ideas board.”

第一句就踩了一个坑，然后把它转化成原则。他告诉我们，你必须 **“Invoke Sites”** 来调用这个插件（像是 `/sites` 这样的斜杠指令），而且必须坚持两件事：

1. **要求它使用真实的模拟数据（realistic sample data）**
1. **要求它“保存待审，不要部署”（save for review, do not deploy）**
这是整节课的第一块金子。大多数人跟 AI 合作时，都会让它一股脑地跑到最后一步——直接部署一个看起来不错的落地页。Greg 阻止了它。为什么？他是这样解释的：**“如果你让它直接部署，它只会做一个漂亮的主页。但如果我一个 prompt 去构建，而它保存待审，我其实是在解除它的‘偏快不偏深’的本能，逼它建造一个真正的产品，而不仅仅是一个门面。”**

当我们从文字上看到这句“building a real product not a homepage”时，应该意识到这是一种新的 prompt 哲学：作者在对抗 AI 的懒惰。AI 更擅长做漂亮的一页展示，而不是深度有逻辑的复杂应用。作为提示者，你是那个逼它变好的产品经理。

**插曲：插件作为增长黑客**

在等待第一次构建时，Greg 打开了一个意想不到的话题——**插件（Plugins）生态**。

他说很多人不使用插件，但插件其实是可以让你的 Sites 从“能用”变成“惊艳”的关键。他提到了几个例子：Figma, Canva, HeyGen（头像视频生成）, Remotion（动画视频框架）。

但真正让人眼前一亮的是他对 **Game Studio** 插件 的评价：**“你通过游戏制造 Buzz，把人带向你的核心产品。”**

他说，Vibe Coding 和人工智能工程化的最大问题是：你建了一个酒吧，却没有客人来。但如果你建一个同样主题的小游戏，它会自动产生传播，然后把流量反哺到你的核心产品里。比如，做一个跟实时新闻挂钩的“送报童”游戏，用户玩得开心就顺便订阅了电子报。

这绝不只是技术 tip。这是一个完整的 **增长思维 + 产品策略 + 生态能力** 的组合拳。Greg 用一个段落，把一个独立开发者的商业闭环打通了。

**[个人感受]**

当他说“这个看板肯定不是最漂亮的，但干净、极简、第一版够用了”时，我能感觉到他呼吸里那个“we shipped something”的踏实感。他不是视觉设计师，他是策略家。他的审美追求的是“功能结构清晰”，而不是“炫酷动效压轴”。

### [第三部分：三位一体——记忆、安全操作与技能（真正的架构武器）] `[章节 10:00 - 15:00]`

**核心观点**

要使一个 AI 应用真正“自主”，它需要三个东西：**记忆（持久化）、安全操作（受控的代理操作）和技能（跨上下文的可复用指令）**。这三者的结合，构成了未来自主应用的操作系统内核。

**深度阐述**

这一部分是整个视频的智力密度顶峰。Greg 着手解决构建产品时最关键的问题：**如何让它不再是一个“一次性演示玩具”。**

**第一步：记忆 - Add Persistent Storage**

Prompt: “Add persistent storage so ideas saved between visits stay saved. Before coding, show me the data model and which records/actions the app needs.”

这里有一个极佳的原则：**在写代码前，要求 AI 展示数据模型**。这种 “Model before Code” 的流程，是专业软件工程的核心习惯。Greg 作为非工程师出身的创业者，深知自己不可能直接写出正确的数据架构。但他用指令，迫使 AI 把架构思路暴露出来。

他说：“对于非技术人员尤其有用，因为你可能根本不知道你的应用需要哪些安全操作——那就直接问 AI：What should they be?”

这句话太强大了。它把“不懂”从弱点变成了杠杆。

他展示的模型结构极其清晰：

- 一条记录（Idea）
- 字段：全卡内容、当前列、用户邮箱
- 操作：List ideas, Add idea, Update idea, Move idea, Score idea, Archive idea, EnsureSeated
Codex 选择了 Cloudflare D1 作为持久化存储。AI 还明确承诺：“我会等你的确认信号，才开始写持久化代码。”

**这就是 safe actions 的概念。**

**第二步：安全操作 - Safe Actions**

为什么要定义安全操作？Greg 用一个类比完美解释：**这像是给 AI 安装了“经过批准的按钮”。**

在 Codex 生态中，你可以在其他聊天窗口中 ping Codex，然后说：“@add idea 这个很好”，它就能直接调取你的 Startup OS 的 API，向你的数据库添加一条新记录。Safe actions 是唯一被允许的交互通道。AI 被限制只能调用这些命名的封装操作（named mutations），而不能直接执行任意 SQL。这完全防止了“AI 失控破坏数据”的噩梦场景。

Greg 反复强调：Codex Sites 比 Lovable 或 Replit 感觉上更技术化。但如果你能坚持住，产出是极具价值的。**“The idea that you can have it self-update... the idea that you can bring in all this stuff from Codex... It's pretty sweet.”**

**第三步：技能 - Skills (跨上下文的永久指令手册)**

Prompt: “Create a Codex skill called Startup Ideas Admin. Explain how to read the board, how to add ideas, move cards, score ideas. Include five example commands.”

技能可能是 Codex 生态里最被低估的武器。Greg 直接指出：“我看到很多人用 Codex Sites，却从来不用技能。我说，等等，这可是最棒的部分之一。”

技能就是一个可复用的指令手册。一旦创建，你在未来任何一个新的聊天窗口里，都可以直接激活这个技能。对话里的 Codex 就会知道如何操作你的专属应用：该读什么 API、有哪些安全操作可用、格式是什么。它实质上是在给 AI 扮演“新员工入职培训”的角色。

**[个人感受]**

Greg 在这一段语气变得非常确定。那种一开始还在犹豫要不要推荐 Codex 的感觉消失了。当他解释 safe actions 和 skills 配合工作时，他的声调里有一种工程师发现优雅架构时才有的喜悦。他不是在背诵产品手册，他是活在一个生态系统里，发现规则后立刻分享。

**[延伸思考]**

这种**Memory + Safe Action + Skill** 的三位一体，实际上是把传统软件开发中的以下概念全部范式化外包给了 AI：

- Memory = Database & Schema
- Safe Actions = API Endpoints & RBAC
- Skills = Developer Documentation & Context Caching
这三者加在一起，等于一个**可被 AI 代理完整理解的微服务架构**。对于非全职开发者来说，这是第一次可以不从命令行创建后端，而是通过自然语言定义架构。

### [第四部分：终局验证——闭环击穿（Prove the Loop）] `[章节 15:00 - 18:00]`

**核心观点**

前五步都是修养。第六步——在一个全新的对话中激活技能，让 AI 操作实时的、已发布的应用——才是真正的证道时刻。

**深度阐述**

Greg 把最酷的环节留到最后。

他说：“最后一步——Prove the Loop。我需要在一个新聊天窗口（不能继承上下文的环境下）做这件事。我输入一个 prompt：‘Use Startup Ideas Admin to add the startup idea `AI Agent SEO Grader for local businesses`. Put it in inbox with a first pass score and next step.’

然后最激动人心的事情发生了。

AI 先做了一个微妙的停顿性思考。Greg 一直在焦虑地等着看 AI 是否真的能打开一个新窗口，加载刚刚定义好的技能。他自言自语：“它说‘I’ll treat this as two pieces. Use the new Startup Ideas admin skill instructions and create a fresh chat thread to do the actual board update.’”

当页面右上角出现新窗口提示时，Greg 兴奋地大喊：“There. Yes!” 那种解放的喜悦透过屏幕就能尝到。

新窗口里，Codex 继续思考：

- “I’m loading the skill.”
- “I’m going to read the board.”
- “I’m going to add the card via safe action.”
- “Verifying by reading again.”
整个过程自动完成。

Greg 读出系统的最终状态：**“Done using only the safe board API. No raw SQL. No generic database writes. No deploy.”**

这意味着它完全遵循了他构建的构架纪律（safe actions）。AI 没有作弊，没有走捷径。它礼貌地、准确地、通过他设计的可靠通道完成了初次操作。

然后他做出最后一步：**Publish**。

网站生成了一个极长、极丑的随机 URL（他略带无奈地承认这是目前 Codex Sites 的缺点，也是他预测很快会改善的地方）。然后他登陆、打开看板，完美呈现了 Inbox 里的新想法卡片。

**“Does it work? Yes. Does it look decent? Yes. Could I get it to a point where it looks beautiful? I can.”**

这句话很重要。他不是在说“好丑”，他是在说 **“核心逻辑运转了，表象是可以迭代的。”** 这是典型的创业者思维——功能完整性优先于审美完美主义。

**[个人感受]**

Greg 在那个时刻的“wow”，是我认为这个视频最珍贵的情绪。

他经历了一个完整的：怀疑 → 尝试 → 深入 → 信任 → 惊讶 → 狂喜的弧线。他不是口播内容。他是真的在做实验，他在真的惊叹。

**“The wow moment for me is building apps where I could open a new chat and say add this idea to my OS. And it shows on the live site updating.”**

说这句话时，他笑了。那不是职业微笑，那是真正被打动后的本能反应。

### [第五部分：成为生活的操作员——软件的未来是自主进化的生命体] `[章节 18:00 - 21:00]`

**核心观点**

技术工具的终极价值不在于帮你把一件事做快，而在于让你彻底不再需要亲自做那件事。Codex Sites，不是工具，是管家。

**深度阐述**

Greg 在最后几分钟里做了一个极其感性且极具格局的收尾。

他回忆起自己早年职业生涯中的一段经历——“我曾经经营过一家网页设计公司，那是我第一份工作之一。我过去常说一句话：‘一个网站是一个活生生的、呼吸的实体。它不是你点一下发布就可以永远消失走开的。’ ”

这段话的核心痛苦是：作为网站所有者，你永远被它绑定。每次更新内容、修改 Bug、适配新的设备，你都要再创造时间、再次打开编辑器、再经过一次部署流程。

**“但我们今天，2026年，已经进入了一个时代——更新、编辑、删除这些事，已经由Agent通过 Safe Actions 和 Skills 替我们完成。”**

Greg 把这场革命的本质精确地归纳为一句话，我要原样呈现：

> **“The real unlock here is to make products that Codex can keep operating for you. That is what excites me about Codex Sites.”**
这句话值得细细咀嚼。它不是关于“用 AI 做网站”。它是关于制造出一个**可以被 AI 持续运营**的产品。你的想象力的输出物，不再是一个静态的网站，而是一个具有自我更新能力的“软件体”。

他最后说：“如果我们能创造出想法、应用和网站，让它们自动更新、自动变好、自主流畅运行……我会觉得这太有趣了。”

你能感到他是发自内心、狂热的。

**[个人感受与最终建议]**

Greg 在结束时的那一句 **“I’m rooting for you. Whatever it is you’re building, I’m rooting for you.”** 听起来可能像是一个 YouTuber 的标准告别，但在这种深入具体的实操内容之后说出口，它有了重量。

他做这个视频的目标，按他自己的逻辑是：“增加你成功的概率——不论你在做什么。” 他不是在教你用 Codex。他是在带你跨过那道从“理解工具”到“理解范式”的门。

**最后的定论：**

- 如果你需要做一个简单的落地页，请用 Replit 或 Lovable。
- 但如果你想制造一个像“活物”一样会自己呼吸、自己成长的产品，并且你愿意生活在 Codex 的生态里，把代码和代理融为一体，那 Codex Sites 很可能是目前唯一能触及那个未来的脚手架。
Greg 的路线图简单而深刻：**Memory → Safe Actions → Skills → Prove the Loop**。记住它，照着做。你要建造的可能不是一个 App，而是一个你智力的自动驾驶系统。

**精华收获**

1. **构建产品的唯一目标不是发布，而是让它能够被 AI 代理持续运营。** 每次构建都要考虑“如果你的 AI 管家三年后依然在为你管理这个产品，它的架构是否支持这种长跑？”
1. **所有的技术概念都可以用语言向 AI 索取：** 不懂什么是 safe actions？直接问“给我展示我数据库需要什么操作”，让 AI 定义给你看。不懂技术不再是壁垒，不懂清晰表达才是。
1. **“保存，不部署”是一个产品经理级别的控制策略：** 在爆发的 AI 生成能力面前，克制部署冲动，一步一步“提交申请确认”，是建立真正产品的核心习惯。
1. **插件生态是你的增长团队：** Game Studio 这样的插件不仅好玩，还可以作为核心产品的流量引擎。不要只把插件看作功能增强器，要把它看作获客模块。
1. **在代理驱动的未来，技能（skill）就是你产品的 API 文档：** 让 Codex 在任何对话中都能理解你的产品，这是真正的“跨上下文持久知识”的实现方式。
