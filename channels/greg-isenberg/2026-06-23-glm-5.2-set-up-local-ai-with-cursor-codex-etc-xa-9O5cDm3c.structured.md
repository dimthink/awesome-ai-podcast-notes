GLM 的特殊之处在于它是开源的。你可以根据设备条件在本地运行它，也可以通过开放模型提供商在云端运行它。它相比 **5.1 版本** 是一个巨大的飞跃。我个人没有测试过 5.1 版本，直接使用了 5.2。但根据我在 Twitter 上看到的和与人们的交流，它在执行类任务（尤其是前端方面）表现相当好。我还没有在后端资源密集型任务上进行过测试，但仅凭感知和报告来看，它表现不错。

## 设置教程：使用 Cursor 与 OpenRouter

**Amir**：关于设置，我个人通过 **OpenRouter 的 API** 在 **Cursor** 中开始使用它。操作方法如下：

1. 前往 **Z AI**（GLM 模型的提供商），获取 GLM 5.2 模型的 API 密钥。
2. 打开你的 **Cursor 设置**，将 API 密钥粘贴到 OpenAI 字段中。
3. 将 OpenAI 端点覆盖为此 API 端点（具体端点需参考官方文档）。
4. 返回模型选项，添加自定义模型 `glm-5.2`（或相应模型 ID），然后你就可以直接调用 GLM 5.2 了。

本质上，你不用 OpenAI 密钥，而是使用来自 Z AI 的 GLM API 密钥，覆盖 OpenAI 聊天补全的 API 端点，然后添加自定义模型协议。

## 设置教程：使用 Codex 与 OpenRouter

**Amir**：另外，你也可以通过 **OpenRouter** 进行设置。如果你使用 **Codex**，可以前往 OpenRouter 获取你的 OpenRouter 密钥，然后在提供商处获取端点，接着进入 Codex 创建一个配置文件，指定要安装的模型为开源模型。Codex 实际上支持开源模型。你可以提供模型的详细信息、上下文窗口等。然后，当你通过 CLI 运行 Codex 时，可以切换到 GLM 5.2。

## 成本对比：本地模型 vs 闭源模型

**Amir**：接下来我们讨论成本。我之前试图计算模型运行（应为模型推理）的 token 成本。如果我们使用约 **50,000 个输入 token** 和 **85,000 个输出 token**，要达到接近 Opus 4.8 的输出水平，使用 GLM 5.2 将花费 **44 美分**，而使用 Opus 4.8 则需要 **2.38 美元**。所以，两者之间大约有 **5 倍**的差价。

**主持人**：2 美元和 44 美分听起来不算多，但当你长期使用这些模型，并且处理大型任务、不想被 token 成本限制时，这就很重要了。5 倍是个大数目。

**Amir**：是的。而且这是基于编码基准测试的平均值以及 Cursor 通过 API 池的收费。但我还想考虑未来：如果我们从 GLM 5.2 已经达到这个水平，未来六个月会怎样？现在开始在你的机器上做前期投资，以便能够下载和运行这些本地模型，也许是个值得考虑的方向。这样当未来版本的 GLM（如 5.3 或 5.5）出现时，我们已经预先做出了计算和设备的投入，未来就能节省更多成本，因为这些模型可能会更加昂贵。我认为我们现在正在看到 AI token 补贴现象：我们从 Claude、Codex 等模型那里获得了更多输出，但你可能也注意到了，我们的使用量比以前消耗得更快，尤其是当 Fable 出现时，我第一天就用完了限额。

**主持人**：完全同意。你说的道理类似于风险投资支持的初创公司历史：Uber 最初补贴乘车，让你上瘾，然后逐渐提价。同样，在 AI 时代，很多 LLM 会先让你依赖它们建立工作流程，然后随着它们上市或成熟，补贴会逐渐消失。所以现在投资本地运行模型可能是个好主意，因为内存不会更便宜，token 也不会更便宜。所以尽早建立和锁定可能是个好主意。

**Amir**：是的，总结两点。第一，类似 **Cursor** 这样模型无关的工具，可以在同一序列任务中运行多种模型，将直接受益于这种成本节省。我不惊讶 Cursor 会直接支持 GLM 5.2 作为模型提供商，让你享受成本节省，并与 Composer 2.5 结合。这就导向了 **模型链（model chaining）**。

## 模型链（Model Chaining）：融合不同模型优势

**Amir**：例如，我之前在本任务中想要优化英雄区（hero section）。我先使用了 **Opus 4.8** 来导入截图，因为 GLM 5.2 不支持视觉能力。所以，我用 Opus 4.8 导入截图，并让它描述看到的内容：“告诉我英雄区的具体前端设计，列出所有细节”。然后我切换到 **GLM 5.2** 去研究这个布局，并实际进行修改。这是一种通过协作规避 GLM 5.2 图像能力限制的方法。我们链式使用昂贵的模型进行思考规划，然后以更便宜的价格获得同等前沿质量的输出。

**主持人**：这很有道理。你的建议基本上是综合利用——在某些任务上使用本地模型，在某些任务上使用最佳闭源模型，最终得到优秀的结果，但不会花费过多。如果你是个“token 最大化者”（像你和我一样，总是尽可能多地利用 AI 输出），这种策略很有用。

**Amir**：没错。而且从企业内部来看，微软的 Satya 曾提到，人力资本加上 token 使用量现在是一个重要因素。很多公司开始取消直接访问 Claude Code API 的订阅，因为成本太高。我们亲眼看到很多公司现在说：“第一年很好，我们有了 AI 采用的命令，token 最大化，这是衡量成功的方式。但现在，我们 token 上花费太多了，如何更有效？”特别是现在，你可以直接看到 token 花费与工程团队产出之间的 ROI：工程师很贵，节省了时间，输出增加了。但当你向非工程团队提供同样的模型时，他们可能只是用来写封邮件，却使用 Opus 4.8 这样的思考模型，这是一个治理问题。我们正在与公司讨论如何建立治理和正确使用模型的培训。我认为模型链是一个重要因素。我毫不意外一年后公司会开始考虑购买自己的机器来运行本地模型，因为他们在 token 上花费了太多。

## 是否需要购买本地硬件？

**主持人**：唱个反调：为什么我不直接用 OpenRouter 一键搞定呢？我在 X 上看到很多人说“买 Mac Studio 或昂贵设备”。听众是否需要购买本地硬件？即使价格翻倍，还是直接用 OpenRouter 加 Cursor 等工具更好？

**Amir**：如果你以为“只要有了合适的机器，就能得到想要的结果”，那并不是这样的。你不需要 Mac Mini 之类的设备。今天就可以开始。我喜欢 OpenRouter 和其他工具，因为它们非常模型无关，让你轻松在云端访问这些模型。它们本地运行模型，你充值 20 美元就能开始使用，设置很简单。我强烈推荐，如果你刚开始涉足这些，并且 token 花费对你是个问题，那就使用这些与模型无关的编程工具，在 OpenRouter 中运行一些 open 模型，开始实验。试试各种组合：比如用 GLM 5.2 执行，用 Composer 2.5 或 Codex 5.5 审查。有很多方法可以变得非常高效，我认为聪明的人都会在不久的将来尝试。

## 对不在意成本的“token 最大化者”的回应

**主持人**：有些人说：“我不在乎 token 成本，因为我认为构建初创公司和进行 AI 套利的机会太大了，成本无所谓。”对那些忽略开源本地 AI 运动的人，你怎么说？

**Amir**：我以前也是这样的人。在我们第一期节目中，你问我“这个要花多少钱？”，我说“不知道，我只是随心花钱”。但现在我的心态变了。现在我注意到我的使用限额用得更快了，成本上升了，我们的团队也在扩大。所以，作为个人，你更容易合理化 token 最大化，这本身有些讽刺。你应该追求 token 最小化，输出最大化才对。所以我的回答是：如果你可以直接证明 ROI（比如花了 200 美元，产出 1000 美元），那没问题。否则，补贴迟早会用完。
