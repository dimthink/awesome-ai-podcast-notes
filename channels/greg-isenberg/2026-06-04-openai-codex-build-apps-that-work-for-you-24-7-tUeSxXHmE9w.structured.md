## 引言：第一印象与学习目标

我看到 Codex 发布了站点功能。当我第一次看到这个消息时，我心里想，这只是 Codex 内部一个更差的 Replit 或 Lovable 版本吗？但越深入研究，我越意识到它确实值得学习如何使用并充分利用。所以，今天是一个完整的教程，主要介绍 Codex Sites 的重大宣布是什么，它如何与竞争对手比较？（音乐）呃，我们将实际使用 Codex Sites 进行现场构建，通过这个过程，我将教你（音乐）呃，基本上是 Codex Sites 的最佳实践。所以，到本集结束，如果你继续看，我将教你如何使用 Codex Sites 构建一个外壳，如何（音乐）添加记忆，如何添加所谓的安全操作（我会解释它是什么意思）（音乐），在节目中，如何创建技能，如何保存（音乐）门，以及如何证明循环，这能使（音乐）它自主工作。让我们开始吧。（音乐两段）

## Codex Sites 与其他平台的对比

那么，Codex Sites 与 Replit 和 Lovable 这些产品有什么区别呢？要我说，Replit 和 Lovable 这类产品如果你想要一个提示就能搞定一切，那它们确实很棒：它们内置编辑器、数据库、服务器、托管，很多还支持域名注册——你可以在其中直接注册域名。这非常棒。但 Codex Sites 不同：如果你生活在 Codex 生态中（我现在也开始生活其中了）。大约一个月前，我在播客中与 Riley Brown 对话时说过：“嘿，说服我使用 Codex。我身处 Anthropic 生态系统。”现在我会说 Codex 已经是我日常使用的一部分了。是的，我仍然使用 Anthropic 生态系统，但同时也在用 Codex。我认为如果你在 Codex 中存放了所有联系人，那么 Codex Sites 真正酷的地方在于你可以用它来构建你的想法、应用等。最酷的部分是它可以自主更新你的应用。

这意味着什么呢？例如，这是我的个人网站，它是一个静态网站。目前有 158,000 人阅读我的新闻简报，但当订阅数达到 160,000 时，我需要手动更新。但使用 Codex Sites，你可以让它自动完成这些更新——自动根据我的内容创建指南，自动添加我正在合作的公司等。这本质上是“自主产品构建”的理念。我之前也谈到过，未来的趋势（实际上现在已成为现实）是智能体自动更新产品。不过，需要注意的是，Codex Sites 目前缺少很多功能：缺少数据库、支付功能、邮件发送、分析系统和密码存储。所以，如果你想要一个更简单的全栈方案，Rapid 或 Lovable 可能更适合你；但如果你对数据库、支付等技术有所了解，生活在 Codex 生态中，并且对自主应用感兴趣，那么 Codex Sites 真的非常酷。

另外有一点：目前（至少据我所知）你还不能将这些网站发布到自己的域名并公开部署。它们只是内部应用，可以与团队共享。但显然，这种情况很快就会改变。

那么，让我们直接进入实战。

## 实战构建：Startup Ideas OS

我想到可以创建一个“创业创意操作系统”（Startup Ideas OS）。基本上就是一个看板，包含几列：收件箱（Inbox）、研究（Researching）、验证（Validating）、构建（Building）和已淘汰（Killed）。每张卡片包含创意、买家（Buyer）、痛点（Pain）、验证（Proof）、下一步（Next Step）和评分（Score）。这对我来说可以作为一个内部工具使用。我们将通过六个提示来构建它。在构建过程中，我会分享一些重要的心得。

### 第一步：搭建基本外壳（Build the Shell）

首先，我们打开 Codex。我创建了一个新项目，打开了一个新聊天。要使用站点功能，你必须调用 `@sites`——它作为一个插件工作。我发送提示：“Build a startup OS。”然后发送。它就像插件一样，你在那里调用它。并非每个人都有访问权限，但我猜测 Plus 用户很快就能使用。我本人与 OpenAI 没有关联，我只是觉得这个产品有趣，如果你也能从中获得价值，我才愿意做这期节目。

关于这个提示，我有几点心得：首先，要调用 `@sites`；其次，要求它**使用真实的样本数据**；第三，**保存审查（Save for Review），不部署**。我注意到有时我提示一个想法，它会直接尝试部署，所以一个很好的技巧就是“保存审查，不要部署”。这样可以解锁构建真正产品服务的能力，而不仅仅是首页。我构建这个的目的正是为了能够自主编辑和运行它——因为那是我理想的状态。

**插曲：插件生态**

（在等待构建期间，我想提及插件。）你可以去插件部分，那里有许多插件可以让你的站点变得更好。很多人没有使用这些插件。想想 Figma、Canva、HeyGen（用于虚拟形象视频）、Remotion。这些插件能让你的站点更互动、更有趣。另一个被低估的有趣插件是 Game Studio。所以，人们在“氛围编程”（vibe coding）和智能体工程（agentic engineering）时最大的问题是，他们构建了东西却没人使用。但如果你构建游戏，就能围绕你的产品制造话题，吸引注意力，然后将那些人引入你的核心产品。当然，这需要根据你的构建目标来定，且更适合消费者应用。比如，你可以创建一个有趣的 Paper Boy 竞争者，基于某个新闻话题进行新闻劫持（newsjack），然后将用户引向你的核心体验（如邮件简报或别的）。关键点是，很多人忽略了这种思路：你可以创建小型应用来吸引用户进入核心体验。

（回到构建）构建结果出来了。看板看起来还算简洁：收件箱、研究、验证、构建、已淘汰。虽然不是最美观的，但干净、极简，对于一个第一版来说已经够用了。好了，让我们继续。

### 第二步：添加记忆（Add Memory）

我们如何添加记忆？Codex Sites 不会自动添加记忆，需要我们来提示。回到 Codex，我复制了以下提示：“添加持久存储，使得创意在每次访问之间保持保存。在编码之前，显示应用需要的数据模型和记录操作。”顺便说一句，这些提示与你在使用 Codex Sites 时的提示类似。

它回复说：“我将在这里使用站点构建指南，因为这是一个带有存储的托管应用。既然你先要求模型，我将检查现有模式，然后展示建议的记录和变更，然后再接触代码。”它说实际上将使用 Cloudflare D1 作为持久存储，一个主要记录类型为 `ideas`。这已经匹配了项目当前方向，OpenAI 有 D1 数据库数据模型。然后它列出了应用所需的记录：每条记录存储完整卡片内容、当前列、所有者邮箱。应用需要的操作有：列出创意、添加创意、更新创意、移动创意、评分创意、归档创意等。它说在编码持久化路径之前等待我确认。

为什么要添加记忆？因为我们需要它成为一个能记住东西的生产力软件。这也是安全操作（safe actions）概念的开端。安全操作是一个解锁，因为我们可以身处其他聊天中，生活在 Codex 里，就可以说“嘿，添加这个创意”，它就能直接添加到应用中，非常酷。所以你会想要创建安全操作，以便充分利用它。

### 第三步：创建安全操作（Create Safe Actions）

我复制提示：“为添加创意、更新创意、评分创意、移动创意和归档创意创建安全操作。”之前我要求显示数据模型和记录操作，是因为你可能不知道应该创建哪些安全操作。所以你可以直接问 Codex 应该有哪些。对于非技术人员尤其如此：你不知道安全操作是什么，就直接问 Codex。

Codex 回复：“我将再次使用站点构建工作流，因为这是一个 Worker D1 服务。我将检查当前操作路由，并将其收紧到一个明确的安全操作边界，这样智能体只能调用指定的变更方法，而不能执行任意 SQL。”你可以开始看到 Codex 的确比 Lovable 或 Replit 或 Bolt 更技术化，但如果你能坚持下去，我认为输出是非常有价值的。比如，你可以让它自我更新，在 Codex 中引入所有这些东西，可以从一开始就使用插件（如 FAL AI 和图像生成、Hugging Face 等开源模型，都内置了）。这真的很棒。

### 第四步：创建技能（Create Skills）

我们将创建技能，以便未来的聊天知道如何使用这个应用。提示：“创建一个名为‘Startup Ideas Admin’的 Codex 技能。它应解释如何读取看板、添加创意、移动卡片、评分创意，并包含五个示例命令。”我认为这是 Codex Sites 最棒的部分之一——创建技能。我看到很多人使用 Codex Sites 却不使用技能，他们错过了很大一部分价值。所以，你最好开始添加一些技能来充分利用你的应用。

Codex 回复：“我将使用技能创建指南，因为你要创建一个新的 Codex 技能。我将读取技能指令，在本地技能目录中创建‘Startup Ideas Admin’，包含看板操作工作流和示例命令。技能名称有效，范围明确。”它成功创建了技能，包含读取看板、添加创意、更新卡片、移动卡片、评分创意、归档创意、五个示例命令。验证通过，技能有效。

### 第五步：保存门（Save Gate）

保存门很有用。我想法就像电子游戏中的检查点，有时会自动保存，但 Codex 不会自动保存。所以最好在创建实时 URL 之前做一个检查点。回到 Codex，我说：“将当前版本保存为 V1 审查。不要部署。确认构建状态、存储选择、访问设置以及我应该审查的确切版本。”顺便说一句，所有这些提示我都会包含在描述中供大家下载，这样他们在构建 Codex Sites 时可以使用这些内容。

Codex 回复：“我将使用站点托管工作流进行仅保存的审查会话。我将验证当前构建，确认存储访问配置，提交最新的安全操作更改，然后保存一个站点版本而不部署。”提示完成。保存审查了，没有部署。构建状态通过，存储选择仍然是 D1，访问设置是自定义。注意在右上角可以看到整个环境，所有更改、本地或 Codex Web 连接、分支等。

### 第六步：证明循环（Prove the Loop）

最后一个提示是证明循环过程。我将以下提示放入一个**新聊天**（重要）：“使用 Startup Ideas Admin 添加这个创业创意：面向本地企业的 AI 搜索引擎优化评分工具（AI agent SEO grader for local businesses）。将其放入收件箱，带有一个初评分数和下一步操作。”看看它是否能做到。它打开一个新聊天了吗？是的。它回复：“我将把它分为两部分：使用新的 Startup Ideas Admin 技能指令，并创建一个新的聊天线程来进行实际的看板更新。我正在加载技能并找到线程创建工具。技能说只使用……哦，它已经打开并创建了聊天。我可以在这里打开它。它说：‘我正在使用该技能。我将确定本地应用如何运行。读取看板。通过安全操作添加收件箱卡片，然后通过读取再次验证。’这正是我想要的。项目很干净，看起来就像一个带有 Worker 和可验证脚本的 Next.js 应用。‘我现在检查 API 实现，以便调用本地服务。应用正好暴露了技能期望的安全路由。’所以它完全按照预期工作。提示完成。它说：‘仅使用安全看板 API 完成。操作层中没有原始 SQL，没有通用数据库写入，没有部署。’速度非常快。

然后我说：“现在发布网站。”它部署了。你会注意到 URL 很长很乱，这是当前 Codex Sites 的缺点，但我想很快就会支持自定义域名。我可以在 Codex 浏览器中打开，需要注册。然后看到我们想要的：收件箱、研究、验证、构建、已淘汰，所有内容都如我所请。现在我可以创建新创意，或者使用安全操作，或者设置每周自动添加的自动化。如果你对自动化和定时任务感兴趣，我可以在未来视频中深入讲解，但基本上你可以让 Codex Sites 帮你做这些。

现在，我们来做一个小结。

## 关键概念总结

好吧，太长不看（TL;DR）版本：Codex Sites 的主要概念有以下几点。

### 1. 记忆（Memory）

你需要要求应用拥有记忆。应用可以保存数据，但如果没有提示，它只是一个演示。所以你必须提示它保存记忆，要求它拥有数据库。很多人开始将 Convex 与 Codex Sites 结合使用，你可以了解一下。随着时间推移，使用 Codex Sites 会越来越简单，但目前你必须提示它做很多事情。

### 2. 安全操作（Safe Actions）

这个概念是指你可以拥有“已批准的按钮”，通过这些按钮可以自动添加、删除、编辑应用，这样你作为人类就不用亲力亲为，无需手动编辑每个网站或应用。我曾经经营一家网页设计公司，那是我的第一份工作之一，我常说网站是一个活生生的实体——不是发布后就可以永远不管的。现在到了 2026 年，智能体正在实际进行更新、编辑、删除，这一切都通过安全操作和技能实现。使用技能，我们看到在本集里它有多大的价值。技能基本上是一个可重用的使用手册，让 Codex 知道后续如何操作应用。所以你需要创建技能来使用安全操作，并且需要将它们存储在记忆中，以便安全地提供正确信息。

### 3. 技能（Skills）

如上所述，技能是可重用的指令手册。通过技能，Codex 知道如何操作应用。

对我来说，（嗤鼻声）“哇”的时刻是构建应用后，我可以打开一个新聊天，说“添加这个创意到我的 Startup Ideas OS”，它就会出现在实时网站上自动更新。一旦发布，我可以直接使用 `@` 命令添加创意，或者让智能体去做，它会全部更新在实时网站上，这真的很酷。我认为很多人会使用 Codex Sites 创建个人网页和小应用，但我认为真正的解锁是构建 Codex 可以持续为你操作的产品。如果我们可以创建自动更新、自动改进、自主的产品，那真的是太有趣了。我认为这个趋势只会越来越大，Codex Sites 正引领潮流。
