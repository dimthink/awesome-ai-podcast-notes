## 从 TUI 到 GUI：2025 年的界面演变

**【Riley】** 在我看来，**2025 年是 TUI（文本用户界面）之年**。我们有像 Claude Code 那样的终端界面。我有点怀念早期使用 Claude Code 的日子，我喜欢终端，真的很有趣。

但是，这个界面对于很多想要从中受益但又不想打开终端或创建那些文件的人来说，并不友好。商业人士只想要一个**更简单的界面**来完成所有智能体工作流。所以，所有这些公司都认定 **GUI 更好**。我们正在从终端界面转向一个更简洁的界面：左边是你的聊天记录，中间是你的智能体，右边是智能体正在处理的内容。

另一种思考方式是：**Codeex 就像是 Claude Code 和 Claude Co-Work 合并到了一个产品里，甚至比那更好**。

我之所以比 Claude 更喜欢 Codeex，是因为 **Claude 做了一个错误的决定：把 Co-Work 和 Claude Code 分开了**。如果它们不这么做，我会更多地使用 Claude。他们似乎认为：商业人士用一个产品，程序员用另一个产品。它们有不同的限制——你无法让 Co-Work 去构建一个应用，你只能去 Claude Code 里做。Co-Work 的权限设置也与 Claude Code 不同。

你知道，我对 **Vibe Coding** 非常热衷。我觉得 Vibe Coding 已经变得极其简单——你想要编码的 95% 的事情，都跟做一份演示文稿一样容易。

我们用这种方法制作了很多发布视频，有些获得了**超过一百万次观看**。以前用其他工具做这件事非常烦人，而 Codeex 是唯一一个你可以直接 `@remotion` 并在应用内打开它的工具。

## 插件（Plugins）与技能（Skills）：有什么区别？

**【Riley】** 现在有很多词：**MCPs、Plugins、Skills、Integrations、Connectors**。老实说，它们之间的区别非常模糊。我把所有这些词都归为“**技能**”，这在我脑子里比较容易理解。

你有一个默认就很聪明的 AI 智能体，它可以搜索网络、创建应用。但如果你不给它任何**技能**，它就只是一个通用模型。你需要让你的智能体能访问你已经使用的工具，才能让它对你的工作流有用。

在 Codeex 里，它们被称为 **Plugins（插件）**。插件更“官方”一些，你需要 OpenAI 批准才能构建自己的插件。而 **Skills（技能）** 是你自己创建的。

我用的一个插件是 **Slack**。它连接到我们的 Slack，我可以问任何问题。我们公司有大约 30 个频道，我可以让 AI 总结所有与我相关的内容。因为 Codeex 记住了我，并且我给过它一份文档说明我认为什么重要，它就能找到重要的东西并提供给我。

你还可以连接到：

- 邮箱（发送/接收邮件）
- Notion
- Sheets
- **Remotion** 和 **Canva**
- **Expo**（可以在 Codeex 内预览移动应用）
如果你在插件里找不到你想要用的工具，你可以自己创建**技能**。只需新建一个聊天，说：“我想创建一个新技能，用来做某某事”。创建后，技能会出现在技能标签页里。

**【Greg】** 这太混乱了。你创建了一个技能，但它却放在了插件标签页里？

**【Riley】** 是的，技能在插件标签页里，有“插件”和“技能”两个部分。我认为这非常混乱，他们应该把所有的合并成一个。而且他们用 `@` 来引用插件，用 `/` 来引用技能，我觉得有点烦人。

**【Greg】** 但是我试过所有这些工具，**这是迄今为止最简洁的版本**。

**【Riley】** 在说技能和插件最简单的区别：插件就像是**更深度的集成**。如果一家公司想创建官方插件，他们会去找 OpenAI。就像以前 ChatGPT 的应用 SDK，是一个更官方的流程。而技能是你自己创建的，本质上就是存放在某个文件夹里的一个 `skill.md` 文件，里面有具体的指令。AI 非常擅长创建技能，你只需要说：“嘿，任何重复性的任务，我想创建一个技能，这样下次可以更快。”然后 AI 会问你一些问题，然后创建技能，它就会显示在那里。

## 自动化（Automation）与新手入门建议

**【Riley】** 假设你得到了一个非常实用的报告，你想每周五都自动运行这个任务。你可以：

1. 创建一个**自动化**（Automation）
1. 设置时间为“每周五上午 9:00”
1. 就这么简单
你只需要做一次，然后告诉智能体“以后每天/每周的这个时间都这样做”。这就是设置重复任务的方式。

一些常见的重复任务：

- 每天两次让 AI 分析 Slack 和邮箱，给你发送一封摘要邮件
- 摘要要非常简洁，并包含相关聊天或邮件的链接，方便导航
如果你对输出不满意，你可以说：“嘿，你给我的报告太长了，我想要更简洁。把它做成 Word 文档而不是 Markdown。并且更新一下周五的自动化。”AI 会帮你完成。

关于**评估（Eval）**的概念：你可以自己设置。这是 GPT-5.5，它会运行任意长时间。你可以告诉它去评估。问题在于，对于主观任务，你需要给它**优秀输出的示例**，让它知道你想要什么。编码任务很容易评判——应用看起来好不好，功能完不完整。但报告就不那么主观了。

**收集尽可能多的优秀示例**，是公司现在能做的最重要的事情之一。把这些示例存储在知识库中（比如 Notion 数据库），AI 擅长去 Notion 数据库里找到正确的文档作为示例。只要你有条理，AI 就能很好地使用它们。

我还听说，未来版本的 OpenAI 模型和 Codeex，你将能够**上传视频**给它。它可以观看视频，精准地看到你是如何完成一个工作流的，然后它就能像你一样控制电脑。

如果真是这样，那么**录制内容的价值**在 AI 时代会上升。这也是一个创业点子。
