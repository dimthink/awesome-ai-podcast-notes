# Conductor：顶尖工程师正在使用的多智能体编码工具

## 产品介绍：什么是 Conductor？

**主持人 (Aaron):** 今天，我很高兴请到了 Conductor 的联合创始人 Charlie 和 Jackson。他们刚刚完成了由 Spark 和 Matrix 共同领投的 2200 万美元 A 轮融资。非常感谢二位的到来。

**Charlie & Jackson:** 谢谢邀请，Aaron。

**主持人:** 请你们多介绍一些。Conductor 到底是什么？

**Charlie:** Conductor 是一款 Mac 应用程序，它能让你**同时运行多个编码智能体（coding agents）**。与在 IDE 中编辑文件或在终端中运行 Claude Code 不同，Conductor 将所有编码智能体整合在一起。通过一键操作，你可以创建一个代码库的独立副本，告诉 Claude Code 去为你工作，之后你可以审查并合并它的工作成果。

## 业务现状与增长

**主持人:** 这非常酷。你们是第一个在一个界面内构建这种多智能体协同编码新方式的团队，而且看起来效果非常好。目前业务状况如何？发展得怎么样？

**Jackson:** 我们是一个小而精的团队，但增长非常迅速。自一月份以来，我们已经**增长了大约 10 倍**。现在，从独立的极客开发者到大型上市公司的工程师，每天都在使用我们的产品。不过，还有很多东西需要构建和探索，我们很高兴能壮大团队。

## 最新发布：Conductor Cloud

**主持人:** 非常酷。你们今天有一些新发布的消息。请告诉我们更多细节。

**Charlie:** 我们非常激动地宣布，今天正式发布 **Conductor Cloud**。在此之前，Conductor 完全在你的 Mac 本地运行。如果你合上笔记本电脑，Claude Code 或你使用的任何智能体都会停止工作。但从今天起，你可以创建一个由云环境支持的工作空间。这样，即使你合上电脑，智能体也会继续为你工作。

**主持人:** 似乎很多 Conductor 的用户已经用满了他们可以同时运行的实例或工作树（work trees）的数量上限。这是不是帮助用户突破当前瓶颈的重大突破？

**Jackson:** 对我来说，我脑子里大概只能同时管理三到五个智能体。我认为这在一定程度上是一个界面问题，这也是我们接下来非常想解决的问题。我们已经证明，同时运行多个编码智能体并保持高效是可行的。但要达到同时运行超过三到五个的下一级别，这需要一个界面上的挑战，我们需要上升到更高层次的抽象来实现它。

**Charlie:** 非常酷。当我们刚开始开发 Conductor 时，我们自己尝试手动完成这些操作，但遇到了很多困难，很难同时用好一两个智能体，这就是我们最初构建这个应用的原因。甚至在更早之前，我们就试图构建类似 Conductor 的东西，但早期的尝试太超前了，人们还无法完全脱离 IDE 工作。

## 产品的演变与契机

**Jackson:** 是的，智能体需要达到一个“甜点区”（sweet spot）。存在一个临界点，你不需要关注它们做的每一件事，但又需要关注足够多的内容，这使得编排和同时运行多个智能体成为可能。在我们早期的实验中，我们克隆了五个代码库副本，在每个副本里运行 Claude Code。后来我们发现了“工作树”（work trees），意识到应该使用它。就这样，我们一步步地为 Conductor 构建了基础组件，以解决我们自己的问题。

**主持人:** 是的，这很酷。我们回到最初，讲讲你们是怎么走到这一步的。你们最初是怎么认识的？

**Jackson:** 我们在大学认识的。我当时是大一新生，Charlie 是大五。

**Charlie:** 是的，在当时，这几乎是最大的年龄差距了。（笑声）如果你那时告诉我我们会一起工作，我会觉得……

**Jackson:** Jackson 当时就是个孩子。对我来说，那太疯狂了。

**Charlie:** Charlie 教我如何健身。他是个好老师，好教练。

**Jackson:** 谢谢。你终于承认了。

**Charlie:** 好的，我再也不会提了。（笑声）你当时试图说服我和你一起上计算机科学课，但作为固执的新生，我拒绝了。

**Jackson:** 不过，我们大学毕业后又重新联系上了，一起做了很多疯狂的项目。Charlie 搬到了旧金山，我在 Netflix 的机器学习基础设施团队工作，Charlie 在 Replicate 负责增长和工程。我们一直一起做很多项目，想尝试一下创业。我们当时完全不知道要做什么，但获得了和你（Aaron）的面试机会，于是我们决定去做。

## 决定共同创业

**主持人:** 谈谈你们决定一起工作和创业的事。你在 Netflix，他在 Replicate。你们当时就知道自己想创业吗？知道想和对方一起做点什么吗？最初“我们应该辞掉工作，真正开始创业”的对话是怎样的？

**Jackson:** 我觉得我们非常享受一起构建东西的过程。我当时甚至对“创业”是什么没有清晰的概念。我并没有创业的梦想，但我真的很喜欢和 Charlie 一起工作。他对创业的了解比我多，这是他感兴趣的事情。我当时也在寻求改变，在家里的卧室工作快让我发疯了。我想找点事情投入我所有的时间。

**Charlie:** 我记得在我毕业后的第一份工作时，我在 Hacker News 上读到一篇文章，说“你生来就不该有老板”，我当时就觉得“是的，我不该有老板”。所以，创业的想法在我脑子里已经存在一段时间了。和 Jackson 一起工作感觉很自然，就像他说的，我们一起做项目很开心，如果能一直这样就好了。我们在大学一起玩过极限飞盘（ultimate frisbee），别不信，那是一个非常非常认真的队伍，我们投入了大量时间，是一支非常出色的团队。对我来说，努力复制那种在精英运动队的感觉——全力以赴，与强大的对手较量——是我们公司文化的重要组成部分。在大学里对手是夏洛特北卡罗来纳大学，现在则是其他许多公司。但为了共同的目标一起努力，真的非常有吸引力。

## 创业初期的想法与 YC 经历

**主持人:** 当你们决定一起创业时，有具体的想法吗？知道自己要做什么吗？

**Jackson:** 我们申请 Y Combinator 的时候，想法是**用 AI 帮你预订餐厅**。

**Charlie:** 那个想法没持续多久。

**主持人:** 我记得你们的申请材料里提到可以预订网球场之类的。

**Charlie:** 是的，事后看来，有些想法挺酷的。比如用 Sonnet 3.5 来控制你的浏览器，但这很大程度上是一个“拿着解决方案找问题”的例子。我们觉得 Claude 能在浏览器里点击很酷，但然后呢？用它来做什么？

**主持人:** 我记得我当时就是带着那个想法面试你们的。你们当时对这个想法有多坚持？

**Jackson:** 我记得我在想，我们应该对这个想法表现出多大的信念。我觉得我们做得还不错，深入思考了很多，包括它的漏洞以及如何做大。这比我们之前做过的任何项目挑战都大，但我内心并没有强烈地感觉到“这就是我们要做的事”。

**Charlie:** 是的。正如我们在 YC 期间多次学到的，仅仅通过思考是无法了解一个想法的。一旦你真正去尝试，你会发现你所有的想法都是错的，至少我们是这样的。

**Jackson:** 是的。甚至有人告诉你“我真的很想要这个东西”，但第二天你把东西放在他面前，他会说“哦，我的朋友会喜欢这个”。这并不一定意味着他们自己真的想要。

**主持人:** 我记得当时面试你们的感觉就是，你们看起来是非常出色的工程师，很可能脑子里有比申请时那个想法好得多的点子。谈谈你们在 YC 孵化期的经历吧，那三个月你们主要关注什么？

**Jackson:** 大概前一个半月我们都在到处寻找想法。我们尝试了各种东西，问朋友，构建随机原型，拿给人们看，结果无一例外都被讨厌了。我们还在构建自己完全不了解的东西。

**Charlie:** 感觉每次导师办公时间（office hours），我们都会说：“好的，我们在尝试这个新东西。”

**主持人:** 我也是这么记得的。感觉有时候我们每三天见一次，你会说：“好，我们三天后再来”，结果下次你来的时候会说：“我们有了个新想法。”

**Charlie:** 是的。你当时面试我们的时候，有没有想到我们要花这么长时间才能想出真正的点子？

**主持人:** 对你们来说，重点不是“想出”点子，而是“找到”那个真正能站得住脚的点子。你们想出了很多点子，其中很多也的确非常有趣。另外，我印象很深的是，你们是**最快速的构建者**。你们构建东西的速度太快了，每三天就能推出全新的想法，这比其他很多公司在整个孵化期内推出一个 MVP 的速度都要快得多。这非常令人印象深刻。

**Jackson:** 这功劳要归于 Charlie。他有令人难以置信的驱动力，能把东西尽快推出。我希望我也能从他身上学到一些。

**Charlie:** 是的，我觉得我们有一种很好的互补关系。工程师有两种原型。我认为自己是一个**黑客（hacker）**，我喜欢快速行动，能做出一些像是用胶带粘起来、身后留下一片狼藉的东西。而 Jackson，我认为他是一个**园丁（gardener）**，他会捡起我身后留下的碎片，然后把它构建成一些健壮的东西。Conductor 的开发过程就是如此：我花两天时间构建了一个 MVP，然后拿给 Jackson 看，Jackson 会说“这里可能有点东西”，没有这种来回互动，我们无法构建这个系统。

**Jackson:** 我记得看到 Conductor 的第一个版本时说：“Charlie，这根本不是我想象的样子，你得把所有这些东西都改了。其次，这东西太棒了，我不敢相信它竟然能工作得这么好，我们应该继续做下去。”

## 找到真正“ sticking”的点子

**主持人:** 谈谈你们找到那个最终能成功的点子的历程吧。你们在孵化期甚至之后尝试了很多想法。

**Charlie:** 我非常清楚地记得，在孵化期进行到一半的时候，我们走进你的办公室，当时我们非常低落。我真的很喜欢面向消费者（B2C）的想法，而 Jackson 喜欢面向企业（B2B）的想法。那时我们已经尝试了大概十个不同的方向。我记得你说了两件事。第一，你说“我认为**开发者工具（DevTools）**会很适合你们”，这让我们茅塞顿开。第二，你当时说的话我们到现在还留着。你说：“在你们墙上的海报上，放上你们自己的脸，然后在顶部写一行字：‘**做出这两个家伙想要的东西**’，就去构建你们自己想要的东西。”我们真的照做了。

**Jackson:** 从那以后，范围就大大缩小了。我们确定了“开发者工具”这个方向。有趣的是，我们当时觉得 AI 编码工具做得还不够深入，这至今仍是我们思考的主线。那年夏天，Sonnet 3.5 刚出来，我们觉得在“什么是可能的”和“人们意识到什么是可能的”之间存在巨大的鸿沟。我们也迷上了一个叫 Aider 的编码工具，它可以说是 Claude Code 的前身。那个夏天，我们真的尝试构建一个类似 Conductor 的版本：人类不应该在 IDE 里手动编辑文件，甚至不应该使用 Tab 补全模型，而应该告诉 AI 要做什么，让它去完成一系列工作后把结果反馈给你。我们试图做一个“后 IDE”时代的用户界面。但问题是它不太行得通，模型还不够好。Sonnet 3.5 还需要太多的人工引导，你无论如何都得频繁打开 IDE。那时候工具调用（tool calling）才刚刚起步。我们对这个想法失去了一些信念，因为模型还没准备好，我们自己也还没准备好去构建它。我们说，先放一放。我们有很多关于这个产品用户体验的想法，但不如先把这些想法应用到聊天（chat）上，因为我们知道我们能做出比现有产品更好的聊天应用。

**Charlie:** 在一次办公时间，你对我们说：“**在找到10个人尝试你的产品之前，你不能 pivot（转型）。**”所以我们决定，就推出一个东西。我们推出一个基础版的聊天应用，让你可以同时和多个模型对话，因为这再次解决了我们自己想要解决的问题。我们在（孵化期结束）几个月后的一月份发布了 **Chorus**。然后在构建 Chorus 和加速构建 Chorus 的开发工具的过程中，我们偶然间发现了 Conductor。

## 从原型到产品

**主持人:** 你们花了多长时间？听起来你们很快做了一个原型，Jackson 看了后说“这不是我想用的方式”。你们是先为自己构建。那么，从开始到找到能让其他用户测试并反馈的“那个东西”，花了多久？

**Jackson:** 大概是从写下第一行代码算起的三周时间。在那之前有几周，我们大量使用 Claude Code，围绕它开发了一些手动流程，并想着“我们需要一个工具”，这背后有巨大的机会。我们当时觉得别人会来构建这个工具，因为我们自己就需要。然后转念一想，“等等，我们可以自己造。”从第一行代码到发布，大概用了两到三周。第一周 Charlie 自己做了原型，效果出奇地好。我们以为再用一周就能发布，但实际又花了两周半来打磨。（笑声）

**主持人:** 是什么让你们相信——毕竟你们已经用其他想法和应用经历过这个过程很多次了——这次你们发现了什么，让你们确信“我们这次搞对了，这个东西能成”？

**Jackson:** 对我来说，第一次是当我给一个智能体分配任务，然后按下 Command+N 给另一个智能体分配任务，接着看到第一个智能体那里出现了一个未读的小圆点，我点回去，发现它**已经完成了**。同时运行多个智能体的想法，在当时哪怕同时运行一个 Claude Code 都还算很新。所以，能同时运行多个并且真正提高生产力，感觉非常神奇。

**Charlie:** 对我来说，另一个试金石是：Jackson 觉得它有用吗？因为我经常会因为很多闪亮的新东西而兴奋，但大多数我给他看的东西，他都会说：“我不知道这能干嘛，我不知道拿它怎么办。”但这次是少数几次他反应为“等等，这对我来说真的很有帮助”。所以这是另一个需要跨过的门槛。

**Jackson:** 是的。我想还有一个阶段，在开发 Conductor 的最初几周，有些问题非常适合 Conductor 来解决，比如有一堆相对独立的功能或 bug 修复要做。我记得有一周我们正好处于那个阶段，我们用 Conductor 飞速完成了大量代码，那种感觉令人振奋。能够**用 Conductor 来构建 Conductor** 对我们来说是另一个神奇的时刻。我几乎就没离开过那个应用，感觉就像在大量咖啡因的刺激下进行“指挥”，而且它真的有效。这又是一个信号。

## 病毒式传播的技巧与心得

**主持人:** 你们是许多“同时运行多智能体”这种模式的先驱。这对你们来说显然很有共鸣，也引起了很多其他人的共鸣。但你们还有一项特殊能力，就是制作病毒式传播的视频。谈谈这是怎么来的，以及你们能分享给其他正在观看的人的任何技巧或秘诀。

**Charlie:** 我的一次“成名作”是在 GPT-4 带视觉功能（GPT-4 Vision）发布那天，我做了一个演示。我写了一个蹩脚的 Python 脚本，它会录制我的网络摄像头，实时观察我，然后用 David Attenborough 的声音克隆来叙述我正在做的事情，仿佛我在《地球脉动》纪录片里一样。我随手把它发到了推特上，结果彻底火了。甚至 David Attenborough 爵士本人也看到了。不幸的是，他并不喜欢，这让我很难过，因为我是他的超级粉丝。

我认为我学会如何在推特上吸引注意力、如何塑造有潜力病毒式传播的帖子的部分原因，来自于我在 Replicate 的工作。我以工程师身份加入，但有段时间我们的主要问题是增长。如何让人们使用我们的 API？首先得让他们知道这个东西。所以，**构建演示应用**，制作展示可能性的视频，是我工作的一大部分。

我认为非常重要的一点是**保持真实（authentic）和直接（direct）**，不要使用公司套话，不要试图表现得比你实际更大。有啥说啥，就像和朋友聊天一样。我认为这对我来说很有效，就像那个 David Attenborough 的视频，就是一个原始的屏幕录制，我展示了我的屏幕，没做什么剪辑，就是屏幕上的一些 Python 代码，向你展示这东西能有多酷。保持语言非常直接和直白很重要。

但我也认为，你需要大量的“练习”，发布很多东西，看看什么能引起人们的共鸣，然后开始学习一个帖子的“形态”。我不知道我解释得好不好。

**Jackson:** 我觉得你从 Aaron 那里得到的一个很有趣的建议是：**观察你自己喜欢什么**。观察你在刷信息流时，会给什么内容点赞，为什么会喜欢它，分析背后的情绪和原因，然后尝试去复制它。

**Charlie:** 是的，这是一个很好的总结方式：**制作你自己会想点击的东西**。类似于 YC 的建议：做出你自己会想要消费的东西。

## 观察顶尖工程师：什么让他们与众不同？

**主持人:** 这是很好的建议。我知道你们经常在城市里进行“自行车之旅”，去见你们的用户，坐下来看他们如何使用 Conductor。这很酷，因为你们可以近距离观察一些世界上最优秀的工程师，他们正在前沿领域做以前不可能做到的事情。我很好奇，在观察这些顶尖工程师工作的过程中，你们学到了什么？

**Jackson:** 我注意到几点。第一，顶尖工程师在**他们的“技能文件”（skills files）上投入了大量思考**。我们自己也这样做，比如从经验丰富的 React 工程师那里学来的最佳实践，或者我们代码库特有的东西，把这些知识写进 Markdown 文件，这样智能体就可以持续从中读取。

第二，可能有点令人惊讶的是，**大多数顶尖工程师的“设置”是多么简单和普通**。我觉得你可以很容易地过度纠结于配置，这让我想起那些拥有超棒 Vim 配置和快捷键的人，他们能用 Vim 做各种疯狂的事情，但不一定能完成实际工作。所以，人们的设置出奇地简单。但把大量心思花在你的“技能”上，这是一个反复出现的主题。

第三，我们正在研究的一个东西，我也观察到的是，拥有我们称之为 **“无插槽区域”（slot free zones）** 的概念。也就是说，你可以让 AI 在代码库的某些部分“自由驰骋”，而在其他部分，则需要人类作为架构师进行更深思熟虑的考虑，而不是由 AI 主导。不过，我们仍在摸索中。

**Charlie:** 是的。一切变化太快了，即使在过去的几个月里。我们发现，即使在当前的 YC 批次中，我们给出的建议也与几个月前有很大不同。这使得现在成为一个非常激动人心的时代。

**主持人:** 以什么方式？

**Charlie:** 你现在可以**更快、更多地构建**。以前，如果你想挑战一个拥有庞大产品表面积的大公司，你会问“你的微小切入点是什么？”。而现在，你可以把整个东西都构建出来。这对编码来说已经发生了巨大的改变。

**主持人:** 未来几年工程领域会变成什么样？

**Jackson:** 几年很难说。谁能知道呢？我比较确信的一点是，模型会变得更聪明，10倍、100倍地更聪明。它们能够运行更长时间而无需你干预。它们会感觉越来越像人类同事。不过，它们仍然会拥有与我们截然不同的“外星大脑”。这也是我们对 Conductor Cloud 感到兴奋的原因之一：智能体将能够运行更长时间。

我认为，我们将需要**让你感觉像一家巨型公司 CEO 那样的界面**，你手下有成千上万个 AI 员工。大多数时候，你像 CEO 一样坐在办公桌前，工程主管给你汇报他们做了什么，市场主管给你更新情况，你保持在高层面上。但偶尔，你需要深入了解细节。这有点像现在的 **PR 审查**，你需要放大来看，询问具体的“员工”为你做了什么。

## 展望未来：Conductor 的下一步

**主持人:** 这又回到了你之前关于“最优秀的人在做什么”的问题。他们能够理解正在发生的一切，去查看智能体在每一层级上的操作，优化掉智能体遇到的瓶颈。同时，他们还会**主动思考**，而不是仅仅对智能体返回的结果做出反应。他们会提前考虑“我需要回答哪些设计问题才能完成这个任务？”或者“这个任务的哪些部分是我关心的？我需要给智能体设定哪些约束？”

**Charlie:** 是的。某种程度上，**那些善于用 AI 构建的人，也是善于与其他人协作的人**，这些技能实际上是可以转移的。在 AI 出现之前是最优秀的工程师，在 AI 时代很可能也是最优秀的工程师。很多东西都是可以迁移的。

**主持人:** 在最近的这次发布之后，Conductor 的下一步是什么？

**Jackson:** 目前，Conductor 在你的笔记本电脑上运行。但我们的愿景是，**Conductor 可以在任何地方运行**，它管理着你组织中运行的所有 AI。你可以从你的手机、笔记本电脑或现有的云基础设施中接入它。

**Charlie:** 我也对下一个抽象层次感到非常兴奋。Conductor 目前在某些方面非常“2025年7月”，那是我们创建它的时间，反映了当时模型的状态。我非常有兴趣**再上一个抽象层次**，让你能够以不同的方式控制模型。我认为在这方面有很多界面产品和实验可以做，我对此非常兴奋。

**主持人:** 非常酷。你认为这将为你的用户、工程师以及任何试图构建产品的人带来什么？
