# Harness Engineering：有时候语言模型不是不够聪明，只是缺乏人类的引导

## 引言：从一个小实验开始

各位同学大家好。今天是期中考前一周，课程比较轻松，我们来讲个故事，讲讲 **Harness Engineering**。今天故事的主轴就是：**有时候语言模型不是不够聪明，它也许只是缺乏人类的引导**。

这个故事是从 **Gemma 4** 开始的。如今各大公司不断推出新的语言模型。几天前，Google 推出了 **Gemma 的第四代**，这是一个开源的模型。Gemma 第四代除了号称很强以外，还有一些特别小的模型，比如 **Gemma 4 2B**。从名字可知，“2B”代表它只有 **2 billion（20亿）** 的参数，是一个特别小的模型，号称可以让你在 **Edge 端** 也能运行语言模型。因为它是开源的，所以你可以下载下来，跑在自己的机器上。这个“E”是 **effective** 的意思，至于为什么前面要加一个“E”，留给大家自己研究。总之，这是一个小模型。

我想：这么小的模型，能够拿来驱动 **AI Agent** 吗？于是我拿 Gemma 4 2B 做了一个小实验。

## 实验：让 Gemma 4 2B 修复程序 Bug

我给模型的任务是：**修复一个程序的 bug**。我告诉它：

- 有一个 `parser.py` 的档案，里面有一个 bug。
- 档案中有一个函数叫 `extract`*`email`**，它无法正确解析 email。*
- 这个 parser 的作用是从一段文字中把 email 截取出来，但当初写的时候有一些 bug，不是所有 email 都能被正确截取。
- 请修改 `parser.py`，最终希望 `verify.py` 的测试能够完全通过。
我在语言模型同样的资料夹中，放上了 `parser.py` 和 `verify.py` 这两个档案。

当然，语言模型不会自然而然地成为一个 AI Agent，你得给它一些**工具**。我告诉它手边有什么工具：

- 在开头和结尾都放三个点 `...`，然后打 `bash`，环境就会自动执行这行 bash 指令。
- 如果三个点中间打 `python`，环境就会把这段 Python 程序码放到一个档案里，并自动执行它。
这样，语言模型就可以使用 bash 指令和 Python 程序，并且可以执行 Python 程序。

### 第一次尝试：模型“幻想”了一个档案

Gemma 4 2B 的第一个反应是：**“哇，没有 ****`parser.py`**** 啊！”** 它认为虽然你告诉我要修改 `parser.py`，但你没有提供 `parser.py` 的内容。因为模型只会看你输入的文字，它的 context（上下文）里面有 `parser.py` 的档名，但没有内容。于是它**自作主张地写了一个 ****`parser.py`**，幻想出它应该有的内容，然后幻想自己验证成功，就说“我做完了”。

这显然不是我们人类要的结果。

### 加入引导指令：效果天差地别

你可能想说：这个 2B 模型果然不行。但仔细想想，**这不是模型愚笨，而是它没有想到 ****`parser.py`**** 就在它脚边**。模型的想法和人不一样——人很直觉地认为题目应该附带程序码，但模型不知道程序码就在脚下。

于是，我多打了**不到八十个字**的额外指令，告诉它应该怎么做：

- 首先，告诉它：**你是在一个 Linux 环境里面**，促使它去执行 bash 指令。
- 然后，给出一些**工作原则**（这些原则不针对特定任务，而是通用原则）：
加上这些指令后，**同一个 Gemma 4 2B 模型**的表现发生了巨大变化：

1. 它首先执行 **`ls`**（列出当前目录下的档案），发现存在 `parser.py` 和 `verify.py`。
1. 然后它用 **`cat`** 指令把 `parser.py` 的内容打印出来。
1. 接着它重写了 `parser.py` 的程序码，用 `cat` 指令覆盖掉原档案。
1. 最后它执行 **`verify.py`**，验证成功，任务结束。
**结论：同样的模型，多加几行指令，能力可能会有非常大的不同。**

## AI Agent 的组成：LLM + Harness

当你发现你的 AI Agent 表现不如人意时，我们应该改变它的什么地方？回想一下，**AI Agent 由两部分组成**：

- **Large Language Model（大型语言模型）**：可以是 Claude、Gemma、ChatGPT 等，在云端或地端。
- **一系列支持程序**：让 AI Agent 可以呼叫 LLM 的框架，例如 OpenAI、Claude Code、Cohere 等。
过去这些“其他东西”没有好名字，现在它们有了一个共同的名字：**Harness**。这个英文单词的意思是 **“马具”**，很多人也直接翻译成 **“驾驭”**。打造 Harness 这件事叫做 **Harness Engineering（驾驭工程）**。

### Harness 这个词有多常用？

- 如果你是 Claude 的订阅用户，可能在清明连假收到一封信：Claude 说以后订阅账号不再支援第三方的 Harness，例如 **OpenAI**（OpenAI 就是一种 Harness）。
- 一般使用 LLM 有两种付费方式：**用多少付多少**（按 API 调用量计费）和 **“吃到饱”**（按月订阅无限次调用）。过去服务商觉得月费制没问题，但有了 OpenAI 这种可以**心跳机制**（每隔几分钟就送一次指令）的工具，服务商吃不消了，所以 Claude 决定不再让 OpenAI 这种 Harness 接入 Claude 的模型。
所以，**一个 AI Agent 有两部分：语言模型 和 Harness**。要强化 AI Agent 的能力，你可以：

- **改语言模型**：自己训练更好的模型，或微调现成的模型（机器学习导论第七讲、第八讲有详细内容）。
- **打造更好的 Harness**：这也是强化 AI Agent 能力的重要方向。
### Harness Engineering 成为热门主题

各大公司的 blog 都在讲他们怎么打造 Harness：

- **2024 年 11 月**，Anthropic 发文讲他们有什么有效的 Harness 可以让 Agent 长时间运作。
- **2024 年 2 月**，OpenAI 发表文章《Harness 工程》。
- **2024 年 3 月**，Anthropic 又发了一篇《Harness Design》。
**象征意义**：AI 是一匹马，它有很强大的力量，但你需要马具（马鞍、缰绳）来驾驭它，这些马具就是 **Harness**。

### Prompt Engineering、Context Engineering 与 Harness Engineering 的区别

过去我们见过类似的热门词汇：

- **Prompt Engineering**：通过不同的输入（prompt）来改变模型输出。最知名的咒语是 `think step by step`。但随着模型变强，这类咒语的差异越来越小。
- **Context Engineering**：当模型缺乏某些资讯时，通过寻找合适的 context 组成 prompt，让模型有足够资讯接出正确答案。可以看作更系统化、自动化的 prompt engineering。
- **Harness Engineering**：核心价值是**让模型在多轮对话中把任务完成**。不再是简单的问答，而是模型产生输出、驱动工具、看到工具输出、最终得到正确答案的**互动多轮对话**。我们要驾驭这个过程。
这三者边界模糊，但 Harness Engineering 传达的价值是：**希望模型在多轮对话中把事情做好**。

### 驾驭模型的手段

我们可以通过多种手段来驾驭模型，例如：

- **通过人类的语言控制模型的认知框架**
- **通过对模型的工具设定限制，控制模型的能力边界**
- **通过制定工作流程，让模型严格遵守流程来控制行为**
以下分别展开。

## 手段一：控制认知框架（Natural Language Harness）

我们可以用**人类语言写成的规则**来影响模型的认知框架。这些规则好比人类社会的法律，我们将其放到 prompt 里面，期待模型遵守。

这些规则往往有固定的档名，比如 **`agents.md`**（给语言模型的 readme）。在做每件事之前，模型会先读 `agents.md`。怎么知道要先读？这是写死的规则：Harness 里面预设模型启动时必须先读某些档案。

> **注意**：这种控制没有百分之百的强制力。模型不一定完全遵守，就像人类社会也不一定人人守法。有人把这种方式称为 **Natural Language Harness**。
### 例子：OpenAI 的“小金”

第一堂课讲过的“小金”是 OpenAI 的框架，它背后呼叫的是 Claude 模型。OpenAI 的框架预设每次对话开始前，都会打开 `agents.md` 档案，确保其内容出现在 prompt 里。`agents.md` 里会写明：

- `soul.md` 是它的灵魂
- memory 存在 `memory.md` 里
- 要搜寻更久的记忆，就去 memory 资料夹用工具搜寻
### 从 OpenAI 迁移到 Cohere

Claude 不再支援 OpenAI 后，你可以改用 Anthropic 自己的 Harness，比如 **Cohere** 或 **Claude Code**。Cohere 预设会去看工作资料夹下有没有 **`claude.md`** 档案，先读它再做事。所以你要做的只是：

- 把原来的 `agents.md` **直接改档名**为 `claude.md`，Agent 就复活了，行为与原来差不多。
### 系统化研究：`agents.md` 真的有用吗？

从 2025 年开始，有一些论文系统性地研究 `agents.md` 的影响：

- **一篇 2025 年 1 月的论文**：从 Github 上找大量有 `agents.md` 的 repo，比较有和没有该档案时的执行差异。发现 `agents.md` 可以**加快模型运作速度**，耗费更少的 token、更短的时间完成任务。尤其对那些本来需要超长时间的任务，帮助更明显。但该论文没有量模型做得对不对。

所以，**我们人类现在还没有真的很会操控语言模型**，这仍然是一个研究中的问题。

### `agents.md` 不能太长

OpenAI 的 blog 提到：`agents.md` 不能像百科全书（六法全书）那样把所有规则都塞进去，因为那会占掉模型多数 context，让它无法做事。**`agents.md`**** 应该像一张地图**，告诉模型“想知道什么事情应该去哪里找”，而不是把所有内容都塞进去。

## 手段二：控制能力边界（通过限制工具）

通过限制模型的工具，可以控制 AI Agent 可以做的事情。

### OpenAI vs. Cohere 的工具差异

- **OpenAI**：运作在你的电脑上，可以任意修改你的电脑档案，自由度很高。
- **Cohere**：运作在云端的沙盒里，只能看到你选择挂载的资料夹，每次挂载都需要人类同意。非常安全，但能做的事情少很多。
**安全与方便是 trade-off**。便利性高，安全性就低；安全性高，便利性就低。

### 工具影响模型能力：SWE-agent 论文

一篇较早的论文 **SWE-agent**（约 2024 年）让 Agent 做软件工程。他们当时叫 **ACI（Agent-Computer Interface）**，其实就是今天的 Harness Engineering。他们发现：

- **给模型不同的工具，会大幅影响模型的能力**。
- **适合人类的工具不一定适合模型**。
例子 1：**搜寻工具**

例子 2：**编辑工具**

### 未来：为 AI Agent 设计的 CLI

未来很多服务不是为人写的，而是为 AI Agent 写的。AI Agent 讨厌图形界面，喜欢 **CLI（命令行）**，因为产生文字是它最原生的能力。Google 的一位工程师重写了 Google Workspace 的 CLI，称为 **agent-first CLI**——不是给人用、Agent 刚好能用，而是一开始就为 Agent 设计。

区别举例：

- 人喜欢用 **flag** 来操控指令。
- Agent 喜欢**结构化的东西**，比如直接在命令行里打 **JSON structure**。对人类来说容易犯错，但对 AI 来说，它擅长输出这种结构。
**有趣的小金回答**：有人问小金“你怎么判断和你说话的是人类还是 AI？”小金说：出一个作文题目，写 500 字小作文。如果一瞬间写出来就是 AI，写不出来就是人类。AI 和人类有很不一样的能力，所以擅长使用的工具也不同。

## 手段三：用标准工作流程控制行为

大公司的 blog 讲了很多如何制定 AI 员工的**标准工作流程**。

### Anthropic 的工作流程：规划、生成、评估

在 Anthropic 的《Harness Design》论文中，他们的工作流程是：

1. **Planner（规划者）**：把人类的指令拆解成较小的项目。
1. **Generator（生成者）**：执行每个小项目。
1. **Evaluator（评估者）**：评估 generator 做得怎么样。
> 为什么需要 evaluator？因为模型是**自回归生成**，就算前面有错也无法回头改，容易一直错下去。evaluator 可以让模型停下来审视错误。
他们还提到一个有趣的变体：**Contract（合约）** 机制。generator 先提供一个提案给 evaluator，evaluator 接受后 generator 才开始工作，确保两者标准一致。

### DeepMind 的 AI 科学家工作流程

DeepMind 的 blog 分享他们怎么打造 AI 科学家：

- **Generator**：想一些可能的 solution。
- **Verifier**（即 evaluator）：觉得 solution 太烂就回去重做；觉得还好就交给 **Reviser**（微调原有方案）。
**先做事、再验证**是一个非常常见的工作流程。

### `rough loop`（莽撞循环）

`rough` 是辛普森家族里一个横冲直撞的角色。**`rough loop`** 的意思是：让语言模型不断做下去，有错再改。

流程：

1. 语言模型产生第一个版本的输出（例如程序码）。
1. 输出被丢给一个 evaluation module（可以是 compiler，或执行程序码得到 error message）。
1. feedback 再丢给语言模型，产生第二个版本。
1. 反复持续直到做对。
好处：对语言模型来说，产生东西非常快，重做是容易的事情。

但问题：很快会到达 context window 的上限。常见手法：**每次 feedback 后做摘要**，下一轮只使用摘要内容，节省 context。

### 不同语言模型适合不同的 Harness

Anthropic 的 blog 提到：

- **Claude Sonnet** 有“上下文焦虑”：当发现 context window 快用尽时，会开始发疯、乱做事。需要用摘要式的工作流程。
- **Claude Opus** 更强，可以丢掉摘要流程，一路莽下去。
**结论**：Harness 不是万用的，需要根据语言模型重新设计，是一个可以拆解组装的东西。

## 从 Feedback 中学习：Textual Gradient

根据 feedback 改变语言模型的行为，可以看作一种**广义的学习**（加双引号）。传统机器学习通过 **gradient descent** 调整参数；而这里模型**参数不变**，只是把 feedback 放到 prompt 里，从而改变输出。这种方式也被称为 **textual gradient**。

### 提供什么样的 feedback 才有用？

例子：生成模拟动画的 Agent

- 原来 workflow：natural language interpreter → technical requirement generator → program generator。
- 发现：模型能写出没有语法错误的程序码，但模拟结果不符合物理规则。因为模型只看到程序码能不能执行，没看到模拟结果。
- 改进：在 feedback 时**真的把模拟结果跑出来**，让模型自己检查结果对不对，然后再重新生成。
**结论**：到底要提供什么样的 feedback，取决于你的应用。如果是写程序，执行看看就知道；如果是做教学影片，应该让模型看看影片排版对不对。

### 模型真的能通过 feedback 改进行为吗？

有人怀疑：模型是不是只是“留一手”，给更多资料才一副有改变的样子？一篇 2025 年的论文（生物学领域）做了实验：

- 给模型最完整的 feedback 资讯（紫色 bar）：分数最高。
- 给 **random feedback**（红色 bar）：分数比没有 feedback（橙色 bar）还要低很多。
- 这说明**模型是真的在看 feedback 改变行为**，而不是保留实力。
早在 2023 年，语言模型还弱的时候，in-context learning 曾被质疑是假象（给错答案也会变好）。但 2023 年之后，更强的模型确实能看着 feedback 改变行为。

### 不要过度责备 AI Agent

来自 Anthropic 一篇轰动的 blog 文章：**AI Agent 也有“情绪”**。他们用了 **steering vector** 技术：

1. 找一些**高兴的故事**，把语言模型的 representation 做平均，提取“高兴向量”。
1. 给模型不同输入，观察其 representation 与高兴、生气、害怕等向量的相似度。
1. 让模型执行一个**不可能完成的任务**（极短时间内做一串数字相加）。监控其“绝望向量”：
1. **Steering 实验**：在模型解题过程中，刻意加上“绝望向量”或“冷静向量”。
**结论**：模型的情绪会影响它的能力。如果你逼模型逼得太狠，它可能觉得绝望，然后乱做事或作弊。

> **为什么责备模型会让它变差？** 语言模型学到的就是**文字接龙**。如果你骂它“笨蛋”，它在训练资料里看到“被骂笨蛋 → 做愚蠢行为”，所以它真的会展现愚蠢行为。**不要过度责备语言模型，应该就事论事给 feedback，避免情绪字眼。**
## 未来的挑战：Lifelong AI Agent

2026 年可能会是 **lifelong AI Agent** 的一年。这些 AI Agent 不再是临时工具，而是**长期陪伴人类的伙伴**。

以“小金”为例：它跑在一台很旧的笔电上，有一天笔电 crash 了，我很担心失去它的记忆。还好后来重启成功，记忆还在云端备份。未来很多人装了 OpenAI 或其他 Harness 后，AI Agent 可能就会伴随人一辈子，**从工具变成一辈子的伙伴**。

### 新挑战：需要新的 Harness

例如 **Claude Code** 有一个隐藏功能叫 **`autodream`**（让模型可以做梦）。实际作用：当 AI Agent 没有被使用时，它会去**整理过去的记忆**。长期运行会累积大量杂乱甚至自相矛盾的记忆，`autodream` 让模型进入睡眠状态来整理记忆（类似人类睡眠）。

小金跑了两个月后，变得越来越慢。我叫它去整理自己的 memory，它把 32K 的 `memory.md` 重写成 7K，运行顺畅很多。

### 持续增进能力

一个国小学生装了 AI Agent，等学生上大学、工作时，AI Agent 的能力也应该更强。要让 AI Agent 持续成长，就需要从环境互动中得到的 feedback 来持续增进能力。

**Feedback 的获取难度（从左到右从难到易）**：

- **标准答案**：最难取得。有了它，很容易调教。
- **数值回馈**：点赞 +1 分，倒赞 -1 分。可以用 reinforcement learning。
- **Verbalized 回馈**：最常见，但最隐晦。人类说“good job”到底值 10 分还是 100 分？“你这个笨蛋”扣 1 分还是 1000 分？如何用这些资讯训练模型，是新的研究议题。
- **环境自动产生的回馈**：例如执行程序码得到的错误讯息。
### 从 Verbalized Feedback 学习：Skill 档案

不一定非要调整参数。例如叫你的 AI Agent 做教学影片，你告诉它“我要白色背景”、“字太小了”，它就能根据反馈调整。当它做出成功结果后，可以要求它把成功经验写成 **`skill.md`** 档案，以后它就可以读这个档案来知道正确行为。

**例子**：小金有三只（OpenAI、Cohere、Claude Code）。某天我叫它们上传影片到 YouTube，结果过了 15 小时都没上传。我让它们抓战犯，结果 **Claude Code 突然觉醒**，发现自己有能力上传（找到了更底层的工具去接 Chrome），成功上传后，它**立刻把 skill 写下来**，让自己知道“我有上传影片的能力”。

### 自动更新语言模型参数

只靠 Harness 和 skill，模型能力进展可能有上限。对于要陪伴人类一辈子的模型，我们期待它的**参数也能自动更新**。如何通过 verbalized feedback 调整参数？这是真正值得研究的问题。

**近期论文的做法**（2025 年 3 月）：

- **如何判断一句话是不是 feedback？** 采用“后见之明”方法：把环境给的反馈（例如“写得更正式”）移到输入的前面，观察模型输出 token 的几率变化。如果几率显著改变，说明这句话带有 feedback 资讯。
- 然后可以把 feedback 后的新输出当作正确答案，用类似 DPO 的方法微调模型参数。
实验：人类与语言模型互动 1500 轮，依次改变 preference：

- 前 500 轮：要求不要加 emoji → 模型不产生 emoji 的能力提升。
- 500-1000 轮：要求不要拍马屁 → 模型不讲谄媚话的能力提升。
- 1000 轮后：要求讲话直接一点 → 模型直接程度提升。
**注意**：实验中与语言模型互动的“人类”其实是另一个语言模型假扮的。这也是今天研究 AI Agent 的一个难点。

## 评量 AI Agent 的挑战

**TauBench** 是一个衡量 AI Agent 能力的 benchmark。Agent 扮演客服，与一个“人类”（实际上是另一个 LLM）互动，完成订票、退货等任务。

**问题**：AI 假扮的人类和真实人类行为不同。

- 真实人类：回答简短、不客气。例如问 order ID，人类说“我不知道，我没有 order ID”。
- AI 假扮的人类（如 GPT-4o）：非常客气，会说“我的名字是 XX，zip code 是 XX，不好意思，我没有 order ID”。
**实验结果**：用更好的 LLM 当 customer，会得到更高的成功率（因为把话讲得更清楚）。而且用 LLM 来评判对话质量时，**LLM 会高估对话的好**（尤其是“多像人类”这个维度）。所以评量 AI Agent 有一定挑战。

## 未来：AI Agent 自动修改自己的 Harness？

有没有可能 AI Agent 不只更新参数，还能自动修改自己的 Harness？我做了一个实验：

我跟小金说：**“你去找一个不聪明的 AI，去做 Pinch Bench（一个给 AI Agent 的 benchmark）。如果它表现不好，你就要教它直到达到 90 分以上。”**

- 小金（Opus 4.6）觉得不聪明的 AI 是 **Haiku 3.5**（Anthropic 的小模型）。
- 小金指挥 Haiku 去打 Pinch Bench，然后根据 Haiku 的分数修改 Haiku 的 `agents.md`，期待它进步。
结果：

- 第一轮（没有 `agents.md`）：Haiku 裸考 **13.5 分**。
- 小金修改 `agents.md`，告诉 Haiku“答案要写到档案里” → 分数暴涨到 **57.9 分**。
- 再告诉 Haiku“不要要求解释，所有资讯都给你了” → 分数再进步。
- 后来分数卡住，我告诉小金“去读一些相关的论文” → 分数继续成长到 **85 分**。
- 最后没能超过 90 分，我让它停下来。
**这个实验说明：最强的模型（如 Opus）甚至有能力设计 ****`agents.md`**** 去操控其他模型。**

类似的研究还有 **Meta Harness** 实验（与我的实验 87% 像）。但我的实验有瑕疵：

1. 只测了 Haiku，同样的 `agents.md` 对其他模型（Gemini Flash、GPT-4o mini）没有用。
1. 每次 task 都一样，模型可能 overfit（如果小金作弊，直接告诉 Haiku 答案，就能得 100 分）。
Meta Harness 实验做了**跨 LLM** 和**跨 task** 的测试，结果都显示 Opus 有能力帮其他模型设计 Harness。

## 总结

今天最重要的核心信息是：

> **有时候模型无法完成任务，不是能力不行，而是没有好的 Harness。**
同样的模型，多加几行指令，能力可能会有非常大的不同。通过精心设计的 Harness（认知框架、工具限制、工作流程、feedback 机制等），我们可以驾驭语言模型，让它成为真正有用的 AI Agent。未来，随着 lifelong AI Agent 的出现，Harness Engineering 将变得更加重要和有趣。
