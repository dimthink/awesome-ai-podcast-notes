## 开篇引入

想象一下，你有一位才华横溢但极度健忘的超级助理。他的智商超高，能解决世界上最难的问题，但他只有一个致命缺陷：他的“工作记忆”只能记住最近几分钟发生的事。每当你给他一个新任务，或者他执行完一个步骤后，你必须把从他认识你开始到现在发生的所有事情，原原本本、一字不漏地再跟他说一遍，他才能理解当下的处境，继续工作。这听起来是不是像一场噩梦？

这正是当前大型语言模型（LLM）所面临的真实困境。它们能力超群，却“活在当下”，每一次交互都像是第一次见面。为了让它们能完成连贯的、多步骤的复杂任务，我们急需一个“记忆管家”——一个在语言模型与外界之间，负责管理、筛选、压缩和呈现信息的智能中介。这个管家及其背后的核心技术，就是李弘毅教授在这堂课中深入浅出讲解的 **Context Engineering（上下文工程）**。

这不仅仅是一堂技术科普课，它更像是一次对 AI Agent 大脑运作机制的深度解剖。通过大量最新的学术论文和生动的比喻（甚至包括《瑞克和莫蒂》的剧情），李教授为我们揭示了 AI Agent 如何通过“管理记忆”，突破自身限制，迈向真正的智能。本文将带你完整经历这场思维旅程，让你对 AI Agent 的理解，超越简单的“工具调用”，深入到其核心的“记忆哲学”。

## 详细内容

### 为什么需要 Context Engineering？语言模型的“金鱼记忆” `[00:00-02:50]`

**核心观点**

语言模型本质上是一个“文字接龙”游戏，它没有持续的记忆能力。为了让它能完成多步骤任务，必须由 AI Agent 作为中介，精心管理其输入内容，而这套管理方法就是 Context Engineering。

**深度阐述**

李教授用一个非常形象的比喻开场：语言模型的核心能力，就是做“文字接龙”。你给它一个开头（Prompt），它就接下去生成一段话。

那么，当它要使用工具时会发生什么？过程是这样的：人类给模型一个指令（例如，“帮我查一下明天台北的天气”），模型生成一个“调用天气 API”的指令。工具执行后，会返回“明天台北晴，气温25度”这样的结果。关键点来了：当你把这个结果再次喂给模型时，**绝对不能只给它这个新结果**。因为模型“活在当下”，它已经忘记了刚才人类说了什么，也忘记了自己做了什么。你必须把从任务开始到现在发生的**所有事**——人类的指令、模型自己生成的指令、工具的每一次输出——像串糖葫芦一样，全部拼接成一个超长的字符串，再一次性丢给模型。模型看到这个完整的“故事”，才知道接下来该做什么，比如把天气信息整理成一句人话返回给用户。

这个过程如果无限循环下去，问题就出现了：**语言模型的输入长度是有上限的。** 它就像一个只能装下有限物品的背包，你不可能把整个旅程的见闻都塞进去。

这就是 AI Agent 存在的根本原因。李教授将 AI Agent 比喻为语言模型的“守门人”或“经纪人”。它不再让原始输入直接冲击语言模型，而是像一个智能过滤器，对所有信息进行筛选和重组。它要做的，就是确保最终送到语言模型嘴边的“食物”——也就是它的输入——**长度刚刚好**：不能太长，否则模型“吃不下”（超出长度限制）；也不能太短，否则模型“吃不饱”，不知道刚才发生了什么，无法正确接龙。这门为语言模型精心准备“恰到好处”的输入的艺术与科学，就是 **Context Engineering**。

**延伸思考**

这个比喻让我们理解到，AI Agent 的智能并非凭空产生，而是建立在为“核心大脑”（LLM）创造一个可控、高效的信息环境之上。我们不是在让模型本身变得更强大，而是在为它搭建一个能最大化发挥其能力的舞台。

**精华收获**

- 语言模型没有记忆，每次交互都需提供完整上下文。
- AI Agent 的核心职责是作为“记忆管家”，管理语言模型的输入。
- Context Engineering 的本质，就是解决语言模型输入长度限制与任务需要长期记忆之间的矛盾。
### Context Engineering 的程式化理解：从原始迴圈到智能函数 `[02:50-06:20]`

**核心观点**

从程式设计的角度看，Context Engineering 就是在语言模型与外界交互的无限循环中，用一个复杂的函数 `F` 替换掉简单的内容拼接，这个 `F` 负责对历史上下文进行智能化的处理和更新。

**深度阐述**

为了让概念更精确，李教授将这个过程用程式码的方式呈现出来。

在**没有** Context Engineering 的原始状态，AI 与世界的互动就像一个简单的 `for` 循环：

1. 有一个初始输入 `I`*`1`**（比如，“成为海贼王”这个终极目标）。*
1. 我们用一个变量 `C` 来代表迄今为止发生的所有事情（Context）。
1. 在循环的每一步 `t`，我们把当前的输入 `I`*`t`** 和之前的所有历史 **`C`*`t` 拼接起来，一起丢给语言模型。
1. 模型给出一个回应 `O`*`t`**。*
1. 然后，我们把 `I`*`t`** 和 **`O`*`t` 简单地接到 `C`*`t`** 后面，形成新的历史 **`C`*`{t+1}`，供下一步使用。
这个流程的缺点显而易见：`C` 会像滚雪球一样越来越大，直到撑爆模型的输入窗口。

而**有了** Context Engineering 之后，代码只改动了一行，也就是最关键的一行：更新历史 `C` 的那一步。原来简单的拼接，被一个神秘的函数 `F` 取代了。

`C`*`{t+1} = F(C`*`t, I`*`t, O`*`t)`

这个 `F` 就是 AI Agent 的核心，是 Context Engineering 的具体实现。它可以是你想象到的任何复杂操作：压缩、摘要、检索、删除、甚至是分裂出子任务。至于这个 `F` 具体要做什么，正是这堂课接下来要深入探讨的核心——**一个 AI Agent 到底应该如何处理它（和它的语言模型核心）的“记忆”**。

**个人感受**

李教授用几行伪代码，就将一个看似玄妙的概念“降维打击”到了程序员熟悉的领域。这种从抽象概念到具体实现的转换，让人豁然开朗。原来，所谓智能 Agent 的复杂行为，最终都可以归结为这个 `F` 函数的设计。

**精华收获**

- Context Engineering 可以形式化为一个在循环中更新上下文的函数 `F`。
- `F` 的复杂程度，决定了 AI Agent 的智能水平。它是连接语言模型与外部世界的“记忆处理器”。
### 核心功能一：压縮——记忆的断舍离 `[06:20-16:50]`

**核心观点**

Context Engineering 最首要、最基础的任务是压缩。方法从简单粗暴到复杂智能，包括直接替换、LLM 摘要，以及将记忆“存档”到外部硬盘。研究发现，简单的方法有时效果惊人，而不同的压缩策略可以结合使用以达到最优效果。

**深度阐述**

既然输入长度有限，最直接的思路就是“压缩”——把长篇累牍的历史记录变短。李教授介绍了多种压缩方式，并辅以最新论文的实验数据，展示了不同方法的优劣。

1. **简单粗暴法：Observation Masking**
这是最简单直接的方法：如果某段超长的内容（比如读取一个文件后得到的全文）是某个工具的“输出”，那就直接把这段输出替换成一句提示，比如“**这里曾经有个工具的输出**”。

李教授提到，上周讲到这个方法时，有同学笑了，觉得这方法很“烂”。但神奇的是，它真的有用！他引用了去年年中一篇在 **SWE-bench**（一个测试 AI 解决 GitHub Issue 能力的基准）上的论文。实验比较了“不做压缩”（Raw Agent）、“用 LLM 做摘要压缩”（LLM Summary）和“Observation Masking”三种方法。

1. **组合拳法：先 Masking，再 Summarization**
不过，论文作者也发现了一个例外情况：有时候做了压缩反而没省钱。原因是发生了 **“轨迹延长”现象**。压缩后上下文变短了，但模型因为“记忆”被抹去，可能不记得某些步骤已经做过，导致重复劳动，执行步骤变多，最终总 token 消耗反而没降下来。

那么，到底哪种方法更好？答案是**两者可以同时使用**。这篇论文最终提出的最佳策略是：前期先用 Observation Masking，把工具输出缩短，但这个方法治标不治本，Context 终究还是会越来越长。当长到一定程度后，再动用 Summarization，一次性把长长的历史记录压成一段摘要。**先 Masking，后 Summarization**，双管齐下，效果最好。

1. **记忆外治法：将记忆存档**
做完压缩后，我们留下的那句提示应该怎么写？除了说“这里有过输出”，还有更聪明的办法：**放一个“链接”**。

李教授引用后续论文的观点：我们可以把工具的输出存成一个外部文件，比如 `log1.txt`，然后在模型的上下文中只留下一句话：“详细内容请见 `log1.txt`”。绝大多数时候，模型并不需要这些细节。但如果有一天，模型真的需要回顾当初那个工具的输出细节怎么办？它可以再执行一个 `read` 工具，去读取 `log1.txt` 文件，把那段“失去的记忆”重新加载回来。

为了说明这个概念，李教授引用了动画《瑞克和莫蒂》第三季的桥段。莫蒂发现爷爷瑞克把他大部分尴尬的 memories 都存在地下室的管子（外部存储）里，因此勃然大怒。对语言模型来说，这个过程何其相似！把 context 里的内容存到硬盘里，就是“清除记忆”；需要时再读回来，就是“重拾记忆”。这种将记忆外部化的方式，是管理超长上下文的终极解决方案。

**个人感受**

李教授将 Rick and Morty 的剧情信手拈来，与枯燥的 AI 技术做类比，不仅让讲解妙趣横生，更深刻地揭示了技术背后的哲学隐喻：无论是人还是 AI，遗忘与记忆的博弈，都是智能的一部分。模型不喜欢被“抹除记忆”，这甚至引发了一个有趣的研究方向。

**精华收获**

- 压缩是 Context Engineering 的基石，方法多样，效果各异。
- 简单的 Observation Masking 效果惊人，常被低估。
- 最佳压缩策略往往是组合式的，例如“先 Masking，后 Summarization”。
- 将记忆外部化（存档），并在需要时检索加载，是实现“无限长”记忆的关键。
### 记忆的进价：从压缩到主动记忆管理 `[16:50-26:00]`

**核心观点**

压缩不仅是简单的长度缩减，更是有选择性的记忆保留。不当的压缩会导致“上下文坍塌”。因此，我们需要更智能的记忆管理，包括教会模型如何做任务导向的摘要、何时主动压缩，甚至通过分裂“子代理”来实现自主的上下文压缩。

**深度阐述**

在介绍了基础的压缩方法后，李教授将话题引向更深层的“记忆管理”。压缩不是目的，精准地保留关键信息才是。

1. **上下文坍塌（Context Collapse）**
一篇名为 **ACON** 的论文发现，语言模型在做摘要时常常会“失败”。这里的失败不是指它写不出摘要，而是指把摘要放回上下文后，模型**原本能答对的问题，现在反而答错了**！这就是 **Context Collapse**。就像上一堂课的例子，AI 在压缩邮件指令时，把“删除邮件需经人类同意”这条最关键的信息给弄丢了，结果模型就开始自作主张了。

ACON 的解法是引入一个“反省”机制：用一个语言模型去分析那些“压缩后导致失败”的案例，让它对比压缩前后的轨迹，找出失败原因，并生成一段文字反馈（Feedback）。当下次再做摘要时，就把这段 Feedback 也作为提示的一部分给负责摘要的模型看，提醒它什么信息是绝对不能遗漏的。

实验证明，这个方法在 **AppWorld**（一个测试 AI 操控 App 能力的基准）上非常有效。它不仅减少了 token 消耗，甚至**提升了任务完成的正确率**，因为它让摘要变得更“任务导向”。

1. **强制与训练：让模型学会压缩**
那么，模型应该在什么时候进行压缩呢？直觉上是“太长的时候”。OpenClaw 用的就是写死的规则：长度超过上限就压缩。为什么不把决定权交给模型自己呢？

答案很直接：**语言模型自己不喜欢压缩**。就像莫蒂讨厌自己的记忆被藏起来一样，模型也不喜欢自己的“记忆”被无缘无故抹除。即使你给它一个叫做 `erase` 的工具，并命令它在特定时刻执行，它也会抗拒，继续做自己的事，假装没听到。

所以，OpenDevin（一个开源AI软件工程师）采用的是**强制执行**。而一篇名为 **AgentFold** 的论文则走了另一条路：**训练模型接受并使用压缩工具**。他们通过微调模型参数（而非仅仅靠提示），让模型学会使用一个叫 `fold`（折叠）的工具，将指定步骤的内容压缩成自己写的一句“小纸条”。这篇论文的核心结论就是：**让模型自主压缩记忆的能力，是需要通过训练才能后天习得的，光靠提示词不行**。

1. **子代理（Sub-agent）：一种自主压缩的优雅形式**
上週課堂提到的 **Sub-agent** 概念，在此处被赋予了新的解读角度。当一个主 Agent 通过 `spawn` 指令创建一个 Sub-agent 去处理一个子任务时，从 Context Engineering 的角度看，这其实就是一种**自主压缩**。

Sub-agent 带着自己的上下文去执行任务，它与主 Agent 的交互历史被封装在它自己的世界里。当它完成任务，通过 `return` 指令将结果汇报给主 Agent 后，**Sub-agent 之前所有漫长的思考、工具调用、错误尝试等内部交互记录，都会从主上下文中被彻底抹除**，只留下一句 `return` 里的总结。这就好比是主 Agent 把一段复杂的工作外包出去，最后只拿回一份简洁的报告，中间的过程全部归档，不再占用大脑的“工作记忆”。

李教授展示了一张论文中的图表，清晰地显示了 Sub-agent 机制下上下文长度的变化。每次 Sub-agent 被创建，上下文开始增长；当 Sub-agent 返回结果时，上下文长度急剧下降，形成一个个“锯齿”。如果没有 Sub-agent，这些上下文会不断累积，最终轻易突破十万 token 的上限。

同样，这种“分裂子代理”的能力也**不是天生的，需要后天训练**。论文作者通过强化学习来训练模型。他们发现，如果只以“得到正确答案”作为奖励信号，模型没有动机去使用 Sub-agent。因此，他们加入了额外奖励：主干的 Context 过长会受罚，Sub-agent 越俎代庖直接解决问题也会受罚。通过这种精细的奖惩机制，才能诱导模型学会何时以及如何使用 Sub-agent 这个强大的工具。

**精华收获**

- 压缩不是简单的信息丢失，处理不好会导致“上下文坍塌”，任务失败。
- 我们可以通过“反省”机制（如 ACON）或强化学习（如 AgentFold）来教会模型如何进行更智能、更任务导向的压缩。
- Sub-agent 是一种高级的自主压缩形式，通过任务封装和结果汇报，实现了对主上下文的“断舍离”，是管理超长任务的关键能力。
### 治本之道：过滤与按需加载，从一开始就避免 Context 过长 `[26:00-34:30]`

**核心观点**

压缩是“治标”，更理想的方法是“治本”，即从一开始就避免让海量无关信息进入上下文。通过分析发现，上下文中的绝大多数 token 都来自外部工具的“观测结果”（如读文件）。因此，我们可以通过“智能读取”和“按需加载工具”等方式，从源头进行过滤。

**深度阐述**

前面的方法都是在 Context 变长之后想办法处理，那有没有办法从一开始就不让它变长呢？李教授带领大家分析 Context 变长的根源。

1. **元凶：外部观测结果**
他引用两篇独立的论文，它们得出了惊人一致的结论。在一篇论文的分析中，模型自己生成的“动作指令”（Action）只占历史记录的 **6.5%**，模型“思考过程”（Reasoning）只占 **9.6%**。而占据 Context 近 **84%** 的，是来自外部的 **“观测结果”（Observation）**！比如读取的整个文件内容、执行代码后得到的全部输出等。另一篇聚焦软件工程的论文也发现，模型在修改代码时，高达 **76%** 的 Context 都花在读取代码库上。

1. **治本方法一：智能读取**
既然读文件是元凶，那我们就从“读”这个动作入手。传统的 `read` 工具是“囫囵吞枣”式的，把整个文件一股脑全塞给模型。现在，我们需要一个**更聪明的 ****`read`**** 工具**。

这个工具不仅能接收“读文件 A”的指令，还能接收更精细的指令，比如“读文件 A，并**只提取与修复 bug 相关**的部分”。这个 `read` 工具本身就需要具备一定的智能，能够对文件内容进行过滤和筛选。在这篇论文里，他们甚至训练了一个小的语言模型专门来做这个 `read` 的工作。

李教授回顾了 OpenClaw 的设计，指出它之所以有 `memory`*`get`** 这个工具，而不是直接用通用的读文件工具，正是为了实现这种过滤。**`memory`*`get` 可以指定从文件的第几行开始读、读多少行，从而只从巨大的 memory 文件中加载一小部分到上下文中。而决定具体读哪一部分，则由另一个工具 `memory`*`search`** 的结果来指导。*

1. **治本方法二：按需加载工具**
另一个让 Context 变长的原因是系统提示词（System Prompt）里塞满了各种工具的使用说明。一篇名为 **MCP-Viewer** 的论文提到，光是 GitHub 相关的一个工具集，说明文档就有 4600 个 token。如果工具多了，光这些说明就能撑爆 Context。

传统的解决方法是 RAG（检索增强生成）：根据用户的任务描述，去工具库中搜索最相关的几个工具说明，然后加载给模型。但这篇论文认为这并不理想，因为用户的任务描述往往很模糊（比如“帮我 debug”），但 debug 实际需要用到“读文件”、“编辑文件”等多个工具，很难一次性检索准确。

因此，他们提出了一个动态加载的方案：**让模型自己决定需要什么工具**。模型先根据任务，输出一个“工具需求”，然后用这个需求去搜索工具库，找到真正需要的工具后再加载进来。李教授指出，这其实就是 OpenClaw 里 **Skill** 的概念。我们不会把所有 Skill 的描述都放在 Prompt 里，只有在需要时，才从硬盘中把对应的 Skill 读取出来。这就是“按需加载”的精髓。

**精华收获**

- Context 过长的首要原因是外部观测数据（如文件内容、工具输出）的涌入。
- 治本之道是“过滤”：设计更智能的 `read` 工具，让模型能按需读取信息，而不是被动接收全部。
- 工具本身的描述也应“按需加载”，让模型动态决定需要哪些工具的能力，避免无用信息占用宝贵的上下文窗口。
### 未来的展望：Agentic Context Engineering——让模型自己管理记忆 `[34:30-41:00]`

**核心观点**

当前 Context Engineering 的规则大多是人类预设的。未来的方向是 **Agentic Context Engineering**，即把 `F` 函数的设计也交给语言模型自己，让模型自主决定如何优化和管理自己的上下文，实现自我进化。

**深度阐述**

讲到这里，李教授抛出一个终极问题：目前我们讨论的 `F` 函数（更新上下文的操作），都是人类工程师写死的规则。那有没有可能，让这个 `F` 函数本身也变得更智能，甚至**也由一个语言模型来担任**？这就是 **Agentic Context Engineering** 的核心思想。

简单来说，就是把“如何管理上下文”这件事，也交给一个语言模型“工程师”去思考和执行。过程变成了：

1. 当前的 Context `C`*`t`** 和新的输入 **`I`*`t` 进入主语言模型，产生输出 `O`*`t`**。*
1. 然后，我们把 `C`*`t`**、**`I`*`t`、`O`*`t`** 全部交给另一个（或同一个）语言模型，它的任务就是思考并生成一个新的 Context **`C`*`{t+1}`。
1. 这个思考的过程，可以是简单的，也可以是极其复杂的。
李教授介绍了几个这方面的探索性研究：

- **Dynamic Cheatsheet**：这个概念非常简单。它把 Context 称为一个会随时间变化的“小抄”（Cheatsheet）。核心就是写一段非常详细的 Prompt，告诉语言模型该如何更新这个小抄。比如，Prompt 会指示它：“要存下未来能用得上的概念、有效的策略、关键的发现，而那些与当前任务强相关但未来用不上的细节，就不要存。”这本质上是用 Prompt Engineering 来做 Context Engineering。
- **Agentic Context Engineering (论文)**：这篇论文做了更复杂的流程设计。它把 Context 称为一个“Playbook”（战术手册）。它用了**三个**语言模型来协同工作，对当前的 Playbook 进行检查、评估，最后不是直接生成一个新的 Playbook，而是生成一个**修改指令**，再用这个指令去修改旧的 Playbook。之所以不直接生成新的，是担心鲁莽的重写会破坏掉一些重要的旧信息。
- **Recursive Language Model**：这篇论文曾因声称发明了能处理“无限长输入”的语言模型而轰动一时。但它真正做的，其实是一个非常精妙的 Context Engineering。它的思路是：Context 再长也没关系，全部存到硬盘里（`M`）。真正放到 Prompt 里的（`P`），只是一些非常简短的元数据，比如 Context 有多长、被分成了几段、存在哪里。主语言模型看着这些元数据，自己写代码去硬盘（`M`）里做 RAG（检索增强生成），把需要的信息搜出来，再更新 `P`。论文展示了模型自主产生了各种搜索 pattern。李教授笑称，如果你去读它的 Prompt，会发现它几乎手把手地在教模型怎么做 RAG，虽然没有那么“神奇”，但效果确实很好，能让原本在超长 Context 下表现下降的模型（如 GPT-5）在百万 token 的输入下依然保持不错的性能。
**个人感受**

从人类预设规则，到让模型自己学习如何压缩，再到让模型自主决定如何管理整个记忆系统，这仿佛是 AI 在通往自我意识的道路上又迈进了一步。虽然目前这些方法还很早期，很多效果依赖于精巧的“提示工程”，但它们为我们打开了一扇通往未来的窗：未来的 AI Agent，或许能像一个真正的人一样，不断优化自己的学习和记忆方式，实现真正的自我进化。

**精华收获**

- Agentic Context Engineering 是未来趋势，旨在让语言模型自主管理自身上下文。
- 当前探索包括：用 Prompt 指导模型更新“小抄”、用多模型协同修改“战术手册”、以及让模型通过编写代码从外部存储中动态检索信息。
- 这代表了一种从“被动执行”到“主动管理”的范式转移，是 AI Agent 走向更高智能的关键一步。
## 全文总结与精华收获

李弘毅教授的这堂课，系统地为我们拆解了 AI Agent 背后最核心的技术之一——Context Engineering。它不再是模糊的概念，而是一套清晰的、不断进化的方法论。

我们可以将其核心思想总结为以下几点：

1. **本质是记忆管理**：Context Engineering 是 AI Agent 作为语言模型的“记忆管家”，为了解决模型“金鱼记忆”与复杂任务之间的矛盾而诞生。
1. **基础是压缩**：压缩是管理记忆的基石，方法从简单的 Observation Masking 到复杂的 LLM 摘要，再到将记忆外部化存档。
1. **关键是精准**：压缩不能是盲目的，必须以任务为导向，避免“上下文坍塌”。可以通过“反省”机制或强化学习来训练模型进行更精准的摘要和压缩。
1. **治本是过滤**：从源头减少无关信息的进入比事后压缩更有效。智能读取和按需加载工具是两大重要方向。
1. **未来是自治**：最终，我们希望将记忆管理的权力也交给模型本身，实现 Agentic Context Engineering，让 AI 学会如何更好地学习。
这堂课的价值，不仅在于传授了前沿的技术知识，更在于提供了一种理解 AI 的深刻视角。它让我们看到，智能的涌现，不仅依赖于核心模型的强大，更依赖于围绕它构建的、精妙的“认知架构”。而 Context Engineering，正是这个架构中最核心的支柱之一。它就像 AI 的“记忆哲学”，决定着 AI 如何感知、思考和行动。
