# AI Agent 的核心技术：Context Engineering 深度解析

大家好，我们开始上课。今天，我们要继续来讲 AI Agent。今天的课程还是比较科普性的，我们把它分成三个段落。

第一段，我们来系统地讲解 AI Agent 背后的核心技术——**Context Engineering**。

第二段，我们来探讨 AI Agent 之间的互动。

最后，我们来讨论 AI Agent 对于我们未来的工作可能造成什么样的冲击。

## 一、引言：为什么需要 Context Engineering？

我们从 AI Agent 的核心技术 **Context Engineering** 开始讲起。等一下这段课程，也许你听了会觉得似曾相识，因为很多 Context Engineering 的技术都已经被实作在 OpenClaw 里面了。今天这段课程比较不一样的地方，是会引用大量的论文。你会发现这些论文都是非常新的论文，几个月前、半年前的论文，而其中许多技术都已经被实作在 OpenClaw 里。所以上週其实很多技术我们都已经有提到，只是今天再从另外一个角度来谈 Context Engineering。

在谈 Context Engineering 之前，这边有一个开场，我们还是跟大家複习一下：**为什么需要 Context Engineering？**

我们知道语言模型就是在做**文字接龙**。你给它一个输入，给它一个 prompt，它就接一段话出来。人类给语言模型一个输入，语言模型给人类一个回应，这个回应不一定是一句话，它可能是一个使用工具的指令。这个使用工具的指令可能会去驱动环境裡面的某一个程式被执行，然后得到工具的输出。

当我们要把工具的输出传给语言模型，告诉它这个工具执行的结果的时候，**你不能够只给它工具的输出**。大家要记得，语言模型是活在当下的，它只管现在的输入，它不管你之前曾经给过它什么。所以，当你得到工具 1 的输出的时候，你要把之前人类给的命令、语言模型自己操控工具的指令，加上工具的输出，**全部接在一起，丢给语言模型**。

有的同学可能会说，这边不是输入三个东西吗？语言模型不是应该回三个回应？你误会这个投影片的意思了。这边是这三段话被接成一段。对语言模型来说，它看到的就是一串非常长的输入，然后它再给一个回应。比如它决定使用另外一个工具，然后它再得到工具 2 的输出。工具 2 的输出要丢给语言模型的时候，**切记不能只给它工具 2 的输出**，之前发生所有的事情都要串成一串非常长的输入，再丢给语言模型。同样的步骤就反覆下去：语言模型说它要使用工具 3，得到工具 3 的输出，再把一串非常长的输入传给语言模型。

这里会遇到的难点就是：**语言模型的输入长度是有限的，它不能够吃无限长的输入**。这就是为什么我们需要 AI Agent。

## 二、什么是 AI Agent 和 Context Engineering？

AI Agent 就是拦截在语言模型跟人类，或者是语言模型要执行的环境之间的一个**介面**。它就像是语言模型的守门人、语言模型的经纪人，它决定语言模型会看到什么。所以来自外界的输入会经过 AI Agent。这个 AI Agent 不一定是 OpenClaw，OpenClaw 只是 AI Agent 的其中一个例子。

今天 OpenClaw 还非常的原始，你可以想成它是**初代的 AI Agent**。也许过几年以后，我们再回头看 OpenClaw，就好像今天你拿着 iPhone 去看过去的 Nokia 手机一样的感觉。我们现看到的只是 AI Agent 的原型，以后一定还会有更多的进展。

OpenClaw 做的事情，或 AI Agent 做的事情，就是**选择给语言模型看的內容**。所以语言模型真正看到的是经过 AI Agent 筛选过的、长度合适的输入。

这个输入不能太长，因为语言模型的输入有上限；但也不能太短，如果太短，语言模型就不知道刚才发生了什么事，就没有办法正确地做接龙了。所以对 AI Agent 来说，它要做的事情其实非常的複杂：它需要产生一个**长度合适的输入**，不能太长，也不能太短。而 AI Agent 帮语言模型管理它的输入，让输入的长度是合适的，这件事情就叫做 **Context Engineering**。

## 四、Context Engineering 的核心技术方法

AI Agent 在这个 Context Engineering 的大 `F` 裡面会做什么样的事情呢？接下来我们就来举几个例子。

### 4.1 核心需求：压缩

一个最需要做的事情就是**压缩**。之所以要做 Context Engineering，最核心的需求就是语言模型的输入不能太长，所以那个大 `F` 裡面最重要的一项功能就是压缩——把本来很长的歷史纪录压短。

怎么压短呢？上週在最后课程结尾的时候，我们有讲到 OpenClaw 裡面是内建 `compaction` 的功能的。它做的事情，说穿了也不值钱：把整个歷史纪录裡面，扣掉 system prompt 的部分，比较久远的歷史纪录通过某一个语言模型把它变成**摘要**。所以本来很长的歷史纪录就变成了一段简短的摘要，然后再继续去接上新的资讯。

4.1.1 压缩方法一：Observation Masking

上週我们也提到 OpenClaw 还有别的处理压缩的方式。比如有一个非常简单粗暴的方式是：如果某一段文字原来是某一个工具的输出，而有时候工具输出会是非常长篇大论的东西，那就把那一段长篇大论改成“**这里曾经有个工具的输出**”就结束了。

上週讲到这裡的时候，我看到同学笑了，你可能觉得这个什么烂方法，这个方法真的会有用吗？神奇的事情是，这个方法还真的有用。这引用自一篇去年（2023年）年中的论文。有人尝试在 **SWE-bench** (Software Engineering Benchmark，软体工程基准）上面，比较用语言模型做压缩，还有单纯把工具的输出换成“这里曾经有个工具的输出”这两个方法，它们在表现上有什么样的差异。

这些实验是做在 **SWE-bench** 上面。SWE 就是 Software Engineering 的缩写，就是要让语言模型去做一些平常软体工程师在做的事情。在 SWE-bench 裡面，语言模型要解的问题就是：给它一个 GitHub Repo，给它一个 Issue，然后它要去把这个 GitHub Repo 的 Issue 解掉，就是一般软体工程师在做的事情。虽然非常有挑战性，不过今天很多语言模型在这样的任务上面都可以做得非常不错。

**实验结果分析：**

- **纵轴**是正确率，**横轴**的单位是用美金，代表在解问题的过程中用了多少 token，花了多少成本。
- **黑色点 (raw agent)**：没有做任何压缩，环境裡发生什么就通通丢到歷史纪录裡。结果显示，虽然多数情况下表现不错，但相比其他方法会耗费更多 token 和成本。
- **红色正方形 (LLM summary)**：拿一个语言模型对上下文进行压缩。结果发现，红色点跟黑色点多数时候比起来是差不多的，代表用 LLM 做压缩其实跟没有做压缩在 SWE-bench 上表现差别不大，也就是说这个压缩其实是蛮有效的。
- **三角形 (observation masking)**：这就是刚才讲的，把工具的输出换成“这里曾经有个工具的输出”。结果发现，三角形的表现很多时候跟 LLM summary 的表现是差不多的。也就是说，与其去呼叫一个语言模型做摘要，跟你直接把工具输出换成一句话，结果是差不多的。LLM summary 并没有在所有 case 都稳定地比 observation masking 好。
- **成本与例外**：多数时候，有做压缩都是比较省钱的。但你会发现有一个例外，有做压缩并没有比较省钱。作者提供的解释是，这发生了**轨迹延长**的现象。当你做压缩以后，上下文变短了，但因为有一些步骤不见了，对语言模型来说，它可能会觉得“我刚才到底执行过这个工具了吗？还没有吗？”，然后重複做了刚才已经做过的事。所以执行的步骤反而变多了，虽然输入语言模型的 context 变短，但它执行的步骤变多了，最后解一个问题所耗费的 token 也就没有下降。
4.1.2 压缩方法二：结合使用

我们刚才讲了两个不同的压缩方式。你可能问说，实际上到底哪一个比较好呢？当然你可以两个同时执行。在同篇论文裡面，它最后提出来的方法就是：**不同的压缩方法是可以同时执行的**。它最后试出来，最好的策略就是：

1. **前期**：先用 `observation masking` 的方法，把工具的输出换掉、缩短。
1. **后期**：但这一招终究会让你的 context 越来越长，就算只是换成一句话，context 最终还是会长。所以长到某一个地步以后，再用 `summarization` 的方式，一次把非常长的输入直接压短。
把这两个方法同时使用，可以得到最好的结果。

### 4.2 进阶：记忆清除与读取

做完压缩以后，这裡要留一个句子代表它曾经被压缩过。但这个句子应该要留什么呢？与其放“这里曾经有个工具的输出”，换别的字眼、放别的符号会不会更有效呢？

在后来的论文裡面，有人说，那我们能不能够在这裡放一个**连结**，放一段纪录说“这个工具的输出，详见 `log1.txt`”？然后你就真的把这个工具的输出存到你的硬碟裡面，存成一个档案叫 `log1.txt`。

这样做之后，可能多数的时候，语言模型都不会再回来看这个 `log1.txt` 裡面有什么，因为多数时候工具的输出可能没那么重要。很多情况下，工具的输出可能是执行 `read`，然后去读了一篇论文，把整篇论文的内容读了进来。可能你根本不需要整篇论文的内容，只需要它的某一段或某一个摘要。所以这些内容不需要一直存留在你的上下文裡面。但如果你不做压缩，直接执行工具，这些非常长篇大论的文章或程式码就会一直存留在你的上下文中。

所以你可以把这些内容存到某一个档案裡面，它就从这个语言模型的输入裡面消失了。语言模型的输入裡面就只留下一句话：“详见 `log1.txt`”。如果语言模型有一天真的很需要知道这个工具当初到底输入了什么、输出了什么，那这个时候它可以再执行一个工具（比如 `read` 指令）去 `log1.txt` 裡面把它需要的内容再读取出来，它就可以重拾它的记忆。

4.2.1 科幻实例：Rick and Morty 的记忆

这让我想到 **《Rick and Morty》** 第三季的其中一段。有一天 Morty 发现，他家有一个地下室，他爷爷 Rick 把他很多的记忆通通都存在这个地下室裡面。他的每一个记忆存在一个一个管子裡面，多数是 Morty 感到难堪的记忆，也有一些是 Rick 本身觉得难堪的记忆。然后 Morty 发现“原来我大半生的人生记忆通常都存在这个地下室的管子裡面”，就爆气，然后跟他爷爷打了一架，两个人记忆就都消失了。就是一个这样莫名其妙的故事。

最近我重看了一下《Rick and Morty》，我现在觉得自从有了 AI Agent 以后，很多科幻电影、科幻小说裡面有的情节感觉就没有那么神奇了。比如 Morty 最新一季裡面有一个**电影制造机**，你输入一个脚本它就输出一个电影。你想想看，现在其实根本就已经有电影制造机了，不是很多现在的短电影都是直接用 AI 生成的吗？所以这个技术现在看起来也没有那么遥远了。

总之，这就是一个**记忆清除、然后再读取**的故事。对语言模型来说，刚才把存在上下文中的内容放到硬碟裡面，之后再读取出来，其实就是它的**记忆**。

### 4.3 记忆管理的完整数学描述

讲到这里，我们其实需要稍微修改一下刚才有关 Context Engineering 的式子。我们刚才说，我们有一个东西叫做 context `C`，然后我们会有一个大 `F` 来更新这个 `C`。如果我们要把记忆的概念加进来，讲得更精确一点，这个 `C` 应该被分成两部分。我们在这边用 **`P`** 和 **`M`** 这两个符号来代表 `C` 的两部分。

- **`P`**** (Prompt)**：会被丢进语言模型的资讯。
- **`M`**** (Memory)**：不会被丢进语言模型，存在你的硬碟中的资讯。
所以，context 其实可以包含存在硬碟中的资讯和可以放到语言模型中的资讯。

左边的演算法跟右边的演算法，除了 `C` 我们告诉你说它裡面有两个 component 以外，唯一不同的地方就是：这个呼叫 LLM 的输入从 `C`*`t`** 变成 ****`P`*****`t`**。这要告诉你说，我们只需要把 `C` 裡面我们準备要給 LLM 看的部給 LLM 看就好，其他部分就存在硬碟裡。

所以当我们在更新 `C`*`t`** 的时候，是把 **`C`*`t` 裡面的两个 component `P`*`t`** 和 **`M`*`t`，分别更新成 `P`*`{t+1}`**、**`M`*`{t+1}`。

- 有时候你更新的是 `P` 的部分（放进语言模型的部分），例如当你执行 `load memory`，把记忆从硬碟中读取出来的时候。
- 有时候你是执行 `save memory`，要把记忆存到硬碟裡，那就更新了 `M`。
这是一个 **Context Engineering 更完整的式子**。

### 4.4 澄清：Prompt 与 Context

讲到这边，我想要对我们常常用的几个词汇做一下澄清。有几个词汇，其实在我上课的时候往往是被混用的，比如 `context` 和 `prompt`。这两个词汇在上课的时候好像差不多是同样的意思，但实际上如果细分起来的话，我认为这两个词汇还是有所不同的。

- **Context**：指的是这个演算法的 `C`。它包含了会被输入语言模型的部分，也包含了存在硬碟中的部分。你可以想像，**Context 是 AI Agent 所经历过的一切事情**。
- **Prompt**：`P` 是真正会被输进语言模型的部分，所以 `P` 是 context 的一部分，是语言模型会看到的部分。这个部分才是 **Prompt**。
所以，**Prompt 跟 Context 是有差别的**。Context 不一定会成为语言模型的输入，只有 Context 的一部分会被作为 Prompt。不过在上课的时候，或者在文献上，或者在你看到别人讨论的时候，往往 Prompt 跟 Context 现在就是混杂成一团，但我觉得这两者其实还是可以做出区別的。

### 4.5 压缩的陷阱：Context Collapse 与 ACON

我们刚才讲到，在做摘要的时候，你其实就是呼叫某一个语言模型，跟它说把这一段记忆做摘要。今天语言模型都有摘要的能力，所以它就会输出摘要来。但是有一篇叫做 **ACON** 的论文，它就发现：语言模型在做摘要的时候，很多时候是会**失败**的。所谓的失败并不是说它没办法产生摘要，而是它产生完摘要、把摘要放到 Prompt 裡面以后，结果本来能够答对的问题、本来做得了的任务，做不了了。

这一件事情叫做 **Context Collapse**。今天在做压缩的时候，损失了一些资讯，如果损失的是最重要的资讯，那语言模型就非常有可能犯错。就像我们上一堂课举的最后一个例子：有一个 Meta 的研究人员让 AI 帮他收信，结果 AI 在做 compress 的时候，把最重要的指令“删除邮件要经过人类同意”压缩变不见了，所以模型就开始不听话了。

所以今天在做压缩的时候，它不是一个普通的压缩，不是一个普通的摘要，我们应该要想办法告诉语言模型，**什么样的资讯是最应该被留在 Prompt 裡面的**。

ACON 的解法

这篇论文的解法是：

1. 它拿另外一个语言模型出来。
1. 它有一些训练资料，这些训练资料是：本来没有做压缩的时候可以做得对，但是语言模型在压缩之后就做不对的例子。
1. 它把这些例子给一个语言模型看，并跟那个语言模型说：“你看，压缩以后结果变差了。你能不能说明一下、反省一下，为什么压缩会变差呢？”
1. 这个语言模型检查了这两个轨迹（trajectory）以后，就会得出它的结论，把这个结论写成 **feedback**。这个 feedback 就是一段文字，没有任何特别之处，完全没有训练模型。
1. 下一次，有不一样的任务进来的时候，再把这段 feedback 给负责摘要的语言模型看，希望有多这段额外的 feedback 可以让语言模型做得更好。
在这篇论文裡面，它也把它的资料集分成训练资料跟测试资料。不过这边要注意，这边的所谓训练资料并没有真的去改变模型的参数，这些训练资料就是拿来得到这个 feedback。然后测试资料会拿这一个 feedback 来强化模型做摘要的能力。当语言模型看到这个 feedback 的时候，它裡面的参数是没有任何改变的，但它只是因为看到这个 feedback，更知道说在做摘要的时候什么资讯是重要的，如果漏了什么资讯之后可能就会任务失败。所以它更能够做摘要，更能够做人要的摘要、任务要的摘要，所以可以表现得更好。

ACON 的效果

ACON 这招真的有用吗？还真的有用。这个 work 是做在 **AppWorld** 上面。AppWorld 就是要让语言模型去操控一大堆的 App，然后来执行一些比较複杂的事情，其实就跟今天 AI Agent 在做的事情是一样的。所以你就知道，今天这些 AI Agent 那么能够执行各式各样的程式、把它们组合起来完成複杂的任务，其实早就有这样子的 Benchmark 在评量 AI Agent 能够执行複杂任务的能力。

- **横轴**是 peak token，也就是在整个 Prompt 裡面 token 最多的时候，多到什么样的地步。
- **纵轴**是正确率。
- **黑色点**代表没有做 compression，token 量当然是最多的。
- 如果只是做一般的 LLM 压缩（图中未标出），在 AppWorld 这个 Benchmark 上面，有时候结果会变差，压缩完以后本来解得了的任务就解不了了。虽然用的 token 量比较少，但是正确率会下降。
- **紫色点**是 ACON 的作法，也就是让语言模型更能够做任务需要的摘要。结果发现，**不只所耗费的 token 变少，表现也變得更好**。
### 4.6 用强化学习训练压缩模型

在 ACON 那篇论文裡面，是完全没有训练模型的。当然你想要训练模型也是可以的，也当然有论文尝试去 **finetune 一个 LLM**，这个 LLM 是专门对 context 做压缩的。

你可能想，那我们要怎么训练这样的 LLM 呢？我们一般训练一个模型的时候，不只要输入，也要有正确的答案才能够训练模型。但是今天假设没有正确的答案，或实际上我们就是没有正确的答案（因为你不知道正确的摘要应该长什么样子，不知道什么样的摘要才是对任务有帮助的），那我们要怎么训练这个模型呢？

实际上在训练这个模型的时候，是用一个 **Reinforcement Learning (强化学习)** 的方法。它是说：

1. 让这个模型产生摘要之后，还不算完，还不知道这个摘要好不好。
1. 接下来再继续去解这个任务。
1. 直到最后，解完任务之后，看有没有做对。有做对就是 positive reward，没有做对就是 negative reward。
1. 用 reinforcement learning 的方法来训练这个做摘要的语言模型。
但其实做摘要的语言模型跟负责做其他事情（比如产生执行工具指令）的模型，其实是同一个模型。所以实际上在训练的时候，你**不只是强化了语言模型做摘要的能力，你其实强化了解整个任务的能力**，包括语言模型根据摘要来解任务的能力。也许今天这个训练，一定不只是让摘要写得更好，可能同时也是让模型更能够读取摘要，更知道怎么从一个简短的摘要裡面去执行它的任务。训练一个 LLM，它的目标是特化在 Agent 的这个情况下，可以做得更好。

### 4.7 何时压缩：模型不喜欢被“遗忘”

讲到现在，我们都还没有讲：**什么时候应该开始压缩？** 直觉上，今天 context 太长的时候就该开始压缩。但是长到什么样的地步应该开始压缩呢？如果你去看 OpenClaw 的话，那裡面就是一条**写死的规则**：反正长度超过某个上限的时候，就开始压缩。

为什么用写死的规则？为什么不让人家模型自己决定什么时候要压缩？因为前人的文献已经发现：**语言模型不喜欢做压缩**。对语言模型来说，压缩就是抹除记忆，它非常不喜歡这件事。就像 Morty 发现自己大半生的记忆都被藏在一個地下室的時候，他非常的生气，殴打了他的爷爷。所以这个语言模型跟 Morty 一样，它不喜欢它的记忆无缘无故就消失了。就算是你在給它一个工具，跟它說“这边有個好用的工具，这个工具會把你过去的记忆某一些地方抹除”，它听了就不高兴，听了就不想执行这个工具。

甚至在这篇论文裡面，他还尝试逼迫模型使用这个工具。他跟模型说，现在有一个工具叫做 `erase`，会抹去模型部分的记忆。然后他说“当我对你说 `reflection`（反省）的时候，你就要执行 `erase` 这个工具”。然后这个模型就开始做事情，做着做着，人类在这边强制输入 `reflection`，然后说“非常重要，你只能做 `erase`，不能做其他事”。语言模型才不做，它就继续做它的事情，它不想要抹除它自己的记忆。

所以语言模型没那么喜欢被抹除记忆，怎么办？有一篇叫做 **AgentFold** 的论文。你会发现在 OpenDevin 用的是强制执行的方法：只要 context 超过一个上限，它就执行一个动作叫 `memory flush`，这是强制执行的，会让模型自己开始压缩自己的记忆。

而 AgentFold 这篇论文做的事情是，它去**训练模型使用压缩记忆的工具**。它把压缩记忆的工具叫做 `fold`（折叠）。这个 `fold` 会吃两个输入：

1. 我们要把刚才整个对话的第几步到第几步做压缩。
1. 压缩完之后，你最好还可以留一个**小纸条**说“这边曾经被压缩过”。这个小纸条内容写什么，也可以让语言模型透过这个工具自己决定。
比如，它可以说“第三步到第四步，刚才是上网搜寻，搜到了一大批的资料，也许太冗长了，我们改成一句话‘上网搜寻，我已经知道我要的资讯，比如台湾最高的山是玉山’”。然后就执行这个 `fold` 指令，把前面的第三步、第四步骤置换成一段文字。

讲到这边你可能又会想，刚才不是说语言模型不喜欢压缩或抹除自己的记忆吗？这边怎么能够？其实这篇论文正好呼应了过去的研究发现——语言模型不喜欢压缩记忆。因为这篇论文的核心就是：**使用压缩记忆工具这件事情，必须透过训练才能取得**。所以他们是微调了模型的参数（这跟刚才前面 ACON 不一样，ACON 教模型做 summary 的时候是没调参数的，这边是得调参数才能做到）。你得逼迫训练语言模型使用这些压缩的工具，它才有办法自己做压缩。

在这篇论文裡面其实又有提到，他们试图 hard prompt 模型，努力的 prompt 模型，看看能不能够在不微调模型的情况下，让模型自己使用压缩工具。他们发现模型很难透过 prompt 的情况下稳定地使用压缩工具。所以**压缩这个能力，是需要另外训练的**。

## 五、Sub-Agent：一种自主压缩行为

我们上週也讲到 **sub-agent** 的概念。sub-agent 其实可以看作是一种**自主的压缩行为**。

1. 当模型到某一个时间，它产生一个使用工具的指令叫做 `spawn`（繁殖）的时候，它就可以产生一个 sub-agent。
1. sub-agent 仍然是跟原来的语言模型互动。sub-agent 的 context 裡面可能有一个 subtask，告诉这个 sub-agent 要做什么。这个 sub-agent 就把它的任务传给语言模型，语言模型给它一个回覆叫它用指令，它可能传回指令导致的工具的输出，然后这个状况就一直下去。
1. sub-agent 可能会有一个动作叫做 `return`。它可以执行一个工具叫 `return`，那 `return` 裡面会告诉主 Agent，现在 sub-agent 它的输出是什么，它想给主 Agent 的资讯是什么。
1. 当 sub-agent 执行 `return` 以后，它就把 return 的资讯丢给主 Agent，主 Agent 就可以继续再跟 LLM 互动下去。
1. **当 sub-agent 执行 ****`return`**** 之后，它之前所做的事情，就通通从这一串 context 裡面被抹除**。你就可以想成，这其实也是自动删除记忆的一种方式。这一整段记忆就被改成了 `return` 裡面写的那句话。所以本来一直到 sub-agent 运作的时候，它的 context 都很长，但是当执行 `return` 以后，等于执行了一个自主压缩，这一段对话纪录就通通不见了。
### 5.1 Sub-Agent 对 Context 的影响

为了让大家可以更直观地了解 sub-agent 是怎麼运作的，我截了这篇论文裡面的一张图。裡面非常清晰地用一个具体的例子，展示了 sub-agent 对于 context 长度的影响。

- 这里有一个很複杂的问题（比如关于一篇 2023 年论文的细节），是个特别拿来考验 Agent 能力的问题。
- Agent 看到这个问题之后，首先开始搜寻相关的文章。每次 Agent 执行一个动作的时候，它的 context 都会逐渐越来越长。
- 但这个语言模型是有产生 sub-agent 的能力的。所以它先产生一个 sub-agent 来搜寻相关的论文，找到相关论文以后，它只把找到的论文的标题传回给主 Agent。所以这个时候，整个 context 就缩短了。
- 然后主 Agent 会再分裂一个 sub-agent 去执行另外一个任务（比如验证作者数是对的）。这时候 context 又逐渐伸长，等找到这个作者的资讯之后，这个 context 又缩短。
当你执行 sub-agent 的时候，对 context 而言，你当然可以看作是有一个主 Agent 带了一堆小弟在工作。但是如果从 **Context Engineering 的角度**而言，所谓的 sub-agent 就是**对 context 做自主的压缩**。每次分裂一个 sub-agent 的时候，就是预示了某一段的 context 之后会被压缩掉。所以每次产生一个 sub-agent，你就可以累积 context；这个 sub-agent 结束的时候，那一段 context 就不见了。你就会看到这个 context 有**锯齿状的上升跟下降**。

它另外还特别说明说，假设如果没有这个 sub-agent，所有的 context 都不断的累积的话，最终会累积到十万多个 token，超过了他们语言模型可以吃的 token 上限。所以对他们来说，能够产生 sub-agent 也是蛮重要的能力。

### 5.2 Sub-Agent 能力的获取

不过就像我刚才说的，语言模型其实不喜欢自主做压缩，所以 sub-agent 这个能力通常不是天生的，它是**后天取得**的，是需要经过训练取得的。但今天很多模型，它都有产生 sub-agent 的能力。当你把你的 OpenClaw 接给 Claude 的时候，它有产生 sub-agent 的能力，但这可能不是一个自然原生的能力，它是需要经过特别的训练才能够具备这种能力。

在这篇论文裡面，他们就有针对产生 sub-agent 的能力做了特别的训练。怎么训练呢？他们是用 **Reinforcement Learning** 的方法。

- 他们用 reinforcement learning 去训练语言模型，希望语言模型可以得到正确的答案。
- 但他们发现，如果只用答案正确与否来当作语言模型学习的讯号的话，它不见得能够学会正确地产生 sub-agent。因为假设你的目标只是得到正确的答案，对语言模型来说，有什麼理由它一定要產生 sub-agent 吗？它就是努力得到正确的答案就好了。

## 六、治本之道：资讯过滤

好，讲到现在，我们刚才都是在提压缩，也就是当 context 过长的时候，把过长的 context 弄短。但是那是治标，我们能不能**治本**，一开始就不要让 context 过长呢？

怎么样不让 context 过长呢？那你就要分析一下，现在到底是什么样的资讯让 context 过长。这边有两篇论文都有做类似的分析，而且分析的结果非常一致。

### 6.1 分析结果：Observation 是元凶

左边这篇论文分析，假设我们没有做 context engineering，就记录在整个对话的历程中，到底模型做了什么事，这些 token 来自於什么样的行为。分析的結果是这样的：

- **Action（动作）**：模型去產生执行工具的指令，这些指令通常很简短，只佔歷史紀錄的 **6.5%**。
- **Reasoning（推理）**：语言模型自己说出来话，这个也很简短，只佔 **9.6%**。
- **Observation（观察）**：佔据了几乎 **84%** 左右的 context。这些 observation 指的是来自外界的输入，比如语言模型读了一个档案，裡面整个档案所有的资讯都变成 context 的一部分；或者它执行了某一个工具，工具有非常长的输出，那些输出变成 context 的一部分。**这些来自外界的输入，才是占据多数 context 的内容**。
另外一篇论文也得到几乎一样的结论。它主要 focus 在 software engineering 上面，让模型去改程式和执行程式。它发现：

- 只有 **12%** 的 context 是在执行程式码。
- 只有 **11.8%** 的 context 是在修改程式码。
- 多数的时候它都在读程式码，有 **76%** 的 context 是花在把整个 repo 裡面的程式码读进来。
所以佔了非常大量的 context。有没有办法直接治本，一开始就不要让这么多的文字进入 context？

### 6.2 方法：更聪明的 Read 工具

有一些论文提出了一些想法：我们也许应该在 **observation 进到语言模型之前，就做一些过滤的行为**。

一般我们在执行读一个档案或读一个文件的时候，常见的做法是：语言模型输出一个指令说“我要读一个 log 档案”，然后你有一个叫做 `read` 的工具，这个 `read` 工具就把 log 档案找到，然后把 log 档案的内容原封不动地、一口气地逼语言模型吞下去。如果这个 log 档案非常大，有时候语言模型就会哽到。

所以怎么办呢？也许我们需要一个**更聪明的 ****`read`**** 工具**。我们也许可以让语言模型在输出指令的时候，它不只说“我想要读 log 档案”，它还说“我想要读 log 档案裡面跟修复 bugs 有关的内容”。希望这个 `read` 指令够聪明，它不只能打开一个档案，还能够从档案裡面找出真正重要的部分。

这个 `read` 指令显然需要有一点 intelligence 在裡面。它去读了 log，从 log 裡面把跟 bug fixing 有关的 content 读出来。语言模型就只 focus 在跟 bug fixing 有关的内容上面。

我说这个 `read` 指令显然需要有一点智能，或者是你需要做比较多的 engineering 去 implement `read` 的这个函式。在这篇论文裡面，他们其实就是训练了一个**小的语言模型**。所以这个 `read` 本身也是一个小的语言模型，这个小语言模型本来就可以吃这个指令，根据这个指令找出合适的内容，再传给主要的 agent。

### 6.3 回顾 OpenClaw：Memory Get 与 Memory Search

讲到这边，我们也可以回顾一下 OpenClaw 是怎么处理 memory 的。在处理 memory 的时候，它其实有两个函式：一个叫 `memory search`，一个叫 `memory get`。

为什么读 memory 还需要一个特别的工具？读 memory 又不是什麼神奇的東西，它就是文字档。用一般的读档函式也可以把档案的内容通通读出来。为什么在读 memory 的时候，OpenClaw 要设计一个特别的工具去读 memory 呢？其实就是为了做到刚才讲的**过滤**。

`memory get` 这个函式，它不是只给它要读的档案，它其实还会给额外的两个数字，代表**从这個档案的第几行开始读起，然后我们总共要读多少行**。所以 `memory get` 是从整个巨大的 memory 档案裡面，只存取一部分出来。因为 OpenClaw 当初在设计的时候，害怕 memory 裡面存了非常大量的资料，把所有资料一次都读到语言模型的 context 裡面语言模型会哽到，所以它就只从 memory 裡面选一小段的內容。

至于要怎麼选一小段的內容？到底一整个 memory file 要取哪裡？那由 **`memory search`**** 的结果来决定**。根据 `memory search` 的结果，还有语言模型自己的判断，去使用 `memory get` 这个工具，只存取 memory 这个档案的一小部分。这个其实也是过滤的概念。

### 6.4 方法：按需加载 (Load on Demand)

另外一个过滤的概念就是**按需加载**。有一篇论文叫做 **MCP-Zero**。

我们一般在让 AI agent 使用工具的时候，你其实就是把使用工具的指令、有哪些工具、这些工具可以拿来干嘛，把它放在语言模型的 system prompt 裡面。这个 system prompt 在有了这些工具的指令之后，可能会非常的长。

MCP 这篇论文裡面特别提到，举个例子，比如有一个使用 GitHub 的工具，足足有 **4600 个 token**。而且这只是使用 GitHub 的工具而已。如果有更多的工具，模型就会直接超过它的 context window 可以承受的上限。

所以怎么办？工具这种东西，应该是要**动态加载**的。过去比较传统的方法是，根据使用者输入的任务再去挑选合适的工具，其实这跟 **RAG (Retrieval-Augmented Generation，检索增强生成）** 的概念是非常类似的。假设这些工具的说明都被存在一个非常巨大的资料库裡面，今天有一个新的任务进来，根据这个任务去启动一个搜寻引擎，去工具的资料库裡面进行搜寻，把相关的工具指令抽取出来，让模型知道有哪些工具可以用。

但这篇论文发现，这不是一个特别好的方法，因为今天使用者的需求往往非常的模糊，不容易根据使用者的需求来判断需要使用哪些工具。比如今天使用者的需求可能是“帮我修改这个 bug”，但是修改 bug 要用的工具不止一个。模型会需要至少先用读档的工具能读档，然后看了这个程式以后，再用一个编辑的工具才能修改这个程式。所以今天虽然使用者只说了“帮我修改这个程式”或“帮我 debug”，但被唤用的工具其实是好几个。你很难直接从使用者问到的问题，让搜寻引擎决定要使用哪些工具。

所以这篇论文提出来的核心想法就是：**我们何不讓语言模型用 AI 动态地决定它自己需要什麼工具？** 让语言模型输出它想要什么。所以语言模型可能在读到任务的指令之后，它想一想，就输出一个任务的需求、一个工具的需求。用这个工具的需求去操控搜寻引擎，让搜寻引擎找出它需要的工具。然后它就可以使用它需要的工具来解接下来的任务。

这件事情，你可能觉得听起来很熟悉。这个其实也是 OpenClaw 裡面所用的 **skill 的概念**。skill 这个东西，它也是按需加载的。我们不会把所有的 skill 都放到 context 裡面，不会把所有的 skill 都放到这个 prompt 裡面。只有在需要的时候，才从硬碟裡面把 skill 读出来，放到 prompt 裡面。这个就是按需加载的概念。

## 七、进阶探索：Agentic Context Engineering

好，讲到这边，我们讲了一些有关 Context Engineering 的想法。但到目前為止，多数 Context Engineering（也就是这个大 `F` 要做的事情）都是**人类决定**的：人类编写好、写死了固定的指令，然后让你的电脑、你的程式按照这些写死的指令来处理 context。

但是，有没有办法让这个 Context Engineering 做得更複杂、更有智慧、做得更好呢？有一个想法叫做 **Agentic Context Engineering**。这个说法就来自于一篇叫《Agentic Context Engineering》的论文。这裡想法的核心就是：**把 Context Engineering 也交给语言模型**，不给人类设计，直接交给语言模型，让它自己想辦法帮自己做 Context Engineering。你觉得本来大 `F` 是人类工程师设计的，现在直接交给语言模型工程师，看看它有没有更好的想法。

### 7.1 核心概念

这个 Agentic Context Engineering 的概念，如果要画成图的话就是这样：

1. 你现在有一个 context `C`*`t`**，加上一个输入 **`I`*`t`，语言模型给一个输出 `O`*`t`**。*
1. 接下来，把 context、输入、输出全部串起来，直接丢给一个语言模型（**Agentic Context Engine**）。它爱干嘛就干嘛，得到一个新的 context，我们叫 `C`*`{t+1}`**。*
1. 然后，前面可能接个 system prompt，后面加个 input，再从语言模型那边得到 `output`*`{t+1}`**。*
1. 然后再有一个语言模型，把 `C`*`{t+1}`**、**`input`*`{t+1}`、`output`*`{t+1}`** 再变成 **`C`*`{t+2}`。
1. 这个步骤就反覆继续下去。
我这边在画图的时候没有把 system prompt 加到 context 裡面，你要把 system prompt 视为 context 的一部分也可以。不过如果你细读这些 Agentic Context Engineering 的论文的话，你会发现实际上它让语言模型自己处理的 context 是全部 context 的一部分。它们通常就是留一个**区块**，这个区块给语言模型爱玩什么就玩什么。但是比较重要的东西，比如 system prompt（裡面包含了这个 AI agent 本身的 identity，真正重要的资讯），这个是不能随便乱动的，所以通常你就固定住它，不要动它。这一系列 Agentic Context Engineering 的论文通常只改整个 context 裡面的其中一部分而已。

### 7.2 案例一：Dynamic Cheatsheet

Agentic Context Engineering 一个比较早期的论文可能是 **Dynamic Cheatsheet**。它就把这个 context 叫做 `cheatsheet`（小抄），只是这个小抄是会随时间变化的。概念非常简单：呼叫一个语言模型，给它一段 prompt，跟它说“我们这个 context 要怎么改比较好”。然后它就把 `context`*`t`**、**`input`*`t`、`output`*`t`** 改成 **`context`*`{t+1}`，就结束了。这边它的核心就是 **prompt engineering**。把这段 prompt 写好，希望语言模型读了这段 prompt，自己知道怎么做 context engineering，也就是用 prompt engineering 来做 context engineering。

这段 prompt 非常的长，仔细读下来，它的核心精神就是：**存下未来能用的东西**。它告诉语言模型说，你不要存一些很 specific 的东西，你要存的是精神概念。比如什麼有效的策略，可以把它存起來；如果你写一段程式，你覺得之后用得上，也把它存起来；关键的发现也存起来。但是非常跟现在这个任务有具体关联的东西，那可能之后都用不上了，就不要存起来。总之，这裡就是用 prompt engineering 来做 context engineering。

### 7.3 案例二：Agentic Context Engineering (Playbook)

像这样的论文现在很多。比如《Agentic Context Engineering》这篇论文裡面，它就做了更複杂的流程。它把它的 context 取了另外一个名字叫做 **playbook**（守则手册）。它希望语言模型看着这个手册做它应该做的事情。

这篇论文实际上的事情我就不细讲了，总之这个 playbook 的演化不是只有一个步骤，要经过**三个语言模型**。这三个语言模型分别做了不同的检查之后，最后产生一个**修改 playbook 的指令**。它不是直接產生新的 playbook，因为它怕直接产生新的 playbook，搞不好一些旧的资讯就被弄坏了。所以它是去修改旧的 playbook。这三个模组合起来，会产生一个修改的指令，用这个修改的指令去修改原来 playbook 的内容，把 `context`*`t`** 原来的 playbook 变成 **`context`*`{t+1}`，变成一本新的员工守则。这是这篇论文做的事情。

### 7.4 案例三：Recursive Language Model

还有另外一篇论文，叫做 **Recursive Language Model**，它其实也是 Agentic Context Engineering 的其中一个可能性。这篇论文曾经一度非常的知名，因为它号称发明了一个新的语言模型，这个语言模型可以吃**无穷长的输入**。

其实，它真正做的事情就是 **Context Engineering**。它做的事情是这样的：

1. 假设现在的 context 真的可以非常非常的长。
1. 那非常长怎么办呢？就通通放到 hard disk 裡面。我们只把这边的 context 拿非常非常小一部分 load 到 memory 的 prompt 裡面。
1. 它这边论文裡面说，这些资讯叫做 **metadata**。比如它只记了现在这个 context 到底有多长，context 被切成几段、存在哪裡等等，非常简短的资讯。
1. 如果用我们这篇论文的符号来讲：
1. LLM 做的事情就是，看着这个 `P`，然后去详细看看它要从 `M` 裡面找寻什么样的资讯。
1. 这个语言模型很厉害，因为它可以写程式。它会写程式去对 hard disk 裡面的内容做搜寻。因为它自己自主地知道要做 RAG，然后它自己知道说，它要去 hard disk 裡面搜寻一些东西出来，把这些搜寻的东西拿出来去修改它的 metadata。所以 `P`*`t`** 就变成 **`P`*`{t+1}`。
这篇论文裡面还有一个章节讨论了这个模型的这些自主產生的 pattern。但是实际上到底有没有那麼神奇，真的是见仁见智。因为如果你仔细去读它的 prompt 的话（因为这些做 context engineering 的 LLM 背后是需要做 prompt engineering 的），就只差没有教语言模型说“你直接做 RAG 了”。它花了蛮多力气不断地暗示语言模型说你可以做 RAG 这件事，所以语言模型就写了一个程式做 RAG。实际读它的 prompt，我觉得也没那么神奇。

不过它的表现是非常好的。它说，你看原来的 GPT-5 是个很好的模型，如果输入越来越长，到某个长度，有些任务它就解不了了（这些 LongBench 就是测试语言模型在 context 非常长的情况下能不能好好运作）。但是加上它的 Recursive Language Model 以后（因为 Recursive Language Model 是一个 context engineering 的方法，可以外挂在任何现有的语言模型上），如果外挂在 GPT-5 上，就可以让本来已经很厉害的 GPT-5 变得更厉害。在输入真的非常长，比如长达 **1M（100 万）个 token** 的时候，仍然可以在这些长 context 的 benchmark 上面做出不错的效果。

## 八、总结

总之，今天就是比较系统化地跟大家介绍了 **Context Engineering**。

- 我们有一个演算法摘要了 Context Engineering 实际上做的事情。
- 我们也跟大家说，其实 context 分成两部分：一部分是 **`M`**，存在你的 hard disk 裡面；一部分是 **`P`**，是真的会当作 prompt 丢给语言模型的。
- 有一系列比较新的研究尝试说，把 Context Engineering 裡面这个最关键的 `F`，能不能不要由人类工程师来设计，也把它交给语言模型。
这个部分要跟大家分享的就是 Context Engineering。
