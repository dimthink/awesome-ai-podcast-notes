## 二、工具混淆问题：DeepSeek 为何表现不佳

**Ahmad**：我有大量的编码经验，我写代码大概有 27 年了。在发布了 300 多个开源项目之后，你会对很多事物有自己的看法。而且我主要做的是非常前沿的工作，很多时候没有文档可以让 AI agent 去查阅。所以那时候，我觉得我的观点比 LLM 能通过 RAG 或其他方式找到的东西更重要。于是我把这种行为编码到了元神经符号架构（meta-neuro-symbolic architecture）里：如果你从我这里学到了什么东西，就把它作为一个技能（skill）为我记录下来。我们开始称它为 **Taste**。比如，如果你发现我经常使用 pnpm，但在链接本地 CLI 时又用 npm global link，那么它就会学到：在几乎所有其他场景下我倾向于用 pnpm 安装包，但在本地链接 CLI 时我会用 npm。这类微小偏好和学习积累最终变成了 **taste 文件**，和技能文件（skill files）很像。你可以理解为 Command Code 会自动从你身上学习——在每个仓库级别学习——从而构建一个技能库。这个库并不啰嗦，它只记录那些反复出现的偏好和模式。这些模式可能来自多个不同的编码 agent。当你把代码合并到 main 分支时，我们就可以触发分析哪些是你的接受、编辑和拒绝。不过，我们今天不聊这个。

你知道，在技术领域有一件很傻的事：你认为是会爆红或被广泛采用的东西，往往不会；而你随手丢出去、根本没在意的东西——比如“谁在乎开源模型啊？”——反而成了热门。今天是 5 月 25 日，大概 25 天前，所有人都在说 DeepSeek 多好多好，也有一堆人说 DeepSeek 多差多差、为什么有人会觉得它好。我当时站的是：“我需要自己判断 DeepSeek V4 Pro 是不是真的和 Opus 一样好。”那时候我们每天处理的 token 量大概在几十亿，现在已经比那时多了 100 倍。我最终发现了一个我称之为 **工具混淆（Tool Confusion）** 的奇怪现象。

我们成功找出了一种确定性方法，来修复开源模型（open models）的工具调用问题。这件事非常棒，很多人已经通过 Command Code 在尝试了。我也把很多这部分工作完全开源了，你可以拿去在任何编码 harness 中实现它。因为，你知道，对我来说这没那么重要。所以，看看你想让我在这个话题上深入多少，我很乐意聊。

**采访者**：好，我觉得我们可以直接深入进去。你有一篇爆红的帖子想分享吧？另外，也许可以先向那些只用过 OpenAI 的人解释一下问题框架——他们可能不知道什么是工具调用问题。

**Ahmad**：你可以这样理解：Command Code 的整体理念是，你应该能够命令你的 LLM 做任何你想让它做的事。这是我们的目标。第一步是构建 Taste，这样我们就可以引导模型走向正确的方向。然后第二件好事发生了：程序员不再写代码了。那我们跟谁去学习 Taste 呢？于是我们开始从模型那里学习 Taste，因为现在是模型在写代码。当你在 Claude、GPT、DeepSeek 上花费超过十亿个 token 时，你会发现整个堆栈中存在大量不一致的地方，尤其是在不同的全栈系统中——比如 DeepSeek 在某些方面（例如 Go 语言代码）表现就不太好。我把所有这些称为 **变体（variations）**。我们发现了这些变体中存在确定性的模式。

当你在运行编码 agent 时，工具调用最基本的情形就是：你试图运行一个 bash 或 shell 工具来探索环境，比如列出目录、读取各种文件。用户可能问了一个关于“这个仓库的身份验证是怎么实现的”的问题，你的编码 agent 正试图用一堆预设的工具来列出所有文件、读取一部分、进行探索，然后回答用户。（笑）但很多开源模型在工具调用方面确实很差劲。而且这种模式并不是特别普遍，而是非常特定于某个模型。比如 DeepSeek V4 Pro 有一种奇怪的“阿尔法男性能量”，它发给你什么，它就认为那是对的。如果你发给它错误的工具调用模式，你返回一个 Zod 错误，它根本不理你，会重复同样的错误平均 50 到 60 次——在十亿 token 的量级上。你会想：“为什么它就是不听错误信息呢？”这不是很常见的事吗？Instructor 以前就这么做。

**采访者**：为什么它们就是不听取错误信息呢？我还以为这是个很普遍的问题。

**Ahmad**：我也不知道。作为一个程序员，我以为如果我的工具调用失败了，我只需要发回一些模式信息（比如 Zod 错误），一个足够聪明的 LLM 就应该能修正。但直觉告诉我（可能是个大胆的猜测），因为很多开源模型是在一个“被告知的东西都是正确”的系统里训练出来的——它们从被认为是高质量的数据中学习，比如从比自己更好的模型那里学习。训练的本质就是你被告知的就是正确的，所以它们的整个天性就是“我告诉你的也是正确的，别想纠正我”。当然，这很大程度上是基于直觉。

## 三、修复逻辑：让 DeepSeek 真正可用

**Ahmad**：我之所以发现这个问题，是因为一件很傻的事。比如说（我可以分享屏幕吗？），我在帖子里也提到了。比如，当模型有一个 shell 工具，它有一些可选参数或参数，它却会发送一些奇怪的东西——比如一个空对象或 null，而 Zod 非常严格，就会返回错误。然后它又会不断重复同样的工具调用。我不再返回错误，而是去修复它。一开始只是大约 **3,200 行修复代码**。可以把它看作数据库迁移：一个迁移对应一个文件。我创建了修复文件——如果它发出了一个 JSON 字符串类型的数据，而实际上我想要一个数组，我就会确定性地把它修复成数组。当我做这件事时，我不仅会返回结果，还会附带一个提示（repair hint），比如：“你应该发送给我这个类型的数据，但这里是结果，你拿着用吧。”就像教人开车，他们快要撞到另一辆车时，你不是先告诉他们正确的操作，而是先帮他们避免事故，然后再解释为什么要这样做。我发现很多模型很喜欢这种修复逻辑，因为一旦你发送了附带修复提示的结果，紧接着第三次工具调用就被修复了。模型突然就变得非常聪明，它理解了：“好的，我拿到了想要的结果，我会这么继续下去。”这种效果在很多地方都体现出来了。比如，它试图读取一个文件，却没有给出偏移量——是从文件顶部读 100 行还是从底部读？那我就做一个判断：这是第一次读这个文件，那就给它最前面的 100 行。然后它很快意识到，“哦，我其实是想读一个日志文件，需要最后 100 行”，就能非常容易地理解，而不是像之前那样平均失败超过 50 次工具调用。这件事对开发者来说应该很明显，但现实是很多开发者实际上在用 Claude Code，他们只是改了基础 API 端点替换成开源模型，并用 Claude Code 作为 harness。而 Claude Code 把很多错误隐藏在 `Control+O` 后面，你根本不知道一个 session 里有超过 50 次工具调用失败。你只是坐在那儿觉得：“啊，为什么 DeepSeek 这么慢？”而对于 Claude Code 来说，他们也有自己的利益不想修复这个——他们根本不关心开源模型，他们不是为开源模型构建的编码 agent。所以这对他们很好，但对普通团队来说，Twitter 上大家就在说“哇好棒”和“怎么这么慢”，这其实总是归结为工具调用 harness 的问题，而不是模型本身的问题。它可以非常傻——比如在发送读取文件路径时，模型会莫名地创建一个 markdown 链接，这种东西是完全可以确定性修复的，你不需要浪费 token 在上面。

**采访者**：嗯，我觉得阅读实际数据并看到你真正做了什么，非常有用。我想知道这是不是只出现在 DeepSeek 上，还是所有开源模型都有这个通病？

**Ahmad**：是的，我一开始也以为只是 DeepSeek。然后我查看了过去 30 天的日志，发现 Kimi 也在做完全相同的事。然后我们修复了 Kimi 模型、MiniMax 模型，现在我们有 **16,000 种不同的修复变体**，覆盖了数千亿个 token。我们现在每天处理大概 6000 亿 token。工具调用失败的数据非常非常重要。总体而言，这能让一个几乎完全不可用的模型（比如 DeepSeek V4 Flash）变得可以和商用模型竞争。我发布了这次更新后，我们的一个投资人，Tom Preston-Werner 基金的一位 GP 说：“你做了什么？为什么 DeepSeek V4 Flash 现在变得这么稳了？”模型的“气质”完全变了，它会用不同的方式做事。不知道你注意到没有，如果你在编码 agent 中启用权限确认，模型反而会变笨；如果你完全绕过权限确认，它们会表现得好很多。即使你不断地“是是是”地接受，你会因为权限确认块的缓慢而导致模型被引导到错误的方向。也许那不是模型训练的方式，但和这里是一个道理：当模型看到更少的工具调用错误，它们会更有创造力，能探索更多，能持续工作更长时间。比如我们有一个用户，他在 DeepSeek 上用 Command Code 跑了超过 **700 亿 token**。我看了他的数据，他用得太多了以至于撑破了我们的使用量统计页面，我们这才发现他。他说：“我经常用 DeepSeek 跑 Command Code，每次 session 超过 12 个小时。”我自己都没试过这么长。所以，当工具调用错误大大减少时，这种修复逻辑真的会让你大开眼界——开源模型其实可以有多好。

## 四、设计 slop 的类似应用

**采访者**：这真的太有意思了。我想问，你们还发现了什么？这方面的后续工作还在进行吗？自从你做了这些研究，你们有没有持续产生类似的洞见？

**Ahmad**：是的。首先，我们可能现在是用在 DeepSeek 上最多的编码 agent。所以我们还在做大量的 DeepSeek 推理。有很多人在 Twitter 上 @ DeepSeek 研究团队。为了证明这一点，我们推出了一个仅 **1 美元/月的计划**，包含 6 亿 token 的免费 DeepSeek Pro 使用，就是为了证明开源模型其实真的很好、正在迎头赶上。我认为这也最终促使 DeepSeek 看到了他们可以降低价格并同样去证明他们的模型真的很好。但与此同时，我也看到很多不同编码 agent harness 公司的人在这个推文下被 @，问“你的东西是不是也在这么做？”而且很重要的是，人们完全可以把这个修复逻辑整个写成一个 prompt，让模型自己修复自己。

不过，最近最有趣的一件事是，我们在 **设计 slop** 上应用了同样的思路。你知道那种所有 LLM 都会犯的靛蓝紫色渐变问题吗？

**采访者**：我猜 Mario Zechner 最近有一条很火的帖子就是关于这个的。你认识 Mario Zechner 吗？

**Ahmad**：是的，是的。他发的帖子里有一个设计 slop 的例子。大家应该都知道。我其实很喜欢紫色，但我想指出那是靛蓝 slop，不是紫色 slop。我们发现在处理了数千亿个 token 之后，我们可以将同样的修复模式应用在设计 slop 上。我们和一些品味非常好的设计师聊了聊，发现这也非常相似：LLM 在做设计时有一套有限的问题模式。如果你能给它们一个 **组合框架**，就可以修复它们的设计思维。比如大概有 10 条大规则，你可以把它们放进一个技能文件（skill file），设计就会变好。我们在不同 LLM（商业或开源）上都看到了重复验证。这和修复工具调用的逻辑本质上是一样的——你引导你的编码 agent“不要做这个、不要做那个”，如果它做了，你确定性地修复。有一个很棒的发现是：第一个模式叫 **工作模式优先组成**。当你让模型“给我设计一个仪表盘”时，它们通常不会考虑设计背后的意图，直接就甩给你三张卡片一行、左边或上边加个边框之类的东西。如果你给它们一个非常简单的框架——比如“我们是在寻找什么类型的表面区域”，实际上就只有七个模式——模型就会表现得非常好。

**采访者**：这七个模式是从哪里来的？是你自己想出来的吗？

**Ahmad**：是的，我和很多不同的设计师聊过。

**采访者**：我只是好奇，是不是有某本书或某种公认的设计原则？

**Ahmad**：你可以这样想：当设计师设计仪表盘时，他们会思考“这是用来监控的表面”，这就是意图。这来自我们和很多优秀设计师的交流，他们就是这样工作的。我们想，如果把这个变成一个 `/design` 技能，看看 LLM 能不能自我修复。我们还发现了一个很有意思的事：我个人不使用 OKLCH，但显然 LLM 很擅长。如果你让它们用 HSL，它们不能很好地控制亮度，人眼很容易看出两个颜色看起来不一样。但如果你强制 LLM 使用 OKLCH，它们就能很好地控制调色板，比用其他色彩空间好得多。这也是我很多设计师朋友喜欢的方式。

**采访者**：这就是我们发明 OKLCH 的原因。每次色彩理论进步，我们还得改 CSS 函数，这对学习 CSS 来说有点烦人。

**Ahmad**：对。但我总有一种挥之不去的感觉（我昨晚刚写了一段相关的东西），对于设计 slop，我们目前只有大概 24 个参考文档、10 个设计坏味道（design smells）和 7 个从不同设计师那里看到的模式。当我们用这些规则设计了一些落地页并让设计师去评判时，他们大概 1.5 秒就能看出“这是 AI 做的，那边是有人类审校过的”。他们发现的差异，我们写下来之后，都是可以确定性修复的模式。比如把 HSL 转换成 OKLCH，或者强制 LLM 这么做。感觉你可以修复 90% 的设计 slop——这根本不是能力差距，而是 **契约（contract）差距**：你的 harness 告诉 LLM 要做什么，和用户真正想要什么之间存在差距。用户总是会说“把设计改好看点，让它更突出”。如果你能给他们一个框架——优秀设计师的设计品味是什么样的、他们会用哪种配色方案、他们会先思考意图再实现落地页——那么设计 slop 就会降到极低。我觉得值得补充一下。

**采访者**：我觉得这是一个非常好的概述。而且对于那些不了解的人，Aman（Ahmad）在设计方面其实也很有发言权。我自己其实都不完全了解你做过什么，但在设计与代码交叉领域，我肯定会听从你的想法。

**Ahmad**：非常感谢。我一直很想成为一名设计师，但从来没有特别擅长——比如我不会素描之类。但我觉得，现在很多人能用 AI 构建任何东西，区分好工作和坏工作的标准变成了设计。所以让每个使用编码 agent 的开发者都能像设计师一样设计，是我非常看重的事情。

**采访者**：是的，我想我们都在变得更通才。有些东西我以前根本不敢尝试，但现在有了 AI 我会去试。你好像想给我们展示一下 Command Code？请吧。

## 五、Taste 系统：自动学习的编码偏好

**Ahmad**：我们现在有两个主要方向。Command Code 本身是一个 CLI，一个全功能的编码 agent，能做其他编码 agent 能做的所有事，支持商业模型和开源模型。但我们发现我们的 PMF（产品市场匹配）更偏向开源模型市场。Claude Code 在工具调用上其实非常宽容，即使 harness 搞砸了，它也能发现错误并自我修复。开源模型不行。但 Taste 系统仍然存在。它大体是这样工作的：我很常见的一个场景是，当我用某个东西时，我希望用 TypeScript、用 TS Up，但模型用了不同的测试框架，而我偏爱 Vitest。你会需要来回调整让编码 agent 做你想要的事。而 Taste 就像一个元神经符号模型，根据反馈，我把它很好地隐藏在了文档某个地方，这样开发者不用去读那些很傻的东西。（笑）我们采用了一种 KL 散度循环：如果 LLM 已经知道某件事，就不应该出现在你的技能或 taste 文件中，因为那是绝对无用的上下文。它做了一系列奇怪的事情：一旦你用 Command Code 构建了一个 CLI、一个 API 或前端项目，它实际上会自动学习一系列规则，并且自动管理。比如你习惯用 Commander 构建 CLI，但在这个项目中你开始使用 Meow，它会帮你替换掉。核心思想是：你的技能是自动学习和自动管理的，并且完全透明——它们存放在你的 Git 仓库里，而不是我们的模型里。这样你在每个 PR 中都能审查它们。你看着会觉得：“嗯，我不想保存这个。”它们永远不会过时。我觉得很多人遇到的问题是因为他们的 `agent.md` 或 `claude.md` 在某个时候包含了错误信息。这是我们宣布 500 万美元种子轮时讨论过的事情：很多人坐下来想规则时，脑子里想的都是特别宏大的东西。规则像……（示意缩放）这是我们的技能文件或规则文件 `.ts` 和 `.md` 的区别。这是你手动写的东西，而 `.ts` 则是从你的提示和编辑中持续学习并存储在同一 markdown 文件中的内容。手动更新靠你记得去做。但大多数时候，我们人类坐下来写那种宏大规则，比如“我总是偏好这个”，而 TS 则包含大量微观决策，不宽泛。比如，当你运行 `up/pr q` 命令时，有时 Emmet 希望你先 fetch main、rebase、在分支上发 PR，然后回到 main 分支。那是我的 PR 工作流，但我的 PR 文件里没写。它看到我一次次回到 main 分支，就会自动修复，哪怕只是一行，却改善了我的工作流。长期来看，这种积累非常有效。比如，有一个人用 Command Code 构建了一个 Android 应用，现在他就能非常快速地构建更多 Android 应用了——因为在构建过程中 Taste 从你身上学习了技能。我们后来对 70 多名开发者进行了一项研究，发现因为 LLM 转向了不同方向而需要手动编辑文件的次数显著减少了。他们用 Taste 引导 LLM 说“不要用这个，不要用 TRPC，用 Hono 做 API 这部分”，他们的编辑或引导次数下降了。

**采访者**：所以，让我问几个澄清的问题。你的网站上写着“一个模型”，但你又说是跨系统可移植的。它到底是一个系统文件？这显然是一种记忆形式。它和 skills 怎么比较？我应该优先用 skills 吗？可以同时使用吗？有很多这类问题。

**Ahmad**：是的，是的。我收到过很多类似问题，还为此写了一篇博客：**Skills vs Taste**。在最基础的层面，Taste 是最高层级的抽象，它管理着你的 skills 和 rules。Skill Taste 是一个自动引擎，为你创建 skills，确保它们不会过时，当然你也可以自己手动编辑。整体上看，大概像这样——我用 Command Code 构建了超过 70 个 CLI，这是我构建 CLI 的 Taste 全部内容，非常小。它知道我用 pnpm 构建依赖，知道我从 0.0.1 版本开始，都是这种小事。我只需要运行 `npx taste pull`（我们也有 taste 包），它就把这个文件拉下来放到你的仓库里，然后你可以让任何编码 agent 遵循我的 CLI 构建品味。然后它会遍历整个列表，知道自己应该用这个、用 Clack 做交互等等。当你不断在上面构建，每个 CLI 项目都不同，taste 文件会在这个仓库里变化。它存在于你的 Git 仓库里，你可以随时自己编辑，它是 markdown 文件。

**采访者**：那么置信度（confidence）也是 LLM 生成的对吧？比如如果你总是那样做，它就会是 1。

**Ahmad**：对，类似这样。我们早期曾尝试隐藏这些学习内容，让人们对比 Claude Code 和 Command Code，开发者会说“哇，这个真好”，但他们不知道到底学到了什么。但我们遇到一个难题：我们不知道如何合并不同的 taste 文件，这非常依赖人工决策。试想有 15 个工程师在一个有上千个分支的 GitHub 仓库里工作，不同分支的 taste 文件差异巨大，我们无法判断你想保留什么。所以我们让它变得完全透明。这其实是在 skills 这个概念出现之前的事，大概是去年十月。当时还没有 skills 的概念，只有 APIs、CLIs 和 MCPs。我们创造了这个，后来 skills 概念才流行起来。所以如果你作为人类手动写规则，你可以直接写置信度 1，因为你清楚自己在说什么。如果规则是从你身上自动学习的，有些很好，有些则一般。比如我总有一个 `--local` 选项，我不希望它显示在帮助命令里，因为只用于调试本地 API。这种细节我永远不会手动写进技能文件，因为太小了。但它能自动捕捉这种小东西。我在 Discord 社区看到，很多人先用一个高质量 LLM（如 Opus 或 GPT 5.5）构建一个项目，生成 taste 文件，然后用超便宜的模型基于这个 taste 文件继续构建。

**采访者**：嗯，我觉得这个系统对我来说是合理的。很好的概述。

## 六、未来计划与开源

**采访者**：如果你有兴趣，大家可以去关注你进一步交流。你们在这个领域已经耕耘了一段时间，很高兴看到你们专注于编码用例。我自己对编码 AI agent 的兴趣也越来越浓。最后还有什么想说的？或者还有什么值得期待的新东西？

**Ahmad**：我可以稍微分享一下我们的路线图。我们很快就会**开源 Command Code**，希望能在旧金山的 AI Engineering 大会上宣布——如果我能处理好那个六年历史的仓库的种种细节的话。

**采访者**：六年？等等，是因为它之前是 CLI 对吧？我明白了。

**Ahmad**：对，我从 2020 年开始构建它，去年才变成产品。我曾经对所有人（包括他们的父母和兄弟）说过：“这绝对不会成为我的产品。”事情就是这么有趣。不管怎样，我们要把它开源，目的是让 Command Code 完全可黑客化。我在这方面花了很多精力。我的背景也与此有关：我在 WordPress 核心贡献了 13 年。Matt Mullenweg 现在是我们的一位天使投资人，他听说我们要开源 Command Code 就联系了我。我们的想法是，无论我们的商业模式怎么走，你都应该能修改 Command Code 的任何部分。另一个我极力主张的想法是：我们不会把这个东西变成一堆模型的“大杂烩”（比如支持 1500 个模型）。我把它比作三种哲学：第一种像 Windows，所有游戏都能跑——我觉得 Open Code 就是这样，所有模型都能用。第二种像 Linux，你构建自己的驱动——Cursor 就是这样，它的扩展性很强。而 Command Code 我想走 Apple 路线：它会内置最好的模型（包括开源和闭源），不会包含所有模型，但它是可黑客化的，所以你可以自己接入本地模型或其他模型。我们很快就会开源，非常快。
