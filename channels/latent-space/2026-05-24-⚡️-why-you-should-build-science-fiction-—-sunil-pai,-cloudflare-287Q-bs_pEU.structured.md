## 开场：使命与最新动态

**主持人：** 今天我们的嘉宾是 Sunil Pai，code mode 的创造者。

**Sunil Pai：** 不，我必须承认 code mode 确实是我创造的。动态隔离……这是我从小就在思考的事情，是我一生的使命。

**主持人：** 我想了解 Cloudflare 最近的所有动态。当然，这些年来我们也一直保持联系。最热门的事情是昨天刚发布的 **Cloud Manage Agents**，我知道你也很感兴趣。这算是爆炸性新闻了，而且你看起来非常兴奋。

**Sunil Pai：** **Cloud Manage Agents** 确实很酷——Anthropic 说这是一个让你构建和部署长时间运行代理的平台。我喜欢 Anthropic 的人，他们对这些事情非常有野心。但当我看到这个产品时，我的想法是：*我想竞争。* 我认为我们可以用 **Workers** 和 **Durable Objects** 做出更好的方案。

**主持人：** 没错，Cloudflare 是这个领域的理想平台。你们早就有 Workflows 了。Cloudflare 现在到底有哪些功能？为什么你不能直接站出来说“我们已经有了这些”，两者的区别在哪里？

## 一、Cloudflare 的核心优势：两大原语

**Sunil Pai：** 我认为，就像其他事物一样，我们正在发现构建软件的新架构——几个小时后我将在舞台上演讲（下午 5:40，请到主舞台）。但有趣的是，有两个原语我认为至关重要，而且我怀疑未来每个平台都迟早会拥有它们。

### 1.1 Durable Objects：有状态的无服务器编程

第一个是 **Durable Objects**，它是有状态的无服务器编程。我认为这是一个真正的创新，是世界上第一个在基础设施层（而非用户空间）实现的 Actor 模型。这意味着你可以启动数以百万计的在后台运行的有状态对象，而且它具备无服务器的特性——无需启动整个虚拟机。

### 1.2 Dynamic Workers：安全运行 AI 生成的代码

第二个是我们几周前刚刚宣布的 **Dynamic Workers**。众所周知，`eval`——评估用户生成的代码或 LLM 生成的代码——一直被认为是一个坏主意。但我们构建了一个可以**在安全环境中以零启动时间运行代码**的东西。我所说的“安全”是指：你可以只暴露你想要的 API，并且控制所有出站流量。事实上，我们推荐的使用方式是默认取消所有出站流量，只使用暴露的端点。

有了这些能力，我们相信可以构建出一种用于 AI 代理的更高效的全新架构。

## 二、Dynamic Workers 的应用：MCP 服务器

**Sunil Pai：** 不久前我们小范围展示的 Dynamic Workers 的一个应用，是我同事 **Matt Carry** 构建的 **MCP 服务器**。Cloudflare 的 API 有 **2600 个端点**，如果你为每个端点都暴露一个工具，那你就完了。你可能会说“我用渐进式暴露啊”，不，我们只用了两个工具调用：**搜索**和**执行**。对每个调用，你都可以提交代码，在隔离区中运行。

比如，你可以提交一段 JavaScript 代码来搜索整个 Open API JSON；第二次调用时，再提交一段执行代码，比如“找到我所有的 Workers，找到它们使用的 DNS 记录，如果它是以字母 Z 开头的，就在一次工具调用中为其应用 DOS 保护”。没有与 LLM 的来回沟通，而且代码是类型检查的。事实证明，LLM 非常擅长运行代码。所以这是一项基础能力，不是你能简单打补丁就有的。这让我对 Cloudflare 在代理软件上的未来感到兴奋。

**主持人：** 你有没有仔细看过 Anthropic 的 Cloud Manage Agents？

## 三、对 Anthropic Cloud Manage Agents 的评价

**Sunil Pai：** 我确实喜欢他们的东西。他们特意指出**管理环（harness）与执行环境是分离的**，因为他们准备迎接一个过去不存在的软件世界。多么美妙的面向未来的视角！顺便说一句，这不只是在构建下一个 CRUD API。我们甚至不知道未来的形状会是什么样。而这些代码需要在沙盒环境中运行，我们确实有关于如何通过管理环来暴露能力和智能层的想法。我认为这非常棒。

**主持人：** 你认为这最终会成为标准吗？像 Cloudflare 这样的公司可能会接入某个标准化的统一层吗？你们对不同路径有什么赌注？每个人都在为未来软件引擎的版本而构建。你有自己的愿景吗？

**Sunil Pai：** 这非常困难。但我脑子里在想的框架是：**还没有人创造出“React 时刻”**。你还记得吗？React 刚出现时，很多人都觉得它很奇怪——你可以用 XML 来描述组件树。2013 年，在 React 的演讲中，有人直接离场，说“Facebook 讨厌 JavaScript”。但就是那个东西定义了此后所有的框架，直到今天 React 依然庞大。

我怀疑现在每个人都在构建自己的“管理环”，每个人都认为“应该是这个形状”、“应该是那个形状”。**需要有人提出原创性思考，构想出一种可跨语言、跨公司、跨基础设施复用的东西。** 我怀疑我们还没有做到这一点。也许答案就是 **Skills**。我觉得 Skills 是终极的翻译工具。它只是一个版本，用的就是英语。规模很好，就是英语。你可以抽象它，也可以独立创建 Skills。

**主持人：** 就这样吗？如果未来说软件就是 Markdown 文件，我会非常开心。漂亮、人人可用、多语言桥接世界。也许答案就是 Skills。但 Skills 最终也会遇到一个问题：你需要越来越具体，到某个点你不禁要问：*我还不如直接用代码表达。*

**Sunil Pai：** 是的，是的。我对自己的东西非常精确。而且 Skills 本身就捆绑了代码，所以还有很多其他因素。

## 四、Slop Forks 事件：与 Vercel 的冲突

**主持人：** 还有其他话题。我知道你很乐意聊各种不同的话题。我们来聊聊 **Slop Forks** 吧。

**Sunil Pai：** 我爱 Slop Forks。

**主持人：** 你和 Vercel 之间发生了什么？我们直入主题吧。

**Sunil Pai：** 你知道吗，我很愿意谈这个。我在这个行业呆得够久，到处都有朋友——现在的、以前的、未来的同事、竞争对手。我把这个行业描述为“旋转门森林”。所以我们一直保持坦诚和友好。事实上，我们很喜欢 Vercel 的 **AI SDK**，我的 Agents SDK 有一大部分就构建在其之上，我觉得我们为它构建了最好的后端。我和 Vercel 的人也有私人关系，比如今天晚些时候我还会和 Nico 一起喝杯啤酒。

但是 Twitter 会把“戏剧性”放大，尤其是管理层之间的。我通常远离这些。不过，竞争对手之间确实有一些紧张感，因为平台本身就鼓励这种对立。事情的经过是：我参加了 **JSCON**，和 Vercel 的 **Javi** 一起玩，他绝对是个 10 分满分的人。我在西班牙玩得很开心。当时我在试玩 Vercel Labs 新出的一个项目 **Just Bash**——一个纯 JavaScript 实现的 Bash，太棒了。我心想：我当然想让它在 Cloudflare 上也能跑。我做了什么？我把 Opus 指向它，然后去吃了个午饭。回来后它就生成了 5000 行代码。我说“好吧，不错”，就直接发布了。我的全部计划是：因为我很尊敬这些人（这里不点名了），我本打算在西班牙好好玩，回家后把代码打磨一下，然后在周一发给一个 PR，附上 issue 说明——就像我对所有其他开源仓库做的那样。

我回去后倒头就睡了。醒来时，我发现 Cloudflare 的管理层在 DM 里问我：“你看 Twitter 了吗？”我上去一看，**Vercel 的 CTO 正在评论我的东西**，大意是“一个家伙的 side project”，然后话锋一转就是 **“Cloudflare 搞了这事”**。我当时的反应是：**我职业生涯要完蛋了。** 一个价值百亿美元公司的 CTO 这么说，我当时真的以为自己要被开除了。那种感觉糟透了。

于是我以最诚恳的方式回应：“兄弟，我本来想和你谈谈，我回去打了个盹……” 好笑的是，我在 **JSCON Fespa** 的演讲现在上线了，在那场演讲里我热情洋溢地赞美了 Vercel 和他们的 Just Bash、sandbox 项目。这就是当时的背景。

然后奇怪的事情发生了：**半个互联网站出来支持我。** 他们都说：“Sunil 是最好的人，为什么要这样对他？” 接下来的 24 小时里，我沉浸在爱意之中。所以，我们来谈谈 **Slop Forks** 吧。

**主持人：** 最近还有一个事件：有人泄露了一个代码仓库——Cloud 代码，有人把它 fork 了并重写以摆脱 Python。我被很多人 @ 问“这是你吗？”，我说不是我。

**Sunil Pai：** 但人们很喜欢那个版本。

**主持人：** 我觉得这在当今环境下有些奇怪。Fork 在我的文化里是一种崇高的尊重标志，也正是 React 社区成长的方式——只需一个按钮。没人靠 fork 赚钱，但它是开源的核心：你应该拿我们的代码去 fork 并做你想做的事，为什么不能把它移植到不同的语言呢？

**Sunil Pai：** 每次有人跟我谈 Agents SDK，说“我觉得我们可以做得更好”，我都会说：“兄弟，放手去做。如果你成功了，每个人都赢了。我不会因此赚得更少。哪怕不成功也无所谓。**如果你是为了热爱而做，那就为了热爱去做。**”

**主持人：** 有趣的是，有人以为自己可以通过用 Python 重写 Cloud 代码来绕过，但一些律师看到了，他们会说：“不，这属于衍生作品。” 这实际上侵犯了版权。

**Sunil Pai：** 关于 Slop Forks，我一直在想：**你也许应该鼓励更多的 Slop Forks**，因为这样你就拥有了依赖权。比如你不再会遇到“LLM 出现问题”或“Exio 问题”，你可以直接把 Exio vendoring 到自己的 Slop Fork 中，这样你就拥有了它。

在 npm 之前，在一切之前，软件是通过 Usenet、电子邮件分发的，你拿到代码，修改，然后分享。现在周期变短了，但 forking 仍然是软件构建的基础。有趣的是，这非常“非代理化”，对吧？人们会纠结“我用微服务还是直接 vibe code 一个副本”。**构建 vs 采购** 已经彻底变了。但在这个过程中没有 fork，没有“去看看前人的工作，fork 它，再适配简化”的环节。我们看不到这种模式，看到的只有“重建它，使用它”，但人们不 fork。

## 五、开源协作的困境

**Sunil Pai：** 我认为还有一个趋势：开源仓库已经变成了 adversarial 的战场。以 Agents SDK 为例，我们不得不关闭贡献通道，现在只允许 issue。我们可以把描述写得很长，但最终我会直接把解决方案贴到 Opus 里。花了你的 token 还是我的 token？我当然更喜欢用自己的 token，因为我擅长提问。而且我喜欢把功劳归于 clanker。现在，开源仓库的 adversarial 程度已经高到让人们几乎害怕收获人气了。我和至少四五个开源维护者谈过，他们都说：“现在外面情况真的很糟。”

**主持人：** 最糟糕的是那些假的安全报告。

**Sunil Pai：** 看起来极其真实，但仔细一看却毫无意义。

**主持人：** 我们今天早上和 OpenClaw 的 Peter 聊过，这是当前头号攻击来源。攻陷 OpenClaw，你就能拿到所有人的远程代码执行权限。

**Sunil Pai：** 是的。

## 六、结语：呼吁原创，构建科幻

**主持人：** 时间差不多了。Sunil，感谢你过来分享智慧。你总是提到“Alpha 思想领导力”，这让我印象深刻。

**Sunil Pai：** **Alpha 思想领导力。** 2018 年 React Rally 上，你演讲了 VR 测试，当时你说“我想先做点 Alpha 思想领导力”。从那时起，我也一直在尝试这么做。你是这个游戏里最棒的人之一。

**主持人：** 这本身就是 Alpha 思想领导力。

**Sunil Pai：** **我真正想鼓励大家的是：做原创。** 在这个疯狂的世界，任何想法似乎都能行得通——clanker 会帮你实现。**甚至不要尝试做渐进式改进。去构建科幻级的东西（build sci-fi stuff）。** 为你家人去做。用 Agents SDK 去做。不要因为你觉得自己能拿到一个大企业的单子而去构建。**你对改变世界拥有巨大的能动性。** 我希望人们去尝试那些连基础设施和 LLM 当下都几乎无法支撑的狂野想法。我们如何才能达到下一个台阶？这就是我对人们唯一的期望。我恳求你：**做原创，有勇气去尝试一个新想法。** 我不需要另一个代理框架，没人需要。

**主持人：** 你应该用 Agents SDK。

**Sunil Pai：** 是的，除了你的。非常感谢。我期待你的演讲。谢谢你来。

**主持人：** 太好了。几个小时后我会在舞台上谈论这些，相当精彩。你可以找到回放。

**Sunil Pai：** 好的，很棒，再见。
