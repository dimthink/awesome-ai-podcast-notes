## 缘起：被成本“劝退”的AI功能和挥之不去的念头 `[00:00-10:00]`

**核心观点**

Simon 创办 Turbopuffer 的灵感并非来自高深的学术论文或宏大的市场预测，而是源于一次具体到令人沮丧的工程实践：为 Readwise 构建一个基于向量的文章推荐引擎，其预估成本高到让这个功能根本无法上线。这个被成本“劝退”的经历，让他开始从根本上思考数据库在云时代的成本结构。

**深度阐述**

故事的起点，是 Simon 在 Shopify 近十年的职业生涯。作为早期基础设施团队的成员，他亲历了电商巨头的指数级增长，也承受过无数次被 on-call 电话从睡梦中叫醒的痛苦。而其中最让他“头疼”、最难扩展的，莫过于 Elasticsearch。他回忆道，在2015年左右，自托管的 Elasticsearch 就像一个难以驯服的野兽，常常无法让底层的 Lucene 按照他们的意愿执行查询，迫使他们不得不在关键时刻 panic-scaling。这段经历让他对传统搜索引擎的架构痛点有了切肤之痛。

离开 Shopify 后，Simon 开始了他的“天使工程”之旅，为 Readwise、Replicate 等朋友的公司提供短期咨询。2022年底，ChatGPT 的发布点燃了整个AI界。当时正在帮 Readwise 优化 Postgres 数据库（主要是调优 auto vacuum）的 Simon，自然也想到了利用这波技术浪潮。他和团队打算构建一个简单的推荐引擎：将用户阅读过的文章进行向量化嵌入（embedding），然后基于此推荐相关内容。

他动手做了一个原型，效果出奇地好。当他用 Readwise 一位联合创始人的数据运行推荐时，结果竟然出现了关于“育儿”的文章，而Simon并不知道对方即将迎来孩子。这个巧合证明了推荐引擎的有效性。

然而，当 Simon 拿出 Napkin（餐巾纸，这里指他用来做粗略估算的习惯）开始计算将这个功能投入生产环境的成本时，一盆冷水浇了下来。Readwise 是一家引以为傲的、精打细算的 bootstrapped 公司，他们当时所有的基础设施（Heroku dynos, Postgres等）每月总开销大约是5000美元。而仅仅为了支持“相关文章推荐”这一个新功能，运行所有文章的向量嵌入和索引，粗略估算下来，每月竟然需要额外的30000美元。这个成本是完全不可行的。

这个结果深深困扰着 Simon。他意识到，**这里有明显的、未被满足的需求，Readwise 完全愿意上线这个功能，只要成本能降低一个数量级**。这个念头挥之不去，驱使他开始深入研究。他自嘲道，当时自己对向量索引几乎一无所知，只知道有很多“向量数据库”公司在融资，市场一片喧嚣。但作为工程师的他，选择了一个最朴素的方法来探索问题——坐下来，打开他那个名为“Napkin Math”的 GitHub 仓库，上面记录着各种硬件性能的基准数据：DRAM 的带宽、SSD 的写入速度、S3 的单连接带宽等等。他反复推演，一个根本性的问题浮现出来：**为什么没有人构建一个数据库，把所有的数据都放在对象存储（如S3）上，只有在使用时才将其“吹”进 NVMe SSD，如果查询频繁，再进一步“吹”进 DRAM？**

这种架构的代价是写入延迟会增加到几百毫秒，但除此之外几乎全是优点。它极度契合现代硬件的特性：NVMe SSD 和 S3 都要求极少的往返次数（round trips）和大量的并发请求。你可以一次性发起上千个请求去S3拉取数据，然后在本地进行最近邻搜索。然而，现有的数据库都不是为此设计的。Simon 越想越觉得这是一个显而易见的空白。

**延伸思考**

这个案例完美诠释了创新的“痛点驱动”本质。AI 的浪潮催生了全新的 workload（将大量数据连接到 LLM），但现有的基础设施无法经济地支撑它。Simon 的经历也提醒我们，很多时候，颠覆性机会就藏在那些被“成本”或“复杂性”劝退的失败尝试里。Napkin Math 不仅是他的个人习惯，更是一种重要的工程思维：从第一性原理出发，用最基础的物理限制去拷问现有系统的效率边界。

## 架构革命：云原生数据库的三重奏与幸运的赌注 `[10:00-20:00]`

**核心观点**

Simon 提出，要打造一个成功的、能服务“地球上每一家公司”的数据库公司，需要同时满足三个条件：一个全新的、普遍存在的 workload，一个现有巨头无法复制的底层存储架构革新，以及一个能够随着时间推移、不断扩展查询能力的演进路线。Turbopuffer 恰好站在了这三个条件的交汇点上，并且幸运地押注了云服务商的技术演进路线。

**深度阐述**

Simon 将自己对数据库行业的思考提炼为一个清晰的框架。他认为，像 Oracle、Snowflake、Databricks 这样的巨头之所以能成功，是因为它们抓住了历史机遇：

1. **新的工作负载 (New Workload)**：大约每15年出现一次。Snowflake 和 Databricks 赶上了云分析和数据仓库的爆发。而当下，这个新 workload 就是“将海量数据连接到 AI”。
1. **新的存储架构 (New Storage Architecture)**：这是最关键的一点，它决定了新玩家能否建立壁垒。Snowflake 的成功离不开 S3 这样的对象存储的普及，这是90年代无法想象的。而对于 Turbopuffer 来说，这个新架构的基石是**NVMe SSD**和**强一致性的对象存储**。
1. **不断演进的能力 (Evolving Capabilities)**：客户将数据放在你这里后，会期望你最终能回答关于这些数据的任何问题。因此，数据库必须从一个单点功能（如向量搜索）出发，逐步实现更多的查询计划。
Simon 指出，正是技术的演进，让满足第二个条件成为了可能。他列出了三个关键的时间点：

- **2017年**：云服务商开始提供 NVMe SSD 实例。
- **2020年12月**：AWS S3 正式宣布实现**强一致性**。这之前，S3 是最终一致性的，这意味着数据库必须自己维护一个复杂的元数据共识层（如 Zookeeper 或 FoundationDB）来保证数据没错，这恰恰是 Snowflake 等前辈的做法。
- **2024年底**：AWS S3 终于支持**Compare and Swap (CAS，比较并交换)** 操作。CAS 是一个原子操作，允许多个节点通过“读取-修改-写入”并检查版本的方式，安全地竞争更新同一个元数据文件，而无需一个中心化的锁服务。
Turbopuffer 的整个架构都赌在了这些云原生特性的演进上。它的设计哲学是极致的“All-in”：所有数据的主副本都在对象存储上，计算节点是无状态的，没有传统的共识层。通过 CAS 操作元数据，通过对象存储本身的强一致性保证数据没错。Simon 形容，即使关掉 Turbopuffer 所有的服务器，数据也不会丢失，因为一切都在对象存储里。

这个激进的架构选择，在实践中遇到了一个戏剧性的挑战。当他们的大客户 Notion（运行在 AWS 上）要求使用 Turbopuffer 时，问题来了：Turbopuffer 最初是构建在 GCP 上的，利用了 GCP 早已支持的 CAS 特性。如果让 Notion 跨云访问，延迟会高得无法接受。是妥协，去构建一个元数据层？还是坚持信念，想办法解决？

Simon 和团队选择了后者。他们的信念如此之强，以至于做出了一个在初创公司看来匪夷所思的举动：**他们自费在俄勒冈州 AWS 和 GCP 数据中心之间铺设了暗光纤**，并通过精细调优 TCP 窗口，将跨云延迟从14毫秒降低到了可接受的水平。当被问到“宁愿做这个也不去写 Zookeeper 吗？”，Simon 的回答斩钉截铁：“对，好多了。”因为 Justine（他的 co-founder）和他都曾 on-call 多年，深知最糟糕的故障就是由多个系统的状态不一致引起的。他们不想给自己创造任何在凌晨3点被叫醒的理由。

**个人感受**

这个故事充满了理想主义的工程师色彩，又混合着惊人的务实和魄力。为了不引入一个复杂的系统组件（元数据共识层），他们宁愿去解决物理网络层的工程问题。这不仅是技术偏好的体现，更是对“简单性”近乎信仰的追求。而这一切，都建立在一个长线判断上：他们坚信 S3 最终会支持 CAS。当被问及这是否过于冒险时，Simon 笑着承认，这其中有很大程度的“幸运”，但他们确实用自己的方式（买光纤、坚持等待）为这个信念押上了筹码。

**延伸思考**

Turbopuffer 的架构故事，是云原生理念的极致实践。它展示了如何将云基础设施的能力（对象存储、NVMe、一致性原语）作为分布式系统的一部分来“编程”，从而极大地简化了数据库自身的复杂性。这也给所有基础设施创业者一个启示：不仅要关注业务代码，更要深刻理解底层云服务的演进路线图，并敢于在此基础上押注未来。

## 新范式：Agentic Search 与 P99 工程师 `[30:00-45:00]`

**核心观点**

AI 的演进正在重塑搜索的形态，从单次的 RAG 调用转向由 Agent 驱动的、高并发的“工具调用”。与此同时，为了应对这种快速变化的复杂世界，Turbopuffer 建立了一套独特的招聘哲学——寻找“P99工程师”。这类工程师不仅技术精湛，更具备“痴迷”、“高 Agency”和“清晰阐述取舍”的特质。

**深度阐述**

随着像 Cursor 这样的 AI 原生应用的发展，Simon 观察到了一个 workload 的重大变化。过去，RAG 就像一个“一次性”动作：在 LLM 开始生成答案前，执行一次搜索，填充有限的上下文窗口。但现在，Agent 可以自主地进行多步推理，搜索不再是起点，而是被 LLM 随时“调用”的一个工具。`[30:00]`

这意味着什么？意味着**并发度的急剧提升**。不再是单个用户等待一个搜索请求返回，而是一个 Agent 为了完成一项任务，可能会并行发起多个搜索请求。例如，Cognition 的 Devin 在做快速上下文的关联时，就可能同时进行8个搜索。这给数据库带来了全新的压力：如何在数据集已经“温”的情况下，以最少的往返次数，处理海量的并发查询。

为了适应这种变化，Simon 透露，Turbopuffer 正在大幅降低查询价格（首次公开提及将降低5倍，甚至更多），以鼓励这种高并发的使用模式。

面对如此快速变化的技术前沿，如何打造一个能始终领先的团队？Simon 提出了他著名的“**P99工程师**”概念。`[35:00]`

这个想法源于他们内部对“人才密度”的追求。他认为这个说法已经有点陈词滥调了，于是他们试图用更具体的方式来描述他们想要的人。他们有一份名为“P99工程师特质”的内部文档，每次面试后，Simon 都会对照这份文档进行复盘。他有一个严格的原则：**默认拒绝每一位候选人**，除非有人愿意“为这个人打一架”。他要求面试官们不只是“还行”，而是要有人“双拳紧握，全力支持”。

那么，P99工程师有哪些特质呢？

1. **有“扭转轨迹”的经历**：他们曾在职业生涯的某个时刻，真正“让计算机服从自己的意志”，做出过超出常规的成果。Simon 举了 Turbopuffer 首席架构师 Nathan 的例子。他仅用了6-8周时间，就带领团队实现了 ANN V3 索引，能够以 P50 40毫秒、P99 200毫秒的延迟搜索1000亿个向量。“他几乎就是凭一己之力，让原本做不到的软件做到了。”
1. **痴迷于“看地图”**：这听起来像个玩笑，但 Simon 是认真的。他发现很多顶尖人才都喜欢在闲暇时盯着地图看，沉浸在探索复杂系统的细节中。这是一种“痴迷”特质的体现，他们会被某个事物深深吸引，并乐此不疲地研究它。他自己就喜欢在谷歌地图上随便找个地方放大，探索加拿大有多大，巴芬岛在哪里。
1. **擅长阐述“取舍”**：P99工程师非常清楚任何选择都伴随着代价。他们能清晰地描绘出决策的“地图”，指出“在这里，你能做A和B，但做不了C”。Turbopuffer 自己就是一个例子：他们清晰地定位自己为一个专注于搜索的引擎，延迟在100毫秒级别，如果你需要极高的事务处理能力，那它就不适合你。这种对自身边界和取舍的清晰认知，本身就是一种力量。
**个人感受**

Simon 的“P99工程师”理论，与他个人特质高度吻合。他自己就是一个痴迷于 Napkin Math、喜欢看地图、能从第一性原理清晰地阐述数据库架构取舍的人。他将招聘标准从模糊的“优秀”拆解为可感知、可讨论的具体“特质”，这本身就是一种高 Agency 的行为——不依赖外部定义，而是主动构建团队文化的内核。

**精华收获**

- **AI Agent 正在改变搜索范式**：从单次 RAG 调用，转向高并发、多轮次的工具调用，这对数据库的查询成本和并发处理能力提出了新要求。
- **“P99工程师”不是指技术百分位数，而是一种思维模型**：它包括高 Agency（扭转轨迹的能力）、深度痴迷（看地图）、以及清晰的取舍思维。这套模型帮助 Turbopuffer 在混沌中维持极高的团队标准。
## 未来之路：专注、演进与热爱 `[45:00-结束]`

**核心观点**

Turbopuffer 的未来蓝图清晰而专注：在巩固向量搜索优势的同时，大力投入全文检索功能，并不断突破规模极限。长远来看，它计划逐步实现更多查询计划，但当前的核心原则是克制，避免因过度扩张而失焦。最后，Simon 对茶的热爱，为这场深度对话增添了一抹充满人情味的注脚。

**深度阐述**

回顾 Turbopuffer 的发展，Simon 将其划分为几个“幕”：

- **第一幕**：专注向量搜索。
- **第二幕**：打造业界领先的全文检索引擎。目前，在由 LLM 生成的超长查询和海量数据集（如 Common Crawl）上，Turbopuffer 已经在某些基准测试中超越了 Lucene。接下来，他们将持续增加全文检索的功能特性，满足那些习惯了传统搜索引擎的用户预期。`[45:00]`
展望未来，Simon 的规划清晰而务实。近期的重点有三个：

1. **全文检索功能**：每月更新 changelog，不断增加新功能，吸引更多用户从传统搜索引擎迁移过来。
1. **规模突破**：让查询千亿级向量或文档变得更便宜、更快速。他们正在研发 ANN V4 和 V5。
1. **完善用户体验**：Simon 自嘲地承认，现在的 Turbopuffer 控制台非常简陋，像一个创始人两年前随便写写，之后别人又随意添补了一些东西的产物。他们计划投入精力，打造一个像“PHPMyAdmin”那样好用的、与数据库深度集成的控制台。`[50:00]`
当被问及更远的“第三幕”时，Simon 显得非常谨慎。他重申了“构建大型数据库公司”的第三个条件：最终要能实现更多的查询计划。他看到了很多同行因为“过于急切”而过度扩张，导致失焦。因此，第三幕的候选方向很多，比如更简单的 OLAP 查询、日志处理、时间序列数据，甚至是图查询（已经有一些客户在这么用了）。但他强调，**“今年最后悔的事，可能就是尝试了太多事。”** 所以，现在的原则很明确：如果不是以“搜索”为核心用例，就不应该使用 Turbopuffer。他们会观察客户的自发使用模式（比如 Cursor 将20TB的 Postgres 数据直接挪到 Turbopuffer 上，只是为了简化分片），从中发现下一个值得专注的方向。`[55:00]`

访谈的最后，话题转向了 Simon 的个人爱好——茶。他带来了自己最喜欢的茶：Yabukita Kamairi-cha，一种用中国方法制作的日本绿茶。他有一个记录了200多种茶的 Airtable，过去15年一直在记录自己的品茗体验。他会为了泡茶随身携带一个精准的温度计，甚至在团队周五的 demo 会上，如果 demo 不够多，他会用分享自己的茶艺笔记来填满时间。`[57:00]`

**个人感受**

Simon 对茶的痴迷，完美呼应了他之前提到的“P99特质”。他会深入到一个领域，建立自己的评价体系（Airtable），钻研细节（温度、产地、家族、季节），并乐于分享这种热情。这让我们看到，无论是在构建数据库还是在泡一杯茶时，驱动他的都是同一种深入骨髓的好奇心和追求极致的“痴迷”。这种对生活的热爱，也让这位技术创始人形象更加丰满和真实。

**精华收获**

- **专注是早期创业公司的生命线**：Turbopuffer 深谙此道，明确将自己定位为“搜索引擎”，拒绝过早地扩张到其他领域。
- **客户是产品路线图的最佳指引者**：他们通过观察“P99客户”如何使用产品，来发现下一步该“全力投入”的方向，而非凭空规划。
- **极致的专业精神可以体现在任何领域**：从数据库架构到泡一杯茶，Simon 的故事证明了，P99 是一种可以贯穿生活与工作的思维方式和行动准则。
