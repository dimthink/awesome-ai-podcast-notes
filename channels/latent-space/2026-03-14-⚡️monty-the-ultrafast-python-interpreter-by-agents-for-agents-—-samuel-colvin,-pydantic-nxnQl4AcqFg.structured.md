**主持人**: 欢迎回到 Latent Space 演播室。今天我们邀请到了 Pydantic 的 Samuel Colvin，他最近刚刚发布了 **Monty**。欢迎你，Sam！

**主持人**: 自从我们上次见面以来，你过得怎么样？Pydantic 和 Pydantic AI 进展如何？

**Sam**: 我想我们上次谈话大概是一年前？那时你们刚刚发布。

**主持人**: 是的，没错。

**Sam**: 嗯，我不想用“一切都很疯狂”之类的陈词滥调，但……确实发生了翻天覆地的变化。当然，我们度过了一段美好的时光。但如果去年这个时候你告诉我，我现在会用 AI 来构建一个可以说是 Python 的“衍生品”，我当时可能会对那些说“不再手动编写所有代码”的人嗤之以鼻。但现在，我认为 AI 非常强大，我为此感到自豪。但这个世界确实很疯狂，你可以在圣诞节期间坐下来，用 Rust 编写出 30,000 行代码，而且它确实强大且有用。

**主持人**: 很多事都变了。我想你们还推出了 Pydantic Logfire，一个可观测性平台。这对一个“AI 原生”的可观测性平台意味着什么？

**Sam**: 我们有点特别，因为虽然大家常把我们和 BrainTrust、LangSmith、Langfuse 等其他 AI 可观测性平台比较，但实际上我们是**基于 OpenTelemetry 的完整可观测性平台**。你可以向我们发送日志、指标和追踪数据。我们的定价比通用的可观测性平台更便宜，而不是贵 50 倍。同时，我们也做很多 AI 相关的事情，比如评估（evals）、提示词 playground、LLM 追踪等。

人们说“AI 可观测性”通常指两件事：**针对 AI 的可观测性**，以及**用 AI 来做可观测性**。我们两者都做。特别是 Logfire 允许你编写任意的 SQL 来查询数据。我们并没有在 MCP 服务器上做什么特别的事，但因为 AI 可以直接编写 SQL 来查找 Bug、展示 P95 最慢的端点，或者基于某些属性进行随机调查，这使得它的能力远超其他平台。这都得益于我们在 2023 年做了一个看似“怪异”的决定：允许用户使用任意 SQL 查询。

**主持人**: 允许任意 SQL，这听起来像个噩梦。

## Monty 的起源：夹缝中的“代码模式”

**主持人**: 你是在圣诞节期间开始做 Monty 的。它的灵感来源是什么？背后的故事是怎样的？

**Sam**: 其实，我几年前就做了一个非常早期的版本，然后完全放弃了。后来，我和 Anthropic 的四位不同的人聊过，他们每个人都独立地对我关于类型安全重要性的观点表示认同。我常说，类型安全对人类很重要，但对 AI 来说至关重要。这四个人都表示：“哦，是的，如果你在链式调用工具（tool calling）或者用代码进行工具调用时，类型安全超级重要。”当第四个人也这么说时，我就知道他们肯定在酝酿什么。

我觉得 Anthropic 很有趣，他们是最神秘的公司之一，但每个人都会因为内部的动向而兴奋，他们又会向你暗示这些动向。他们和我说话的都是开发者，而不是营销人员。

于是我开始着手构建。果然，在 12 月，他们推出了**程序化工具调用**的功能，后来又有了用代码调用 MCP 服务器的功能。Cloudflare 大概发明了“代码模式”（Code Mode）这个词，并大力推广。

后来，我和一位关注沙箱领域的投资人聊过，他估计**70% 的沙箱调用本质上就是这种“工具调用”或“升级版工具调用”**，比如渲染图表或执行计算，这些任务不需要完整的计算机使用能力，但编写代码来解决又非常强大。

所以，Monty 试图在两者之间找到一个位置：

- 简单的工具调用（tool calling）：非常安全，易于实现，不需要外部基础设施，但对 LLM 来说表达能力有限。
- 沙箱（sandboxes）：表达能力更强，可以做更强大的事情，但设置复杂。成本其实是最小的，真正的挑战在于复杂性。对于初创公司来说，把信用卡塞进 Modal 或 E2B 很容易，但对于需要自托管一切的大型金融机构，这些方案就行不通了。对他们来说，为自己的 Agent 添加“代码模式”基本不可能。对企业客户而言，沙箱并非易事。
另一个巨大的优势是**延迟**。如果你有一个运行在同一进程内的 Python 解释器，启动时间会非常快。在热循环中，从代码到执行结果可以在 800 纳秒内完成。实际运行时，执行代码或下一步协程也只需个位数微秒。而创建一个 Daytona 沙箱需要 1 秒钟，差距巨大。

## Monty 的设计哲学：权衡与定位

**主持人**: 我想这是个好时机来共享屏幕，展示一下你那个很棒的对比表。显然它是一个 Python 解释器，所以可以运行 Python？你打算演示什么？

**Sam**: 好的，我共享屏幕，可以谈谈这个对比表。我会坦诚地讲其中的权衡，因为这一切都是设计问题。

（共享屏幕）

这是 GitHub 仓库，已经有 6000 颗星了，对于一个开源项目来说非常棒。上周下载量达到了 27,000 次，虽然对维护 Pydantic 的人来说不算多，但也是个不错的数字。

对我来说，Monty 是从类型安全到代码执行的整体安全的逻辑延伸。但这是一个拥挤的领域，人们首先会想到 Pyodide——一个 WebAssembly 的轻量级 Python 解释器。

**我们确实犯过一个经典错误：最初使用了 Pyodide**。我非常尊重 Pyodide 及其维护者 Hood，但事实证明，它非常不适合在浏览器之外运行 Python。我们之前维护了一个叫 `mcp-run-python` 的东西，推动我们构建 Monty 的动因之一，就是人们不断报告它的安全漏洞。

**Pyodide 的问题在于：**

- WebAssembly 本身并不隔离，如果用 Node.js 运行，你可以导入 JS 模块并执行任意 JS 代码来访问主机。
- 你不得不在 Deno 中运行它，并设置文件系统和网络限制。但如果要下载包，又必须暴露一些文件系统权限。
- Deno 无法控制内存使用，恶意代码可以耗尽你的机器资源。
- 即便在 Deno 沙箱内，恶意代码也能“污染”那个服务器，导致每次调用都需要杀死并重建 Deno 沙箱，延迟比完整的沙箱还要差（比如运行 `1+1` 需要 2.8 秒）。
- 安装也很重，Deno 有 50MB，Pyodide 包有 12MB。
相比之下，Monty 只是一个单一的 Rust 二进制文件，可以用 `pip` 或 `npm` 安装。已经有 PR 在支持 Dart 和 Kotlin 了，安装非常简单。最终，你可以在任何能运行 Rust 的地方运行它。

**主持人**: 那么，Monty 做出了哪些权衡？

**Sam**: Monty 最大的缺点是**它不是完整的 CPython**。我们是自己从头实现一切。

- **标准库支持有限**：我们支持部分模块（如 `asyncio`、`dataclasses`），但无法安装第三方库（比如 NumPy 或 Pydantic 本身），也永远不会直接支持 CPython ABI。
- **功能尚不完整**：目前甚至还不能定义类（classes），不支持 `match` 语句（但未来可能会支持）。
- **手动实现标准库**：我们必须手动选择和实现我们需要的标准库部分。
**主持人**: 这听起来像是一个艰巨的任务，甚至有点“自我牺牲”——你们做了一个运行时，但它本身却不能运行 Pydantic？

**Sam**:（笑）我明白你的意思。关键在于，你可以通过**宿主函数（host functions）**调用外部功能。比如，你可以注册一个 `fetch` 方法来发起网络请求，或者通过外部函数调用 Pydantic 来验证 JSON 数据。所有这些强大的包并不需要在 Monty 运行时内部运行。

## AI 驱动的开发：100 倍速实现

**主持人**: 你提到了用 AI 编写 Rust 代码。能否谈谈你的开发体验？

**Sam**: 我想实现 Python 的 20 多个内置函数。我让 LLM 去实现它认为需要定期使用的标准库函数。我们做了一点规划，然后它就开始了，跑了 2 个小时。一个经验丰富的开发者需要几周才能完成的工作，在几个小时内就完成了。

我认为有四件事，如果能同时满足，LLM 的效率提升不是 3 倍、5 倍，而是 **100 倍**：

1. **内部实现为模型所熟知**：比如如何实现堆、如何实现字节码解释器。
1. **外部 API 为模型所熟知**：你不需要解释 Python 应该如何工作，它已经在权重中“知道”了。
1. **单元测试极其简单**：只需断言“是否和 Python 的行为一致？”，模型可以近乎自动地修正自己直到测试通过。
1. **无需争论 API 设计**：当把字符串和整数相加时应该报什么错？参数顺序是什么？这些都由 Python 定义好了，人类无需争论。
这四点，就是为什么现在大家都在用 Rust “克隆” Redis 这类项目。因为 LLM 知道内部逻辑、知道外部接口、单元测试简单、且没有 API 设计的“ bikeshedding”。如果任务符合这四点，开发速度能快 100 倍。

比如 Monty 上有一个待合并的 PR，实现了 `math` 模块的 50 个函数。我还没细看，但有人用 LLM 生成了它，并添加了 800 个测试。

## 编码工具的选择：Claude、Codex 与 Gemini

**主持人**: 你刚才提到用 LLM 写代码。你主要用什么工具？Claude Code？有什么给想尝试类似项目的人的建议？

**Sam**: 我主要用 **Claude Code** 和 **Gemini CLI**。

如果把它们比作超级英雄：

- **Claude Code** 就像典型的美队，相当能干，大部分事都做得对，有点过度自信，但还行。
- **Codex** 就像一个神经质的、极客风的“Q博士”，对不重要的小事非常挑剔。
- **Gemini** 则像**小丑**，一个精神不稳定的疯子，偶尔能做出不可思议的工作，但有一半的可能会删掉你所有文件。所以我几乎从不给它修改文件的权限。
**主持人**: 我很少遇到 Gemini CLI 的用户。

**Sam**: 举个例子，我会把分支的 diff 直接传给 Gemini，让它给我写一份审查报告。它不被允许编辑任何文件，只会生成报告。然后我把报告交给 Claude Code，让它去实现。Gemini 的速度非常快，因为它可能把整个 PR 的 diff 打包成一个请求，让模型花一两分钟处理，然后返回报告。而 Codex 会逐步、细致地调查所有变更，可能需要半小时。所以 Gemini 的初审能让我更快地发现一些问题。

**主持人**: 有道理。有人认为 Google 拥有世界上最大的私有代码库，Gemini 很可能在其上训练过。所以在 Monty 这种非常技术性的领域（比如如何最佳实现字节码解释器），Gemini 能知道一些其他模型不知道的东西。

**Sam**: 是的，这也是我尝试使用它的原因。有时候它确实能给出一些神奇的建议。

## 技术演示：用 Monty 构建 Agentic Web Scraper

**主持人**: Monty 的未来方向是什么？有商业化的考虑吗？

**Sam**: 我给你看一个例子，或许能让大家更清楚地了解我们在做什么。

（再次共享屏幕，展示 Logfire 中的 Agent 运行轨迹）

这个例子的目标是**从 AI 公司的网站上抓取模型价格**。讽刺的是，这些靠爬取互联网发家的 AI 公司，自己却把网站数据保护得严严实实。

这是一个 Pydantic AI Agent。我们给了它一个 `open`*`page`** 函数（使用 Playwright），它返回一个 **`Page`** 类型的实例。这个 **`Page`** 类有一堆方法（如 **`goto`**、**`click`**、**`fill`**），我们在 Monty 内部暴露了这个数据类，所以 Monty 代码可以直接调用这些方法。*

另外，Monty 内置了类型检查器。在运行任何代码之前，它会先用这些类型存根（type stubs）进行类型检查。这意味着，如果 LLM 生成的代码类型不对，就不会运行，这给了 LLM 信心。

我们还提供了一个 `BeautifulSoup` 函数，返回一个 `Tag` 类型，同样有一堆方法可以查询 DOM。

（运行示例）

你会看到系统提示词里告诉了它能用什么 Python 代码，以及允许调用的函数类型提示。然后 LLM 开始编写代码去获取 Claude 文档的价格页面，用 BeautifulSoup 解析表格，并进行一系列调查。

这里它遇到了一些错误，比如它想当然地认为需要 `asyncio.run()`，但忘了导入 `asyncio`。因为类型检查失败，它被告知需要修正。下一次运行，它就能正确执行了。

最终，它会返回模型价格的摘要。整个过程花费了 102 秒，成本 30 美分。

**关键在于**，你可以在 LLM 返回价格信息的同时，要求它返回一个**最优的 Python 代码**来再次执行此任务。下次运行时，直接给这段代码，它就能立刻成功。这实际上为应用层创造了一种新的状态——你可以把它看作是“内存”，但它远比内存复杂，它包含了**Agent 优化的当前状态**：一部分是代码，一部分是模型选择和设置，一部分是系统提示词。

这正是我们在 Logfire 中努力的方向：**超越“评估”，让 Agent 能够随着时间自我改进**，无论是完全自主地，还是通过用户审查和点击接受。

## 关于商业化与未来展望

**主持人**: 最后一个问题：Monty 有商业化的打算吗？

**Sam**: Monty 本身不太可能推出托管版本，因为它的意义就在于你不需要使用托管服务。但**在其之上构建服务**有很大的空间。比如，你可以选择在你喜欢的语言中使用它，但我们也会提供让性能更优的服务。

很多人，尤其是做可观测性的，最终都会向往这种“自我改进的 Agent”的理想状态。现在我们在 Pydantic AI 中引入了**可序列化 Agent** 的概念（应该已有公开的 PR）。你可以用一个 TOML 文件完整地定义一个 Agent，包括模型、系统提示词、所有能力（如代码模式、MCP 服务器注册等）。这样做的好处是，你现在有一个单一文件可以运行优化，理论上，这个文件可以运行所有已注册的能力，而 Monty 则负责执行该进程中所有的任意代码。

**主持人**: 关于用 TOML 定义 Agent，我有保留意见。任何标记语言本质上都是 DSL，你最终会需要为它实现分支、变量和循环，结果就是得到一个实现得很糟糕的通用语言。

**Sam**: 我同意你的观点。如果真的走到那一步，我们还是会回归 Python，或者用 Monty 来处理一部分。我认为 Monty 本身就是答案。

**主持人**: 感谢你的时间，恭喜你为开源和 Agent 运行时找到了一个新的切入点。感觉你正在 AI 原生时代做着类似 Deno 和 Bun 对 Node.js 做的事，构建 Agent 真正需要的东西。

**Sam**: 我跟 Anthropic 的朋友 David Sora 展示了这个项目，他说：“没错，这正是每个人都需要的。”最大的问题就是怎么用它赚钱。（笑）但 Bun 也没担心这个，对吧？

**主持人**: （笑）没错。

**Sam**: 我想我们会做得不错。关于这个有个有趣的故事，但不便在此公开。

**主持人**: 好的。那我们就聊到这里，伦敦见！

**Sam**: 好的，非常感谢！
