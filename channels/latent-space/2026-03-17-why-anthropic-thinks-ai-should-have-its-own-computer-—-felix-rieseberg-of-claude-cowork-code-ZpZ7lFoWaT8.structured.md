# 赋予AI自己的电脑：Anthropic对AI Agent未来的大胆设想 —— 专访Claude Code / Cowork产品负责人Felix Rieseberg

**导语**：当大多数AI公司致力于将一切应用迁移至云端时，Anthropic却反其道而行之，坚信AI的未来与你的本地电脑息息相关。在Latent Space的这期播客中，主持人Alessio和swyx与Anthropic的“成员技术职员”（Member Technical Staff）Felix Rieseberg进行了一场深度对话，揭开了Claude“同事”Cowork的神秘面纱。这款在短短10天内（基于大量已有原型）诞生的产品，不仅仅是Claude Code的“用户友好版”，它代表了一种更宏大的愿景：**给AI一台它自己的电脑**。本文将带你深入理解Cowork的设计哲学、技术内核，以及它将如何重塑我们的工作方式，从整理桌面到规划税务，乃至影响整个劳动力市场。

---

## 一、 什么是Claude Cowork？不仅仅是“用户友好版”的Claude Code

**主持人Alessio**： 对于还没听说过或尝试过Claude Cowork的听众，你能先给我们一个最顶层的介绍吗？

**Felix Rieseberg**： 好的。简单来说，**Claude Cowork是Claude Code的一个更用户友好的版本**。它的工作方式是，我们有一个非常强大的Agent框架——Claude Code。但在去年12月，我们注意到越来越多的人在使用它，即使他们并非技术人员，不习惯在终端操作；或者，即使是技术人员，他们也开始用Claude Code处理非编码类工作负载，比如管理开支、填写收据、整理知识库等。当时出现了一股Obsidian热潮，很多人喜欢用这种方式。我们希望抓住这个机会，同时也将这种能力带给那些不熟悉终端、不知道如何用`brew install`安装软件的人。

所以，Cowork就是**在一个虚拟机中运行的Claude Code**。我们给它增加了一些缓冲、更多的护栏，让它更安全、更方便，特别适合那些不希望每天一上班就先打开终端的人。

**主持人swyx**： 这很有意思。你把它定位为“更用户友好的版本”，但我个人感觉，它反而像是一个更强大的高级工具，因为它似乎能更好地与你的电脑、Chrome浏览器以及其他工具链集成。这可能是我的 perception bias？

**Felix**： 说实话，我不认为你错了。这也是我过去两周一直在思考的事情。当人们听到“用户友好”时，常会误解为“功能精简的傻瓜版”。但恰恰相反，**Cowork实际上是一个超集**。这让我想起了大概10到12年前，我在微软开始研究Electron和跨平台技术时的情况。当时Visual Studio Code的第一个用例是作为一个网站，最初的描述也是“Visual Studio Code是Visual Studio的一个更用户友好的版本”。同样，也有人质疑说“这不是给严肃开发者用的”。但我个人认为，VS Code之所以能取得巨大成功，“可破解性”和“可扩展性”起到了很大的作用。你可以将VS Code接入几乎所有的工作流，在上面做扩展非常容易。我认为Cowork可能也有类似的潜力：它非常易于扩展，易于融入你的工作流。所以，**便利性是我们作为开发者努力的目标，但用户最终的价值在于将它恰当地映射到他们实际的工作中**。

## 二、 诞生故事：10天与无数原型

**主持人Alessio**： 去年年底，你们观察到了Claude Code在非技术用户中的使用高峰。那么，从观察到决定打造Cowork，这个设计过程是怎样的？毕竟你们只用了10天就构建出来了。

**Felix**： 在Anthropic，我们一直在思考如何让那些习惯于用Claude聊天问答的用户，能够享受到AI“执行任务”的强大能力。我们围绕这个目标做了很多原型，最早可以追溯到一年半以前。Anthropic内部有一种“原型-演示优先”的文化，我们有很多从未公开发布的内部原型。

而Cowork的诞生，实际上是我们从众多原型中挑选出了正确的组件组合而成的。我想这是个很重要的说明，每当人们提到“10天”这个数字时，我想强调的是，我们并不是从零开始的。就像你建一个网站会用React一样，Cowork也是建立在许多已有的组件之上的。

在决策路径上，我认为我们生活在一个**执行成本极低**的有趣新时代。过去，产品经理可能需要去拜访潜在客户，以低带宽的方式试探他们的问题和需求，然后写规范、做设计、再执行。而在Anthropic内部，我们现在更倾向于“**别写备忘录了，直接动手建一个吧！**”我们会把所有候选方案都快速构建出来，然后挑选最好的那个。

对我们产品影响最大、对用户也最关键的一个决策，就是我们如何看待你**本地电脑**的价值。很多人在思考，这个AI实体最终应该运行在你的电脑上，还是应该运行在云端？这其中有很多权衡。但我认为，我可以随时从任何地方下载一个文件，然后直接放进Cowork里处理，这是一个巨大的优势。

## 三、 核心哲学：为什么AI应该拥有一台“自己的电脑”？

**主持人swyx**： 我们来构建一个关于Cowork的清晰思维模型。它是基于Claude Code，运行在一个虚拟机里，并且能与你的电脑和Chrome深度集成。你能解释一下这些主要组件是如何协同工作的吗？

**Felix**： 好的，你基本上已经说到了要点。我认为有几个东西在Cowork里非常有价值。

- **虚拟机的力量**：目前，我们运行一个轻量级的虚拟机，并把Claude Code放进去。这样做首先是出于安全和保护的考虑。但即使我们先不谈安全，就假设你完全不在乎，**给Claude一台它自己的电脑本身就是一个极其强大的想法**。在Anthropic，我们在设计架构和用户体验时，发现一个很有用的方法就是**把Claude拟人化**。你可以这样想：如果它是一个人，你会怎么做？
我给我父亲举过一个例子，他至今仍坚持用聊天的方式来处理编码任务。我问他：“如果你是一个开发者，你的雇主告诉你，你不需要电脑，他们只会通过邮件把代码发给你，你再把改好的代码用邮件发回去。你觉得这样行吗？”这在石器时代也许能应付一些简单文件，但肯定非常低效。所以，通过虚拟机，因为这是一个Linux系统，Claude Code可以相对自由地安装它需要的任何东西——Python、Node.js等等。我们当然有严格的网络出入控制，你可以用自然语言明确告诉系统你允许什么、不允许什么。但至少，我们不需要再去问一个做市场或法务的人“你允许我安装Homebrew吗？”因为这个问题的含义和答案太复杂了，难以判断。**虚拟机给了我们一层很好的抽象，让Claude变得非常强大**。

- **系统提示词的微调**：围绕虚拟机，我们每周都在增加一些东西，让Cowork在某些任务上比单纯的Claude Code表现更好。这些东西大多存在于**系统提示词**中。我们会根据对你工作的推断，引入一些提示，让它更高效。
- **与Claude和Chrome的深度集成**：你可能会注意到，随着模型越来越好，很多人面对大量的MCP连接器时感到手足无措。而Claude和Chrome的集成之所以强大，是因为我们可以直接跟Claude和Chrome代理对话，让它直接为你做事。
**主持人swyx**： 所以，你的观点是，**整个硅谷可能都低估了本地电脑的价值**。看看我们，为什么大家还在用MacBook，而不是iPad或Chromebook？这就是本地电脑的价值所在。当我们将Claude视为一个应该对你**极其有用**的实体时，我认为这个实体需要能访问你所能访问的**所有工具**。否则，它就会在各种复杂的情况下受到限制。

**Felix**： 完全同意。针对这个问题，有两种解决路径。一种是，我们一点一点地把你电脑上的东西都搬到云端。我个人，作为一个使用大量工具的人，实在没有耐心去给另一个工具授予所有权限，并持续维护这些权限的更新。第二点，也是我仍在思考、没有答案的问题是：**把整个工作环境都搬到云端，究竟意味着什么？** 比如，如果一键就能克隆你整台电脑到云端，你真的愿意吗？我不确定所有人都会愿意。

还有更上游的技术和社会问题。举个简单的例子，作为一个桌面应用，理论上只要我们想，在你授权的情况下，我们可以在你的电脑上做很多事情，包括读取你的Chrome cookies。我们可以解密它们，然后把它们放到云端。这看起来是个很酷的解决方案，我们就能在云端处理你所有任务了。但问题是，很多网站，包括银行，如果检测到同一个账户从两个不同地点同时登录，就会直接锁定你的账户，你只能跑去银行网点处理。所以，尽管我们对“Agent”这个词已经感到厌倦，但整个社会体系、技术规范都需要慢慢适应和赶上。在此之前，作为Claude的开发者，我让Claude最高效的方式，就是**让它运行在你工作的地方——你的本地电脑**。

## 四、 用户故事：从整理桌面到构建“自动化帝国”

**主持人swyx**： 我想给你展示一下我实际是怎么用Cowork的。这是我运行Latent Space Discord频道的方式。起初我并不完全信任它。

- **第一次尝试**：我先让它帮我手动从Zoom下载所有录制的视频并上传到YouTube，这是一个非常繁琐的过程。它真的做到了！
- **深入一步**：然后我让它把下载Zoom的部分也接过去，它开始压缩文件，甚至能通过查看视频的单个帧来为视频命名，以便自动上传。感觉它在取代我的工作。
- **创造“技能”**：最棒的是，我让它把重复性、一次性、需要人工引导的任务，打包成它自己可以**复用的“技能”**。它会写技能，这些技能会进入上下文，出现在底部的技能栏里。现在，我每周都能使用这些技能。
- **不断进化**：我甚至告诉它，有些任务可能需要拆分成几个子任务。于是它就创造了一个“技能拆分”的技能，把一个父技能拆分成三个子技能，然后还有一个父技能来编排它们。
- **终极体验**：还有一个例子是，我懒得去读YouTube API的文档来搞程序化上传，我就直接让Cowork去做。它帮我搞定了Google Cloud的一切，我根本不在乎过程。它真的太棒了。
**Felix**： 看到你这样使用，我感到非常兴奋和有趣。关于“技能”，最棒的一点就是它们**非常容易创建**。任何人都可以创建技能，甚至一段文本消息也能成为一个技能。而且它们可以高度个性化。你刚才描述的，其实就是人们上手Cowork的典型路径：从一个你平时需要手动点击完成的知识工作开始，尝试让AI去做；然后你觉得，嗯，再进一步会怎样？再进一步呢？随着你对它的信任增加，不断扩展它的工作范围，同时教会它如何取代你的部分工作。这有点像在玩《异星工厂》（Factorio），从一个极小的自动化开始，一旦成功了，你就不断地往这个自动化帝国里添加东西，让自己的生活越来越轻松。

**主持人swyx**： 我个人最喜欢的“技能”是，每天早上Cowork都会查看我的日历，确保没有日程冲突。人们经常临时或忘记安排会议。我在自定义提示里给了它很明确的指示：比如，如果是Dario（Anthropic CEO）安排的会议，就别尝试去重新安排了。还有一些规则关于我更在意哪些会议，哪些可以推迟等等。正是这些**微小但贴心的事情**，让人们真正喜欢上这个产品。

**Felix**： 是的，当我们刚推出Cowork时，在推特上最火的一条用户反馈是“帮我清理桌面”。这听起来有点傻，好像不需要模型来做这件事。但当你真的把自己的桌面或下载文件夹交给它时，效果惊人。它会说：“你有这么多份Term Sheet，还有8份办公室租赁合同的副本。”你会觉得，好吧，别对我大喊大叫了。虽然是个小任务，但正是这种切入点，让用户开始产生信任。

**主持人swyx**： 而且它还会在执行计划时，通过“询问用户”的界面来和你互动，比如问某个文件是不是明显的垃圾文件，是否可以删除。这种交互非常清晰，我可以在旁边微调它的指令，比如“别这么做，别那么做”，体验非常好。

## 五、 深度对比：Cowork vs. Claude Code

**主持人swyx**： 我发现我使用Cowork的频率甚至超过了Claude Code。你之前提到要我把“规划”（planning）这个概念从对Cowork的理解中拿掉，因为Claude Code也有。但我在Cowork中感受到的体验确实不同，它总是会先给我一个九步计划，我可以编辑它、给它反馈，然后看着它执行。这感觉比Claude Code更有长远规划的能力。

**Felix**： 你的感觉很准，这可以说是两者兼有。如果Claude Code的开发者坐在这里，他们可能会说，我们也有所有这些功能。确实都有。但我认为，人们倾向于交给Cowork的任务，通常**时间跨度更长**。这是第一点。第二点，因为任务时间更长，也就会更模糊。我们在系统提示中**确实会引导Cowork更频繁地使用“规划”工具和“询问用户”工具**。我们希望它能设想出不同的场景，向用户明确需求，而不是埋头苦干四个小时后，给一个错误的结果。你可能正是捕捉到了这种差异。

在评估标准上，两者也是不同的。Claude Code非常优化于编码任务，我们评估它主要看它在典型软件开发工作中的表现。而Cowork的评估则更偏向于典型的**知识工作**，比如金融、法律办公室里的那些任务。我个人的用例是管理我的个人事务，比如处理我的按揭贷款，或者为我和家人规划旅行。我们基于这些用例来评估和调整Cowork。通过修改系统提示、调整工具，让它在不同方向上更优。所以，**Claude Code会更擅长写代码，而Cowork则更擅长非编码任务**。当然，随着模型代际的更新，这种差距是否会一直存在，我也不是很确定。

## 六、 技术内幕：虚拟机、Windows支持和“瑞士奶酪”安全模型

**主持人swyx**： 我知道你最近在Windows支持方面做了很多工作。这其中有什么有趣的技术故事吗？让大家能理解你们在沙箱技术上投入了多少心血。

**Felix**： 是的，虚拟机这部分其实非常酷，尽管有很多权衡。比如有人问我，为什么Cowork占用了10GB磁盘空间？其实这只是Mac OS显示字节的方式有问题，我们实际通过压缩镜像中的空白空间，并不会真的占用10GB。但更关键的是启动速度。

在Windows上，我们使用的是**Windows Host Compute System**，也就是和WSL2（适用于Linux的Windows子系统）一样的底层技术。我们必须仔细分离虚拟机运行的系统空间，确保只有Claude能与之通信，而其他乱七八糟的应用程序无法访问它。我们还在做很多底层的优化，比如上周我们就在编写一个新的网络服务/驱动，以优化Claude访问互联网的方式，并确保它能适应公司内部复杂的网络环境（比如数据包检测、SSL解析等）。

我们之所以要费这么大力气构建这个VM，核心目标是让Cowork能在**大多数用户的默认电脑配置上高效运行**。那台电脑可能没有Python，没有Node.js。如果我们去掉这两样，Claude在你的电脑上效能将大打折扣。而通过虚拟机，我们给Claude提供了一个内置了所有必要环境的“自己的电脑”。

即使在像Anthropic这样的公司，IT部门也会严格控制你能安装的软件。而我们的方案让IT部门的工作变得更容易：他们可以把**“Claude的电脑”和“用户的电脑”分离开来管理**。对于Claude的电脑，你只需关心数据泄露、潜在恶意行为等问题。一旦你控制了网络和文件系统层，你就无需再担心Claude可能会写一些非常有用的Python脚本——你真正担心的是，一旦用户电脑安装了Python，任何人都可以用它做任何事。但把Python放进虚拟机，风险就大大降低了。

这背后是一种**“瑞士奶酪”安全模型**的思考。我们不能要求用户批准每一步操作，那产品就没法用了。我们需要建立一个系统，让用户可以放心地**授权并走开**，相信AI不会搞砸。我们不能等到100%的模型对齐才行动，而是需要像其他行业一样，通过多层防护来构建可信赖的系统。

## 七、 行业影响：谁将被取代？初级员工的未来

**主持人swyx**： 这可能会是个有点尖锐的问题。你认为哪些行业会因AI而消失？

**Felix**： 我不确定我是最适合预测哪个行业受冲击最大的人。但我可以说，在Anthropic，我们作为一个群体，**深切关注这些工具对劳动力市场的影响，特别是对初级员工的影响**。当我们谈论自动化掉那些我们觉得烦人的、不想花时间去做的工作时，这些工作在很多行业里，原本是交给初级员工去做的。我认为，担忧这会对进入就业市场的年轻人造成什么影响，是完全合情合理的。

**主持人swyx**： 我有一个解决方案，一半玩笑一半认真——**给他们创造“模拟工作”**。以软件工程为例，一个初级工程师工作两三年，其中真正学到东西的时刻可能就那么几个，其他日子都是在重复或低效劳动。现在，我们可以用AI和这些模型来“加速”他们的职业生涯，模拟出早期工作的各种场景，让学习体验变得超级密集。比如，公司要开发一个分布式系统功能，按传统你得学三个月。现在我们可以模拟一个完整的项目环境，让你在一周内“速通”整个流程，获得核心经验。重复一年，你就相当于拥有了三年的项目经验。听起来有点傻，像是在创造“假工作”，但这其实很像上大学。人们付费去学习如何工作。未来可能会出现Jane Street的模拟器，你想去那儿工作？先在里面模拟三个月。所以，这方面是存在可能性的。

对于我自己更熟悉的工程领域，我看到的是，资深工程师的加速效应非常明显，他们的生产力提高了，提供的价值也更大了。我一直在想一个现象：即使在这一切发生之前，我就非常尊重滑铁卢大学的毕业生。他们加入我的团队时，总是感觉比那些从未在真实产品环境中工作过的名校毕业生准备得更充分。滑铁卢大学的学生花大量时间在不同公司实习，他们就像一个去过20家公司的初级工程师。这让他们比其他新毕业生优秀得多。我在想，这或许可以成为未来的一个模型：当初级员工的价值受到影响时，我们或许可以借鉴这种方式，把学习和成长的阶段**压缩得更紧凑、更有价值**。

## 八、 未来展望：Cowork将走向何方？

**主持人swyx**： 最后，我们来聊聊Cowork的未来。你对它的发展有什么规划？

**Felix**： 我觉得我们还处于非常早期的阶段。我们会持续快速地迭代，你可以预期**每周都会有一个小新功能，甚至是大功能**。我会继续在“你的电脑”上加倍下注，让你和Claude在你的电脑上都能更高效地工作。我们正在开始思考那个更深层次的问题：“你的电脑”究竟意味着什么？是你面前这台？是你电脑上的一个虚拟机？还是在云端的某台电脑？

第三点，我们非常兴奋的是，我们将继续努力，慢慢引导那些习惯了“问问题、拿答案”的用户，让他们一步步学会**把更大、更独立、更长期的任务交给Claude**。你可能已经在我们发布的功能中看到这种投资了。

**主持人swyx**： 比如，让Cowork能远程控制另一个Cowork？这个功能快来了吗？

**Felix**： （笑）好问题。很快就来。如果你想继续在你电脑上押注，这显然是个方向。

**主持人Alessio**： 还有一个关于“多人模式”的问题。比如，我和同事各有各的Cowork，它们之间能交流吗？或者我同事机器上有个文件或任务进度，我想知道，并据此更新我的任务。你们会做这方面的东西吗？

**Felix**： 这个问题非常吸引我，它又回到了那个“哪些脚手架会消失”的问题上。我在想，到了某个阶段，我们是不是就直接给这些Agent分配一个Gmail账号、一个Slack账号？它们就可以像人类一样，用我们现有的工具来互相交流。我们的财务团队一直在努力做很好的办公套件集成。曾经我们做了很多技术，只为让Claude能在Google Docs里留下有用的评论，现在它可以直接做到了。或许这就是答案。关于Cowork Agent之间如何交互，我仍在思考：是我们为他们构建一套超级定制的通讯协议，还是直接跳到终点，给每个Agent一个Slack账号，这样它们就自然具备了“多人”协作能力？

**主持人swyx**： 想象一下未来的组织：销售人员跟工程团队的Agent对话，工程团队的Agent再跟市场和产品团队的Agent交流，所有这些Cowork把决策整理好，然后交给人类去审批。怎么让我的Cowork有权限去批准另一个Cowork的任务？怎么决定哪些需要我亲自过目？这些都是很有趣的问题。

**Felix**： 这又回到了一个核心：强大的功能和令人感到不适的边界往往只有一线之隔。比如，技术上我们可以通过蓝牙LE让两台电脑知道它们彼此靠近，从而推断你可能在和同事协作。你会看到Cowork这么做吗？大概不会。但确实存在很多我们从未尝试过的、非常有创意的解决方案。
