这不是什么官方外交辞令。这位身居高位的部长洋洋洒洒地写了一整篇技术指南，详细拆解他是如何在一个树莓派上部署并运行你的项目，还自己捣鼓了一套融合了 Karpathy LLM Wiki、Embedding 存储和记忆系统的“第二大脑”。他甚至在文中直接点名感谢了你的名字。

“他显然是在深度极客式地玩耍（nerding out）。”Gavriel Cohen，那个被点名的项目——NanoClaw——的创始人，在 Latent Space 的采访中回忆这段经历时依然难掩兴奋。这个微小的、看似偶然的事件，却成为了一场关于企业 AI 代理未来走向的深刻思考的催化剂。在这场对话中，Gavriel 不仅剖析了一个开源项目如何阴差阳错地被全球顶级决策者拥抱，更系统性地提炼出了一份关于「自主工作代理」的完整蓝图：**从安全架构到商业模型，从工具哲学到代码开源治理的未来。**

## 详细内容

### 一、事件：当外交部长成为你的「首席架构师」 `[对话节选：故事开端]`

**核心观点**
一次真实世界中的高端个人化使用案例，让 AI Agent 的「杀手级场景」从模糊变得清晰——它不是什么宏大的自动化工厂，而是一个贴身的、不断进化的**个人知识中枢**。

**深度阐述**

Gavriel 回忆，那本是他推出 NanoClaw 后的第一个完整周末。他正在岛上度假放松，X 上突然炸开了锅。有人转发了一条来自新加坡外交部长 Vivian Balakrishnan 的 Facebook 长文。这位部长不仅盛赞 NanoClaw 的简洁与隐私设计，还完整公布了他基于 NanoClaw 搭建的个人系统架构：使用了基于 Karpathy 风格的 LLM Wiki、结合 Embedding 的语义记忆系统，以及一个叫做 **Nemon** 的小众记忆管理工具。

Gavriel 当时查了一下 Nemon，发现它只有 43 个 GitHub Star。也就是说，这并不是什么大公司或主流工具的背书，而是一个技术背景深厚的高级官员，在用自己的审美和需求，从开源社区里精心挑选工具，亲手搭建了一套也许只有他自己能完全搞懂、但对他本人来说极其高效的“第二大脑”。这位部长从 NanoClaw 第一版迁移到第二版，写了一份完整的迁移指南，觉得不错，就顺手分享了出来。帖子迅速在 X 上传播，两人开始直接交流。

“他说，‘如果你来新加坡，我很乐意接待你。’三天后，你们（Latent Space）就跟我说要在新加坡举办活动了。”

这个故事的意义远超其戏剧性。它给了 Gavriel 一个极其重要的信号：**在所有人都追逐“用 Agent 自动化整个工作流”的宏大叙事时，真正的价值峰值可能藏在“给每一个个体创造一个可信、私密、不断学习的 AI 延伸”里。**

Kovid Goyal 敏锐地指出，这正是很多企业高管在使用各种 Claw（OpenClaw, NanoClaw, Hermes）的真实状态——他们将代理当作一个私人知识管家，而不是一个工作流的替代者。这彻底改变了 Gavriel 接下来的产品路线图选择。

**精华收获**
一个外交部长级别的用户背书，不仅仅是一个社交媒体热点。它揭示了一个深刻的范式迁移：当代 AI Agent 的终极“杀手应用”，很可能不是取代人的工作流自动化，而是**赋能每一个知识工作者构建属于自己、且只属于自己的“知识宇宙”**。这种“赋能个人”的路径，远比“替代流程”更易落地，也更具粘性。

### 二、抉择：当「个人第二脑」战胜「自动化工厂」 `[对话节选：理念探讨]`

**核心观点**
在「团队管理 Agent」和「Agent 赋能个人」两条路线之间，NanoClaw 选择了后者。原因只有一个：**使用 Agent 本身就是一种需要学习的高级技能，而个人第二脑是学习成本最低、收益最直接的切入点。**

**深度阐述**

Gavriel 坦诚地分享了他在设计阶段面临的巨大岔路口。他和团队一直以来都在两条并行但本质不同的路径上探索，为企业引入 AI 代理：

1.  **Agent 工厂模式**：团队一起管理一堆代理，将其嵌入工作流，自动化某个特定流程（如法务团队的合同草拟自动化）。
2.  **个人 Agent 模式**：公司里的每个个体拥有自己专属的代理助手，帮助自己做决策、整理信息和执行任务。

在很长一段时间里，他们没有明确的优先级。NanoClaw 团队既在内部用两种方式使用 Agent，也在与设计伙伴同时探索两条路。直到新加坡外交部长的故事出现，这场“路线之争”才有了终局。

“部长的使用场景让我彻底看清了一件事：**如果你想在企业里推广 Agent，第一步必然是给每个人发一个私人的 Agent。**”

为什么？Gavriel 给出了一个反直觉但极具洞察力的解释：

“人们（包括高管）总是犯一个最大的错误：他们想扔一个任务给 Agent，然后走开，指望回来就能看到一个完美的成果。但现实是，你必须投入精力。你必须学会如何与它共舞——调整指令，调整技能，管理上下文窗口。就像训练一个实习生。”

而这个学习曲线，在“个人第二脑”的场景中是最低的。你不需要让 Agent 去执行涉及复杂跨部门协作的流程。你只需要让它帮你**收集信息、整理笔记、生成摘要**。

“把它变成你的维基百科。你不需要它给你现成的成品，你只需要它不断地吸收你世界里的信息，构建一个关于你的内部知识图谱。当你在询问‘我这周最重要的几件事是什么？’时，它能基于几份文件生成一份精准的列表。这才是今天 Claude 类 Agent 真正的杀手场景。”

Gavriel 还分享了他的个人实践：他并没有用什么高深的 RAG（检索增强生成），而是用一种更简单的 **LLM Wiki** 方法。他让 Agent 读取 Karpathy 关于 LLM 的博客，然后说：“读这个，我们聊聊。”同时，他让 Agent 遵循一个简单的核心指令：“创建一个 Markdown 文件，任何实质性内容都要保存下来。要么放在 `cloud.md`，要么放在一个特定的 Markdown 文件里。必须有一个地方保存它。”

当然，这不是完美的。Kovid Goyal 提出了一个几乎每个 Obsidian 深度使用者都会遇到的问题：“我用了这种方案，但代理会创建大量重复的文件。它不知道已经有了相似的内容，于是创造了另一个版本的真相。我现在有两个版本的真相。”

Gavriel 赞同道：“你需要额外的指令让它创建一个文件索引。并且，你必须有一个后台进程，每天或每隔几天去检查这些记忆文件，寻找并标记重复项。这确实是个大坑，我们的 Nana 内置的记忆系统目前只是一个简化的版本。但路径是正确的——**用检索（Retrieval）去解决个人化背景信息是不靠谱的。你需要的是一个结构化的、逐步写下来的‘生活知识库’。** ”

**延伸思考**
这给了所有在构建 AI 助手的人一个巨大的启示：**RAG 对于开放式的、高度个性化的知识工作场景，可能并不是终极答案。** 因为当你问“我这周该干什么？”时，没有哪个语义搜索引擎能直接回答你。但如果你的 Agent 有一个类似 LLM Wiki 的“大脑”，里面写着你每个项目的时间线、本周的会议记录、以及你设定的优先级，它就能像人类助理一样，翻阅这几份文件，给你一个综合的回答。**这是一个从“检索匹配”到“推理阅读”的转变。**

### 三、基石：从零开始，为「安全」重构整个架构 `[对话节选：中段偏前]`

**核心观点**
AI Agent 的安全不是靠外部防火墙包裹，而是必须从设计之初就在架构层面进行**绝对隔离**。NanoClaw 的核心理念是：让代理永远管不了自己的钥匙。

**深度阐述**

为什么新加坡外交部长选择的是当时才上线不久的 NanoClaw，而不是功能更丰富、社区更大的 OpenClaw？

Gavriel 指出了一个在 AI 产品中极易被忽视但至关重要的问题：**代码库的健康度与安全架构的严谨性。**

他本人并非一开始就想自建。他最早使用的是 OpenClaw，并且将其深度嵌入了自己正在运营的 AI 营销原生企业中。仅仅两天，OpenClaw 就相当于一个销售经理，开始管理整个销售线索管道。功能确实强大，但也正因为要用于商业，Gavriel 开始仔细审视它的底层。

他看到了一个庞杂的代码库、无数密集的依赖项，以及一个致命的问题：**所有消息都以明文记录（plain text）**。

“如果你把它用于个人用途，感觉还好。但如果我要把它连接到我客户的数据库和业务流上，我的整个公司都要建立在这个系统之上……它不够稳定，不够安全。我感觉我无法在上面构建一个严肃的业务。”

于是，Gavriel 没有去改 OpenClaw 的代码，而是做了一个极致的决定：**从零开始，全部重写。** 他带着一个清晰的目标：**极致的可读性、极简的依赖、以及根本性的安全隔离。**

这种理念催生了 NanoClaw 独特的三层安全模型，这或许是目前公开可见的最具实操性的 Agent 安全架构指南：

**第一层：运行环境隔离**
整个 NanoClaw 运行在一个独立的虚拟机上，或者一台专属的 Mac Mini 上。但这只是第一步。

**第二层：代理运行时与消息桥接隔离（核心创新）**
连接 Slack、Discord 等渠道的“消息桥”和“路由器”，必须与执行实际任务的“代理运行时”严格分离。这里的做法是：**每一个 Agent 运行在自己独立的容器中。**

**第三层：凭证保险库与零信任（最关键）**
这是 NanoClaw 最富洞察力的设计。**Agent 的运行环境中绝对不能存放任何凭证。**

“这非常关键。大多数有用的场景，Agent 都需要接触外部输入。比如审查一个 PR，那就是一个完全未经消毒的输入。任何人都可以向开源仓库提交一个 PR。如果 Agent 环境中藏着一个 API Key，一次简单的提示注入（Prompt Injection）就可能把钥匙交出去。”

正确的做法是：Agent 容器内部没有任何钥匙。Agent 发出的所有外部请求，都必须通过一个**凭证保险库（Vault）** 进行代理。Vault 根据预设的访问策略（Access Policies）来决定是否为这次请求注入凭证。并且，对于敏感操作（“发送一封邮件”），必须触发**人在环（Human-in-the-loop）** 审批。审批通过 Slack 的按钮实现，你可以看到 Agent 想要发送的内容，然后决定批准或拒绝。

Gavriel 用一种近乎坦诚的口吻说：“我自己对于什么样才是正确且合理的架构有一个直觉——要做成可读的、安全的。但我不太相信自己不会搞砸，所以我把它开源，让全球的安全专家来审查。最后的结果是：没人质疑过这个核心方法论。大家都觉得这个路径是扎实的。”

**精华收获**
如果要在 AI Agent 的安全上记住一句话，就是**让 Agent 永远管不了钱（凭证）**。把 Agent 当作一个随时可能被污染的“前端”终端，所有对真实世界的操作请求都必须经过一个拥有强策略和审批流的“中间代理层”。这种架构思想，是任何试图在企业内部署 Agent 的人必须尊重的铁律。

### 四、商业：从「工具提供者」到「AI 工程合伙人」 `[对话节选：中段偏后]`

**核心观点**
AI Agent 的创业机会不在于卖软件许可证，而在于提供一种**动态且持续的服务**——帮助客户完成那个跨度极大的“最后一公里”。

**深度阐述**

NanoClaw 的诞生充满了偶然。Gavriel 最初全身心投入的是一家 AI 营销原生企业。NanoClaw 只是他的个人周末项目，以 MIT 许可发在 Hacker News 上。直到 Karpathy 发了一条推文，项目彻底爆火。

“有一个很好的团队，但我们在一个岔路口：是做营销代理，还是做 NanoClaw？Karpathy 的点赞似乎直接给出了答案。社区的力量和热情太大了。”

于是，NanoClaw Co. 正式成立，如今拥有一个 10 人左右的团队。

最令 Gavriel 意外的是用户的构成。他发现，早期的深度采用者（不仅仅是 NanoClaw，包括 OpenClaw 和 Hermes）大多是大公司的 CEO 和高级管理者。这些高管沉迷于自己搭建完美的工作流，但他们遇到了一模一样的问题：“我有三台不同的 Agent 配了不同的记忆系统，简直酷毙了。我想把这套东西推广给我的整个团队。**但我可不想变成 IT 网管，每天帮他们排错。** 你们能来帮我搞定吗？”

这个需求直接定义了 NanoClaw 作为公司的商业模式：**托管部署服务。**

这背后是对企业采用 AI 本质的深刻洞察。Kovid Goyal 指出了那个几乎不可能完成的鸿沟：“他们会跟你提 SSO、BGP 对等、本地部署、和他们的凭证管理系统对接……”

Gavriel 完全认同：“是的。企业有很好的工程师，有很好的 DevOps，有很好的安全团队。但如果没有**AI 工程**那块拼图，他们就无法确信自己搭建的东西是安全、正确和最合理的配置。”

**因此，NanoClaw 扮演的角色不是软件公司，而是一家“AI 工程合伙人”。** 他们的流程是：

1.  **初期深度介入**：AI 工程师进驻，联合企业的 DevOps、安全、IT 团队，完成初始搭建。将 NanoClaw 对接企业内部凭证管理、监控、安全系统。
2.  **交接**：初始的 API 集成完成后，IT 和安全团队能够接手新增的数据源集成。
3.  **持续管理服务**：这才是重头戏。Gavriel 用一个极佳的对比总结了 AI 与传统软件的区别：
    > “传统企业软件，比如 CRM，你部署在服务器上，放那儿五年不碰，它一般还能正常跑。但 AI 代理绝对不行。你构建的核心在持续变化。你不能只运行 GPT-4.6 然后坐等三年。你必须升级到 4.8、4.9、5.0。每次升级都改变规则。模型厂商在推出新功能、新能力、内化记忆。为了始终站在前端，你必须持续更新。这就是我们的价值。”

目前，已经有超过 100 家公司主动联系 NanoClaw，希望在其基础设施上部署 Agent 系统。

**精华收获**
AI Agent 的企业的落地，揭示了纯粹的 SaaS 模式的一个盲区。当底层模型半年一次大升级、提示工程技巧日新月异时，一个“配置好就遗忘”的软件是无意义的。**真正的价值在于一个能够跟随模型发展共同进化的“AI 工程服务”。** 这或许预示着 AI 时代 IT 外包和咨询行业的一个全新进化方向。

### 五、未来：当开发变成「从 Wiki 拉取上下文」 `[对话节选：后段]`

**核心观点**
Git 和 Pull Request 的黄金时代正在被 AI 代理革新，未来的开发规范可能会变成 **「Prompt Requests」和「Wiki-Driven Development」**。

**深度阐述**

对话的最后，话题转向了一个困扰 Gavriel 的现实问题：**开源项目的管理困境。** 他是 NanoClaw 这个开源项目的维护者，而社区正在用 AI 代理向他发起一场“混战”。

问题出在 GitHub。Gavriel 的代理工厂（Agent Factory）本身就在 GitHub 上通过 PR 运作。代码生成 AI 的流行，使得任何人都能毫不费力地生成并提交大把的 Pull Request。这听上去是开源社区最幸福的烦恼？

“不，简直是一场噩梦。庞大的 PR 量需要极大量的审查（triage）工作，去理解它是否真正对齐项目的目标。”

Gavriel 提出了一个极具前瞻性的解决方案，这个想法甚至让 Kovid Goyal 当场击节：

**“不再有 Pull Requests。只有 Prompt Requests。不要给我你的代码，给我你的用例（Use Cases）。”**

这句话背后的逻辑极其精妙。如果将 Bug 报告、功能请求等所有外部输入，都看作是一种对项目的“上下文补充”，那么它们就不应该以零散的代码 diff 形式出现。它们应该直接流入一个**活的维基（Living Wiki）**。

Kovid 立刻补充了他的观点，并与之共鸣：“所有的变化应该持续不断涌进来，变成一个 Wiki。开发者不是从 Issue 列表开始工作，而是从这个 Wiki 中拉取上下文进行开发。Wiki 就像是一个缓冲池。你解决一个问题，从中拉取；你完成一个提交，把新的上下文推回 Wiki。”

Gavriel 透露，他们已经在为自己的代理工厂构建这样一个 Wiki。“在每次合并、每次提交时，我们都在向 Wiki 添加内容。未来的开发，将是这样一幅图景：在实时开发的过程中，同步构建和更新着项目 wiki。Bug 所在的段落，直接链接到项目的相关部分。新功能的需求，也存在于这个 Wiki 中。开始一次开发时，你从 Wiki 拉取上下文；完成后，你将新的上下文推回 Wiki。这是一个有机的生长体。”

**延伸思考**
这个对话触及了一个更深层的变革。当 AI 能够生成代码时，“写代码”本身已经不再是瓶颈。真正的瓶颈变成了 **「定义问题」和「对齐意图」**。Prompt Requests 的本质，是将人类从繁琐的代码表达中解放出来，专注于更高阶的意图沟通。而 Wiki 则变成了这个意图沟通的沉淀池。**未来懂行的开发者，不是一个好的代码提交者，而是一个好的 Wiki 贡献者。**

### 六、栖息地：Slack 是国王，SMS 是噩梦 `[对话节选：末段]`

**核心观点**
AI Agent 的使用体验，严重受限于其运行平台的 API 丰富度与生态开放性。

**深度阐述**

这是一段轻松但有深刻洞察的讨论。当 Gavriel 被问及人们应该在哪里使用 Agent 时，他给出了一个非常明确的“消息平台分级列表”。

*   **S 级：Slack。** 几乎没有竞争对手。API 全面，UI 成熟，极其适合构建复杂的交互式工作流。Gavriel 甚至倾向于把一切都放进 Slack 里跑，包括代码合并审批。
*   **A 级：Discord 和 Microsoft Teams。** 功能强大，但各有各的生态限制。
*   **C 级：Telegram。** 功能不错，但在构建复杂的 Business UX 方面受限严重。Gavriel 坦言自己很喜欢 Telegram 的使用量，但无法把它作为主力。
*   **D 级：WhatsApp。** “体验非常非常有限，很难做点什么。”
*   **F 级：SMS。** “天哪，那简直是地狱。你需要一个独立的手机号码才能收消息。我不能发消息给自己，因为我需要一个独立的号码去接收它。我放弃了。”

这个看似玩笑的分级，实际上给所有构建企业级 AI Agent 的人提供了一个标准：**选择一个平台，就是选择了一个 Agent 的能力上限。**

## 精华收获

这场对话提供的不仅仅是一个产品故事的记录，更是一份沉甸甸的行动蓝图：

1.  **起点是“人”不是“流程”**：别急着用 Agent 取代工作流。先试着用它来赋能团队里的每一个人。个人的第二大脑是目前门槛最低、价值最确切的切入点。
2.  **安全是信仰，不是功能**：绝对信任 Agent 是一个危险的幻觉。将 Agent 设计成永远不能触碰凭证的零信任终端，通过保险库代理和人类审批来控制它的行动半径。
3.  **卖服务，不卖软件**：AI 底层变化太快。客户需要的不是一份安装指南，而是一个能陪他们走过每一个模型大版本的 AI 工程团队。
4.  **用 Wiki 管理混乱**：当你被代码生成的 PR 淹没时，不要试图修复流程，而是重构系统。用活的 Wiki 代替死的 Issue 和 PR，让意图成为核心资产。

关于未来，Gavriel Cohen 的一句话或许最能概括这场对话的精神内核：
**“不要给我你的代码，给我你的用例。”**
