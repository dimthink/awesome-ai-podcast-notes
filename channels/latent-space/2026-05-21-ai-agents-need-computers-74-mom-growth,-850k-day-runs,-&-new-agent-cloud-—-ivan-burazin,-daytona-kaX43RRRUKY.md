## 开篇引入

想象一下，你开发了一个 AI 代理，它不只是回答问题，而是真正帮你操作电脑——打开浏览器、登录系统、导出数据、运行代码。问题在于：今天的 API 覆盖率远不够，大多数企业应用仍是封闭的“遗留系统”，而即使是现代 SaaS，也总有数据无法通过纯接口获取。此时，代理需要的不是一个简单的代码执行器，而是一台**完整的、可随时启动、可持久化、可弹性扩展的计算机**。

这正是 Daytona 正在做的事情。这家由 Code Anywhere 创始人 Ivan Burazin 领导的初创公司，在 2023 年底完成了一次关键转型，从面向人类开发者自动化环境，转向为 AI 代理提供“可组合计算机”。结果令人瞠目：**74% 月环比增长、每天处理超过 85 万次沙箱启动、客户从个人开发者到顶级 AI 实验室，而最大的瓶颈竟然是——CPU 不够用了**。

在这期 Latent Space 的深度访谈中，Ivan 不仅揭示了 Daytona 的产品逻辑与技术架构，还分享了他对基础设施市场趋势、开源策略、代理未来工作负载的独到见解，甚至包括对“代理人将取代所有白领工作”这一宏大叙事的务实拆解。无论你是正在构建代理的开发者，还是关注 AI 基础设置的投资人，这段话都能让你重新理解“计算机”在 AI 时代的含义。

## 详细内容

### 1. 代理需要一台真正的电脑 `[对话开头部分]`

**核心观点**

AI 代理需要的不只是代码执行环境，而是一台有持久状态、可被“使用”的计算机——就像人类需要笔记本电脑一样。

**深度阐述**

对话伊始，主持人直接抛出了核心洞见：代理已经具备了很强的推理能力，但它们仍然缺乏对现实世界的操作入口。Ivan 用他亲身经历的例子说明问题——为了准备一份董事会报告，他尝试将所有数据源（ClickHouse、PostHog、QuickBooks）通过 API 连接，却发现总能拿到不完整的数据。最终他做了一件事：给代理一台 macOS 虚拟机，为它创建公司账户，然后命令它“登录这些网站，导出数据”。这个小小的实验让他彻底信服：**对于知识工作，API 永远不够用，代理必须能够像人类一样操作 GUI 和遗留系统**。

Ivan 进一步将视角放大：全球有约 1 亿知识工作者，仅在美国就涉及 10 万亿美元薪酬。其中 56% 集中在医疗、政府、金融等依然深度依赖 Windows 的行业。这些应用不被重写（成本过高），所以任何试图实现大规模流程自动化的方案，最终都必须落实到“代理操作计算机”这一层。他估计，如果代理能自动化这部分工作的 40%，那就是每年约 10 万亿美元的价值体量。这不是 Daytona 直接获得的价值，但却是整个市场的终极天花板。

**个人感受**

Ivan 在讲述自己“用虚拟机做调研报告”的经历时，带着一种兴奋又无奈的语气。他反复强调：“我花了很多 token 就是为了这件事，但任务确实完成了。”这种亲自参与的使用体验，让他对“计算机使用”这一方向形成了极强的信念。

**延伸思考**

Ivan 提到，Windows 的沙箱化潜力巨大——目前只有 AWS 和 Azure 能提供 Windows 虚拟机，但启动时间需要 3-5 分钟。而 Daytona 开发了专门的 Windows 沙箱（秒级启动，支持即时快照、分支），这恰好瞄准了那个最大、最冷、最难啃的企业自动化市场。如果后续 macOS 也能解决授权与并发限制，整个“遗留系统自动化”的叙事将变得更真实。

### 2. 两个连续创业者的“婚姻与离婚” `[创业经历部分]`

**核心观点**

Ivan 与联合创始人之间超过 15 年的合作关系，以及从浏览型 IDE 先驱到 AI 基础设施的进化，是 Daytona 得以在关键节点快速 pivot 的关键因素。

**深度阐述**

Ivan 的故事开始于 2000 年代初：他和搭档先是做服务器虚拟化与服务外包，随后发明了全球最早（之一）的浏览器内 IDE——Code Anywhere。那时没有 VS Code、没有 Docker、没有 Kubernetes，他们必须自己写下整个技术栈。Code Anywhere 吸引了约 300 万用户，但因过于早期无法规模化，最终以天使投资退出的方式结束。

紧接着他们创办了开发者大会 Shift，正是这场会议（以及主持人 Swyx 关于“本地主机已死”的文章）激发了 Swyx 开始自己的活动业务。Ivan 回忆道：“我当时向你征求意见，你建议我成立公司，还愿意当顾问。我那时太专注了，拒绝了你的提议——后来我非常后悔。”

然后，在 3 年前，他们重启了自动化开发环境的项目，名为 Daytona（最初仍是面向人类工程师）。直到 2023 年底，他们观察到 Devon（早期版本）等行业样本开始出现，才意识到真正的机会在于为 **代理本身** 提供基础设施。于是元旦之夜，Ivan 用“vibe coding”拼出了一个最小可行产品——虽然被 CTO 评价为“绝对不能给别人看的垃圾”——但经过两周强化，最终赢得了第一批“难缠”的客户。

**个人感受**

Ivan 用“我们像结婚又离婚”来形容与联合创始人的关系，带着幽默和深刻信任。这种长期合作使得在 pivot 时，团队能快速达成共识、互相信任技术判断，而不需要漫长的决策过程。这种“老团队”的优势是他认为 Daytona 能以 25 人规模应对大规模客户的核心原因之一。

**延伸思考**

Ivan 的故事也反映了一个模式：许多成功的基础设施公司来自过去失败或过于超前的项目。Code Anywhere 的教训——构建一体化的 IDE 堆栈——恰好为 Daytona 提供了自主编写调度器、快照系统的能力，使得他们能从底层构建出与主流 VM/容器方案不同的性能特点。

### 3. 关键转折：当客户焦急地打电话要一个 API Key `[转型洞察部分]`

**核心观点**

2023 年底，Daytona 的早期 demo 让客户从“这东西不行”迅速变为“不给 access 就要打电话催”，标志着他们找到了正确的原始“计算机抽象”。

**深度阐述**

Ivan 详细描述了 pivot 经过。在 2023 第四季度，他们将 Daytona 顺手绑定在开源项目 OpenDevon（当时叫 OpenHands）上，作为 SaaS 运行。没人真正使用，但很多做代理的人联系他们：“我知道你有个能跑代码的东西？我需要那种环境。”于是 Ivan 和团队开始为这些建造者提供试用。

最初的反馈却是：“你的沙箱是在 EC2 或 VM 上吗？这东西不好用。”他们发现，市场已经有很多基础设施（EC2、GKE、Fargate），但用来运行代理时体验很差——冷启动慢、不能持久化状态、不能动态调整。

Ivan 花了大量时间听所有相关播客、读论文、参加活动，最终在元旦夜里独自拼出了一个新模型：它能在**60 毫秒**内启动一个 Linux 沙箱，底层直接跑在没有虚拟化开销的裸金属上，并提供即时快照与恢复。第二天 CTO 虽然嫌弃前端简陋，但后来重写后，他们给最初的 20-30 个“挑剔客户”做了演示。

结果出乎意料：所有 demo 电话都远超预定时间（15 分钟变 25-30 分钟）。所有人表示“我们想要这个”。第二天，如果有人没收到 API Key，就会直接打电话或发消息催促。Ivan 说：“我做了多个公司了，从没经历这种事——客户会因为得不到 access 而追着你。这意味着我们做对了，他们根本找不到替代品。”

**个人感受**

“他们想要的这个产品似乎根本不存在，而他们迫切想要。”Ivan 在讲述这个时刻时，语气中充满惊讶和确定的兴奋感。他后来补充：“更让人震撼的是，市场的体量——未来每一个可能存在的代理都需要这样的计算机。这个市场可能等于所有个人电脑市场，也就是一千亿级以上。”

**延伸思考**

这个转折点也揭示了基础设施市场一个重要特征：当新一代工作负载（AI 代理）出现时，旧基础设施（通用 VM、短暂容器）有根本性缺陷，而正确的“原始抽象”需要从第一性原理重建。Daytona 的定义是“持久、高速、可迁移的裸金属沙箱”，这与当时主流的“临时、不可变”容器思路完全相反。

### 4. 架构说明：为什么 Daytona 那么快？ `[技术细节部分]`

**核心观点**

Daytona 的设计围绕三个需求：**绝对快的启动速度（60ms）、持久化状态（暂停/恢复）、大规模并发（50,000 实例在 75 秒内）**。他们通过裸金属、自有调度器、本地快照传播实现性能优势。

**深度阐述**

Ivan 解释了与 Lambda 和 EC2 的类比：如果把 EC2 比作始终运行的台式机，Lambda 就是每次开始都从零配置的临时任务。代理需要的是“能随时合上屏幕，再次打开后还是刚才的状态”的体验。Daytona 同时提供这两种特性。

技术层面，他们提供了多种隔离级别（Docker 加固版 + Sysbox 提供接近 VM 的安全性、Firecracker 微 VM 等）。但性能核心来源于：

- **裸金属运行**：没有中间 VM 层，IOPS 极快，内存与 CPU 直接访问。
- **自有调度器**：从代码任意时代继承的自建调度算法，而非 Kubernetes 或 Nomad（对于持久化场景不够优化）。
- **本地快照**：每个模板/快照在裸金属节点的 NVMe 上预存，启动时直接“挂载”，无网络延迟。
他们还在评测网站（如 compute SDK）上长期稳居第一，单实例启动时间 0.1 秒，但 Ivan 强调基准测试只是最低门槛——更重要的是功能集、并发能力和客户支持。

**关键数据**

- 单沙箱启动：60ms（含网络往返）
- 并发 50,000 个沙箱：约 75 秒（而其他提供商可能需要 2,000 秒即 30 分钟）
- 当前最大客户每天运行约 85 万次沙箱
- 现有请求：50 万并发（相当于同时运行 50 万个 CPU）
**个人感受**

提到技术细节时 Ivan 表现出对工程团队的骄傲，尤其是 CTO 能从 2008 年的“原始”方法中找到现代解。他指出 Daytona 的利用率目前平均只有 15%（因客户工作负载极度尖峰），但他们努力通过客户协作来平滑需求，避免过度配置。

**延伸思考**

这种“速度+持久”的组合形成了对现有主流方案（AWS EKS、GKE、Fargate）的降维打击。对于强化学习训练或大规模评估（需要极快迭代），GPU 等待时间也是成本。Ivan 说：“如果你因为 CPU 慢而让 GPU 闲置，那实际上是在浪费更多钱。” 因此沙箱启动速度直接影响整体训练成本。

### 5. 两种工作负载：月亮的阴晴面 `[工作负载部分]`

**核心观点**

代理基础设施面临两大类工作负载：**跟随太阳的长期运行代理**（类似于人类使用模式）和**从天而降的突发尖峰**（RL/eval）。不同的模式对容量规划和弹性提出全新挑战。

**深度阐述**

Ivan 将客户需求分为两大阵营：

1. **背景代理 / 长期运行代理**：如 Devin、Lovable 这样的编码代理，以及 Harvey（法律 AI）。它们的使用模式很像人类：上午活跃，夜间降低，周末低谷。这种负载较容易通过“跟随太阳”的 Geo 调度来提升利用率。
1. **研究者 / 评估 / RL 训练**：如各种开源模型实验室。它们的特征是完全不可预测——可能连续几天无负载，然后突然要求 10 万并发，运行几小时后销声匿迹。使用模式几乎是“方波”。
Daytona 发现，仅仅六个月前，RL 工作负载占比几乎为 0，现在已上升到约 50%，且仍在加速。这也导致了非常低的平均利用率（15%），但峰值可达 90%。如何应对尖峰？有两种选择：过配置（成本高），或“即时计算”——溢出时从其他云提供商临时拉取裸金属（但速度慢）。对于需要 GPU 的团队，CPU 慢就意味着 GPU 空闲成本，所以优先选择能确保快速弹性的供应商。

Ivan 感叹这是基础设施领域前所未有的新问题：“我们和 Neon（数据库）、Parogen（搜索）这些公司聊，所有人都在面对同样的超尖峰负载模式。”

**个人感受**

当他指出“所有构建代理基础架构的团队都有相同问题”时，声音里既有共鸣也有苦恼。这种模式在人类时代是非常罕见的：你通常不会让某个网站的使用量瞬间飙升 10 倍再瞬间消失。但 AI 代理的“实验型”工作本质就是如此。

**延伸思考**

这个问题也引出一个推论：**未来云服务的形态可能会围绕“弹性+地理分散”重新构建**。类似于 Cloudflare 利用全球分布节点调度用户流量来平滑峰值，专门为代理设计的云需要更智能的资源调度。Daytona 未来可能会变成“代理流量路由器”的一部分。

### 6. 市场格局：与 Kubernetes 竞争，与 Twilio 类比 `[市场定位部分]`

**核心观点**

Daytona 在市场上与 EKS/GKE 等托管 Kubernetes 竞争，但最终更像“针对代理的 Twilio”——提供简单的 API 和 SDK，而不是复杂的集群管理。开源策略则是为了扩大社区与信任。

**深度阐述**

Ivan 认为，潜在的竞争对手数量众多（各种 VM 提供商、Serverless 平台、其他沙箱玩家），但 Daytona 的最大“可比对手”是自我管理的 Kubernetes。当客户测试过 Daytona 后，“他们再也不想回到 K8s”——原因在于：K8s 需要大量 operator 与界面交互来管理沙箱，而 Daytona 只提供一个 API Key。从消耗角度看，它更像 Stripe 或 Twilio。

关于开源：Daytona 的核心调度器、完整沙箱栈都在 AGPL 3.0 下开源。Ivan 明确承认 AGPL 并不被所有人都视为“纯粹开源”，但他们这样做是为了避免被直接复制，同时仍保持透明度。开源对他们的影响主要集中在“让客户更容易做初始评估”、信任构建以及间接影响云产品采用。但他表示：“开源并没有显著推动我们的增长，今天的增长几乎全是 PLG（产品驱动增长）通过云产品自然实现的。”

**关键数据**

- Daytona 现有 25 人团队（13 人合作超过 7 年）
- Slack Connect 连接超千个群组
- 企业客户从接触到签约最快 5 天（传统通常 2-3 个月）
**个人感受**

Ivan 坦言“开源”作为策略在今天更多是早期优势，现在客户更看重实时响应和支持。他甚至分享：“我们很多客户是‘推着我们通过采购流程’，他们太需要这个产品了，以至于愿意帮我们跳过审查。”

**延伸思考**

Ivan 对基础设施生态有长期思考，比如他判断“GitHub 并不是为代理时代构建的”。他提到一家客户的做法：每 5 秒将整个仓库状态 dump 为一个 JSON 并上传 S3——没有 diff，没有分支，仅仅是最新全量快照。这种野蛮但快速的做法说明现有代码版本工具无法满足代理频繁迭代的需求。他认为未来会有“代理优先的代码管理工具”诞生。

### 7. 计算机使用的未来：重回 RPA 时代，但更快 `[知识工作与 Windows 部分]`

**核心观点**

Ivan 认为最“热”的新方向是“计算机使用”（Computer Use）——让代理像人一样通过 GUI 操作任意软件。这将是下一代 RPA，但关键在于为代理提供快速的、可脚本化的 Windows 和（理想情况下）macOS 沙箱。

**深度阐述**

主持人提到 RPA（机器人流程自动化）在 COVID 期间有过一波炒作，但后来热度下降。Ivan 认为现在时机成熟了：模型能理解多模态界面、能推理复杂步骤。但核心瓶颈仍然是底层计算机的可用性。

Daytona 已经开始提供 Windows 沙箱：不是传统意义上启动 3-5 分钟的 EC2 Windows 实例，而是**一个真正的沙箱**——利用即时快照和 fork，2 秒内即可使用。这是他们花了大量精力投资的方向。

他还透露即将提供 macOS 沙箱，但 macOS 有严重的授权和技术限制：每台机器只能同时运行 2 个虚拟机、每 24 小时只能换一次用户、不能跨机实时迁移快照。这使得大规模代理化 macOS 几乎不可能。他调侃：“除非有人像 Linux 那样 clean-room 重写一个兼容层，否则 Apple 会亲手毁了云代理规模的潜力。”

**个人感受**

主持人分享了自己尝试用 Daytona 自动做报表的经历——用虚拟机、登录各网站、导出数据——并感叹“即使是用着最现代工具的创业公司，依然需要计算机使用。那高盛这样的传统巨头，根本不可能用纯 API 来解决。” Ivan 随声附和，语气里既是共鸣又是验证自己策略正确的满足。

**延伸思考**

Ivan 还提到一个更大的愿景：“未来每个代理都会需要自己的计算机，就像每个人类需要电脑一样”。他引用半分析师 Dylan Patel 的观点：GPU 缺货之后，CPU 将变成下一个瓶颈。因为代理需要大量 CPU 来运行浏览器、处理脚本、执行传统业务逻辑。这是对“算力”概念的一次快速迭代：不只是训练，还有无所不在的推理与操作。

---

### 8. 创始人的坚持：一切都会痛 `[文化部分]`

**核心观点**

Ivan 信奉“一切都会痛”——健身、减肥、创业——如果你不能享受痛苦，那就不适合做创始人。他有家庭、两个小孩，却经常跨洲飞行；他要求团队 007 级别的响应速度，但自己也做表率。

**深度阐述**

主持人专门提及 Ivan 早年一条回怼批评者的推文：“我连你名字都没听过，圣诞假期别发了，去放松吧。” Ivan 回复：“这就是为什么我做不到。” 这种拼命风格不是刻意为之，而是天赋。他承认：“我让人 007 加班。我妻子说 996 已经是懒惰了。”

然而，他强调这种文化不是硬性推行的，而是来自老团队的自发共振。25 人中有 13 人合作超过 7 年——他们都是经过“没钱、必须拼命活下去”阶段的战友。新人也经过严格滤网，不是所有人都能适应。

在家庭与事业平衡上，他诚实地说：“很痛，非常痛。我会想念女儿。但这就是签下的契约。我的妻子承担了很大一部分，我感恩。你必须有支持你的伴侣，并且自己也要甘心接受错过。”

**关键数据**

- 每周工作 7 天，无时差响应
- 客户可以在 Slack 上直接要求“5 分钟开一个 huddle”
- Ivan 本人每月在克罗地亚和旧金山之间飞行
**个人感受**

这番话毫不回避代价。没有“平衡”的漂亮话，只有实事求是的取舍。主持人也分享自己对“996”的调侃，两人在这个话题上有趣的共鸣。Ivan 说：“我妻子抱怨我不在家，但我告诉她：我正在 build 一个东西，它会变大，到时候你也会觉得值得。” 这种极端的自驱和对痛苦的接受，也许是持续创造增长所需的心态。

**延伸思考**

每个成功的基础设施公司背后都有类似的“疯狂阶段”——AWS 初期团队全体睡在服务器机房，Cloudflare 早期工程师 24 小时抢修路由。Daytona 现在正处在“让大型客户相信一个 25 人公司能支撑千万级沙箱”的信任建立期，需要的正是这种不计代价的付出。

---

### 9. 对热点现象的冷思考：重新核算 AI 公司收入 `[创始洞见部分]`

**核心观点**

Ivan 对市场当下的“AI 公司通过转卖 token 获得重新加速收入”现象持怀疑态度。他认为，真正健康的增长应来自 API 消费带来的实际增值，而不是简单的 token 套利。

**深度阐述**

他说：“我现在看到不少‘AI 公司’把自己包装成 SaaS，但其实是卖 API Key，利润率糟透了。市场却给它们高估值，好像这是经典的 SaaS 重新加速。这是错误的。” 他的逻辑是：经典 SaaS 毛利高、粘性强；转卖 token 不仅毛利低，而且客户随时可以更换底层模型。

他继续引用“Allbirds 转型做 GPU”和“厨房用品公司推出 AI 滤镜”等例子，说明很多传统公司只是借 AI 蹭一波涨势，并非真正的 AI 基础设施。

他真正的建议是：“把你的产品暴露为 API，按消费收费。让代理直接调用你的核心数据，而不是提供又一个封闭的 agent 界面。Salesforce 的 CEO 刚刚宣布所有产品都暴露 API——这就是未来。”

**个人感受**

这一段中 Ivan 的语气略带戏谑和担忧。他明确表示“我不想贬低任何人，每个创始人都有权争取融资，但我认为我们将迎来一次冷水澡：很多 AI 公司的收入可能很快回落。”

**延伸思考**

这个观点也呼应了“代理优先”的基础设施思维：不要做代理的应用，而做代理的（数据/计算）基础设施。Daytona 自己的定位就是如此——卖的是“计算机”而非“应用”。

---

### 10. 未来蓝图：成为“Agent 云”的基础设施 `[结尾展望部分]`

**核心观点**

Ivan 认为会有专门的“Agent Cloud”出现，包含沙箱、网络搜索、数据库等专门为代理优化的服务。Daytona 希望成为这一新云的计算层，但不是数据中心提供商，而是运行在现有设施之上的优化调度层。

**深度阐述**

他提到自己公司在研究的下一个代理原始抽象：“不仅仅是一个计算机沙箱，还有更多。现在市场只看到了冰山一角。”

关于 GPU 沙箱：已经开始提供，但主要是用于训练中的 3D 渲染或需 GPU 的强化学习场景。他强调，Daytona 仍专注在 CPU 与 GPU 的组合上，不会直接去建数据中心——目前投资数据中心的经济不适合他们，利润率太低，需要巨大资本。

对比类似公司：他认可 Railway 的做法，并提到 Cloudflare 的“Agent 组合”理念非常棒（尽管技术上限受限于 Workers 模型）。他认为未来会有一家或几家“Agent Cloud”，类似于 Stripe 之于支付。Daytona 完全有机会成为其中之一。

**关键数据**

- 当前 74% 月环比增长
- 整个基础设施市场（为代理提供计算）也在以每月 40% 左右速度增长
- Ivan 预言“很可能明年拥有 CPU 本身将成为一种 go-to-market 策略”——当弹性带宽不够时，提前储备 CPU 成为一种竞争壁垒
**个人感受**

在最后的问答中，Ivan 对“你预计到年底会变成什么样”这个问题有些犹豫：“我不敢说出来，因为它太疯狂了。” 但他肯定表示，如果市场其他条件不变，现有的增速足以让他们几个季度内跃升到一个惊人的规模。他最后笑着对主持人说：“谢谢这次访谈，我知道你会把所有好东西都写在 show notes 里。”

**延伸思考**

Daytona 的快速增长恰好反映了 AI 代理从概念走向生产级应用的关键瓶颈：**需要为代理配备足够聪明且快速的计算机**。当所有人都把注意力放在模型能力上时，基础设施层面的限制正在显现。如果未来的“Agent Cloud”真的成形，那么 Ivan 的“电脑即服务”可能是比 GPU-as-a-Service 更底层的护城河。

## 精华收获

1. **代理的基础设施根本不同**：代理需要的不是临时代码运行器，而是持久化的、可交互的、能模拟人类操作的“完整计算机”。速度与状态的结合是关键。
1. **市场比你想象的大**：每个未来存在的代理都可能拥有一个或多个“计算机”。这个市场不仅是开发环境，而是整个 PC 市场（千亿级）加上云市场。
1. **工作负载完全新形态**：长期背景代理（温和波动）与 RL/Eval 代理（极端尖峰）同时存在，容量规划技术需要彻底重写。
1. **开源是杠杆，但不是全部**：开源降低初始信任门槛，但真正驱动增长的是产品体验和极度响应式支持。25 人团队的高速文化不可复制，但值得学习。
1. **计算机使用将是 killer app**：遗留软件不会消失，代理通过 GUI 操作电脑将成为 RPA 的终极形态。Windows 和（若授权改善的）macOS 沙箱将直接开启 10 万亿市场。
1. **警惕 token 套利的假象**：AI 公司需要提供真正差异化的 API 消费价值，而不是买 token 再卖 token。基础设施玩家更需要思考自己的不可替代性。
---

## 延伸思考

- **人才与信任代际传递**：Ivan 的老团队经验再次证明，在极端不确定的早期硬件+软件初创公司，“老战士”组合是存活和快速 pivot 的关键。
- **AI 基建不是只有 GPU**：CPU 将成为新瓶颈，而且它对应用层的影响更为直接。如果代理的数量继续以月 74% 增长，那么未来英特尔和 AMD 的产能可能会像现在 NVIDIA 一样被预订数年。
- **从“工具”到“环境”**：代理开发者不再关注单个 API 调用，而是关注一个完整、可编程的操作环境。Daytona 定义的“可组合计算机”可能正是下一波基础设施的雏形。
---

## 精华收获

- 为代理提供“真正的工作计算机”而不是“临时代码容器”是 Daytone 高速增长的核心原因。
- 工作负载格局正在改变：长期运行代理、RL 暴增、对 CPU 可穿戴性的需求前所未有。
- 开放式文化（开源 + Slack 直接响应）可以取代传统企业销售流程，但需要创始人亲自下场。
- 对市场的冷思考：不要盲信“卖 token 的重新加速”，真正的价值在于可消费的 API 基础设施。
