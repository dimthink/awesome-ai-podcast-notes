# 从“五美元预算”到“内存基准”：SuperMemory 如何重新定义 AI 的长期记忆

## 嘉宾与背景

**主持人：** 好的，今天我们在一间远程演播室，与 Super Memory 的 Dhravya Shah 在一起。欢迎来到 Latent Space。

**Dhravya：** 谢谢你的邀请。

**主持人：** 你在技术圈的时间线上已经活跃了好几年了。我刚刚发现你是在 2023 年推出的 Super Memory。感觉时间比那更短，但你也确实做了一段时间了。

**Dhravya：** 是的，我在这行已经待了太久了。Super Memory 经历过很多种不同的产品形态，但现在它把所有东西都整合到了一起。我会再详细讲这个。总之，经历了很多。

## SuperMemory 的演变：从消费级应用到上下文基础设施

**主持人：** 好的。那么，让我们回顾一下你在 Product Hunt 上的首次发布。你最初的产品是什么样子的？

**Dhravya：** 让我们从那里说起。Super Memory 最初是一个**消费级应用**，旨在成为你的一个更好的书签和笔记工具，并附带一些其他功能。例如，你可以跨大语言模型（LLM）保存你的记忆。那是很久以前的事了。所以我们有一次 MCP（模型上下文协议）的发布，然后又有一次针对应用本身的发布。早期，它只是一个应用，你可以把东西保存到应用里，也可以从应用里获取东西，就这么简单。

后来我们想，MCP 现在是个趋势，所以我们应该基于它做一个 MCP。那时我个人还在实习，在 Cloudflare 工作。这其实是作为我参与的一个名为 **Bispace** 的项目的一部分而做的副业。你可能听说过 Bispace，它是一个很多开发者利用夜晚和周末时间进行构建的地方。他们真的会推动你去做出东西，并与用户交流等等。

**主持人：** 那就像一个 Z 世代的加速器。

**Dhravya：** 没错。所以我参加了 Bispace 的第三季和第五季，并在那期间将 Super Memory 开源了。这个项目彻底火了，成为了 2024 年增长最快的项目之一。我当时一直在发推文，说我在 Cloudflare 上运行它，每月预算只有 5 美元。它那时仍然是一个消费级应用，但我一直在添加功能。

早期，业界对“记忆”的理解方式，包括我们自己，就是简单的**检索增强生成（RAG）**：你只需嵌入内容，放入向量数据库，就完事了。但后来我深入钻研了这个难题。当时（即使是现在，除了像 Turppuffer 和 Chroma 等少数例外）的向量数据库，对于我所达到的规模来说，要么变得太慢，要么运行成本太高，无法真正扩展。因此，我不得不寻找其他有创意的方法，让它对我来说运行得非常便宜。

我一直在推特上分享这些，有一条推文说：“我在 5 美元的预算下运行 Super Memory，却拥有 10 万用户，这是我的架构图。” 那是一张完整的架构图，那条推文也火了。那是 Super Memory 第一次出名的事件。但这并没有让人们对该应用本身产生兴趣，反而让人们对**我正在构建的基础设施**更感兴趣。

**主持人：** 我记得你开源了它，然后我当时就想，“哦，是的，我得标星和复刻这个项目，因为如果我以后需要这个，我就需要有这个设置。”

**Dhravya：** 完全正确。所以在开源之后，它在几周内就涨到了 1 万颗星。但后来我意识到，很多公司开始联系我，他们说：“嘿，你能帮我们搭建我们的记忆系统吗？” 我也去其他记忆公司工作过。这变成了一个我做了大量关于**上下文工程**咨询工作的事情。我意识到，所有这些公司都有非常相似的用例，只是侧重点略有不同。他们不仅需要检索，还需要真正的用户理解。我们稍后会谈到这一点，比如什么是现代记忆，AI 应该如何记忆。

然后我意识到，要让一个大语言模型真正擅长理解用户，你必须处理四件事：

1. **知识更新**：你必须使过时的知识失效，并在其基础上构建新的知识。这与存储向量不同。
1. **时间推理**：你必须让代理（agent）理解时间的概念，以及事物是如何随时间流逝而变化的，以便进行个性化。
1. **遗忘机制**：需要有能力忘记那些完全不再相关的事情。
1. **用户画像**：记忆不仅仅是一次检索调用。大语言模型需要拥有一个非常小的用户画像，在每一次交互中都能利用它。
所以我一直在将这些组件构建到 Super Memory 和所有这些应用中。然后我意识到，这里有一些非常困难的基础设施挑战需要解决。这既涉及到**提取层**——如何真正训练一个小模型来提取关于用户的信息，也涉及到第二层——**知识图谱**。

我们构建知识图谱的方式与通常的方式不同。人们通常有**三元组**的概念。

**主持人：** 三元组。

**Dhravya：** 他们有“对象-谓词-对象”。

**主持人：** 确切地说，“实体-关系-实体”。

**Dhravya：** 没错。我们认为三元组实际上会导致性能下降，因为你必须大量遍历它们才能获取到任何信息。

**主持人：** 这是每个做图谱的人都会学到的教训。

**Dhravya：** 完全正确。例如，要找出“Dhravya 喜欢什么食物”，你必须先到达实体“Dhravya”，然后遍历一层或两层才能发现“Dhravya 喜欢食物，享受这个东西”等等。这会导致极其缓慢，而且完全没有必要。所以我们构建了我们自己的知识图谱，构建了我们自己的提取管道。我意识到，这个消费级应用可能不再有用了，这个核心基础设施现在更有价值，这就是我们的核心业务，也就是我们现在提供的服务。

**主持人：** 令人惊叹的更新。那么，**今天的 Super Memory 是什么**？是一个云业务吗？是一家风险投资支持的公司吗？你是一位年轻的创始人，请向大家多介绍一下公司本身及其现状。

**Dhravya：** 好的，我应该先做个自我介绍。我是 Dhravya，目前 20 岁。在此之前，我曾在包括 Cloudflare、Hightouch 等几家公司从事代理（agents）和数据库方面的工作。我有过两家公司被收购的经历，然后来到了美国。我再次打造了 Super Memory，去年筹集了**300 万美元**。从那时起，我们成为了一家主要开源的、由风投支持的云业务公司。我们正在努力为 AI 时代构建**上下文基础设施**。

我所说的上下文基础设施是指代理（agent）可能需要的任何外部上下文，包括单纯的检索、用户理解、记忆，以及自我改进、自我学习、个性化等。我们通过这一款单一工具——Super Memory——提供所有这些功能。因为我们组织数据的方式，我们可以自动提供所有这些方面，因为每个记忆用例都需要不同的实现方式。我们为开发者提供服务，提供一个 API 供人们使用。

但我们仍然有一个面向消费者的应用，其理念是让你拥有自己的记忆，可以在你使用的所有东西中利用它，或者你可以将 Super Memory 添加到你的代理（agent）中，使其在记忆事物方面做得更好。例如，OpenClaw 或 Cursor 的插件当时很火，它们就是我们提供的面向消费者的插件。

## 聚焦 OpenClaw：为什么它的内存很糟糕，以及修复方法很简单

**主持人：** 确实，看着很令人印象深刻。我想很多人会把 OpenClaw 放在我们今天讨论的标题里。我们就直接进入 OpenClaw 的话题吧。关于 Super Memory 还有更多可以深入探讨的，但当下的话题是，有人遇到了 OpenClaw 内存的问题。

**Dhravya：** 每个人都在抱怨。

**主持人：** 然后他们会说，“好吧，那你是怎么做的？”接着就会出现各种不同的答案。我就想，哇，这问题真的还没解决。

**Dhravya：** 是的，没错。

**主持人：** 那么，请告诉大家到底发生了什么。

**Dhravya：** 好的。我们先谈谈 **OpenClaw 目前处理内存的方式**。我为此准备了一个小小的 Cursor 制品。OpenClaw 与它的 .md 文件或其他任何内存插件配合使用时，它本质上依赖于**工具**来搜索它准备的这些 .md 文件。比如，“关于 API 我决定了什么？” 代理（agent）会决定是否要搜索（有时它不会决定搜索，这可能是最大的问题），然后它会得到一些结果，进行一些评分，最后返回结果。而这种搜索目前是针对 .md 文件或内存文件夹进行的。因此，如果代理（agent）决定不明确记住某些事情，那么这些信息就永远不会出现在结果中，因为在注入时你不知道该记住什么。

这里有一系列问题：

- 因为这些文件是静态的，它们无法处理**更新**。
- 它们无法处理我之前提到的**遗忘机制**。
- 没有**时间推理**能力。
- 你无法查看文件的特定部分，因为你不知道什么信息在什么地方，还有其他类似的问题。
这就是为什么每个人都在抱怨 OpenClaw 的内存。令我惊讶的是，当人们抱怨 OpenClaw 的内存，甚至当 OpenClaw 增加内存功能时，人们并没有深入研究它是如何工作的以及为什么它很糟糕。所以我们做了深入的挖掘，然后思考什么是做内存的正确方式。我们已经为 Cursor、OpenClaw 等开发了插件，所以我们对于代理（agent）应该如何拥有状态（stateful）有很多经验。

我们基本上将基于**工具**的方法转变为基于**钩子（hooks）**的方法。这种方法实际上是从 Super Memory 图谱中读取数据，该图谱保持了内容的**新鲜度**，能够处理更新等。现在，对于每次工具调用或每条用户消息，一个钩子会自动注入少于 **2000 token** 的动态信息。所以整个上下文中的 Super Memory token 永远不会超过 2000 个。可选地，如果代理（agent）需要更多信息，它也可以进行一次工具调用。我们内置了**矛盾解决**、**时间推理**等功能。因此，无论何时，信息都保证是真正新鲜的。这就是要点，但我们还可以深入探讨，因为这张图里没有包含我们做的一些其他事情。嗯，这就是为什么我们的内存更好，简单来说。

**主持人：** QMD 是来自 Tobias Lütke 的，非常令人印象深刻。他想在本地做所有事情。而你显然是 Cloudflare 的拥护者，你打算在 Cloudflare 上做吗？所以这基本上是两种不同的思路。我认为基于钩子（hook）的方法可能是正确的，因为模型肯定会犯错。当我在 XAI 里使用 Grok，它不去搜索推文时，我会非常沮丧。我使用 Grok 的唯一原因就是它能搜索推文。

**Dhravya：** 是的，你必须明确要求它使用这些工具。而大多数 OpenClaw 用户甚至不知道工具的存在，所以他们不想去考虑工具的问题。

**主持人：** 你认为这是因为他们不专业吗？很多人在工程意义上很专业，但在代理（agents）领域并不专业。我们现在可能身处一个巨大的泡沫中，他们不会知道幕后到底发生了什么，甚至根本不会去看。

**Dhravya：** 是的。

## 文件系统记忆 vs. 动态记忆

**主持人：** 除了这个，还有一个关于内存的讨论，我觉得 Cursor 和 Claude 那些人一直在说，就是那种基于文件系统的内存。你只是把一堆东西转储到你的文件系统，然后封装一下。OpenClaw 有一些压缩、日志、心跳之类的功能。你对此有什么看法？

**Dhravya：** 我认为在许多情况下，**文件系统内存实际上是正确的方法**，对于大多数 Cursor 用户来说尤其如此。但你仍然面临同样的问题：你必须**知道要记住什么**，才能让它记住。你必须明确地告诉它“请记住这个”或“请提起那个”。而且它受到文件本身的限制。文件可能会变得越来越长，这是个问题。所以你必须在多个文件之间进行拆分，而这些文件仍然没有任何更新逻辑，除非你明确让代理（agent）去做。遍历这些文件也真的很慢，所以你需要进行**代理式发现（agentic discovery）**来找到回答任何问题的上下文。

例如，如果我要它“实现这个工作流程”，它必须首先查看它在内存中知道的所有东西（所有工作流程等），然后才能得出答案。我认为这是一个很大胆的观点：一方面是**个性化**，另一方面是**代码索引**。Cursor 确实做了代码索引，但 Cursor 没有做。我相信这也是因为他们没有动力去为用户节省 token 成本。他们只是将索引等成本，转移到了用户运行代理（agent）所花费的 token 和时间上，而这正是他们想要的。所以理论上，你可以添加代码索引和个性化，效果会好得多，速度也会快得多。

我这么说不是空口无凭。我们昨天刚跑了一个基准测试。我发表了一篇非常技术性的报告，也引起了一些关注。我们针对三个对象运行了一个长基准测试。结果截图在一条关于 Cursor 的推文里。嗯，**Cursor 的方法表现最差，OpenClaw 稍好一点，而 Super Memory 得分最高**。你可以看到，这是一个相当显著的差异，几乎达到了 **50%**。

**主持人：** 让我澄清一下，这是你对 Cursor 方法的复现，对 OpenClaw 方法的复现，还是你实际使用了 Cursor？

**Dhravya：** 不，这是我们的复现。所以这基本上是我们自己实现的。

**主持人：** 这很重要，我得说。

**Dhravya：** 是的，这确实很重要。而且，我们也认为基准测试不应该被完全信任，也确实不能完全信任。所以我们构建了这个开源的基准测试评估平台，叫做 **Memory Bench**。这是我们实现的，完全模拟了它们的确切方法。事实上，我们深入研究了 OpenClaw 的混合方法，使用了相同的公式，完全相同。你可以查看 PR 来了解具体做了什么。简而言之，Memory Bench 是一种方法，可以在相同的 BL 规则下，针对任何基准和任何评判者，对任何提供商进行基准测试。这让你能够量化所有事情：不仅包括内存系统的质量，还包括延迟、成本以及其他启发式指标，如 top-K 召回率、NDCG 等。所以这是一个非常好的内存基准测试方法，而且使用起来非常简单。我实际上邀请每一个竞争对手来使用它，因为这实际上也让我们的竞争对手更容易与 Super Memory 进行比较。

## 内存基准测试的现状与挑战

**主持人：** 是的，我认为这是个非常好的基础设施。既然我们在讨论基准测试，你对 LongMemEval 有什么看法？

**Dhravya：** 是的，我认为我们在内存基准测试方面还没有达到理想状态，背后有很多原因。我可能也会写一篇详细的文章来谈这个。我认为 **LongMemEval 是一个非常好的基准测试**，因为它测试了正确的东西。数据集很棒，各方面都很好。除了一个事实：你可以通过记住尽可能多的东西来在 LongMemEval 上获胜。**得分与你提取的记忆数量有直接的相关性**。这看起来似乎也不坏，但在现实世界中，你会花费更多成本。你不想提取所有东西，因为提取本身不是免费的。为了在基准测试中获胜而这样做是可以的，但在现实世界中，人们并不期望记忆以这种方式工作。LongMemEval 也没有测试遗忘机制。你怎么能测试某件事是否应该被遗忘呢？它测试了更新，但没有测试遗忘和其他一些东西。尽管如此，我们仍然使用 LongMemEval，因为这是我们目前最好的选择。

还有 **LoCoMo**。LoCoMo 真的很差劲，而且不只是我觉得，几乎所有人都知道 LoCoMo 不是个合适的东西。原因是 LoCoMo 本质上是在测试检索容量，而不是在测试真正的记忆功能。你基本上可以把所有上下文都 dump 进 LoCoMo，然后在最新的模型上获得 100% 的分数。它的数据集只有 30 个问题，非常小。其次，它没有测试知识升级、时间推理、多会话、所有其他重要的事情，比如偏好。它除了检索之外什么都不测试。而且在 Gemini Flash 之类的模型里，问题本身和会话都足够短，你可以把整个会话都放进上下文窗口，它就能直接工作。LoCoMo 现在也已经很旧了，三四年前它可能还有意义，但现在模型已经好多了。

还有 **ConvoMe**，我其实挺喜欢的。它来自 Salesforce，测试了更多**助理型用例**，比如助理应该同时做 RAG 和记忆等。这相当不错，有时两者都需要，这也是我稍后想谈到的。但这就是我对基准测试领域的看法。

还有一个重要的事情是，目前没有关于**用户画像**的基准测试概念。所有这些基准测试都非常侧重于**检索**，侧重于事实性信息。你本质上是想获取某些东西，然后回答问题。但这并不是所有的用例。我很快展示一张幻灯片，这会让事情变得非常清楚。幸运的是我准备好了。传统上，检索是这样发生的：你得到一个查询，检索一些东西，然后基于这些生成答案。但我们认为，这对于大多数**非字面意义的问题**是行不通的。比如“为我找到最好的显示器”。如果我从未谈论过显示器（很可能没有），它什么也检索不到。它会给出一些通用的答案。但代理（agent）可能需要在所有时候都知道很多关于我的默认信息，比如“用户是 CEO”。同时，它也需要知道一些** episodic 信息**，比如“用户最近融资了”，“上周搬进了新办公室”。在这种情况下，它可能会推断：“哦，你整天写代码，而且刚搬进办公室，所以这款显示器可能是适合你的，虽然有点贵，但适合你。” 这只有在你真正拥有一个非常小的（比如 500 token）关于用户的信息才能做到，这个信息同时捕捉了静态信息和一些最近发生的事件。我们在 Super Memory 中也内置了这一点，但目前没有办法对其进行基准测试。

**主持人：** 是的，这基本上就是**个性化**，对吧？这有点难。

## OpenClaw 内存的其他问题与混合模式

**主持人：** 我想就 OpenClaw 相关的话题做个收尾。人们在使用很多东西时还会遇到什么问题？比如**密钥泄露**、如果做了相似的事情导致的**上下文混淆**，你知道，在检索和准确性或召回率和准确性之间总是有权衡的。

**Dhravya：** 是的。我认为 OpenClaw 目前的做法基本上是发回对话中最后 15 条消息，并利用这种来回的上下文。这种方法本身并不理想，因为你没有利用任何缓存 token，所以实际上你支付的费用是正确方法的 **10 倍**。这是一个 OpenClaw 本身的问题，我们对此无能为力。

另一个问题是，人们让它访问他们的密钥等敏感信息。作为一个内存系统，我们必须确保在任何情况下**没有任何秘密信息最终流向我们**。很多 OpenClaw 用户也连接了浏览器之类的东西，还有心跳机制。心跳每小时都在发生，这些心跳信息也最终进入了内存，造成了一些**信息污染**。有些人想将他们 OpenClaw 的记忆与 Cursor 的记忆一起使用。我们其实不希望这样，它们应该是隔离的，是用户自己的东西。但用户想要这样，所以我们提供了配置来支持。

否则，Super Memory 有时可能不会给出正确的结果。这就是为什么我们最近为 OpenClaw 推出了一项新功能，这对我们来说是一个巨大的学习经验。它叫做**混合模式**。在内存系统中，你总是需要选择到底要记住什么，这是很重要的。你想保持信息更新、新鲜，你想遗忘一些东西。但有时你并不知道用户会问什么。有时用户会问一些没有被记住、当时看起来不那么重要的事情。那么如何处理这些情况呢？我们的方法是，因为我们同时做记忆和**托管式 RAG**，如果记忆存在，我们可以提供记忆。但如果用户没有记住某些东西，我们有一个后备方案，即返回**原始文本块**。如果 Super Memory 没有记住它，它会返回原始的块。然后在后台，它会把这个信息也添加到记忆中。这样，我们基本上做到了，正如我们所说，只要用户在问，Super Memory 就应该**几乎永远不会忘记任何东西**，因为我们总是给代理（agent）提供正确的上下文。

**主持人：** 太棒了。刚才我们是在说混合模式吗？

**Dhravya：** 是的，这就是混合模式。本质上，我们会首先返回记忆。如果有任何匹配的原始文本块，我们也会返回那些，以确保代理（agent）拥有足够的信息来回答问题。目前没有其他提供商这样做。

## 成本的重要性

**主持人：** **成本**有多重要？我的感觉是人们对成本很不敏感。但显然，任何真正的工程师都会关心缓存命中之类的问题。我不知道，人们到底在不在乎？

**Dhravya：** 我认为人们实际上**非常关心成本**。我们又身处一个巨大的泡沫中。我们的很多用户，他们确实关心成本，因为他们会将内存系统与他们使用的大语言模型进行比较。如果一个内存系统是每百万 token 10 美元，而他们用的大语言模型是每百万 token 2 美元，他们就会觉得这不合理，太贵了，不应该用这个。这确实发生在我们身上。我们希望确保用户能满意地使用记忆功能，而不是为此担忧。对我们来说，我们已经优化了我们的基础设施，使我们每百万 token 的成本只有 **2 美分**，因为我们有自己的模型、自己的数据库、自己的一切。所以我们也能向用户收取非常低的费用，我们确实这样做了。
