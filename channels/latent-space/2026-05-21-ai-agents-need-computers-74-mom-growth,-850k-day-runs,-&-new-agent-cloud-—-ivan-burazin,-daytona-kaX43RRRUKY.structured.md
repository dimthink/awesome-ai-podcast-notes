# AI代理需要计算机：月增长74%，日运行85万次——Daytona CEO Ivan Burazin访谈

在进入今天的正式内容之前，主持人首先对听众表达了诚挚的感谢。他表示，正是因为听众的选择和支持，他们才能持续带来高质量的AI工程、科学和娱乐内容。虽然几乎每天都有赞助商找上门，但多亏了足够多的订阅用户，节目才能保持无广告的可持续发展。主持人唯一请求大家做的一件事就是点击订阅按钮，这是他对听众提出的唯一请求，但对他和整个团队来说意义非凡。他承诺只要大家订阅，他们就会不断努力让节目变得更好。

接下来，访谈正式开始。

## 从Code Anywhere到Daytona：一段创业旅程

**主持人：** Ivan，我们认识很久了。你当初是怎么找到我的？

**Ivan Burazin：** 是我主动联系你的。我和联合创始人一直在思考一个问题——我们是 **Code Anywhere**（首个基于浏览器的IDE）的联合创始人，很早就在思考“本地主机（localhost）应该消亡”这个理念。我看到你写了一篇关于“本机（Localhost）”的文章，所以就联系了你。那时我刚换了一份工作，负责开发者体验（DX）方面的事情，而你在那个领域很有见地，所以我向你（以及其他几位专家）请教了关于如何推动这件事的关键点。你很友好地接了那通电话。我记得当时还迟到了，因为我和家人在意大利度假。我觉得很过意不去，但你非常友善。

**主持人：** 我友善是因为我也经常迟到别人，所以……谁没有错呢？另外，你创办的 **Shift** 会议是我创立 **Infobip** 的灵感来源之一，谢谢你给了我那种“你也可以创办并销售会议”的推动力。

**Ivan Burazin：** 我记得你一开始还要求我给创始人股份，但我当时太专注了所以拒绝了，现在想想真该拿那些股份（笑）。不过，我们也不是风投那种模式。

**主持人：** 我觉得你令人印象深刻的地方在于，**Code Anywhere** 是你一直在努力的东西，中间你暂停了一段时间，创办了 Infobip，然后又回来做了 **Daytona**。请给我们讲讲整个故事。

**Ivan Burazin：** 好吧。很早以前，我和联合创始人就一起共事。我们有点像“结了婚又离了婚，又再婚”。有人甚至直接问我的联合创始人是不是我的伴侣（笑）。我们确实一起创办过多家公司。从 Code Anywhere 到 Shift 会议，再回到 Daytona。最初我们做的是堆叠服务器、虚拟化、路由器等底层技术，那是一家服务型公司，后来卖掉了，开始专注于我联合创始人发明的首个基于浏览器的IDE。在 **Heroku** 之后（他们做了一小段时间），我们是最早的之一。我们之后才有 **Cloud9**、**Replit**（我们停掉后他们成功了）、**Nitrous.IO** 等。但当时太早了。那时 **VS Code** 还没出现，**Kubernetes** 也没有，**Docker** 刚刚起步。所以我们不得不用整个技术栈自己构建一切，这也成为了我们今天在 Daytona 中使用的关键经验。当时大约有 300 万用户使用 Code Anywhere，主要靠天使投资。我们没有达到那种规模，但最终偿还了所有投资者。大约三年前，我们开始了类似的东西——**Daytona**，但最初是面向人类工程师的自动化开发环境，也就是 Code Anywhere 的底层技术栈。然后去年一月，我们进行了一次**硬转型**（hard pivot），转向了现在的沙箱（sandbox）产品。

**主持人：** 那确实是一次历史性的转型。我本人投资了 Code Anywhere，也投资了 E2B，然后你们两家都转向了同一个方向。

**Ivan Burazin：** 你投资了 Daytona 吗？你是我们的第一个投资人。如果没有你的支票，我们可能不会做成。我们必须先让你登船，你是那个推动者。

**主持人：** 那是因为你把我放到了你的演示文稿里（笑）。我写过那篇“本地主机终结”的文章，所以所有做这个领域的公司都联系了我，然后那些收到演示的风投也打电话来和我讨论。

**Ivan Burazin：** 现在这一切终于发生了，只不过可能是面向非人类用户。

## 产品核心：面向AI代理的可组合计算机

**主持人：** 那么今天的 Daytona 到底是什么？请给一个简洁的描述。

**Ivan Burazin：** 我穿着印有“它运行AI代码”（It runs AI code）的T恤。我们的品牌很一致也很简单。但可能很快要改，因为这只是我们做的一部分。我们发过五六千件这样的T恤，人们穿着很自豪，因为它不是在宣传我们，而是在宣传用户自己。不过这也只是子集，因为人们提到“运行AI代码”时，通常只会想到小的代码执行隔离环境，发送代码得到输出。而 Daytona 实际上是**面向AI代理的可组合计算机**（composable computers for AI agents），市场称之为**沙箱**（sandboxes），但这可能有些误导。

**主持人：** 是的，因为沙箱通常被认为是演示或测试环境，而不是生产环境。

**Ivan Burazin：** 没错。想象你面前的笔记本电脑、那边的图形工作站（我妻子是建筑师，用带高端显卡的Windows电脑做3D渲染）——计算机有不同的组成方式。我们坚信，现在的以及未来的AI代理将需要所有这些不同组成的计算机来执行不同类型的任务。我们通过API提供这些能力。

## 市场验证：74%月增长与850K日运行

**主持人：** 市场正在爆发。你们报告了 **74%的月环比增长**，而且已经持续了一段时间。不只是你们，所有的计算提供商都在增长。我不想用“计算提供商”这个词，你不一定同意，但确实是这样。你们的增长是自然的PLG驱动，同时企业级也在加速。

**Ivan Burazin：** 去年一月我们转型时，洞察来源于2024年底。那时我们用 **Devon**（你给了我访问权限，我应该不可以说出来）和 **Open Devon**（现在的 OpenHands）做了个演示。我们想：“这好像是个新东西，不如用我们为人类开发环境做的自动化产品，加上 Open Devon，作为一个SaaS推出。”结果并没有很多人注册使用，但很多正在构建AI代理的人联系我们说：“嘿，我的代理需要计算沙箱、运行时，随便你叫什么。” 我们发现这是个新市场，我们有基础设施和产品，可以切入。但我们很快发现，人们不喜欢我们一开始建的东西，它不工作。很多人说：“这和EC2、VM有什么不同？”但我们给了20-30个人测试，他们都说不满足需求，会崩溃。我并不是AI专家，我是基础设施背景，所以我决定去学习所有相关的播客（包括你们）、博客、参加会议，了解整个市场。

**主持人：** 你想点名感谢哪些有用的资源吗？

**Ivan Burazin：** 总体而言，我看了很多不同侧重的播客：你们、No Priors、Bill Gurley的（当时还在）、20VC（动态很不同）等等。但我关注的不仅仅是计算市场，而是整个AI代理基础设施市场。通过和客户交流、参加会议、阅读白皮书，我终于在**新年前夜**半“氛围编码”（vibe coded）出了第一个最小可行产品。我凌晨三点才睡，把代码发给了我的CTO。他早上看到说：“这绝对是垃圾，不要给任何人看。”但想法是好的。他花了两个星期，做出了一个更简陋但能用的原型。然后我们联系了所有之前说“你们这东西不行”的人，给他们演示。每个15分钟的电话都延长到了25-30分钟，所有人都说“我们需要这个，我们想要访问”。但当时没有登录，只是一个API密钥的测试版。第二天如果我们没发给他们，每个人都会回头追问：“我的API密钥呢？”所有人都想要。我**从未经历过**这种事——如果你不给他们访问权限，他们会直接打电话给你。他们**现在**就要。这说明他们需要的东西不存在，或者他们没有找到，而他们非常想要我们建造的东西。我们意识到自己搞对了。再想想市场有多大：人类工程师的市场已经很大了（比如GitLab等），但未来将存在的每一个AI代理的市场呢？那会有多大？我们决定全职投入这个方向。

**主持人：** 当时还不是可组合的。只是一个你可以定义CPU、内存、磁盘空间的Linux盒子。不能有多种操作系统、不能动态调整、不能加GPU等。

**Ivan Burazin：** 对，但那只是第一个版本。我们从一开始就使用**裸金属**（bare metal）。

## 技术解密：裸金属与自研调度器带来的速度优势

**主持人：** 给不了解背景的听众解释一下，普遍的路径是什么？

**Ivan Burazin：** 大多数提供商运行在虚拟机（VM）之上，或者使用 **Firecracker**（微虚拟机）。他们通常的状态是：机器的硬盘不是沙箱的一部分，而且沙箱通常有生命周期，是抢占式的。我们的想法是：AI代理会像人类一样——你不希望你的笔记本电脑在你工作完成之前被关闭，你希望合上盖子再打开，状态保持不变。代理也需要**暂停和恢复**，需要**有状态**且可长期运行。同时代理也需要**速度**。所以我们需要的是一种结合了Lambda和EC2的东西：又快又有状态且长期运行。当时我们不知道别人是怎么做的，只是从第一性原理出发。我们看了Kubernetes，不够好；看了Nomad，也不行。于是我们用之前在Code Anywhere自研调度器的经验，由CTO重新实现。第三个联合创始人看到代码时说：“这是什么？这是2008年的风格！” CTO说：“没错。” 这就是为什么Daytona超级快。我们在**裸金属**上运行，有自己的调度器，直接使用底层机器的CPU、内存和磁盘，所以IOPS极快，没有网络延迟（像EBS那样）。同时，快照和模板也预加载在裸金属机器的NVMe硬盘上。当你从模板或快照启动沙箱时，会直接定向到该快照所在的裸金属机器，然后启动本地机器，完全没有网络延迟。这就是我们从第一性原理出发为AI代理设计的计算机。

**主持人：** 有个平台叫 **Compute SDK**，你们在上面表现非常好，TTI（Time to Interactive）很低。这对你们来说是相关的基准吗？

**Ivan Burazin：** 我不确定，基准每天都在变。但我们确实是长期以来的第一名，现在出现了不同的沙箱定义、隔离模式等。例如有平台直接在S3上运行，类型很不同。但我们的启动速度极快：**60毫秒**（包括网络延迟）启动一个；如果需要同时启动5万个，我们大约需要**75秒**；而其他一些公开数据可能需要2000秒（30分钟）。我们目前对能同时运行的实例数量没有硬限制，因为我们拥有自己的裸金属。我们最大的客户每天运行约**85万次**，接近一百万。我们还收到了一个创建50万个并发实例的请求，相当于50万个CPU同时运行。

## 工作负载类型：后台代理与RL/评估的迥异模式

**主持人：** 人们通过这些沙箱运行什么样的工作负载？是GB级内存还是存储？哪种资源最贵？

**Ivan Burazin：** 最贵的是CPU，其次是内存，然后是磁盘。我们目前不收取网络费用（但网络成本正越来越大）。我们主要区分两种工作负载：一种是**后台代理**（background agents）或长期运行的代理；另一种是**RL（强化学习）和评估（eval）**。它们的模式完全不同。后台代理（例如 **Cognition**、**Lovable**、**Harvey** 等）的使用模式类似人类：在一天中，中午使用率最高，午夜最低，周末低。地理上，美国占大头，但我们有亚洲、欧洲和美洲用户。有趣的是，按用户城市排名，第一名是**新加坡**，日本东京也排在前列，这在以前的技术周期中从未发生。巴西也很多。这种负载跟随太阳，你可以预测容量。但对于RL和评估，负载极其不稳定。研究员们可能会一次性需要一万、五万、十万个CPU，全速运行然后停止。某人可能在午夜启动一个运行然后明天醒来收结果。这种形状像“方波”一样，无法预测。所以我们必须锁定承诺（commits）来预留容量。目前Daytona的平均利用率只有**15%**，非常低，因为负载很尖峰，但最高能到90%。我们正在思考如何像Cloudflare那样通过在地理上移动负载来优化。

**主持人：** 对于RL/评估类负载，你们从零开始，现在占比已经达到50%。

**Ivan Burazin：** 对。这些负载通常使用我们的声明式镜像构建器（declarative image builder），基于快照进行并行运行。客户告诉我们，他们绝对偏爱这种模型。而且在这个场景中，我们主要与**托管的Kubernetes**（EKS、GKE）竞争。一旦体验过Daytona和EKS的对比，他们再也不想回去。原因包括：易用性（像Twilio或Stripe一样的API和SDK）、速度、动态调整大小防止OOM（out of memory）。另外我们有多种隔离层（默认是经过Sysbox加固的Docker容器，安全等级相当于VM，但仍然是容器）。它们可以在沙箱内运行Docker compose甚至K3S（轻量级Kubernetes），解锁大量其他平台无法支持的工作负载。

## 客户生态与开放源代码

**主持人：** 你们是开源软件（OSS）公司，很多竞争对手不是。很多人担心保持开源会被直接fork。

**Ivan Burazin：** 我们的原始产品是开源的，这对我们帮助很大。开源现在更多是帮助我们上云产品的消耗，而不是直接转化。我们的商业模式是**开源核心**（Open Core）：Apache 2.0协议的核心代码，加上企业版专有代码。转型后我们没时间重新思考，就用了AGPL 3.0，这是一种折中：企业可以使用但必须开源自己的衍生品。纯开源信仰者认为这不完全合规，但除了功能标记后面的Windows支持、GPU等，大部分代码都是开源的。我们的调度器等等都在那里。有竞争对手说：“你们是真正的开源，可以看到一切。”这对社区有吸引力。通过开源，我们能让更多开发者在集成Daytona时查看仓库，获得更多上下文，间接有助于增长。不过开源作为增长引擎的作用在AI时代已经不那么显著了。

**主持人：** 关于GitHub，你们有考虑过替代品吗？

**Ivan Burazin：** 我有一些想法。当时很多客户是编码代理（coding agents），他们都在试图解决版本控制问题。GitHub对于内循环（inner loop）来说太慢了，而且它本来是为外循环（outer loop）设计的。有人甚至直接把整个代码库倒成JSON推送到S3，不做diff。这说明需要一种面向代理的第一性原理的版本控制新方案。我认为未来Git还会存在，但会有新的代理协作层出现。另一个有趣的点是CI/CD。现在大家生成大量PR，CI成为瓶颈。我们和一家公司聊过，他们一天有1000个PR，CI排队严重。有人建议用沙箱做CI runner，我们正在探索。

## 计算机使用与Windows沙箱的未来

**主持人：** 聊聊计算机使用（Computer Use）吧。Windows沙箱是你们的大赌注。

**Ivan Burazin：** 是的。我们花费了大量投资在计算机使用上，但尚未完全公开。现在的AI模型很先进，但大部分知识工作仍然锁定在遗留的Windows应用中，这些应用不会很快消失。我们估算，全球知识工作者约1亿（美国），工资总额约10万亿美元。如果自动化比例达到40%，那就是4万亿美元的市场。要解锁这些价值，代理需要拥有带Windows环境的计算机。目前唯一的选项是EC2或Azure上的Windows，启动需要3-5分钟。我们创建了真正的Windows沙箱：启动在秒级而不是毫秒级，但有点快照、分叉等沙箱特性。我们希望这将解锁巨大的价值。

**主持人：** 我认为这是RPA（机器人流程自动化）的下一波浪潮。我当时很兴奋，但为什么微软自己不大力做这个？

**Ivan Burazin：** 萨提亚（Satya）昨天才发了“每个代理都需要一台计算机”的帖子。微软有Power Automate，但生态不同。我们也会提供Mac OS沙箱，但Mac OS的许可限制很严：每台机器只能运行两个并行VM，而且每24小时只能分配给不同的用户。安全方面，内存快照只能在同一台物理机器上恢复。所以跨机器迁移困难。我们的定价会不同，功能也会受限。但如果你要进行Mac OS上的RL训练，后续模型的迭代需要运行环境，所以Mac OS沙箱的需求仍然存在。我觉得苹果在限制授权模式上有点“搬起石头砸自己的脚”。

## 创业建议：家庭与工作的平衡

**主持人：** 我对你个人很佩服，你平衡了年轻家庭和创业。你认为那些因为有家庭而不敢创业的人应该怎么办？

**Ivan Burazin：** 我的家人有时在这里，但我经常出差（克罗地亚和美国之间）。陪伴时间少，这是一种牺牲。你必须意识到，一切都会痛。去健身房痛，减肥痛，创业也痛。你要学会享受痛苦。如果你的伴侣支持，你可以做到，但必须接受缺席的事实。我的孩子将来会知道我的故事。我对家庭有目标有规划，但确实需要另一半承担很多。

## 市场趋势：SaaS加价转售Token不可持续

**主持人：** 你有很多犀利观点。别人发推说“你说的对，但你太小众了，圣诞节就休息一下吧”，你回复“这就是为什么你做不到”。还有什么关于创业生态的看法？

**Ivan Burazin：** 最近一个看法是：市场给那些**加价转售Token的SaaS供应商**给予了过高溢价，我认为这是不对的。因为你的定价依赖于基础模型的价格波动，而且你的收入重新加速并不是真正的有机增长。另外，你的数据还是被孤立在客户自己的系统中。我主张：应该把**所有产品功能通过API暴露出来**，按API调用量收费。Salesforce的CEO Benioff三天前发推说“Salesforce的所有产品都已经通过API暴露”，我当时震惊了：这才是真正的重新加速，基于API消费的真实收入。我认为会有一次“冷水淋浴”，当人们意识到所谓的AI收入是转售Token而非真正的SaaS价值时，泡沫会回落。

**主持人：** 所以本质正确，但很多公司只是试图变成“AI公司”。

**Ivan Burazin：** 没错，我并不贬低这样做的人，每个公司都需要争取估值和下一步融资。但我们作为社群不应该宣扬这是神奇的脱困之路。

## 展望未来：AI代理的云

**主持人：** 你们会提供GPU吗？

**Ivan Burazin：** 会，但不是用于推理，而是提供带GPU的沙箱，比如3D渲染或CAD上的RL训练。我们已经在路上了。

**Ivan Burazin：** 目前我们在托管商那里用裸金属机器。从毛利率角度看，自建数据中心需要大量资金和风险，只换取几个百分点的提升，所以现在不划算。但我们的架构支持未来这样做。

**主持人：** 你如何看待Daytona的终局？是新一代Heroku、AWS，还是新的Stripe？
