这场对话不只是关于 xAI 内部的技术迭代，更是一次关于计算、数据、注意力压缩与交互未来的长途跋涉。从 Flipbook 到 NeuroOS，从视频扩展到 Reference to Video，Ethan 描绘了一个“生成式界面前端 + 确定性后端”的未来，而他认为最终点燃这一切的，不是更大的模型，而是更聪明的语言智能。

## 详细内容

### 1. 从 Nvidia 到 xAI：一个关于计算的赌注 [章节1]

**核心观点**

世界模型需要像语言模型一样遵循缩放定律，而扩大规模需要远超 Nvidia 的计算资源。这推动 Ethan 从 Cosmos 的领导者转向 xAI，在那里他获得了几乎无限的 GPU，却面临从零搭建整个团队的挑战。

**深度阐述**

在加入 xAI 之前，Ethan 在 Nvidia 领导了 **Cosmos 世界模型** 的开发。Cosmos 是一系列旨在模拟物理世界的视频基础模型，为机器人研究提供通用基础。但当他完成第一版训练后，他意识到一个关键事实：**视频模型存在与语言模型类似的缩放规律。**（原文：*“I realized this as this thing also has a scaling law similar to language model.”* [在讨论 Cosmos 时]）要进一步提升性能，必须更大规模地扩展模型和数据，而 Nvidia 内部的 GPU 虽多，但研究院的环境仍不足以支撑他认为所需的计算等级。于是他选择在 mid 2025 加入了 xAI，那里拥有被称为 Colossus 的巨型集群。

加入时的状态令他印象深刻：**“没有基础设施、没有数据、没有模型，只有几个工程师。”**（原文：*“At that time I joined by the time when XI was about to build video models and in multimodel models. There were no no infra no data and no model. And just as just a few engineers.”* [在讲述转职时]）Ethan 将这种情况比喻为第二次建造——第一次是在 Nvidia 用一年时间搭建 Cosmos，第二次则是用三个月完成 Grok Imagine。他反复强调，xAI 强有力的数据、模型推测和底层基础设施支持，极大地加速了开发。

**个人感受**

Ethan 坦言，从零开始反而是某种自由：“**每个人的天赋都很高，而且彼此非常接近，朝着同一个目标。**”（原文：*“Everyone everyone were very strong and clever very close with each other towards a common goal.”* [描述团队文化时]）没有冗长的会议，每天只有一次异步同步，剩下时间全部用来构建。这种“马拉松”般的节奏也让他意识到保持健康的重要性。

**延伸思考**

许多实验室在建立之初会陷入“先组建大团队再分工”的陷阱，但 xAI 的做法是保持小团队、高带宽、低沟通成本。结合 Ethan 后来的观点——**模型的大部分进步来自抓 bug 而非新算法**——我们或许可以得出一个反直觉的杠杆：在顶尖人才构成的紧凑团队中，迭代速度比人数更关键，而高效的基建则能将迭代周期从周压缩到天乃至小时。

**精华收获**

- 跨实验室跳槽的首要驱动因素应当是**计算资源的量级和稳定性**，因为缩放规律决定了顶点。
- 小团队+高迭代+坚实基建是短期内构建前沿模型的最优组合，甚至可以弥补起步阶段的数据和算法空白。

### 2. 三个月从零到 Grok Imagine：迭代与 bug 的隐形战争 [章节2]

**核心观点**

从构建图像模型到视频模型再到联合音频，Ethan 的团队在三个月内完成了 Grok Imagine 0.9，而其中最大幅度的性能提升并非来自新架构，而是来自对数据管道和训练流程中微小 bug 的修复。

**深度阐述**

当一个团队从无到有构建生成模型时，通常会经历几个阶段。Ethan 在 Cosmos 和 Grok Imagine 中都遵循了相同的基本流程：

1. **构建图像模型作为基础**：图像模型比视频模型训练成本低、图文映射更密集，是后续扩展的基石。
1. **生成合成文本 - 视频对**：互联网上的视频标题或描述通常与视频内容无关（例如一段雪山视频的标题可能是“今天真开心”），因此必须通过 VLM（视觉语言模型）或人工标注来为视频配上精确描述。
1. **训练一个压缩器（Tokenizer/VAE）**：将像素压缩到连续的隐空间，将图像或视频的 token 数量降低到可处理的长度。
1. **训练扩散 Transformer**：在隐空间上进行去噪训练。
然而，Ethan 强调，真正决定模型质量的是**迭代速度和抓 bug 的能力**。他直言：“**很多时候改进不是来自新算法，而是来自发现数据管道和训练管道里的小 bug。**”（原文：*“a lot of the improvements does not come from new algorithms. It comes from finding small bugs here and there in the data pipeline in the model training pipeline.”* [谈论迭代经验时]）而迭代速度的衡量标准是“每天可以进行多少端到端实验”。

在 xAI，强大的基础设施让这套流程极快运转：从获取新数据、设计算法、在小规模上训练新模型、到评估结果，周期被压缩到极致。Ethan 认为这是三个月奇迹的关键。

**个人感受**

这种“抓 bug”取胜的方法听起来反直觉，因为通常我们以为更多人力可以更快找出所有 bug，但 Ethan 指出另一种现实：**团队内沟通带宽是稀缺资源。** 如果人太多，信息流变慢，bug 反而更难暴露。而代码模型在 mid 2025 年还不是那么强，Ethan 回忆当时尝试用大模型写代码：“**它第一次构建东西确实快，但扔给你几千行意大利面代码，我自己无法维护，模型自己也搞不清怎么改。**”（原文：*“it produced codes that are kind of difficult to maintain even though like the first time it builds something extremely fast, but it gave the like a spaghetti code thousands of lines that I couldn't maintain and the OM itself couldn't figure out what's what's wrong and how to improve on top of it.”* [谈及早期代码模型]) 但到了访谈时，他认为代码模型已经好了很多，甚至带来了新的瓶颈——以前写一个实验需要几周，现在只需几小时，导致**计算本身成为迭代速度的新瓶颈**，因为你有太多想法需要尝试，但 GPU 有限。

**延伸思考**

Ethan 提到的“迭代速度 = 每天实验次数”与很多高效研究组织的理念一致。有趣的是，随着代码模型能力提升，从“人工实现慢”变为“计算资源不够试”，这意味着实验室需要重新考虑资源分配：不是去抢更多 GPU 用来跑成品训练，而是建立一套**即时实验系统**，让半自动化的代码生成 + 小规模快速训练 + 自动评估形成闭环。

**精华收获**

- 捷径不存在：成功的生成模型依赖基本功（数据标注、压缩器、合成配对），这些步骤没有魔法跳过。
- 开发中的“bug 修复”贡献通常被低估，而小团队由于沟通高效，在这方面可能更有优势。
- 当代码模型将实现成本降为零时，**算力分配逻辑**需要重新思考——应预留一部分算力给实验性小规模训练，否则会浪费模型自举能力。

### 3. 视频模型的真实成本：不仅仅是 GPU [章节3]

**核心观点**

训练一个顶级视频模型的总成本（包括存储、传输、特征向量存储）实际上与同规模的 LLM 相当，甚至更偏向 IO 密集。许多人在计算 GPU 算力时忽略了存储带来的惊人开销。

**深度阐述**

当被问到“视频模型的训练费用大概是多少”时，Ethan 给出了一个出乎意料的类比：**“视频模型的总成本与语言模型相当，大约属于中等规模 LLM 的水平。”**（原文：*“Surprisingly video models is like the the cost is very is comparable to language models and I'll be the largest scale as language model maybe like a medium scale language models.”* [在讨论成本时]）但构成却非常不同：

- **存储成本极高**：假设有 10 亿个视频，每个 5MB，就需要 5 PB。再加上 VAE 压缩后的连续特征也需要同等规模的存储，整体超过 10 PB。他给了一个估算：仅将 5 PB 视频存储在 AWS S3，每月就要 10 万美元；而数据入站通常免费，但出站（比如下载到训练集群）的费用更高（5 PB 出站约 23 万美元）。
- **IO 瓶颈远大于常规 LLM 训练**：由于数据量巨大，每次训练都需要大规模 I/O 操作。Ethan 透露在 Cosmos 中做了大量优化来避免 IO 阻塞。
- **GPU 计算成本**：视频模型的参数量通常很大（如 LTX Video 有 19B 参数，属于稠密模型；或混合专家模型总参数近 100B）。训练的 token 数（压缩后的视觉 token）也达数万亿。因此实际 GPU 小时数与 LLM 类似，但架构效率可能更低（因为扩散模型的去噪步骤较多）。
不过，Ethan 也指出**推理环节的成本可以通过“步数蒸馏”大幅降低**。扩散模型通常需要 100 步甚至上千步生成一个样本，而步数蒸馏可以让模型用 4 步或 8 步就能达到相近质量。基本原理是：教师模型（原始大模型）要学习的是互联网上极其复杂的数据分布；而学生模型只是拟合教师输出的简化分布，所以更容易。Ethan 说：“**步数蒸馏的模型只是尝试从教师学习，而教师的分布比整个互联网简单得多。**”（原文：*“The step distill model is just trying to learn from the teacher. The teacher is a model and the size is fixed. The distribution is much simpler than the whole internet.”* [解释蒸馏为何有效时]）

**个人感受**

Ethan 在谈到成本时有意无意地表达了对“云服务器陷阱”的警惕。他说“也许建自己数据中心的同时也要建自己的存储”，暗示 GPU 数据中心的设计者通常没预料到视频模型对存储网络的需求。他还说：“**这些成本加起来比很多人想象的要高，我自己在 Cosmos 时也低估了。**”

**延伸思考**

成本结构的变化可能会推动一种新的基建模式：**超大规模视频模型实验室正在从纯 GPU 集群转向 GPU + 大容量 NVMe 存储 + 高带宽内部网络的一体化设计。** 同时，蒸馏技术的成熟让推理成本大幅下降，这意味着服务层（如 Grok Imagine API）可以提供极低价甚至免费的实时生成，为生成式 UI 等新应用铺平道路。

**精华收获**

- 视频模型的实际总成本 = 存储 + 传输 + 训练 GPU。这三者中，存储常被低估。
- 推理成本的下降路径是步数蒸馏，目标可达到 4-8 步甚至 1 步。
- 基础设施的设计必须考虑数据密集型工作流，否则 IO 会成为不可见的天花板。

### 4. 图像模型是视频模型的前提，但智能来自语言 [章节4]

**核心观点**

要做出好的视频模型，必须先训练一个强大的图像生成模型。而 Ethan 最大胆的断言是：视频模型（扩散模型）的“理解能力”几乎完全来自语言侧（如 prompt 改写器），视频扩散本体只是“笨重”的像素解析器。

**深度阐述**

在 Cosmos 和 Grok Imagine 的设计中，存在一个**不对称结构**：

- **语言侧较大**：例如在 Cosmos 中，prompt 改写器（基于 Llama 或 Mixtral）参数比视频扩散模型本身还大。改写器的任务是将用户简单的指令（如“一只猫”）扩展为极尽详细的视频描述，包括背景、运动、光照、视角等。
- **视频扩散侧“很笨”**：Ethan 解释，因为训练时使用的配对都是极其详细的描述（“你要描述得一个盲人读了也能在脑中重建视频”），所以模型学会了**精确按照输入描述执行**。如果用户只说“一只猫”，模型可能生成一个白背景静止的猫——因为它没有被告知更多细节。
视频模型的**真正理解能力来自那里的语言智能**。当 Ethan 说“**每次我们看到视频模型有提升，实际上大部分增益来自语言模型，而不是视频模型本身**”（原文：*“every time you see there there's some improvement on these models I would say mostly the the gain comes from language model not not coming from the the video model itself”* [章节4]）时，他颠覆了“多模态融合”的主流叙述。举例来说，GPT-4o 生成的图像之所以惊艳，不是因为扩散头有多强，而是背后的语言模型花了大量时间推理、搜索、规划布局。本质上，视频生成正在变成“语言智能决定质量”的代理问题。

进一步的，Ethan 指出，即使不进行联合训练，单独使用一个聪明的语言模型作为 prompt 改写器也能大幅提升效果：“**在 Cosmos 里，我生成一个快乐的绵羊，不重写的话看起来像 CGI；重写后就变得非常漂亮。**”（原文：*“I generate a happy sheep and if without any rewriting it it looks so CGI and after rewrite it looks it it looks so beautiful.”* [举例说明改写效果]）

**个人感受**

这个观点对于计算机视觉研究人员可能是令人沮丧的。Ethan 自己也承认，他曾专注于视频模型的架构改进，但逐渐意识到瓶颈在语言端。这或许也是他最终决定离开 xAI、专注于语言模型研究的原因之一。

**延伸思考**

如果语言智能确实是视频生成提升的动力，那么未来的方向应该是**让语言模型学会更精准地调用和编排生成工具**，而不是试图把所有能力塞进单一扩散模型。这直接导向了 Ethan 提出的“视频代理”概念。

**精华收获**

- 当前最好的视频和图像模型本质上是 **“语言规划 + 扩散执行”** 的分工结构。
- 优化重点应从扩散架构转向 prompt 改写/代理框架。
- 视频扩散模型本身对于提示的“字面性”意味着**你需要把想象过程外包给语言模型**。

### 5. 压缩的策略：帧级 vs 时间压缩，以及实时性的代价 [章节5]

**核心观点**

视频编码时可以通过利用帧间冗余来高效压缩（时间压缩），但这会牺牲实时交互能力；框架级的逐帧压缩虽然 token 膨胀，却能为未来的交互式世界模型铺平道路。两者的选择取决于应用场景。

**深度阐述**

视频模型在将视频像素压缩到隐空间时，存在两种主要策略：

- **逐帧压缩（帧级 VAE）**：对每一帧图像单独压缩，缺乏时间维度上的压缩。优点是**低延迟、可即时响应**——因为模型可以逐帧输出，用户无需等待完整序列铺满才能看到结果。缺点是 token 数量激增（大约是时间压缩方案的 4 倍）。
- **时间压缩**：利用帧与帧之间的相似性，将多个时刻的 token 合并为一个。Ethan 举例说，例如 W.A.L.T. 2.1 使用了 8×8×4 的压缩比，即原本 4 帧的时间 token 被压缩为 1 个。这大幅减少了上下文长度，但引入了**固有延迟**，因为它必须累积足够的帧才能开始解码。对于需要极低延迟的交互场景（如实时游戏），这种延迟是致命的。
Ethan 以 Flipbook 和 NeuroOS 为例，说明了为什么逐帧压缩的路径可能对未来更重要：**“如果你想要一个世界模型让人能够实时互动，你必须逐帧生成，以便模型能立即对用户输入做出反应。”**（原文：*“If you do it frame by frame you can the model can respond to any user requests immediately.”* [在比较两种压缩方式时]）

世界模型的“实时性”要求响应时间从几十毫秒到几百毫秒不等，而时间压缩的累积延迟不符合该需求。不过，Ethan 也指出，对于非交互的大规模视频生成（像电影制作），时间压缩可以极大节省推理成本。

**个人感受**

在讨论这一部分时，Ethan 表现出对设计决策中取舍的清醒认识。他甚至提到一个具体案例：在 Grok Imagine 的视频扩展功能中，为了支持长上下文，他们没有选择对前面所有帧进行时间压缩，而是保留全精度 token，但通过选择性的上下文管理来限制序列爆炸（比如 Reference to Video 机制）。

**延伸思考**

压缩策略的选择实际上反映了对最终应用的假设。交互式世界模型（如可玩的生成式 Doom）需要逐帧处理，而离线合成则可以用更激进的压缩。值得注意的是，语言模型也面临类似的分支：是用稀疏注意力还是全局注意力、是精确留存还是压缩摘要。Ethan 在对话后半段提出，**语言模型未来也会像视频模型一样，需要学会自动管理自己的上下文**，比如知道什么时候应该剪裁、压缩或检索历史。

**精华收获**

- 视频压缩不只是技术问题，而是交互范式的选择：**帧级压缩 = 可交互，时间压缩 = 高压缩率 + 延迟。**
- 世界模型的实时、长视界、交互性要求它选择帧级路径，但必须配合有效的上下文管理（如 Reference to Video）。
- 类似地，语言模型未来也需要实现“模型自主上下文管理”才能突破长上下文瓶颈。
---

### 6. 生成式 UI：从像素到用户意图的直接桥梁 [章节6]

**核心观点**

当推理成本降到足够低时，我们不应再用代码来描述界面，而应让扩散模型直接从用户意图生成像素。这将带来界面设计的完全个性化，生成式 UI 将成为人与 AI 交互的最高带宽接口。

**深度阐述**

Ethan 畅想了一个“生成式界面”的未来。他以 Flipbook 为例：一个模拟 Web 浏览器的界面，所有 UI 元素（文字、按钮、图片）全部由图像模型在用户交互时即时生成。用户可以点击文字链接，模型理解意图并即时生成新页面——这些页面背后没有真正的网页代码和数据，全部都是模型“想象”出来的。

更极端的展示是 NeuroOS，它用视频模型模拟一个完整的操作系统，用户可以在其中打开 Firefox、运行 Doom，甚至用快捷键关闭标签页。整个系统是实时生成的视觉流，只有用户交互事件是真的，操作系统本身不存在。

Ethan 的核心论点是：**“先有代码、再有二进制、再渲染为像素”这一链条将被打破，未来我们直接用用户意图生成像素。**（原文：*“So why don't we have like a user instruction to the pixel directly?”* [描述生成式UI理念时]）他对比了两种方式：

- **传统开发**：用户需求 → 产品经理 → 前端代码 → 后端逻辑 → 二进制 → 像素输出。
- **生成式 UI**：用户意图 → 推断模型理解需求 → 扩散模型生成像素（前端） + 语言模型执行逻辑（后端）。
这种范式的优势是**极致的个性化**。Ethan 举例：“也许每个人都点开邮箱看到相同界面，但我想让我的邮箱看起来像 TikTok，左右滑动邮件；或者我想去掉 Instagram 故事上的点赞按钮。生成式界面可以立刻实现。” 他甚至预言：“**未来，前端的扩散模型将负责展示，后端的大模型和代码模型将负责确定性的逻辑运算。**”（原文：*“So in the future we might have much more powerful OM and coding models running behind the scene and in the in the front end the diffusion model will actually be the front end to show stuff to you.”* [在描述未来架构时]）

他承认目前（2025-2026年）这种方式的推理成本还太高，但他提到 H100 约每 GPU 小时 1 美元，一个人每天使用 8 小时、每月 240 美元，比云主机还贵。但**计算成本每 12-18 个月降低 100-1000 倍**（结合了模型效率和芯片进步），成本很快就会不是问题。

**个人感受**

Ethan 在描述时充满激情，说他相信生成式 UI 是“人类与 AI 交互的最高带宽，直到 Neuralink 到来之前”。他同时幽默地指出：“对于网页设计师来说，这可能会引发可访问性问题——屏幕阅读器该怎么处理完全生成的界面呢？”但这种担忧并未减弱他的兴奋。

**延伸思考**

这实际上是对“代码是否冗余”的激进追问。在 LLM 能直接写代码并渲染的今天，为什么还需要代码？答案可能是**可验证性、稳定性和效率**。生成式 UI 牺牲了确定性和可审核性，但换来了无限的灵活性。未来的平衡点或许是：确定性后端处理逻辑，生成式前端处理外观，两者通过标准的协议交互。

**精华收获**

- 像素直接生成比代码-渲染链更快响应个性化需求。
- “扩散前端 + 代码后端”的架构可以结合生成式界面的灵活性和确定性逻辑的可靠性。
- 推理成本曲线下降节奏极快，我们可能在 2-3 年内进入生成式 UI 的实用阶段。
---

### 7. 定义世界模型：实时、交互、长视界 [章节7]

**核心观点**

Ethan 给出了一个清晰的操作性定义：世界模型 = 实时 + 交互 + 长视界视频生成。这一框架将视频生成、游戏模拟和机器人控制统一在同一维度下。

**深度阐述**

当被问到“世界模型与视频生成有何区别”时，Ethan 提供了他自己的版本（免责声明：“我不参与‘世界模型究竟是什么’的争论，我只说我理解的”）。他认为世界模型应该具有三个要素，按顺序逐步实现：

1. **交互性（Interaction）**：用户可以通过键盘、鼠标、甚至语音与世界模型交互，模型必须合理响应。Flipbook 和 NeuroOS 是初步的交互示例，但还远未成熟。
1. **实时性（Real-time）**：从用户输入到视觉反馈的延迟必须极低。Ethan 以 CS:GO 玩家为例，“他们要求 300fps 以上，即低于 3 毫秒。即使对要求更宽松的数字人对话，也需在 200 毫秒内回应，这依然很难，因为 VAE 的时间压缩会引入固有延迟。”
1. **长视界（Long horizon）**：生成的内容持续数分钟甚至数小时，且保持长时间的一致性。当前模型只能生成几秒到几十秒，而长视频的难点在于上下文管理。
Ethan 认为，**真正的世界模型最终将是我们眼前看到的操作系统模拟、游戏模拟、甚至物理世界模拟的统一体。** 而实现它的第一步就是解决“长视频扩展”问题。

他以 xAI 的 Video Extension 功能为例：通常的视频生成模型只能生成一段固定长度的视频。创作者有时尝试将上一段视频的最后一帧作为下一段的起始帧，“但如果你这样做几次，质量就会严重下降，因为它无法在整个视频上保持长期上下文。” Grok Imagine 的 Video Extension 则将**历史上所有生成的视频 token 都放入上下文**，因此可以保持角色和场景的一致——但它带来了 token 爆炸的新难题。

**个人感受**

Ethan 在解释世界模型时，流露出了他作为 Cosmos 前领导者对“模拟”的偏爱。他强调“交互性”是核心，但承认目前的视频生成模型离满足“交互”还有很大距离。他认为实时光栅化（每秒数十帧）仍需多年优化。

**延伸思考**

Ethan 的定义实际上将“世界模型”从学术概念降维成了容易衡量的工程指标：延迟、最长生成时长、交互输入种类。这种定义的好处是：**你可以逐步优化的，每一步改进都直接对应更接近“真实世界模拟”。** 与语言领域的“世界模型”讨论（如递归自我感知）相比，他的定义更接地气。

**精华收获**

- 世界模型的三个可量化维度：实时性（≤200ms 甚至更低）、交互性（多维输入）、长时性（分钟级）。
- 视频扩展功能是通往往世界模型的第一步，但需要从单帧条件升级到全长条件。
- 长视频一致性是当前的核心难题，解决方案包括选择性上下文（如 Reference to Video）。
---

### 8. 视频扩展与 Reference to Video：长上下文的两条路径 [章节8]

**核心观点**

为了生成长视频，当前存在两条互补的技术路径：**全上下文扩展**（Video Extension，保留所有历史 token）和 **参考对象注入**（Reference to Video，将关键帧作为条件）。Ethan 领导了这两条路径在 xAI 的落地。

**深度阐述**

Video Extension 的基本思路是：当你生成第二段视频时，不仅仅是把前一段的最后一帧作为条件，而是将**前一段所有视频的压缩 token 都喂入模型**。Grok Imagine 已经实现了这一功能。Ethan 透露：“它可以记住谁在说话、出现了什么物体，以及所有之前的上下文。” 但缺陷是：token 数量会线性增长。在 Cosmos 中，5 秒视频约有 50-60k 个 token，所以 50 秒就是 500k+ token，很容易达到百万级。

Reference to Video 则是不把所有历史都塞进上下文，而是允许用户上传最多 7 张图像作为参考条件（如角色照片、物体图片、场景背景），模型在生成时主动将这些参考对象的特征复制到新视频中。这类似于“选择性长期记忆”，避免了全量扩展的序列爆炸问题。Ethan 展示了把马斯克、爱因斯坦和自己拼在一起的生成图片（“Optimus Einstein myself”），并说这是他们内部解决长文本问题的中间方案。

他进一步展望了未来的“上下文管理”机制：**模型应该自己学会选择何时引用历史、何时忽略，而不是依赖 heuristics（启发式规则）。** 像 Frame Pack 这样的论文已经开始探索“远距离帧压缩”的启发式方法，但 Ethan 认为这些最终会被模型自己吸收，类似语言模型中“工具调用结果自动修剪”正在从 harness 层下沉到模型层。

**个人感受**

Ethan 对长视频生成的执着显而易见。他说“Video Extension 很受创作者欢迎”，但他也承认这只是一个向最终世界模型进发路上的垫脚石。在谈到 xAI 不怎么宣传这些功能时，他带着一点无奈：“我觉得 xAI 的沟通风格比较低调，但我们真的做了很领先的事情。”

**延伸思考**

上下文管理问题正在视频和语言两个世界里同时发生。语言模型的 long context 方案（如 RAG、按需检索、窗口滑动）与视频的 Reference to Video 有类似的结构：把外部存储或参考点引入生成流程。未来或许会统一为“可微分上下文管理器”——模型以可学习的方式决定何时阅读/保留/丢弃信息。

**精华收获**

- Video Extension 保证一致性，但代价是序列长度暴涨。
- Reference to Video 提供高效的外部参考，但不解决长范围动态变化。
- 最终解决方案是让模型自我管理上下文，而不是依赖外部硬编码。
---

### 9. 音频生成的隐蔽挑战：模式对齐与连续性 [章节9]

**核心观点**

音频生成（尤其是音乐和语音）面临两重困难：一方面，现有的 VLM 几乎无法理解音频内容（尤其是音乐细节）；另一方面，语音与音乐的建模方式不同（语音偏向离散 token，音乐高度连续）。模态对齐（音频-视频-文本三者精确时间同步）是巨大的工程挑战。

**深度阐述**

Grok Imagine 0.9 被 Ethan 称为“首个大规模部署的音视频联合生成模型”。但他坦言，音频部分的构建比图像和视频都更棘手。

- **语音与音乐的差异性**：语音可以被视为带有韵律的离散语言文本（近似 token 序列），但音乐是连续的、高频、密集的，无法用离散 token 建模。Ethan 说：“**你可以和语言模型说话，但让它唱首歌，效果通常很差。**”（原文：*“if you ask them to sing a song or something they typically is not very good”* [在描述音频生成难度时]）
- **缺乏音频标注能力**：大多数 VLM 只能理解图像和视频，几乎不能理解音频。要生成合成数据标注音频-视频对齐，需要大量人工标记。而即使是人类，要描述一段音乐的细节（节奏、音色、各乐器轨道）也极其困难。Ethan 用图像类比：“就像我们之前要求把图像描述得让盲人也能重建一样，现在我们需要把音频描述得让聋人能重建——这太复杂了。”
- **时间精度对准**：视频和音频必须以时间步为单位精确对应。对于文本与图像的对齐，可以松散（“大约描述内容”）；但视频和音频需要精准的逐帧对齐。这要求模型设计时必须考虑时间意识，这是传统语言模型不具备的。
Ethan 暗示，未来的模型需要具备“时间感知”能力，这与长视频的另一个难题（长时一致性）是一体两面的。

**个人感受**

在讨论音频时，Ethan 显得有点审慎。他提到 Grok Imagine 的音频部分是他投入最大精力的区域，但仍然对效果不够满意。“我们在持续改善，但模音视觉联合一致性依然是最难的部分。”

**延伸思考**

音频生成在某种程度上是视频生成的“感官补充”，但当我们要达到真实世界模拟（世界模型）时，音频是必不可少的。有趣的观察是，当前语音对话系统已经相当成熟，但“让 AI 完美合唱歌唱”却很困难。这种不对称提示我们：连续信号建模（音频、视频原始像素）仍是深度学习稳固但未完全攻克的前沿，而离散信号处理（文本）更像已经攻下的高地。

**精华收获**

- 音频生成的难点是连续性（音乐）与离散性（语音）的矛盾，以及对时对齐的硬要求。
- 目前的 VLM 在音频理解上极为薄弱，完全依赖人工标注，构建全面的音频数据集是首要瓶颈。
- 音频-视频联合建模要求模型必须有时间意识，这比文本-图像的对齐更严格。
---

### 10. 视频代理：当语言智能成为驱动引擎 [章节10]

**核心观点**

Ethan 最重要也最具突破性的观点是：视频生成的未来不在于更大的扩散模型，而在于**以语言模型为大脑、以生成工具为手脚的视频代理架构**。大部分可见的质量提升来自语言模型的思考、工具调用和迭代修正，而非扩散本身的进步。

**深度阐述**

视频代理（Video Agent）是指将语言模型作为核心控制器，让它调用多种工具（视频扩散模型、图像编辑模型、ffmpeg、图像处理库等）来**多步协作完成复杂视频制作任务**。Ethan 认为这是“视频生成的 Copilot 时刻”——就像从自动补全（Copilot）进化到完全自动的代理（Devin/Codex）一样。

他指出当前很多用户对 Grok Imagine 的视频编辑功能有误解：例如用户要求“把这段视频扩展成一分钟”，实际上视频编辑工具只能局部替换、删除或过渡，不能凭空生成长内容。但视频代理可以理解这个意图，然后拆解成多个子任务（生成多个片段、拼接、添加转场、检查一致性等），逐个调用工具完成。Ethan 说：“**你甚至不需要联合训练模型。拿一个前沿推理语言模型，给它一个 harness，让它可以调用 Grok 作为工具。这就是你的视频代理。**”（原文：*“You can just take an offtheshelf Frontier reasoning model, slap it on a harness, give it Grock as a tool, that's it. That's your video agent.”* [在讨论视频代理实现方式时]）

他还展示了 Grok Imagine Agent Beta 的实例：在录制过程中，代理正在后台生成一段 36 秒的视频，它不是直接用视频模型生成，而是先规划：生成关键帧、图像模型生成图像、再用 ffmpeg 拼接、添加字幕。整个过程像一个人一样工作。

他由此导出结论：**“视频模型的智能增益主要来自语言模型”** 这一论断的实操版本就是视频代理。如果你接受这个观点，那么意味着“继续去优化扩散架构”的边际收益会越来越低，而“教语言模型更好地使用视频工具”才是真正的增长点。

**个人感受**

主持人听了这个观点后说：“这有点令人失望——好像模型到头了，我们只能加持 agent 了。” Ethan 表示同意这种“焦虑”，但他强调这其实是激动人心的：工具本身已经很强大，缺的是编排它们的智能。这也印证了他后来所说离开 xAI 的理由：“我想做更多语言模型端的研究。”

**延伸思考**

视频代理和最近兴起的“计算机使用”（CUA）是一致的——它们都在探寻智能体能动性。如果我们训练的语言模型能熟练操作视频工具，其实也就相当于它能操作任何一个软件。因此，视频代理的发展可能反过来推动通用 agent 能力的成熟。不过，这种“代理调用”的缺点是：很难保证生成时长和质量稳定，因为分为多个步骤。Ethan 的乐观在于，“当视频质量跨过可用性门槛后，企业愿意花更多钱，因为代理虽然昂贵，但效果远超端到端生成。”

**精华收获**

- 视频代理架构 = 推理语言模型 + 一组确定性/生成工具。
- 质量提升的主引擎在于语言模型的推理规划能力，而非扩散模型本身。
- 视频代理是“AI 视频 Copilot”阶段，最终目标是全自动生成（与代码代理如 Claude Code 类似）。
- 投资视频代理方向比单纯改进扩散模型目前可能更具产出比。
---

### 11. 离开 xAI：当研究兴趣与公司目标不再一致 [章节11]

**核心观点**

Ethan 离开 xAI 的原因很坦率：公司快速变化的目标限制了他想做的特定研究（尤其是纯粹的语言模型研究），而他坚信未来的瓶颈在语言智能侧。同时他承认这对他个人来说是一次职业风险的权衡。

**深度阐述**

当主持人最后问“为什么离开 xAI”这个辛辣问题时，Ethan 给出了相当直接的答案：

- **研究的自由受到内部优先级变迁的限制**：“公司目标变化很快，这是 xAI 的特点，也是很多大实验室的共同点。但具体到我想做的研究——更多在语言模型侧——在 xAI 内部无法充分展开。”
- **对自己职业路径的清醒认知**：他回顾了自己从计算机视觉（ResNet 时代，与孙剑等人合作）到自监督学习、再到大模型缩放的世界模型，最后到语言模型的转变轨迹。他意识到，视频模型（尤其是扩散模型）的发展已经进入平台期，现在真正的提升点是在语言模型如何为视频模型“注入智能”。因此他决定向语言模型侧移动。
主持人对此表示不解：“你是在‘降级’吗？因为大部分人留在实验室是为了更多计算资源，你却放弃了现成的集群。” Ethan 回应：“我过去十年里做过好几次大转型，每次的底层技能都是相通的——训练大规模模型的核心原理高度一致。对我来说，这不是那么大的跳跃。”他也坦陈：“当然有职业风险，但我比较自信曾经从零做成过几次。”

**个人感受**

Ethan 在整个对话里都表现出一种“工程师-研究者”的双重气质。他反复说“我喜欢研究大于产品工程”。在聊到他离开 Nvidia 是因为需要更多计算，离开 xAI 是因为需要更多语言研究自由时，显示出他是一种典型的“问题驱动型”人才——问题在哪里，他就去哪，基础设施和知名度反而不是优先因素。

他还提及自己十年前曾申请顶尖博士项目全被拒，只能进入工业界；但这一“失败”反而促使他走上了一条更适应他风格的道路。他说：“**很多人会以为做 CV 的就不能做 NLP，但其实很多核心规律是共通的。**”（原文：*“I feel actually within within ML it's actually easier to switch... a lot of the core principles how train large models are largely the same.”* [在讨论职业转型时]）

**延伸思考**

Ethan 的故事反映了一个主流趋势：**计算机视觉出身的顶级研究者，正在集体向语言模型研究迁移。** 这不是因为 CV 不再重要，而是因为他们曾经认为视觉理解需要专用架构，现在却发现语言能力可以反向赋能视觉。Ethan 的这条路径——CV → 世界模型 → 语言模型——可能是一代研究者的缩影。

**精华收获**

- “算力”和“研究自由度”是决定人才流动的两大变量。
- 技能树在训练大规模模型上是通用的，变换领域不是从头开始。
- 职业轨迹遵循“问题导向”：找到当前最大的瓶颈（语言智能），而不是捍卫已有的工具集。
---

### 12. 对语言模型的预测：自治上下文与递归自我编程 [章节12]

**核心观点**

Ethan 认为未来语言模型将学会自主管理自己的上下文窗口，包括增加、压缩、删除信息。他进一步推测，模型可能会在测试时递归地修改自己的 agent harness 代码，从而实现真正的无限上下文。

**深度阐述**

在对话尾声，主持人问 Ethan 对语言模型接下来一年的预测。他给出了两条线索：

1. **上下文自主管理**：从视频模型的长上下文困境出发，他类比到语言模型：“语言模型目前对自己的上下文长度没有感知，它不知道已经用了 80%。未来它应该能意识到，并且在接近限制时决定压缩或丢弃什么。这不是通过外部的 agent harness 修剪，而是**内化为模型自身的能力**。”
1. **自我编程的 agent**：更激进的是，他想象语言模型读取自己的 agent harness 代码并在测试时自主修改。例如，如果 agent harness 代码很短，可以放在系统提示里，模型在运行中生成对自己的子 agent 调用的逻辑、修改上下文策略。Ethan 说：“**想象一下，模型可以在测试时在线修改程序，就像人类改写自己的工具那样。**”（原文：*“It might be very interesting to to see that the model can like a program the model can program itself online in test time.”* [在对语言模型未来的预测中]）
这实际上是 agent 层面上的“自我改进”，也是一种 **递归世界模型**——模型必须意识到自己拥有世界模型，并能更新它。Ethan 表示这与他常说的“世界模型必须也是递归的”一脉相承。

**个人感受**

主持人注意到这一想法非常激进，但 Ethan 并不认为短期内不可能。他提到目前的 open code 和 pi 已经允许一定程度上的自我修改，但“还有很多工作要做”。他那种平和地说“我觉得未来一年会在这个方向有很多突破”时的笃定，像极了前面谈论视频代理时的语气。

**延伸思考**

模型在测试时修改自身代码的好处是：无需训练新模型就能适应新任务或环境。但风险同样明显——可能产生不可控的演化。如果语言模型有能力改写它的推理循环，那么对齐和安全将面临全新挑战。有趣的是，Ethan 在讲完这个预测后，整个对话也接近尾声，没有详细探讨安全。但这是一个值得持续追踪的信号。

**精华收获**

- 语言模型接下来的突破可能是**内化上下文管理**：不依赖外挂修剪，而是模型自觉控制。
- 更长远的是“测试时自编程”：模型通过修改 agent harness 来实现自我改进。
- 这可以看作是“世界模型 + 递归意识”在语言领域的自然延伸。
---

## 精华收获（全篇总结）

Ethan He 的这场对话不是一篇零散的幕后花絮，而是一张连贯的技术蓝图：

- **视频智能的源头是语言**。那些让我们惊叹的生成作品，背后是语言模型在“思考”、规划、描述、迭代调用工具。
- **世界模型的核心要义是交互**。它的实现路径不是单模型端到端，而是一个由视频代理（语言模型+工具）组成的系统，逐步逼近实时、长视界、可交互的模拟。
- **架构选择应服务于应用目标**：如果需要交互性，就要选择帧级压缩并承担 token 膨胀；如果追求低成本海量生成，时间压缩和蒸馏才是出路。
- **职业选择遵循问题驱动**。Ethan 从 CV 转向世界模型，再转向语言模型，每一步都是为了站在当下瓶颈的一侧。
- **未来已来，但分布不均**。生成式 UI、视频代理、自我上下文管理，这些概念已经 demo 级可用，唯一阻止它们渗透的是一年一两个量级的推理成本下降。而当成本到位，我们会看到“像素即界面、语言即智能”的新范式全面铺开。
也许 Ethan 最值得记住的不是某个具体模型，而是他对“模态统一”的解构：真正统一的不是模型架构，而是智能的语言核心——它学会去看、去听、去生成、去使用工具，而所有“视觉”和“音频”只是它调用的外围设备。从这个意义上说，**世界模型的内核，一直是语言。**
