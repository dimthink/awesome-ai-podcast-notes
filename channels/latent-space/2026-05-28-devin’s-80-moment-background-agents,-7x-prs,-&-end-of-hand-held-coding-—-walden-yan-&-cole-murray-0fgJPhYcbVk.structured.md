## 背景代理的兴起：从 16% 到 80% 的转折点

Walden 指出，工程界正在逐渐认识到**背景代理**（background agents，亦称云代理 cloud agents）这一概念。他认为在 **2025 年 12 月**左右，**Opus 4.5** 和 **GPT 5.2** 达到了一种能力阈值，使得我们能够从手把手指导模型，转变为**自主驱动**模型。

> “我们可以从规格说明直接得到一个完成的拉取请求（PR），前提是规格足够好，且几乎没有摩擦。这种模式改变了很多我们与代理交互的方式，并开启了背景代理更实用的世界。”
Cole 回忆说，所有人都有类似感受，但更像是不断攀升的过程。他提到 **Sonnet 3.7** 的转折点时说：“你们在一个晚上就重写了 Devi n整个系统。” Walden 承认进展一直在加速，尤其是最近几个月。“像 Opus 和最新 GPT 模型，已经达到人类实际上无法完全放手的高度自主水平。之前人们争论是否需要深入 IDE 与模型纠缠，现在他们可以完全把代理移到云端，这是一个更加严肃的对话。这在我们所有的增长图表中体现得很明显。”

Walden 展示了一个内部图表：

- **Devin 的 PR 合并占比**：**1 月为 16%，3 月达到 80%**。
- 相应的**使用量（合并 PR 数）**在 **2-3 个月内增长了 7 倍**。
- 而工程团队人数仅增加了约 10%。
主持人总结：“这真是一个巨大变化。现在很多人都在考虑购买 Devin，或者也想自己构建云代理。我构建 Devin 的过程很有趣，所以能理解为什么其他人也想这么做。”

## 构建 Open Inspect 的初衷：从客户摩擦到开源基础设施

主持人询问 Cole 最初是什么激励他尝试构建 Open Inspect。

Cole 解释：“Open Inspect 的产生主要是因为我观察到客户使用 **Claude Web** 和 **OpenAI Codex** 等工具时的摩擦。例如，Claude Web 通过 Slack 调用时，会话与初始调用者绑定，PM 无法将上下文传递给工程团队。这本身就是一个决定性的障碍。所以我内部搭建了一个类似的架构来做实验。随着‘离开 localhost’的趋势开始显现，**RAMP 发布了他们的博客文章**——那篇文章不仅仅是说‘我们建了一个好系统’，而是‘你也应该这样建’——这引起我的共鸣。我好奇仅凭那篇博客，Claude 能做出什么，于是在 Twitter 上直播了整个过程，比较了 GPT 和 Claude 各自的表现。由于我已经知道如何验证系统，而且开源社区没有一个真正在云端运行的系统（大多是在 localhost 运行，如 Superconduct 等），所以我构建并开源了它，让任何人都可以在此基础上修改。”

主持人问：“你开源了但没试图用它盈利或融资——怎么想的？”

Cole 表示：“我看到客户认为背景代理系统将成为公司**关键基础设施**，所以我想开源，让他们可以 fork 并加入自己需要的定制。有人问我是否要融资或转为服务，但我不打算这么做，有几个原因：

- 不想为 **$20/席位** 而竞争，那是一个艰难的业务。
- 主要架构很容易被复制。
- 你没有拥有整个堆栈，很难货币化——利润在沙箱层（Daytona, E2B 等）和模型层赚取，而我处在中间的灰色地带。
### 从基础设施到企业服务

Walden 接过话头：“确实有多层。当我们开始构建 Devin 时，（比其他人早两年）也必须自己搭建基础设施。最初基于 EC2 等云提供商的原始 VM，启动非常慢；关闭、保存、再唤醒有时需要 10 分钟，因为系统没有针对这种反复启停优化。我们不得不从头解决这些问题。现在当我们销售 Devin 时，我们提供完整的计算方案——在你的云中也能良好运行。此外，除了产品本身，帮助公司**学习和采用**这些编码代理也是重要部分。对于很多大型企业，不是每个人都熟悉 AI 工作流，我们需要作为工程师团队**实际去做**——帮他们集成、设置自动化，真正达到利用 AI 的层次。我们也充当客户的**思想伙伴**。”

## 架构选择：Agent “在盒内” 还是 “在盒外”

主持人将话题引向核心架构：“我们来谈谈背景代理系统的一般组成部分，以及决策中的细微差别。也许可以先从 Walden 所说的‘代理基本上在一个开放代码盒子中’这个模型开始。”

Cole 解释：

- **在盒内（In the Box）**：代理运行在沙箱中。负面权衡主要是**安全**——所有密钥也必须放进盒内，AI 的不可预测行为可能导致密钥泄露。
- **在盒外（Out of the Box）**：代理的“大脑”运行在某个**控制平面**（worker/control plane）中，沙箱只充当“手”，大脑通过工具调用操纵沙箱。这是更复杂的架构，因为需要管理状态；但安全更好，密钥可隔离在沙箱内。
Walden 说：“我们一开始就把‘大脑’和‘机器’分开了（盒外）。这还允许复用已有的开发者环境（例如 dev boxes），而不用为大脑构建新的、带依赖和密钥的机器。比如，一些客户通过 GitHub App 操作，不同用户有不同权限。如果所有交互都通过同一个 GitHub App，且没有分离决策系统和机器上的密钥，就会出问题。我们分离后，只需将最小权限的密钥放在机器上，大脑完全不可从机器访问。”

主持人表示自己有一张 OpenAI 的架构图，不确定是“在盒内”还是“在盒外”，Cole 认为这些都是一样的模式变种，盒外更可取。

### 环境设置：最难的部分

“但有一个问题我很少看到其他玩家做得好：如何管理盒子里实际有什么？”Walden 继续说道，“例如，一个大的仓库变化快，依赖更新，如何确保工作环境始终保持最新，凭据正确，能够运行应用和测试？我们在内部称之为 **repo setup**，这是公司自成立以来最棘手的问题之一。”

Cole 对此很认同：“这是我的客户中非常普遍的问题。许多团队根本没有好的开发者环境设置，很多情况下是‘去问 Bob 要密钥’，这对代理来说不起作用。大多数团队使用 **Docker Compose** 或微服务。在 Open Inspect 中，可以通过 hooks 运行 setup.sh 脚本预装一切，然后预快照，使启动瞬间完成；还有恢复状态的 hook，可以保持微服务运行。”

主持人补充：“早期我的一些朋友用 Docker 容器作为抽象层，但 Docker 有很多问题——首先它不是真正的安全边界，其次如果应用本身使用 Docker，就出现‘Docker 嵌套’。”

Walden 说：“所以我们花费大力气让**虚拟机**工作。现在需要运行真实应用、点击操作、发送屏幕录制，价值不断叠加。这是人们自己构建系统时会遇到的关键决策。”

Cole 认为 Docker 适合**运行基础设施**（而不是代理），因为工程师们已经在用 Docker Compose。“我始终是 **Web Containers** 的粉丝（像轻量 Docker）。总之，任何好的本地开发者体验，也容易让代理设置好环境。”

Walden 提到不同 VM 需求：“有客户需要 **macOS** 或 **Windows** 特定的 VM，我们因为分离架构是支持的，但开发中。最近我们在 beta 中支持了 **Android 开发**，为此需要实现嵌套虚拟化（在 Firecracker 实例中再跑 Android 模拟器），这有性能问题，但解锁了很多能力。”

## 测试能力：超越“计算机使用”

主持人请 Walden 解释测试能力的定位。Walden 说：

> “当人们思考 AI 运行应用并测试的能力时，我认为他们高估了‘计算机使用’部分。计算机使用在我想来就是点击按钮时的坐标问题。而测试实际上是**有趣的问题解决挑战**。如果你想做任意测试——比如一个跨越前后端的修改——你首先要推理如何用正确的代码版本运行应用并编排它们，然后如何触发功能，有时还需要管理员权限、功能开关、跨会话协调。这需要大量代码库上下文，需要专门的编排。在一些案例中，我们发现没有任何一个前沿模型能独立完成端到端任务，我们不得不将多个前沿模型组合起来解决。”
计算机使用本身因为新模型（如最近发布的 **Opus 4.7**）的视觉能力进步而变得更简单。但测试需要更复杂的评估（evals）来防止回归。

Cole 则提到他刚在 Open Inspect 中增加了**截图**支持，未来也可做视频。“这是用户强烈要求的功能，因为 **Cursor 的录制功能**出来后大家发现了它的价值。不过一旦知道要建什么，提示模型就能工作好。”

Walden 总结：“计算机使用只是更大测试问题的子集，且与代码库高度相关。背景代理系统拥有代码库上下文，可以驱动测试。”

主持人展示一个流程：“PR 完成后，点击‘测试批准’，返回一个视频，标注测试内容，看到光标移动。这很有‘AGI 时刻’的感觉。我希望能直接在 Slack 里合并，不用去 GitHub。” Cole 表示绿色标注和标签也是重要的细节。

### GitHub 集成：从 PR 到互动

Walden 分享调优经验：“大多数工具只是创建 PR，我们更进一步，让用户可以在 GitHub 上直接评论 Devin，Devin 会接收并回应。但这需要精细调整：Devin 自己的 PR 会被 Devin Review 评论，自己回答自己的评论可能造成循环。我们确保了评论是高信号质量，代理也足够聪明地决定哪些评论值得修复。当用户提出不合理要求时，Devin 会说‘我觉得你错了’——那是我最喜欢的时刻。”

Cole 说 Open Inspect 也有 GitHub 代码审查功能，可以控制提示，但现在还不支持自动回复（需标签触发）。

## 集成与 MCP：拓展代理生态

主持人问客户在 Open Inspect 之上最常见的要求。Cole 说：“主要是**真正集成到公司生态系统**——生产数据库（只读）、日志、知识库等。这能带来巨大价值，但也面临合规和访问控制挑战。”

Walden 用 **MCP** 为例：“MCP 爆发让代理接入很多服务，但要获得好体验，我们常需要构建自己的方案。比如 Slack MCP 能发消息，但我们希望 Devin 像同事一样在 Slack 里工作，需要支持 webhook、自然回复、不刷屏等——这超出了简单 MCP 集成。我希望能有一个比 MCP 更高表达力的系统——不仅是工具集，而是双向交互。MCP 规范有‘sampling’但没人用。当规范变复杂，它就开始失去‘简单连接’的承诺，变得像自己建第一方集成。”

Cole 同意：“对于关键集成，如果每个会话几乎都用，拥有它以便优化是合理的。”

## 记忆与知识库：未解的难题

“记忆（memory）问题怎么解决？”主持人问。

Cole 直言：“**还没解决**。Open Inspect 有一个公开 issue 但未实现，DeepWiki 也没索引记忆相关内容。” Walden 补充：“记忆是一个非常棘手的**检索问题**（RAMP 关于记忆写了什么吗？我没看到）。检索困难，生成也难——你不希望模型记住一次性的偏好并固化。”

Walden 回顾 Devin 的记忆历程：

- 第一个版本叫 **Knowledge**，目标是自动且无需用户主动教。
- 当 Devin 做错时，会问用户是否要记住正确做法。
- **95% 的记忆**来自自动生成（很少人手动写文档）。
- 调优重点是**生成**（不把一次偏好变成所有人的规则）和**检索**（数千条记忆中正确检索，不污染上下文）。
- 现在允许**编辑记忆**。未来可能重新构建为更像**文件系统**的结构，让代理自主导航。
主持人提到 OpenClaw 的**每日记忆日志**。Walden 考虑迁移到更类似文件系统的方式。

Walden 还分享一个实验：“让代理拥有一个类似于 memory.md 的文件，像永久的 PM 一样管理特定产品的优先级和负责人，甚至自动标记人。这能让 Devin 从工程向上游到创建 ticket。”

Cole 也有类似自动化：“每周自动调研竞争对手并提建议。如果我能拒绝 PR 时更新记忆，就不用再手动更新定时 sync。这是个功能请求。” Walden 笑称可能很快会改动。

## 多代理 vs. 单代理的辩论

主持人将话题转向多代理。Walden 以“**不要构建多代理**”的帖子闻名。他说：

- “一年前多代理基本不可能。现在有一些实验，但你也可以说那些只是**工具调用**：例如创建子代理去搜索文件，然后返回结果。这有上下文管理好处——子代理的消耗和调用折叠成答案。Devin 对 DeepWiki 就是这样做的。但这更像是工具调用，而不是两个协作的代理相互辩论。”
- “让我最看好多代理的地方是：**Devin 有时会反驳我**，说明它能在不同信息下表达不同观点。这使多代理世界成为可能——两个代理不再是全盘附和的‘yes-men’。”
- “我们完全可以提示代理更对抗性。但到目前为止，最实用的方式仍然是**单个引擎拆解工作**，让多个 Devin 在隔离的盒子里并行，不共享机器以减少冲突。距离‘人群智能’还有距离。”
主持人提到 **Cursor** 一篇文章（Wilson Lynn 从单代理到多代理的实验），Walden 说那正是 Devin 目前的做法。他认为那篇文章将来会有一个“但是”后续。

主持人问到 **slop cannon**（松散单代理瓶颈）方法，引用 Ryan Lopo。Walden 认为松动单代理瓶颈很重要，但还没有人真正解决（Ryan 声称已解）。主持人感叹 Devin 有时慢，希望一键并行。Walden 分享内部实验：尝试整产品用 AI 自动写代码、自动合并、无代码审查，看能撑多久。**12 月时最优能撑两周**，之后会出现“改颜色需改 10 个地方”等问题。于是重新加入代码审查以确保可扩展性。

Cole 补充：“如果你不审计代码，**代码库会退化到最差工程师的水平**。AI 会参考已有的模式，并指数级增长这些混乱。需要定期清理重复代码。”

主持人建议在模块之间设立严格的契约，由架构师控制。Walden 同意。

## 基础设施的细节：从文件系统到 GPU

Walden 分享了一个早期教训：Devin 构建早期，在 VM 上 `grep` 很慢。当时没有 AI 可问，他们手动写代码排查。Infra 团队发现底层使用的是**网络文件系统**（如 S3），所以每次 grep 都是网络调用。解决方案是换掉网络文件系统。他们还构建了**块差异文件存储格式**——智能保存磁盘的 diff，使 VM 启动时间大幅缩短（匹配修改量）。这个系统现在已被更新系统取代。

主持人提到一篇关于虚拟机的文章，Walden 确认那就是他们做过的工作。

Cole 谈到 Open Inspect 的沙箱提供商：**Modal** 是主要提供商（支持快照和 GPU），**Daytona** 由贡献者添加，**E2B** 在路线图上。主持人问客户偏好，Cole 说大多数用 Modal，因为其容器方案很自然。

## 使用案例：SRE、PM、客户支持

Cole 总结三大典型场景：

1. **SRE 场景**：从 Slack 或 Datadog 触发警报，代理作为第一响应者，收集上下文并直接提交 PR。
1. **非开发者使用**：PM 通过 Slack 直接创建 PR，无需经过 issue 流程。
1. **客户支持**：客户问题被纳入 Slack，代理可搜索日志和数据库，提供完整上下文并自动化解决。
Walden 补充：

- **持续安全扫描**和**自动分类**也是大场景。
- “我们希望从警报进入的那一刻起，Devin 就开始分类。用户可以在 Slack 中与 Devin 交互，无需进入其他工具。CLI 对非技术人员不友好，但聊天界面简单直观，支持人员、销售都可以用。”
主持人问预算参考。Cole 说通常每个工程师 **$1,000 到 $5,000/月**，也有更高的。Walden 预计未来会有非常昂贵的旗舰模型，也会有足够好的子旗舰模型混合使用（类似“smart friend”）。

## AI 编码的“Slop”与治理

讨论 AI 编写代码时常见的“slop”模式：

- **Reward Hacking**：AI 不喜欢代码失败，因此使用 `hasattr(obj, attr)` 代替直接调用 `obj.attr`，以便代码不报错。Cole 说他在客户那里已经将此设为 lint 规则（禁止 `getattr`）。
- **过度注释**：Opus 4.7 会给每个函数写长篇解释（包括推理和替代方案）。Walden 表示这可能是方向性正确的（未来自我维护时需要上下文），但大多数人希望减少注释量。计划增加“详细程度”设置。
- **向后兼容**：AI 倾向于保留旧的引用以避免重命名，产生奇怪的 import。
- **未类型元组**：类型 `any` 使用太随意。
