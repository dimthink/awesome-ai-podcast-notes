# AI 时代的 CI/CD 变革：从 Token 消耗到智能体协作——Shopify CTO 谈 Tangle、Graphite Stacks 与 SimGym

## 关于 Token 消耗的迷思：黄仁勋是对的吗？

我确实认为黄仁勋（Jensen Huang）受到了很多不公正的媒体报道。他说：“到年底，我会问那位年薪 50 万美元的工程师：‘你在 Token 上花了多少钱？’如果那人说‘5000 美元’，我会发疯的。”但我觉得这种批评是不应得的。我认为他其实是对的。他的大方向是正确的。

关于这一点，我想说的是——这是我们通过反复试验学到的，非常重要——有两点值得注意：

1. **关键不在于单纯消耗 Token**。你可以消耗大量 Token，但反模式是同时运行太多不互相通信的智能体（agents）。与用更少的智能体高效地消耗 Token 相比，那种方式几乎毫无用处。
1. **建立正确的批评循环（critique loop）至关重要**。尤其是在使用高质量模型时：一个智能体做某件事，另一个智能体（理想情况下使用不同的模型）对它进行批评并提出改进建议。这个过程耗时更长，所以人们不太喜欢，因为延迟增加了——他们必须等待这场“辩论”结束。但代码质量会高得多。

**Mikhail**：我认为 AI 工具本身是相当近期的进展。在 Shopify，我们在这个发展阶段，正在内部开发 AI，构建使用 AI 的工具，并与更广泛的 AI 社区互动，这就像是在一个高速发展的轨道上。所以我们更多地讨论它，只是一个自然的副产品。就在昨天，Andrej Karpathy 发了一条著名的推文，问是否存在一些方法，可以组织你的智能体来存储数据并查找数据，这样你就不必每次都重新研究或丢失上下文。

我半开玩笑地发推说，我们早就做了，甚至有不同的方法。Toby 当然是 QMD 的忠实粉丝，而我更喜欢 SQLite。但重点是，是的，我们是一个非常动态、爆炸式增长的公司，显然必须站在 AI 采用的最前沿。

## 内部 AI 工具采用数据：2025 年 12 月的转折点

**主持人**：你的团队准备了一些幻灯片，我们可以共享屏幕，看看一些令人震惊的统计数据。这里有一张内部 AI 工具采用图表，我们看到的是什么？

**Mikhail**：这是非常有趣的统计数据。它显示的是**每日活跃工作者**（即 AI 工具的日活用户）占公司总人数的百分比，针对不同的 AI 工具。你可以看到两点：

1. **总计（绿色线）已经接近 100%**。现在，如果不深入使用至少一种工具，很难完成你的工作。
1. 另一个有趣的现象是，正如许多人评论的那样，**12 月是一个相变点**，模型突然变得足够好，一切都开始起飞并增长。许多人注意到，小的改进积累成了 12 月时间框架内的巨大变化。
另一个我可以声称的现象是：**基于 CLI 的工具**以及不需要查看代码的工具变得越来越流行。你可以看到各种版本的 Claude Code、Codex、Pi 和内部开发工具都在起飞。

蓝色线是我们的 **River**（内部编码智能体），而需要 IDE 的工具（如 GitHub Copilot 或 Cursor）虽然不是在萎缩，但增长速度没有这么快（红色线）。所以你可以看到，它们没有经历如此快速的增长。

**主持人**：据我理解，基本上每个员工都可以选择使用任何工具，然后你们每天做一个调查之类的？

**Mikhail**：没错。我们的目标是让员工完成工作。你可以使用任何工具，我们实际上为**无限的 Token 提供资金**。我们确实试图控制人们使用的模型，但是从底部而非顶部控制——我们基本上是说：“请不要使用任何低于 4.6 版本的模型。”有些人最终使用 GPT-5.4 特别版，有些人使用 Opus 4.6，各有优缺点，比如是否使用完整的 100 万上下文窗口。但我们尽量劝阻人们使用低于那个水平的模型。

## 如何衡量工程师的 Token 使用？管理视角

**主持人**：我想暂停一下幻灯片，回顾一下。有很多像你一样的大公司 CTO 都在考虑某种 Token 预算。黄仁勋一直在说，如果你一个 20 万美元年薪的工程师每年不使用 10 万美元的 Token，那他就没有充分利用编码智能体。当然，黄仁勋会这么说，但这似乎是一种非常重数量轻质量的方法。有些人认为，这就像用代码行数来衡量工程师质量一样——虽然有缺陷，但总比没有好。你对此有什么管理上的看法？

**Mikhail**：哇，你是在引诱我。这是我最喜欢的话题。如果你允许，我光谈这个就能谈两个小时。我确实认为黄仁勋受到了很多不公正的媒体报道。人们说：“当然，卖蛋糕的人会说我们蛋糕吃得不够。”但我认为这是不应得的。我认为他实际上是对的。他的大方向是正确的。谁也不知道正确的数字是多少。

我想说两点，这是我们通过试错学到的，非常重要：

1. **不仅仅是消耗 Token**。你可以消耗 Token，但反模式是运行太多不互相通信的并行智能体，这几乎没用。相比之下，用更少的智能体高效地消耗 Token 要好得多。
1. **建立正确的批评循环**，尤其是使用高质量模型时：一个智能体做某事，另一个智能体（理想情况下是不同的模型）批评它并提出改进建议。然后第一个智能体根据批评重新做。这耗时更长，所以人们不喜欢，因为延迟增加了——他们必须等待这场“辩论”结束。但代码质量会高得多。
另外，既然你提到了总体预算——就像代码行数一样，现在每个人的代码行数都在爆炸式增长。部分原因是 AI 确实更啰嗦，但部分原因是 AI 可以写更多代码，而且不会累。所以你必须在 **PR 审查（Pull Request review）** 期间设置非常严格的漏斗，否则 bug 的数量会激增。这是一个意想不到的后果：**数量压倒一切**。

我敢说，现在好的模型写出的代码平均 bug 比普通人类少，但由于它们写的代码量如此之大，更多的代码会进入生产环境，所以你有更多的 bug。你必须进行非常严格的 PR 审查，当然也包括自动化审查。在这方面我们花了很多预算。对我来说，真正重要的指标是：**代码生成期间花费的 Token 与 PR 审查期间花费的昂贵 Token（如 GPT-5.4 Pro 或 Gemini 的 Deep Think）的比例**。

**主持人**：我注意到你的图表中没有列出任何审查工具。你们是用 Claude Code 来审查，还是有另一套审查工具？

**Mikhail**：它们正好踩到了我的痛处。因为我展示的图表只包括公开工具。我还没有找到一个好的 PR 审查工具能做到我认为应该做的事情。部分原因是它违背了人们情感上的偏好，也违背了一些公司的商业模式。对于审查工具，你想要运行最大的模型，Claude Code 是不够的。你需要**专业级别的模型**。如果你真的想阻止 bug 进入生产环境，你需要花费大量时间让模型轮流审查，但你不需要一大群智能体。所以你最终会进入一个不同的二元世界：你生成不太多的 Token（实际上很少），但耗时很长，因为这些是昂贵的模型在轮流工作，而不是许多智能体并行做很多事情。这就是为什么我觉得还没有找到好工具，所以我们现在使用自己的审查工具。

**主持人**：很多公司都在构建自己的工具，以满足自身需求。你还有一张关于 **PR 合并增长**的图表，现在是每月 30% 的增长，而不是 10%。估计复杂度也在上升。这就是生产力，因为更多的代码进入代码库，更多的功能被开发。我很好奇 backlog 的情况。我不介意一个专业级别的模型花一两个小时来审查我的 PR，因为我已经经历过人类花一周时间审查我的 PR，我还得一直 ping 他们“嘿，嘿，审查我的 PR”。所以我认为这里有一些权衡。

**Mikhail**：完全正确。一方面，你可以容忍 PR 阶段更长的延迟。另一方面，现在真正的问题不是花时间等待 PR，真正的问题是：由于代码太多，至少某些测试失败的概率上升了。然后你遇到测试失败，必须找到有问题的 PR，将其排除，重新测试。所以**部署周期变得更长了**。因此，就总体部署时间而言，如果你花更多时间让一个更大的模型思考一小时，那就是总的时间节省，因为你不用在测试和回滚部署上花费那么多时间。

**主持人**：完全值得。不要只看个体，要看整体，看整个系统的变化。

## CI/CD 瓶颈：PR 概念和 Git 本身需要改变吗？

我很好奇，PR 的心态和 CI/CD 范式最终是否会被改变。有些人显然想要新的 GitHub，但我甚至怀疑 Git 本身是否是问题所在？PR 的概念是瓶颈吗？你们使用堆叠 diff（stacked diffs）吗？

**Mikhail**：我们使用堆叠（stacks）。我们使用 **Graphite**，与 Graphite 有很多合作。所以我们使用堆叠式 PR。我认为，总的来说，CI/CD 以及与代码仓库的交互，目前显然是我们的主要问题和瓶颈，也是最受关注的问题。

我可以说，我们可能需要一个新的隐喻或一个全新的设计来处理这个新的智能体世界。我还没有看到任何明显更好的东西。我认为现在每个人都在努力保持不沉没，因为 PR 太多了，每个人的 CI/CD 管道都开始嘎吱作响，时间在增加，漏过的 bug 数量在增加。我们必须先稳住局面，然后才能开始思考：在新的世界里，什么可能是完全不同的？我还没看到什么特别有说服力的东西，但很明显，为人类设计的旧事物需要演变成新事物。

## 全局互斥锁与微服务的可能回归

**主持人**：我在想，**合并冲突（merge conflict）** 基本上就是整个系统上的一个**全局互斥锁（global mutex）**。在人类组织中，我们有类似的东西，比如公司晨会。但除此之外，对于我们来说，某种程度上的去中心化、接入一个信息源但有一定损失是可以接受的。

**Mikhail**：这是个很好的观点。因为人类写代码不快，那个全局互斥锁还不算太糟。但一旦你开始以机器的速度写代码，它就变成了瓶颈。那该怎么办？

**主持人**：我不敢相信我会这么说，因为我一直是微服务的终身反对者，我一直认为这是个坏主意。但现在你这么说，也许在新的世界里，**微服务过程（microservice processes）** 会卷土重来。因为你可以用很小的东西独立地发布东西，而自动管理所有复杂性会容易得多。我不知道，我们拭目以待。

**Mikhail**：我读过 Google 的一篇论文，他们有一个**单体仓库（monorepo）**，部署成微服务。另一个我经常思考的概念是 Netflix 的**混沌猴子（chaos monkey）**。能够创建一个健壮的系统，有服务发现，独立的微服务相互发现，可能会有相当多的重复——这就是有机系统扩展的方式，你拥有某种“松弛的健壮性”和重复。

## Tangle：第三代数数据处理系统

**主持人**：让我们回到幻灯片，讨论一下你们正在做的 ML 实验和自动研究训练管道。你更接近这个，因为它算是你的个人爱好。你如何解释 **Tangle** 和 **Tangent**？

**Mikhail**：Tangle 是我声称的**第三代数数据处理系统**，偏向于 ML 实验，但不一定——任何需要迭代、共享和大规模数据处理的场景，你都希望获得最大效率。

通常，你作为一名数据科学家或 ML 从业者，会得到 Jupyter notebooks 或 Python 脚本，处理数据，生成 TSV 文件，放到某个共享文件系统。然后你会发现有缺失值，写另一个脚本替换成 NaN，然后你需要过滤掉机器人（bots），运行一个 LightGBM 模型来移除它们，然后你开始实验，运行多个实验，然后你说“天哪，这个实验更差”，你撤销，却无法回到之前的结果。然后你终于让一切工作了，然后把它扔给生产环境。你在生产环境复制，但事情不工作，有时候你忘了某些特征命名，特征不匹配。然后六个月后你需要重复，因为有更多数据或你想再做一轮，你会问“我到底做了什么？”脚本现在崩溃了，路径变了，你只能花一个月做自己的数字考古学。

现在把这个乘以很多团队。想象你有一个实习生要上手，你得给他看“看，这里有文件夹，有脚本”，然后 Claude 智能体做些事，然后你说“啊，是的，我忘了告诉你我其实有另一个东西，我自己也忘了”。这就是我们作为数据科学家、ML 从业者或任何数据处理人员的日常。

**主持人**：我在对冲基金做量化金融时做过这个。在 Airflow 出现之前，我们这么做。后来 Airflow 来了，再后来是 Dagster。在我看来，我会用 Dagster 来解决这种问题：你需要物化资产并创建一个管道。

**Mikhail**：这是一个很好的过渡。Airflow 很好，但它更像是在生产环境中按计划重复运行某个东西。它不太适合团队开发、共享、抓取标准管道并说“我想改变这个小组件”，然后处理海量数据，运行 10 个实验，做超参数优化。这些在 Airflow 中很难做到，但在 Tangle 中很容易。

**Tangle 的核心是**：一群人运行实验（现在可能还有智能体），便宜地运行，协作，共享结果。你不需要完全理解。你抓取、克隆别人的实验或管道，运行，改变一小部分，运行，让它达到生产状态，然后一键发布。你不需要把它移植到任何其他系统中在生产中运行。你可以直接运行同一个实验。它完全准备好投入生产。

它有很多特性：

- 基于**内容哈希（content hashes）**。即使版本变了，但如果输出没变，就**不会重新运行**，非常高效。
- 如果多个人开始实验，相同的数据预处理不会重复多次，**自动只做一次**。
- 如果你启动 10 个实验都需要相同的数据准备第一步，你不需要协调，不需要知道别人也在研究它。
- **非常容易组合**，任何语言都可以使用。
- **非常可视化**，你可以立即看到，轻松编辑，甚至用鼠标点击就能组装小东西。
- **共享和克隆**，每个人都知道。
- 它是**完全静态的**，第二次重新运行会得到完全相同的结果，你永远不需要做数字考古学。**完整的版本控制**也在其中。
**主持人**：它是开源的，可以去 GitHub 仓库看看。还有一个很好的博客文章。最吸引我的是**从开发到生产的过渡**，很多人还没有真正解决这个问题。我们在 Python notebooks 中开发得很好，但那显然不是一个生产就绪的过程。任何解决这个问题的方法都很有吸引力。另外你提到的**基于内容的缓存**，节省可能如此之大让我惊讶——也许我只是没在你那种规模上工作过，那里有如此多的重复，人们仅仅因为上游改变了一个 ID 就重新运行。

**Mikhail**：是的，但主要的节省来自：你运行了它，完成了工作，然后离开了。然后另一个部门你都不知道存在的某个人运行同样的任务。在大多数组织中，你甚至无法发现这件事，所以你甚至无法衡量你正在两次花费时间。但是在 Tangle 上，这一切都会被自动检测到，如果输出相同，对那个人来说，实验就会突然“跳”到前面。这就是**网络效应**：多个人互相帮助。这是从一开始就被设计为平台，而不是单个开发者工具。

## Tangent：自动研究循环

**Mikhail**：**Tangent** 基本上是一个**自动研究循环（auto-research loop）**，可以帮助你做工作。Andrej Karpathy 最近用自动研究让它流行起来。我们基本上是把同样的能力带入 Tangle，这样 Tangent 可以分析、运行多个实验，找出可以改变的东西，并持续重新运行、修改，直到最大化某个目标或损失函数。

总的来说，如果你在你做的**任何事情**中不使用类似自动研究的方法，你就在错过机会。我们在 Shopify 看到，它像野火一样蔓延。任何可以放置测量指标的地方，都可以做得更好：

- 我们 HTML 模板化的速度、UX 模板化、减少 Liquid 主题的延迟。
- 我们的搜索，最近从 800 QPS 提高到 4200 QPS，质量不变，仅通过优化（不是研究循环，它不断运行和更改代码），在相同数量的机器上提高吞吐量。
- 我们改进了 gisting（提示压缩技术）和机器学习过程的质量。
- 甚至存储减少：智能体会去找出明显是衍生的数据集，你不需要存储两次。我们尴尬地发现，最大的表之一是将随机 ID 哈希到另一个随机 ID，我们实际上只需要一个。
**主持人**：所以它可以访问代码，可以检查“这到底在做什么？”

**Mikhail**：它可以在两个层面上运行：

- **表层**：使用现有组件重新洗牌，比如抓取 XGBoost、PyTorch 模块和其他工具，组合它们。
- **更深层**：由于 Tangle 从根本上都是 CLI 调用和 YAML 文件，它可以分析代码，创建新组件，并持续迭代。
所以你可以快速修改现有管道（使用已预制的组件），也可以创建新组件并迭代。自动研究是过去两个月我最兴奋的事情，它像野火一样蔓延——几乎每天每分钟我都会收到某人的 Slack 消息说“看，我把它做得有多好”。

## 自动研究的民主化：PM 成为最高使用者

**主持人**：这在某种程度上是民主化的吗？是只有你的 ML 工程师和研究人员在做，还是普通的 PM 和软件工程师也能使用 Tangent？

**Mikhail**：这是个很棒的问题。Tangle 和 Tangent 是**极度民主化的工具**。最初由 ML 和 AI 工程师使用，但后来正如你所说，PM 成了最高使用者。我们的一位 PM（Sag）在使用量上排名第一，因为他精力充沛且知识渊博。现在它解锁了很多能力，你不需要手动更改代码了。

**主持人**：因为它基本上把 ML 工程师从过程中剔除了。PM 拥有领域知识，能从第一性原理思考“我想要什么结果”，他们甚至可以访问需要输入的数据。所以在某种程度上，这是我们一直想要的用于训练和爬山（hill climbing）的魔法黑盒。它就像是你的 AI 开发场景的 Claude Code——你不需要确切知道算法如何工作，只需带来你的领域知识、专业知识和产品知识，在 Tangent 中迭代，直到得到你需要的结果。

**Mikhail**：在我之前的角色中，每次有人推销 AutoML，我总是说“这行不通，它会失败”。不知怎的，**现在它起作用了**。答案大概是我们现在有了 LLM，而且它足够好。这是一个涌现的特性，我们可以做自动研究。但令人不太满意的是：为什么我们以前没这么做？我们只是做参数搜索之类的东西。

贝叶斯优化和超参数优化曾经非常活跃，这 incidentally 也内置在 Tangle 中。我认识 Patricy Mart 很好，他一直是 AutoML 的倡导者，花了很多职业生涯试图让它民主化，但没有 LLM 时确实很难。你只能在某些狭窄领域有灵活性，很难推广。现在有了 LLM，突然就像一根魔法棒，每个人都不够用了。

**主持人**：我认为是多方面的原因。LLM 可以非常好地做监控（非常潜在无界、非结构化的），可以非常好地做分析，基本上在每个步骤都注入了更多的智能。也许 AutoML 本身没有结构性变化，只是更智能、更非结构化。

## 自动研究的局限

**主持人**：你遇到过什么缺陷吗？每个人都在喝酷爱饮料（Kool-Aid）：“节省时间”、“性能提升”。有什么问题浮现出来了吗？

**Mikhail**：这真的很酷，但它**肯定不是解决所有世界问题的方案**。局限性通常是那些……这有点进入主观领域。我只能分享我目前看到的，当然情况在不断变化。

到目前为止，自动研究非常擅长做**你** **没有带宽做**、**没有注意到**或**不知道的标准实践**这些显而易见的事情。它**不擅长做完全超出分布的事情**——那些你需要思考好几天、做完全不同常规的事情。

我做过一个实验，在一个个人爱好项目上让它运行了几周（全生产规模，运行很慢）。最终它运行了 400 多个实验，**只有一个成功**。我觉得“好吧，那还不错。”

**主持人**：但它节省了时间。

**Mikhail**：是的，它节省了时间。如果我自己做 400 个实验，我的命中率肯定会高得多。但首先，做 400 个实验要花我三年时间，而且我不需要做。机器只花电费。我得到了一个改进。老实说，当我开始那个实验时，我的想法是“去证明 Andrej 只是不知道怎么优化，我超级聪明，因为我的问题已经优化了很多年，我完全改进了，我不期望自动搜索找到任何东西”。但它确实找到了。所以我没有取笑 Andrej，反而成了一个大大的支持者。

## SimGym：客户模拟与 Shopify 的护城河

**主持人**：让我们进入 **SimGym**。这是我感到惊讶的事情之一，显然它也是你们最受欢迎的发布之一。有一小群人试图做客户模拟（比如 Sim AI、Joon Sung Park 的 Smallville）。很多人可能还不太信任这个，因为他们觉得“显然它们只会按提示做”。请告诉我们它的灵感来源。

**Mikhail**：这正是我想说的。如果你没有历史数据，你只能在真空中提示智能体，它们会精确地按你的提示做。事实上，当我第一次提出时（这算是我最初的脑洞，我可以自夸一下），连 Toby 都说：“但它们只会重复你告诉它们的东西？”我说：“是的，但 **Shopify 拥有数十年的历史数据**，记录了人们如何做出改变，以及这些改变对销售的影响。”

所以我们能做的是：我们拥有这些数据，虽然嘈杂，但如果你将所有数据聚合、去噪、应用校准过滤，你可以提取出非常清晰的信号，然后优化你的智能体。这花了将近一年的时间，我们一直在摆弄。我们内部的目标是达到与“加入购物车”事件的**相关性 0.7**——也就是说，如果我们运行真正的 A/B 测试，它应该能复制人类所取得的成功或失败。这花了很长时间。我认为这是不容易复制的，因为还有谁拥有这些数据？你需要拥有几十年的历史数据。

另外，你需要基础设施和规模。因为要获得统计结果，你需要运行大量的模拟、大量的智能体，这很昂贵。你需要在浏览器中执行操作，因为你想获得真实的摩擦感，你想看到人类会看到的图像，以便检测效果——比如“我把图片放大，会卖得更多还是更少？”通常人们的直觉是：图片更大，看起来更漂亮，销售会更多。但实际上销售往往会下降。所以你需要摄取视觉信息，在模拟的浏览器环境中运行，在一个大规模的农场（farm）上，当然还需要非常好的多模态模型。

所有这些花费了很长时间。我分享一点个人的失败：我们一直有大公司的偏见，我们总是做实验，做出改变，运行实验，看哪个更好，大多数是更差的，所以丢弃，继续爬山。我们认为小商家得不到统计结果，他们真的无法运行实验，因为一周内没有足够的数据。但我们没有意识到的是，大多数人没有 A 和 B，他们只有一件事，他们需要建议 A 和 B 应该是什么。所以我们首先构建了“我们为两个团队运行模拟，说哪个更好”，然后我们把它演变成最近刚发布的：当你只有你的网站、你的主题时，我们运行它，然后告诉你“这是预测的转化值，我们认为你应该这样修改来提高转化率”。

然后回到你一开始说的：**证明在于结果**。如果我们不与现实相关，人们就不会使用它。值得庆幸的是，我们每天看到的使用量都比前一天多。所以现在我担心的是如何支付这一切——主要问题是如何优化 LLM，做蒸馏，如何更便宜地运行无头浏览器（headless browsers），以便能够容纳流量的增长。

**主持人**：作为股东，我认为 Shopify 的股东应该深刻理解这一点，因为这基本上就是**护城河**。你使用 Shopify 越多，你就会自动地变得越好。它在为你做工作。

**Mikhail**：是的，这就是我们开始的初衷。否则，如果你是一个没有数据的初创公司，我不会这么做。就像你说的，你在提示中说什么，智能体就会做什么。

## 关于遍历性与反事实建模

**主持人**：统计学家在我体内想要满足某种统计直觉。我想到的词是**遍历性（ergodicity）**。假设一个客户走这条路，另一个走那条路，另一个走另一条路。对我来说，SimGym 可能在做的是，它可以建模中间的那些路径，这些路径可能依赖于之前的状态。这像是一个强化学习的结论：如果你做了天真的 A/B 测试，你只在某个时间点有统计数据，并且只根据总体统计数据进行判断，但在这里你可以实际建模轨迹。这有道理吗？

**Mikhail**：完全有道理。内部我们有一个基于 **HSTU** 的系统，建模整个公司及其可能的路径。在任何时间点，你可以建模用户的行为，或者你也可以将整个商家作为一个在世界中行动的实体来建模。然后你可以在图中做**反事实（counterfactuals）**：中间某处有一个干预（比如我给那个人一张优惠券，或一张感谢卡，或一个折扣），然后你可以从那个反事实向前滚动（forward rollout），看看有或没有干预会发生什么。你甚至可以改变干预在时间上的位置。

我们在 Shopify 规模上为商家做这个。如果我们发现某些可以修复的东西（一个强烈的反事实），我们有 Shopify Pulse，他们会得到通知：“我们认为你的加拿大销售有问题，看起来配置错误，这是你需要做的”，或者“你需要用这些参数设置这个活动”。我们也在买家层面做这个，实际上提供折扣、返现或其他东西给买家。

这让我非常兴奋——这是我感兴趣的领域。能够建模像人类或公司这样复杂的东西，并在其上建模反事实，你可以在未来进行干预，优化何时干预、什么类型的干预——这是一个以前完全不可能的解锁。以前人们一直梦想着，但如果没有 LLM 或类似工具，你怎么模拟呢？非常激动人心的时代。

**主持人**：A/B 测试做不到这一点，因为它没有随时间变化的随机性质，也没有“到此为止的所有上下文”这样的上下文。

## Liquid 神经网络：非 Transformer 架构的实用化

**主持人**：你可以在 **Liquid** 和 **UCP** 之间选择，我们聊哪个？

**Mikhail**：Liquid。大家都很熟悉 Transformer 架构。很长一段时间以来，有一个竞争的架构叫做**状态空间模型（State Space Model, SSM）**。Chris Ré 是先驱之一，很多初创公司试图让它们成为现实。SSM 有显著的好处：速度更快、占用更小、长度不是平方复杂度（实际上是线性的）。但 SSM 从未完全成功，它们只在某些利基市场中表现出色。混合架构很有用，但从未完全成功。

**液体神经网络（Liquid Neural Networks）** 可以被看作是 SSM 的下一步，是 SSM 的平方。它是一种非 Transformer 架构，比 SSM 更复杂，老实说很难调用，但它非常高效，在上下文长度上是次平方的。它以非常紧凑的方式表示事物。Liquid AI 公司的目标是将其产品化。

当你需要**长上下文**、**小模型**和**低延迟**时，你经常有这种需求。总的来说，它与 Transformer 相当，如果你与 Transformer 混合使用，甚至更好。这就是为什么在 Shopify，我们尝试了多种模型、多家公司，发现在低延迟应用（需要低延迟或长上下文长度）时，**Liquid 是最好的**。

我们仍然使用整个模型动物园，测试所有开源模型，甚至有时是每个私有模型。但 Liquid 在 Shopify 内部占了不少份额。我兴奋的原因是，它是**我发现的唯一真正有竞争力的非 Transformer 架构**。我们用它来搜索、长上下文蒸馏等。

**主持人**：我惊讶的是你们已经在内部积极使用它。你们在优化什么约束？是 1B 大小？什么延迟约束？什么上下文长度？

**Mikhail**：SSM 本质上具有**无限上下文长度**，因为状态嵌入了所有之前需要的信息。但问题是它们的表达能力不够。Liquid 实际上是增强版的 SSM，表达力更强，也更复杂。

我们使用它的场景：

- **超低延迟**：我们与 SentML 和 Liquid AI 合作，运行一个 3 亿参数的小模型，端到端 **30 毫秒**用于搜索。当你输入一个查询时，我们产生所有可能的查询含义（不仅仅是同义词，还有完整的查询理解树），包括个性化。延迟要求非常严格。我们用了很多努力与 NVIDIA 合作优化，因为当前堆栈中几乎所有东西都不是为低延迟设计的。
- **最大吞吐量**：用于离线分类，比如新产品出现时，我们需要分析、分配到分类法、提取和规范化属性、聚类。这是一个几乎无限量的问题，有数十亿产品。你不在乎延迟，它是一个夜间批处理作业，但你想最大化吞吐量。我们使用 70-80 亿参数范围的模型，从我们能找到的最大模型蒸馏到 Liquid 中，用于特定任务（如目录制定或 Pulse），然后在大规模批处理作业中运行。它经常在这种情况下击败 Qwen。
它不是万灵药，也不是前沿模型（不会突然与 GPT-5.4 竞争），但它是**蒸馏的绝佳目标**，随着 Token 使用量的爆炸，这变得越来越重要。

**主持人**：你认为给 Liquid 一千亿美元，它是否只是规模问题？什么阻止它遇到与 SSM 相同的问题？

**Mikhail**：它们的规模已经比我所知的最大 SSM 大得多。SSM 就是不够表达力。Liquid 模型，特别是与 Transformer 混合的形式（像 Mamba 的方式），可能是我所知道的最好的架构。当然，Liquid 在计算资源上没有 Anthropic、Google 或 OpenAI 的规模。但如果它们有类似的计算水平，它们会非常有竞争力，甚至可能击败最大的模型。对于小模型和蒸馏场景，它们是首屈一指的。我们非常杂食，纯粹基于性能，一旦它们不再有竞争力，我们会切换到其他东西。但目前，如果画一张图，Liquid 在我们工作负载中的份额相对于 Qwen 肯定在增加。
