# ⚡️ 马特·波科克（Matt Pocock）：为什么工程基础在当下更加重要

## 从 Total TypeScript 到 AI Hero 的创作者之旅

**主持人**：我们最初建立联系的时候，你正要开始你的创作者之旅和开发者之旅。你创办了 **Total TypeScript**，做得非常好。而现在你在运营 **AI Hero**，并且快要完成一门 **Claude Code** 课程了。

**马特·波科克**：是的。如今我做以两周为一期的课程，因为 AI 发展得太快了。要发布一门自定进度的课程并保证永久更新是很困难的。所以没错，现在正是我这门两周课程的最后几天。

**主持人**：太棒了。那是不是意味着，就在你开课的时候，Claude Code 的源代码泄露了？

**马特·波科克**：是啊。你知道那感觉如何吗？我就像是受了诅咒一样。因为我之前做过一门关于 **AI SDK** 的课程，是基于 **AI SDK V4** 的。结果在开课第二天，他们就发布了 **AI SDK V5**——一个巨大的破坏性更新。而这次，我想是在开课第二天，他们发布了百万 token 的上下文窗口……也可能是在开课之前。然后现在，Claude Code 的源代码又泄露了。

不过幸运的是，这门课程其实并不是真的在讲 Claude Code。我在构建它的时候就注意到这一点了。它主要讲的是**软件工程基础**——这也是我在这里要做的演讲。这正是我现在的关注点：回到那些经典的老文本，找出哪些是有价值的，并把它们翻译到 AI 时代。

## 为什么 YouTuber 能成为出色的会议演讲者

**主持人**：所以大家可以去听你的演讲。你还做了一个 workshop。你是我们三位做了双倍场次的人之一。因为你是伦敦的大人物，我想确保大家有足够的机会接触到这些内容。我常常觉得，很多会议不敢让同一个演讲者讲太多场。但我有一个特别的论点：**YouTuber 能成为特别出色的会议演讲者**，因为他们靠说话为生，一直在讲。我叫你 YouTuber 的时候你会觉得尴尬吗？还是说这只是虚荣心作祟？

**马特·波科克**：我其实……我的定位是——我是一名**老师**，我真的很努力地……有一整类 YouTuber 是我并不想互动也不想成为的，就是那种“评论员”（pundit）。我不是那种试图预测未来的人。我只是在努力教学。

**主持人**：你不会每两周就换一个最喜欢的观点。

**马特·波科克**：没错。不像其他某些 YouTuber。［笑声］我是说，那种内容确实表现很好，很难不落入那些陷阱，因为那些数据真的很漂亮。但是，是的，我在努力教那些**持久的、有道理的、能让我自己感觉良好的东西**。

## 核心论点：代码依然重要——甚至更加重要

**主持人**：工程基础和所有这些。那么，你内心深处最珍视的基础是什么？

**马特·波科克**：我会在这里稍微复述一下我的演讲内容——给个小小的预览，大家之后可以在网上找到完整版。我的论点是：现在有一种观点认为，**代码已经不再重要了**。我们处在一个你不需要真正思考代码、也不需要在构建应用时把代码记在心里的时代，因为你只需要用英语来描述，然后让 AI 像编译器一样把它转成代码。你不需要关心代码本身，它只是一个编译目标。

但我发现，**每当我尝试这样做、每当我试图忽略代码的时候，最终都会得到一个可怕的烂摊子**。

于是，为了尝试优化这个“编译器”，我回到了那些经典的著作——比如《**极限编程**》（Extreme Programming）、《**程序员修炼之道**》（The Pragmatic Programmer）、《**软件设计哲学**》（Philosophy of Software Design），还有**领域驱动设计（DDD）** 等等。我在那些书里发现的是：哦，我完全可以把这些东西直接放进我的提示词（prompt）里。

然后我开始越来越深入地探索，意识到：“好吧，这里面的东西比我想象的要丰富得多”。而且，**所有我们为人类而做的那些事情，对 AI 也同样有效**。实际上，把代码放在心上，对你正在构建的东西保持高度意图——不一定是要审查每一行代码，但仍然要对架构和模块有所理解——这真的会带来回报。因为：

- **如果一个代码库对人类来说易于修改，那么对 AI 来说也会易于修改。**
- 这意味着你可以工作得更快。
- 因为一个对人类来说难以修改的代码库，对 AI 来说只会更糟。
## 窄腰架构（Narrow Waist）与深模块（Deep Modules）

**主持人**：我最近一直在思考“**窄腰**”（narrow waist）这个概念。我不知道你有没有听说过，它来源于古老的互联网架构哲学。你可能听说过七层模型，窄腰指的是第三层和第四层——基本上 TCP/IP 和 HTTP 主导了那里，除了 WebSocket 之外几乎没有其他替代方案。

我认为，**在代码库中引入窄腰的概念，可以有效地控制“垃圾输出”（slop）**。你可以说：“我不关心这里面发生了什么，但进入这里的由我定义，从这出去的也由我定义。你在里面随便怎么做都行。”

**马特·波科克**：是的。在我的脑子里，这和以前 MVC（Model-View-Controller）的分解方式有一些对应关系。以前人们叫它“模型-视图-控制器”，今天我管它叫“模型-视图-爪”（Model-View-Claw）。我在运营 AI Hero 的过程中也看到了很多类似的东西。AI Hero 现在是一个 9 个人的公司，每个人负责这头大象的不同部分，我必须确保大家能协同工作，产出一个让任何人都看不到底层粗糙边缘的会议。

所以，部分原因是……我管着网站、管着演讲者、管着内容。但所有的东西都必须从我这里流出去，再到——“你需要什么音视频？你需要什么酒店？你需要什么其他的住宿安排？”——这其中有着巨大的协调难题。我甚至不觉得这还算是编程了，这完全是**系统**，以及你如何建立这些系统。

**马特·波科克**：你说的这个——在任何系统中拥有自包含的“盒子”的想法，让我们以代码库为例——这是个很迷人的想法。**约翰·奥斯特豪特（John Osterhout）** 在他的《**软件设计哲学**》中谈到了这一点，就是“**深模块**”（deep modules）的概念——用简单的接口实现大量的功能。有点像“端口与适配器”（ports and adapters），对吧？你有一个端口，它与其他端口通信，等等。

你说得完全正确：当你有一个这样的代码库或系统设计时，你可以设计接口，对此非常有意，并且作为人类去理解它，然后**把实现——盒子内部发生的事情——委托出去**。这就是我使用 AI 进行编码的最佳方式。我猜想，对于你所做的——把它放到一个组织中，那也会非常有效。

## 领域驱动设计（DDD）与通用语言（Ubiquitous Language）

**主持人**：我在两端都看到了这一点。我也为高盛（Goldman Sachs）和 Devin 做咨询，当你面对数万名工程师时，情况完全不同。看到这些真是件美好的事情。你提到了**领域驱动设计（DDD）**。我是那种年纪大到经历过那个时代的人。我们昨天实际上刚有一个来自宜家（IKEA）的人做了 DDD 的演讲或 workshop。你觉得 DDD 会流行起来吗？它是否足够重要，以至于我应该更多地推广它？

**马特·波科克**：我认为 DDD 对于架构师和处理大型系统的人来说是一个非常吸引人的概念。大多数近期才开始软件之旅的人可能从未接触过它。这是一个很迷人的想法，因为 DDD 是一套**构建块和一套实践**，它们都整合在一起。由于它已经存在了很长时间，它已经存在于这些模型的“潜在空间”（latent space）中。这意味着你不需要发明一套全新的术语，你只需要挂上这个框架、这个系统——它非常灵活、非常可组合，并允许你以这种方式构建系统。

正因为如此，它非常有吸引力。使用这套**预制的框架来与模型对话**，让模型因为理解这套框架而精确地明白你在说什么，这感觉非常对。而且 DDD 本身就非常关注输出，以及让代码与语言对齐——这其实就是 DDD 的本质。你本质上是在尝试创建一个**统一的领域模型**，这样——我现在是这么看的——这样 **AI 和你就能够使用同一种语言进行沟通**。

**主持人**：太对了。这真的、真的非常有用。

**马特·波科克**：举个例子来说明一下。我在 **mattpocock/skills** 上发布了一组技能（skills），这个仓库非常火，有大约 1.3 万颗星。其中一个技能就是“**通用语言**”（ubiquitous language）技能。通用语言本质上是一个概念，你创建一个文档，明确定义你正在讨论的术语。

假设你是一位客户，而我是为你构建系统的人。当我说“mole”这个词的时候：

- 我们可能正在讨论一个健康应用，研究的是“痣”（mole）；
- 或者我们可能正在讨论间谍，需要理解“内鬼”（mole）在那种语境下是什么意思；
- 或者可能是动物——我们可能正在构建一个动物园应用，指的是“鼹鼠”这种动物。
任何在成熟项目上工作过的人都知道，项目会伴随着一堆晦涩的行话。所以我有一个技能，它可以扫描你的代码库，找出所有这些术语，并与你一起精炼它们。我发现，**每当我在写提示词时，我都会打开这个文档**，因为它意味着我可以更清晰地进行提示，与 AI 进行更好、更高质量的对话，因为它正在使用**我的领域模型**。

**主持人**：你会把这个文档放在你的 `AGENTS.md` 或 `CLAUDE.md` 里吗？还是说它只供你自己参考？

**马特·波科克**：我会在 `AGENTS.md` 中放一个对它的引用。对于放太多东西进去我其实非常谨慎。但 AI 知道它在那里，如果需要查看什么就可以看到。但大多数时候，当 AI 在仓库里搜索时，它会使用那些术语，然后那个文档就会出现在搜索结果中。

## 揭秘：如何成为一名优秀的AI教育者

**主持人**：更广泛地说，关于 AI 趋势，我认为你在解释和教学方面真的非常出色。你是怎么做到你做的这些事情的？你的视频表现那么好。我觉得英国口音肯定有帮助。

**马特·波科克**：［笑声］

**主持人**：而且我觉得你在 Twitter 上也一直非常出色。尽管人们说我很擅长这事，但我跟你比起来差远了。

**马特·波科克**：嗯，首先我自己也不太明白这事。但我知道的是，**在成为开发者之前，我做过 6 年的声音教练**。我发现，当我成为一名开发者后——我是从最底层开始的，从初级到中级到高级等等——**沟通能力**始终像是我口袋里一个荒谬地超强的技能，而别人都没有。

我认为在过去几年里，我选择做的事情是**真正地收窄我的焦点**。再说一次，我不是想当一个评论员，我只是在努力教学。这意味着我需要把我的时间分配在**学习这些东西**和**找到合适的措辞来解释它们**上。这也是为什么回头去看那些经典文本如此有用——我现在有了所有这些模型，可以用它们来向人们解释概念。

是啊，我不知道。我不知道为什么我这么擅长这个。大多数时候我并没有觉得自己很擅长，大多数时候我只是觉得**我看到了可以改进的地方**。

## 课程创作流程：从探索到执行

**主持人**：也许你可以带我们走一遍 Claude Code workshop 或者其他你做过的项目。像这样的课程，准备过程是什么样的？你的研究流程是什么？你会舍弃什么？

**马特·波科克**：好的，假设我们有一个“探索”（explore）和“利用”（exploit）的阶段。我先搞清楚这个 workshop 可能是什么样的。我最近的这门课程花了大约 **2 个月**的时间来制作。成片大约是 4.5 小时的经过高度剪辑的视频，但还有大量的练习和大量的内容，大概有 100 个不同的单元。

我从零开始——什么内容值得展示？这个环境里有什么有趣的东西？我的切入点是什么？然后我在一个 **Obsidian** 知识库里做大量的东西，产生大量的想法。本质上就是一个**卡片盒笔记法**（zettelkasten）的实践，如果你听说过的话。

这些零散的笔记会开始凝结成一个计划。我实际上还有一个自己构建的自定义应用来规划这些东西。这意味着我可以开始看到各个部分、看到信息的分组。从那里开始，我对每节课进行优先级排序（P1、P2、P3），根据它们对课程流程的重要性来决定。然后我主要录制所有的 P1 内容，基本上就构成了这门课程。**大量的内容被留在了剪辑室的地板上**，因为我实在塞不进去，或者它们有各种问题。

这种方法，以及发展出这样一个理念——**每节课只教一件事**，**知识之间的依赖关系必须极度清晰**，并且**你必须在不 overwhelm 学员的情况下持续挑战他们**——这大概是我长期做这件事积累下来的本能。只要你脑子里保持这个想法，就很难犯大错。

## 教育方式的演变：知识、技能与智慧

**主持人**：你感觉到人们的学习方式有什么变化吗？因为现在你可以直接问 AI、问 ChatGPT。这大概是一个更广泛的教育学概念：你是给学员更多练习吗？是不是更加动手实践了？

**马特·波科克**：你要教给人们的东西可以分为三种：

1. **知识**：通过讲授（lecture）来教——这是事物如何运作，这些是术语。
1. **技能**：通过互动练习来教——这是如何去做，亲自尝试一下。
1. **智慧**：这是最难教的。对我来说，通过 workshop 中的小组讨论是推荐的方法。但智慧是很难获得的。
实际上，**与 AI 协作这件事，可以通过和 AI 对话来获得一点智慧**。这在以前任何学习机制中都是做不到的。书籍大多是知识层面的，你能从中汲取一点智慧，但你需要自己付出一些努力才能达到。

但我注意到的是，**我越是倾向于用 AI 进行实验性的内容，就越会把人们从我的材料上推开**。我发现实际上，**传统的方法**——给我讲授（lecture）——对今天的大多数人仍然有效。大多数人仍然想用传统的方式学习。

还记得创作者经济浪潮火热的时候，Maven 平台主打互动讨论，但结果证明人们只是想要讲座（lecture）。他们可能想要一点点作业，但很多人甚至连作业都不做。

**主持人**：［笑声］没错。

**马特·波科克**：所以我得出的结论是，我已经找到了最终的形态——那就是只有讲座。

**主持人**：［笑声］

**马特·波科克**：但我在自己的材料上其实非常谨慎，会真正**强迫人们去做功课**。即使在我的 TypeScript 课程中，我也会设计成让你被直接抛到一个问题面前，必须自己解决，然后再给你知识。这种“先问题后知识”的方式感觉非常好。

## TypeScript 与 Python 在 AI 工程中的竞争

**主持人**：有件事我在创办 AI Engineer 时注意到了。我当时相当有意识地保持了 50% Python、50% TypeScript 的比例。但现在情况变了。我认为**今年是 TypeScript 在 GitHub 调查中超越 Python 的一年**。我没有预见到这一点。我认为 Python 更具表现力，尤其是在后端有用。而 TypeScript 则有我们一直以来的依赖问题。你觉得 **TypeScript 会赢吗**？当然，你可能有偏见说“是的”。

**马特·波科克**：我当然有偏见。我的信息回音壁是 100% TypeScript。再说一次，我不是什么评论员。TypeScript 拥有的是**极其丰富的框架和工具生态系统**，还有 Vercel 的 Next.js 和 Cloudflare 的 Next.js……我不是在讽刺，但你不能在任何对话中忽略 TypeScript，尤其是在构建应用的对话中。看起来**所有在构建这类东西的人——所有在构建聊天应用的人——绝大多数都在做 TypeScript**。至少我观察到的是这样。

**主持人**：是的，是的。我认为**TypeScript 可能会赢得 AI 工程**，这是我没有预料到、也没有看到的事情，我甚至不知道如何对此进行定位。因为这对我选择推广或押注哪些框架确实有着很重要的意义。

**马特·波科克**：是的，这确实很有趣。

## 未来方向：超越 AI 助手的框架与方法论

**主持人**：AI 领域还有哪些其他让你特别着迷的话题？你马上就要结束 Claude Code 的探索了，下一步是什么？

**马特·波科克**：嗯，我认为这是我做过的最成功的一门课程。在过去的 18 个月里，我一直在努力寻找自己的定位，努力弄清楚自己适合哪里。我甚至曾押注于“TypeScript 会吃掉 Python”这个想法，为此还做了一个 TypeScript 的评估（evals）框架叫 **Everlight**，但它的开发已经停滞了。你看，人们其实对评估并不感兴趣，**评估（evals）一点也不性感**，现在没有人对做评估感到兴奋。

**主持人**：我们这里会有很多人不同意这个观点。［笑声］

**马特·波科克**：我的意思是，根据我和所有我聊过的人以及我在相关内容上的产出，评估并没有引起什么反响。至少人们现在还不怎么接受它。

**控制反转**（inversion of control）将会变得非常重要——**把更多的控制权交到开发者手中，而不是交给“ harness ”**。

**马特·波科克**：我认为这有一个权衡。以 Claude Code 为例，直到最近你还看不到它的内部。你会觉得 Claude Code 外面有一个“盒子”，你不需要往里看。这很让人安心，因为你会想“好吧，这个 harness 想让我做什么？顺着它的方向走，不要太担心里面发生了什么。”你得到的是**易用性**，但你失去的是**可观测性**（observability）——你失去了控制内部正在发生的事情、调整内部正在发生的事情的能力。

这个权衡对我来说非常有趣。比如，看看 **Pi**，它是由微小的、基础的原语组成的，你需要从零开始完全构建。这意味着，如果你能很好地使用 Pi，你就**拥有对系统的完全控制权**，并且可以**观测到系统的所有部分**。但与此同时，你也有了更多需要维护、需要关心、可能出错的东西，而且这一切都取决于你自己。所以，这个权衡就是我下一步肯定想探索的东西。
