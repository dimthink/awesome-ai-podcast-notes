## 从 C 轮融资与艺术装置说起

主持人：今天节目邀请到 Modal 的 CTO **Akshat Bubna**，一起聊聊。恭喜你们获得 C 轮融资！昨天派对上那些艺术装置太棒了，看到产品被放在底座上，旁边就是罗丹的作品。

Akshat：哈哈，是啊，我们搞了很多艺术装置，把产品放在底座上，旁边是罗丹，那种感觉很有趣。

## 起源故事：不是从 GPU 推理开始的

主持人： Modal 刚起步时并不是一家 GPU 推理公司。你是怎么开始的？请回顾一下起源故事。

Akshat：我第一次见到 CEO Eric 是通过一位投资人。当时 Eric 已经在思考构建一套新的运行时。他的出发点在于：为什么工作流编排产品那么难用？因为必须跑在 **Kubernetes** 上，而 Kubernetes 难以管理，不适合突发性负载，自定义镜像麻烦，开发者体验极差。对于不熟悉这段历史的听众，我们两年期曾采访过 Eric，那里面有更多故事，包括他从 Spotify 离职等细节。我第一次注意到 Eric 是在 Data Council 上他的演讲，主题正是你们后来做的无服务器容器堆栈。那对我来说是“必须认真对待模型”的时刻，但当时还不确定这些是否真的需要用在数据管道上。

主持人：最初你们想的是构建一个更好的运行时，它本身就是一个非常有用的原语：无服务器函数可以解决很多问题，比如 ETL、任务队列、各种突发性处理。同时你们也把它作为原语去构建一系列产品。也许数据工程会先出现，但当时你们也在考虑推理——不过那时还是传统的推理，比如计算机视觉、XGBoost 之类。你们在 ChatGPT 发布前一年就加上了 GPU 支持，只是没想到后来会变得那么重要。

Akshat：是的。关键还是当时的工具没有一个兼顾出色的开发者体验和针对计算密集型工作的支持。我们看到的工作负载需要大量资源，需要频繁向上向下爆发，而 Kubernetes 是为慢速扩缩容设计的，更适用于 Web 服务器。而且工作运行的环境越来越专业化，有时需要加速器，有时需要不同镜像，这是跨很多公司的普遍需求。

## 装饰器模式与“锁定”之辩

主持人：我写过一篇关于自配置基础设施的文章——把所有东西用装饰器放在代码旁边，跟代码放在一起。这在你们早期故事里是否重要？

Akshat：非常重要。我们真不希望用户花大量时间写 **YAML**。你可以把做的事浓缩在代码里，让开发更具表达性和动态性。所以装饰器模式一直是重要部分。

主持人：但有人会说这是 **DSL**、你会被 **Modal** 锁定。

Akshat：我们从未真正被这样推回过。因为 Modal 的优点是你完全可以带来自己的代码，DSL 只是配置层——指定用什么硬件、怎么伸缩，但代码仍然属于你。这一点即使在做推理的现在也一直很重要。

## 从开发者体验到代理体验

主持人：如果今天重新开始构建，开发者体验当然重要，但很多东西已经变了：让代理去做，用 cloud code、codex 实现工具。代理原生原语和传统的自己做很不一样吧？

Akshat：我们其实已经改了 **SDK** 团队的方向，从关注开发者体验（**DX**）转向关注**代理体验（AX）**。我们认为 DX 的好处同样适用于 AX：为什么要让代理去读几百个 Kubernetes 文件、写连类型都没有的 YAML？它完全可以在装饰器里做少量修改，就获得自配置运行时，并能实时看到变更效果。从我们接触的客户来看，代理在 Modal 上比在其他底层上运行要快得多。

主持人：因为你又把基础设施需求和运行它的代码放在了一起。

Akshat：但一个反面的论点是：现在没人看代码了，所以这样做没意义。不过我们仍然认为**可观测性**非常重要。我们很多信息推到 **CLI**，让代理自己根据信息去调查，但你仍然需要人类去解读发生了什么、做出判断。这或许比看代码本身更重要了。

主持人：你可以把代码当作黑箱，通过观察它的行为和输出去提示变更。

Akshat：所以我觉得我们需要一定的克制，不要非要发布一个新原语并专门化，而是保持通用。别人问你“你是做什么的”，你可以回答“我们这也能做那也能做”。

## Modal 的定位：为 AI 应用构建的云平台

主持人：我们想问：Modal 到底是什么？你们做的事太多——沙箱、GPU、各种。你怎么回答？

Akshat：**Modal 是一个为 AI 应用从头构建原语的云平台**。目前它覆盖推理、训练、批处理、沙箱工作负载。我们还在构建更多。

主持人：你没说 Web 服务器，所以那种永远在线、大规模的 Kubernetes 场景仍然有存在空间？

Akshat：绝对有。我们不和 Render 这类公司竞争。我们的差异点在于需要专用计算、需要频繁扩缩容的工作负载，它们形态不同。

主持人：你们跟很多初创公司一起创新，比如博客里提到的 Cognition、Decagon、Ramp。这些公司跟你们共同演进，AWS 不会跟客户这么直接合作。

Akshat：是的，我们团队小，行动快，工程师直接跟客户一起解决问题。我在 Cognition 入职第一周就看到有人穿 Modal T 恤，我问他为什么在这儿。他说他是 Modal 派驻的，因为沟通延迟太高了，所以把一个人放在那里。

主持人：顺便说，我三年前用过你们，当时我在做 Small Developer 工具，受 cloud one 启发。我找到 Modal 做突发计算，体验很好。我还出现在你们的董事会仪表盘上，因为我那篇工具在 Hacker News 上火了，给你们带来了一波流量。对我而言，Small Developer 就是 “protocognition”——回头想想，如果坚持下去，也许就能画出完整的技术树。

Akshat：我当时同时跟一些需要沙箱的客户交流，那还是 2023 年。我们在 2023 年 5 月就构建了沙箱，那时候还没人知道这玩意儿会火。第一个示例是把 Small Developer 放进循环里，让代理自我迭代。

主持人：那时模型的循环和自纠错能力还很差，工具调用也不行，十几次迭代后模型就会发散。但现在回头想，“好吧，跟三年前的我对话，答案就是：收集所有失败案例，构建基准，构建强化学习环境，然后以百亿美元卖给 Meta，再训练个模型以 600 亿美元卖给 Elon。这真是个印钞机。但当时很难有这样的信念。”

Akshat：事后看一切都太明显了。

## 沙箱的爆发与弹性扩展

主持人：沙箱功能在去年才真正爆发。

Akshat：是的，中间有几年平静期。但沙箱的价值一直很明显：在我加入 Modal 之前，有个黑客马拉松认识的朋友，他想运行一个托管不了的小模型。他说 Modal 很好用，把 Hugging Face 链接丢进去，就能启动 GPU 沙箱，用完关闭，几天后再打开它仍然存在。到今天大家依然需要这种能力。

主持人：虽然工作负载形态变了，核心问题一样：从 0 到 1 的弹性，以及大规模如何从 10 个 GPU 快速扩展到 1500 个 GPU。

Akshat：我们最大的用例是**弹性推理**。我们最早的产品市场契合点是为自定义模型做推理，避开了 LLM 领域。我们服务 Sunno（音频）、Runway（视频）、机器人、计算生物公司，他们用自己的模型训练后再部署，而 Modal 是最好的部署后端。这些公司流量模式多变：某一天发布就激增，而且不只部署一个模型，常常跨区域部署多个模型，自动伸缩变得更复杂，还要兼顾每个区域的扩缩容时差。

主持人：有一批推理提供商（Fireworks、Together、BaseTen）已经在这个领域开创了针对语言模型的独特市场。

Akshat：我们真正特长的是**自动伸缩**，这不是每个平台都能做到的。我们还在技术上通过 **GPU 快照**（GPU snapshotting）深度优化，能快速冷启。

主持人：对于强化学习，做 rollout 时需要 10 万个沙箱。这种突发性极高。

## 持续学习与新原语

主持人：有没有看到持续学习的早期迹象？比如你们的一些客户已经开始尝试？

Akshat：我们有幸服务了一些前沿客户，他们用我们的原语做有趣的事情，如持续学习。如果需求更多，我们也会将其正式加入产品。

主持人：继沙箱之后，你们添加的下一个重要原语是什么？

Akshat：我们深入了 LLM 推理，因为我们意识到自动伸缩、区域路由等优势在其他地方还不被提供。我们过去的黑盒模式限制了模型层面的优化，所以现在我们已经能够达到前沿模型性能。我们还把很多工作开源了，比如基于块的猜测解码器 **Dlash**。接下来我们想帮助人们训练自定义猜测解码器，并让草稿模型能随数据分布自动演化。

## 猜测解码详解

主持人：（请解释猜测解码）

Akshat：**猜测解码**（Speculative Decoding）是用一个小模型（草稿模型）一次预测多个 token，然后用大模型验证这些 token。因为如果逐个 token 生成受内存带宽限制，但批量验证草稿模型的 token 能显著提高计算效率。只要草稿模型的 token 被接受（接受长度足够高），速度就能提升 2-4 倍，而且没有质量损失——大模型永远不会接受比它自己差的 token。

主持人：所以你们在 Dlash 上做了更块级的猜测解码。

Akshat：对，Dlash 是预测一个块而非单个 token，我们已全部开源。

## 自动端点：无代码的推理部署

主持人：介绍一下自动端点（Auto Endpoints）？

Akshat：通过 UI 或 CLI 创建一个端点就能使用，内置所有优化（如 Dlash）并完全透明：我们给你代码，你可以自己跑，如果你想，可以随时脱离黑盒进入完整的 Modal 体验。未来我们还希望实现草稿模型自动根据分布演化，无需人工介入。

主持人：如果我自建推理（用某个模型、某个推理引擎、某些 GPU），Modal 额外提供了什么价值？

Akshat：我们开源并向上游贡献，但我们的团队具有显著专业知识——如果现有方案不满足，我们能帮用户优先获得性能。另外，我们的端点更弹性：有真正缩放至零、真正的突发能力。实际上，这对用户比单纯找 GPU 跑模型代码重要得多。

主持人：从自建到 Modal，差别不仅仅是选引擎组合这么简单，背后的权衡不是所有人都一眼能看透。

Akshat：是的，运行生产级推理本身就是很困难的工程，哪怕先搁置自动伸缩，还需要处理尾部延迟、保证每一条请求至少一次投递等。我们一直在创新。

## 你会做什么，不会做什么？

主持人：Modal 正在变成完整的云平台，你开始进入别人的领地。哪些是你们不会做的？

Akshat：我们想跟住用户，提供一切协同良好的平台。目前聚焦**模型生命周期**和**代理生命周期**：从数据准备到训练到推理，再到代理需要部署的后台能力（沙箱、存储、网络等）。我们不和 Render 竞争永远在线的 Web 服务器场景。

## 基础设施演变：从 8:1 到 1:1

主持人：在黄仁勋的 GTC 演讲中我注意到一个点：AI 工作负载的推理拐点——过去 GPU:CPU 比例是 8:1，现在因为代理的频繁阻塞和 CPU 调用，变成 1:1，瓶颈不断在 GPU 和 CPU 之间来回切换，你必须协置一切。

Akshat：是的，这也是 Modal 的吸引力之一——我们有一个跨越 **17 个云提供商** 的容量池，擅长在全球运行各种云容量。

主持人：但你们没有自己的数据中心。

Akshat：没有。我们的差异点在软件层，资本轻使我们行动快。目前很多人在建数据中心，我们与他们有效合作，专注让我们特别的部分。

## 可靠性层与全球容量

主持人：17 个云提供商也包括一些本地小云，可靠性不一。你们怎么处理？

Akshat：我们花了大量时间构建自己的**可靠性层**：如果 GPU 从总线上掉下来，用户工作负载不受影响。这让我们能利用比个体用户多得多的容量。

主持人：这让你可以成为“所有云的超级云”。

Akshat：对，这正是我们的理念。同时协置需求让用户有时需要精确的位置来放 CPU 或 GPU——比如与欧盟的数据同域或追求低延迟。我们都能满足。

## 网络创新：侧车、覆盖网络与 RDMA

主持人：似乎关于网络方面有很多可谈的。你们支持侧车（sidecar）、沙箱的多容器、网络控制、覆盖网络、RDMA 等。

Akshat：对。沙箱现在支持 **sidecar**，一个沙箱可以是一组容器。为了控制出站网络，用户可能需要中间人代理来记录强化学习的日志、控制流量、注入凭证，这些我们已构建。我们还看到用户需要横跨多节点的沙箱间通话，所以支持 **IPv6 覆盖网络**（I6PN），让同一工作空间的 Modal 容器能通过私有 IPv6 地址直接通信。这原是为我们的分布式训练产品做的原语——为一个函数加装饰器就能获得 GPU 集群和 RDMA 网络，运行真正的无服务器分布式训练。但我们看到人们把它用于其他用途。

主持人：有个著名的系统教授 John Osterhout 提交的关于 RDMA 实现的手稿。我怎么也想不到一个操作系统大牛会关心 RDMA。但这反映出一个趋势：如何高效地在 GPU 间移动 KV 缓存或权重，对 RL 等至关重要，这本质上是一个调度和移动内存的系统问题。

Akshat：是的，我们建的覆盖网络是 TCP 层的，因为做 RDMA 密钥交换之前需要它。有人发现了 TCP 部分也很有用。

## 多节点训练与后训练

主持人：多节点训练在 Modal 上是针对什么样的规模？

Akshat：不是大规模预训练，而是很多中小规模的后训练（post-training）——比如人们用中等规模 Qwen 模型做后训练以提升推理质量。这对我们来说是完美匹配。研究者做小规模跑，弹性是最关键的；这实际上是现在 AI 研究的瓶颈：你只需要给你的模型一些 GPU。我们有一篇关于自动研究（Auto Research）的博客提到了 Modal 是很好的基体。

主持人：现在的自动研究到底在什么程度？是科学展览级别吗？

Akshat：现在还是这样。我们内部的训练和推理团队都用这种模式：有个内部叫 “auto inference” 的仓库，代理会自动发起一系列的配置尝试，甚至运行 Nvidia nsight 剖析器，调整配置，最终找到最佳方案。甚至自动更换 GPU 类型。

## 语言模型写 Modal 代码：Modal Bench

主持人：LM 生成 Modal 代码的效果如何？

Akshat：在 Cloud 4 之前不太好，现在能一次生成正确代码。我们在构建 **Modal Bench** 来测试更难的任务：比如在出问题时如何查看日志并更新代码——这需要推理能力。我们根据这些缺口把能力加入到调优技能中。

## 短缺与容量规划

主持人：你们是否面临 GPU、CPU、内存的短缺？

Akshat：高速增长迫使我们做更好的主动容量规划。我们有一个“**计算策略**”团队负责这个工作——包括不同区域、不同 GPU 类型的混合、预测自己的容量、对供应链演变做出判断。良好的单位经济学以及解锁客户价值依赖于此。我们也在构建一个**批处理层**：不关心延迟的用户可以用更低价格，在 24 小时内得到结果。

主持人：这种 API 在前沿实验室中有但似乎不热门？

Akshat：我们看的需求主要不是 LLM，而是计算生物等领域的用户，他们要做很大的批处理任务，不关心何时完成。

## 未来方向：模型后训练、区域路由与代理原语

主持人：下一步的主要方向是什么？

Akshat：为我们的用户打造更易用的原语。比如数千家公司将后训练自己的模型并部署开源模型，我们在构思最佳产品形态，包括从训练 gym 到自动前沿端点。涉及不同的垂直领域：实时音频视频应用，所以我们构建**区域路由**和回退，让 GPU 尽可能靠近用户以降低延迟。在代理端，我们与客户紧密合作，他们需要的东西变化很快——除了沙箱和持久文件系统，还有其他生产代理所需的，我们正在思考。

## 代理权限：硬边界还是 LM 中介？

主持人：代理的权限方面，有些人觉得可以用 LM 来授权（如 cloud code 跳过权限、命令白名单），但也有调侃“LLM OS 核心就是 LLM 本身”。你的看法？

Akshat：你在用目标和循环与洪水做斗争。我对沙箱层用 LM 授权持怀疑态度，因为你需要硬边界。当然，可以和软性措施结合（比如审核和协调）。硬边界永远是需要的。

## 元编排层与竞品关系

主持人：如何看待 OpenAI、Vercel、Databricks 推出的元编排层（meta harnesses）？

Akshat：我们很高兴和 Anthropic 等实验室合作，Computer use 是一个很好的起点。但当你要构建生产级代理（比如 Ramp 的会计代理），你就需要对计算原语有更多控制：如何处理文件、如何快照和恢复、如何控制网络，甚至可能需要 GPU。这时就需要专门的沙箱提供商，这就是我们扮演的角色。我们对编排层不发表意见，不管它跑在哪，都可以用 Modal 沙箱。

## CI 市场与 GitPod / Coder

主持人：如何看待 GitPod（改名为 Coder）？他们有很强的工程实力，卖给了 OpenAI。你研究过吗？

Akshat：我团队和他们聊过，技术非常强。Modal 在 CI 市场也很看好：因为编码代理越来越多，它们会跑更大量的 CI。很多时间浪费在制品准备和依赖处理上。我们通过快照和恢复可以大幅提升效率。

## 应用领域：不止大模型

主持人：你们只服务大模型吗？

Akshat：绝对不是。我们服务药物发现、计算生物、机器人等公司。这让我们更加多样化。我们的目标是围绕好的原语，为所有需要弹性计算的公司服务。

## 与 Replicate 对比

主持人：有一个曾经的竞争对手 Replicate，你们对它的反思是什么？

Akshat：我们一直避免成为模型 API 服务，因为那常常服务于业余爱好者市场，粘性不足。我们致力于服务构建产品的公司，它们需要更多灵活性而不仅是 API。我们的例子从来不是“开启这个模型，给你 API 令牌”，而是代码。用户可以任意定制。如果一家公司要在计算生物等领域构建产品，它们会选择代码层。

主持人：什么情况下会从“API 调用”变成真正的产品？用户加了一些代码，那到底加什么才算有实质？

Akshat：有些公司就是想深入那一层，构建差异化的东西。比如早期 Ramp 训练自己的 tokenizer，用 lava 替换分词器；比如 Cognition 使用自定义模型架构。他们需要在代码级调整，无法通过一个 API 实现。

## 视频生成代理的发展

主持人：我预测未来会出现一种编排视频模型的代理，让视频长度从 10 秒变成 6 分钟。

Akshat：我们已经看到有公司在用我们的 GPU 沙箱做视频操作代理。我原本以为方向是编辑，但没想到或许是视频长片段生产。

## 语言支持：Python 和 TypeScript

主持人：Modal 会支持其他语言吗？

Akshat：Python 是我们的第一个 SDK 语言，但现在也有 Go 和 TypeScript SDK。运行时是用 Rust 编写，不绑定 Python。推理和训练仍是 Python 为主，但代理使用 TypeScript SDK 更多。Python 和 TypeScript 已经主导，我们不会很快拓展其他语言。还有两种语言是英语和 prompt（笑）。

## 结语：从 DX 到 AX，并继续深化

主持人：最后，Modal 用代理体验（AX）取代了开发者体验（DX）作为团队重点。光靠更好的代理体验能建成像你们这样的多独角兽公司吗？是否还需要别的？

Akshat：这是我们身份的核心部分；不只是代理怎么用 CLI，还包括多快能启动新服务、在生产中用起来、迭代时间多短。我们相信这仍会很重要。至于 DX vs AX 是否需要根本不同的做法，我们认为重合度是 0.9。我们主要变化是构建了 Modal Bench 来找出代理的短板，并加入产品支持（比如将日志和指标从 UI 也搬到 CLI 中）。简单如此。

主持人：好的，非常感谢！我看到你们成功的原因：专注且执行力出色。

Akshat：谢谢，我们还有很长的路要走。
