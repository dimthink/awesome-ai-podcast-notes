**开篇引入**

想象一下，你拥有一个智商高达170的天才大脑，它能思考复杂的相对论，能写出优美的十四行诗。但问题是，这个大脑被装在一个玻璃罐里，没有手，没有脚，甚至没有眼睛和耳朵。它能做什么？什么都做不了，只能空转。

这正是当前大语言模型的困境。我们惊叹于GPT-4、Claude 3.5的推理能力，但当试图让它们“做点实事”——比如订一张机票、重构一个代码库、甚至只是整理一下电脑桌面——它们往往会显得笨拙不堪。模型的智能是大脑，但缺少了让大脑与世界交互的“身体”。

这个“身体”，在AI Agent领域，被称为 **Agent Harness**（智能体框架）。最近，Anthropic 的明星产品 Claude Code 源码泄露，为我们提供了一个解剖这个神秘“身体”的绝佳样本。主持人 Koji 邀请到了23岁的连续创业者、谢尔AI创始人来新璐，他既是这个领域的深度实践者，其开源教程 **Learn Cloud Code** 在 GitHub 上已收获超过5万颗星。

这场对话不是一场枯燥的技术讲座，而是一次关于“如何为硅基生命打造躯体”的思想探险。我们将一起拆解 Harness 的三层架构，探讨为什么“Bash is all you need”，理解“Auto Dream”记忆机制，并最终窥见一个“零人公司”遍地开花的狂野未来。

## 01. 为“大脑”披上机甲：重新定义Harness `[对话 00:00 - 05:00]`

**核心观点**

Agent 的上限并非仅由模型智能决定，更取决于其 Harness 的设计。模型是“大脑”，而 Harness 是赋予其行动能力的“身体”或“机甲”。

**深度阐述**

当我们谈论 Agent 的智能时，很容易将目光全部聚焦于模型本身。毕竟，一个笨模型什么也做不好。但来新璐提出了一个更精细且极具启发性的观点：在模型智能达到一定基线后（比如智商120-170区间），真正的差距体现在“Harness”上。

他做了一个生动的比喻：**模型是人的智商，而 Harness 是我们去健身房锻炼、学习舞蹈、修习武术，甚至穿上外骨骼机甲。** 机甲极大地扩充了我们的能力边界，让我们能做到仅凭肉身无法完成的事。对于 Agent 而言，这个“机甲”就是 Harness。

主持人 Koji 抛出了一个核心论断：“Agent 的上限是由 Harness 决定的”，并追问这是否意味着模型智能的提升不再重要。来新璐对此表示“赞成一半”。他明确指出：

> “Agent 的智力的上限提升，肯定还是模型层面的发展嘛……但是我们现在看到大部分模型智力也都比较够用了。” `[对话 03:30]`
这揭示了当前阶段的本质：我们正处于一个模型能力“够用”但尚未“完美”的时期。关键在于，如何将现有模型的潜力通过工程手段最大化地释放出来。一个设计糟糕的 Harness，就像一个糟糕的指挥官，会让一个聪明的士兵（模型）无所适从，甚至做出错误决策。而一个优秀的 Harness，则能让一个普通士兵发挥出数倍的战斗力。

由此，来新璐给出了一个极其简洁且有力的定义：

> “模型以外，都是 Harness。” `[对话 02:50]`
这句话廓清了概念的边界。所有围绕模型，用于让其能够感知环境、做出决策、执行动作并记住经验的系统、工具、流程和架构，都属于 Harness 的范畴。它不再是一个模糊的流行词，而是一个可以被拆解、设计和优化的工程系统。

**个人感受**

来新璐作为一名23岁的创业者，其表达充满了技术理想主义和敏锐的洞察力。在谈到 Harness 时，他眼中闪烁着兴奋的光芒。他并不沉迷于对模型本身的神化，而是务实地将目光投向工程实践，这种“让技术为我所用”的务实精神，是优秀开发者与普通用户的分水岭。

**延伸思考**

这个“机甲”比喻也暗示了一种可能性：未来，同一个模型可以通过更换不同的 Harness，瞬间从一个“代码专家”变成一个“市场分析师”。Harness 将成为模型能力的“应用商店”，不同的场景和任务，对应着不同的、预置好的能力框架。模型的竞争将逐渐从“参数规模”转向“生态适配性”。

**精华收获**

- **重新定义 Harness**：不要把它想得太玄乎，它就是“模型以外的一切”，是让模型能行动起来的工程体系。
- **当前瓶颈转移**：在模型智能相对充裕的今天，构建 Agent 的瓶颈已经从“如何让模型更聪明”部分转移到了“如何为模型设计更强大的Harness”。

## 02. 拆解“机甲”的三层结构：能力、环境与治理 `[对话 14:30 - 19:00]`

**核心观点**

一个完整的 Agent Harness 可以拆解为三层：底层的执行能力层、中层的上下文环境层、上层的编排治理层。

**深度阐述**

为了将 Harness 从一个抽象概念落地到具体设计，来新璐提出了一个清晰的三层架构模型。理解这三层，是理解一切复杂 Agent 系统的钥匙。

**第一层：执行能力层 (Action Layer)**

这是最直观的一层，即“给模型提供什么工具”。模型本身无法直接操作世界，你需要给它“手”和“脚”。这些工具包括：

- **文件系统操作**：增、删、改、查、搜索文件。这是几乎所有 Agent 任务的基础，尤其是编程类 Agent。
- **浏览器操作 (Browser)**：让 Agent 能像人一样浏览网页，获取信息，这是进行深度研究 (Deep Research) 的必要能力。
- **语言解释器**：如 Python 或 Node.js 环境，让 Agent 能够直接运行和测试代码。`[对话 31:00]`
来新璐认为，配齐这三类工具，就能满足95%以上的 Agent 任务需求。

**第二层：上下文环境层 (Context/Environment Layer)**

这是容易被忽视但极其关键的一层。它定义了 Agent 工作的“环境”和“状态”。包括：

- **System Prompt**：给 Agent 的基础人设和行为准则。
- **Skills**：预置的“技能包”，类似于给 Agent 的说明书或最佳实践模板。
- **Memory**：记忆系统。这是最复杂的部分，因为模型的上下文窗口有限，无法记住所有历史。如何让 Agent 拥有长期记忆，并在不同会话间传递关键信息，是这层的核心挑战。
来新璐用一个生动的例子说明了这层的重要性：当你需要多个 Agent 接力完成一个任务（比如设计芯片），上一个 Agent 工作到一半，上下文窗口满了，它必须把工作“交接”给下一个 Agent。这时，它需要写一份“交接文档”，清晰地说明“我做到了哪一步、遇到了什么问题、接下来需要做什么”。这份“文档”的格式、存储和加载机制，就属于上下文环境层的核心设计。`[对话 24:30]`

**第三层：编排与治理层 (Orchestration & Governance Layer)**

当任务复杂到需要多个 Agent（甚至成百上千个）协作时，这层就变得至关重要。它主要解决两个问题：

- **编排 (Orchestration)**：如何组织 Agent 们协作。是串行？并行？还是按照某种复杂的 DAG（有向无环图）结构？比如，设计CPU芯片时，可以同时让多个Agent并行编写不同的功能模块。
- **治理 (Governance)**：如何管理不同 Agent 的权限和职责。比如，负责测试的 Agent 不应该拥有修改代码的权限，否则它可能会为了通过测试而“作弊”，偷偷 hack 掉失败的测试用例。`[对话 27:00]`
**个人感受**

来新璐用“CPU芯片设计”这个高复杂度案例来串讲这三层，非常具有说服力。这表明他对 Agent 的理解已远超简单的“调用API”，而是进入了“分布式系统设计”的层面。他像一个精于架构的工程师，将混乱的 Agent 开发过程，梳理得井井有条。

**延伸思考**

这三层架构与人类社会的组织方式惊人地相似。第一层是“工具和技能”，第二层是“工作记忆和情境意识”，第三层是“管理制度和组织架构”。这或许并非巧合——我们正在以人类社会的运作模式为蓝本，为 AI 构建社会组织。

**精华收获**

- **结构化理解**：提供了一个可操作的框架来分析和设计任何 Agent Harness。
- **分清主次**：别只盯着第一层的工具，第二层的“交接文档”和第三层的“权限治理”往往是决定系统成败的细节。

## 03. Harness的进化论：告别“流程图”，拥抱“CLI与文件系统” `[对话 05:30 - 12:00]`

**核心观点**

基于 Prompt 节点和状态图的传统 Agent 开发范式（如 LangChain、Prompt Flow）已逐渐过时。新一代、Agent-Native 的范式是“Bash is all you need”——为 Agent 提供一个类 Unix 的命令行和文件系统环境，给予其更多上下文和更少的控制。

**深度阐述**

来新璐毫不客气地指出，过去一两年流行的 Agent 开发框架，如 **LangChain、LangGraph** 等，其核心理念已经“越来越不适用”。这些框架本质上是一种 **“基于规则的状态机”** 思维：开发者预先想好 Agent 可能遇到的各种情况，用节点（Node）和边（Edge）组成一张流程图，规定 Agent 在不同条件下如何流转和路由。

这种方法的致命缺陷在于 **“假设模型不够聪明”**。它试图用程序员的确定性逻辑，去框定一个充满概率性的智能体。当模型能力较弱时，这种方式提供了必要的“脚手架”。但当模型能力突飞猛进时，这种脚手架反而成了镣铐，限制了模型自主决策和探索的能力。

那么，新一代的范式是什么？来新璐给出了一个振聋发聩的口号：

> **“Bash is all you need.”** `[对话 10:00]`
这背后是深刻的洞察：

1. **训练语料的天然优势**：Unix/Linux 的命令行接口、Bash 脚本、文件系统，是互联网上最丰富、最标准、最鲁棒的训练语料之一。模型在预训练阶段已经看过数十亿行相关代码和文档，它“天生”就更懂如何在一个类 Unix 环境中工作。
1. **极致的灵活性与组合性**：CLI 工具（命令行接口）遵循 Unix 哲学——“只做一件事，并把它做好”。通过管道 (`|`)、重定向等机制，简单的 CLI 工具可以组合出无限复杂的功能。这比一个专为某个任务定制的 MCP 服务器要灵活得多。
1. **从“控制”到“赋能”**：旧范式是“少一点上下文，多一些控制”（Less Context, More Control）；新范式是 **“更多的上下文，更少的控制”（More Context, Less Control）**。`[对话 39:00]` 你不再需要写死 Agent 的每一步，而是给它一个丰富的环境（文件、历史、命令行），给它强大而通用的工具（Bash, Vim, grep），然后相信它能自己找到解决问题的最佳路径。
来新璐甚至分享了他的亲身经历：他曾使用 GitHub 的 MCP 服务器来管理代码，但发现任务成功率不高。后来他换成了 GitHub 官方的 CLI 工具（命令行接口），效果出奇地好。他分析原因认为，CLI 的使用范式在模型预训练语料中浩如烟海，而 MCP 作为一个全新的协议，模型对其理解远不够深刻。`[对话 52:00]`

**个人感受**

听到“Bash is all you need”这句话时，作为开发者会感到一种返璞归真的震颤。我们曾热衷于构建各种花哨的可视化编排工具，试图用GUI来驯服AI。但最终，可能最原始、最强大、最与AI“心性”相通的，恰恰是那个黑底白字的终端。这是工程哲学的一次胜利。

**延伸思考**

这个趋势将如何影响未来的软件开发？或许未来每个应用、每个服务的首要交付物，将不再是网页或APP，而是一个设计良好的 CLI 工具。开发者们需要学习如何为 AI 而不是为人设计接口。人的图形界面追求直观和美观，而 AI 的 CLI 接口则追求结构化、可预测和语料丰富。

**精华收获**

- **范式迁移**：从“给 Agent 画流程图”转向“为 Agent 构建一个模拟的 Unix 操作系统”。
- **简单即强大**：不要盲目追求新协议和复杂抽象，回归到模型最熟悉、最本质的交互方式（CLI/文件系统）可能才是正途。

## 04. 记忆的“梦境”：Claude Code 源码中的瑰宝 `[对话 38:30 - 45:00]`

**核心观点**

Claude Code 源码泄露揭示了其在上下文管理上的精妙设计，尤其是其“Auto Dream”（自动做梦）机制，通过异步 Agent 定期回顾、整理和修正记忆，实现了长期记忆的自迭代。

**深度阐述**

Agent 的记忆一直是个老大难问题。简单地将所有对话历史塞进上下文，不仅会迅速撑爆窗口（尽管256K已很大，但对复杂任务依然不够），还会导致关键信息淹没在海量噪声中。Claude Code 的解决方案，为我们展示了教科书级别的设计。

来新璐将其总结为两套并行的记忆更新机制：

**1. 实时增量记忆 (The “Stop” Hook)**

每次 Agent 完成一轮任务后，系统会触发一个名为 `Stop` 的钩子。这个钩子会创建一个 **“分叉 Agent” (Forked Agent)**。

- **工作原理**：这个新 Agent 会继承原 Agent 的全部系统提示词和 KV Cache（键值缓存，一种注意力机制的计算缓存，可大幅提高重复推理的效率）。它不负责执行原任务，而是专门负责“复盘”。它会扫描刚刚结束的这轮对话，判断有哪些值得长期保存的信息（比如用户偏好、项目关键决策、学到的新技能等），然后将这些信息以结构化的 Markdown 文件形式，更新到文件系统中。这些文件的头部甚至还有 YAML 格式的元数据（如描述信息），方便后续检索。`[对话 41:00]`
**2. 定期深度整理：“Auto Dream”**

这是更加惊艳的设计。大约每隔一天，当对话会话（Session）超过5个时，系统会自动触发一个后台 Agent 进行“深度整理”。

- **为什么叫“做梦”？** 这个过程很像人类睡眠中的记忆巩固。它不会处理实时信息，而是对过去一段时间（如最近的所有会话）进行“重放”和“压榨”。
- **它做什么？**
这个“做梦”的 Agent 是异步的、低优先级的，它默默地在后台运行，不断地优化 Agent 的“认知地图”。

**个人感受**

这个“Auto Dream”机制充满了诗意。我们不再仅仅是编程，而是在创造一个具有某种“生物学”特征的智能体。它会“休息”，会“做梦”，会在睡眠中整理自己一天的“经历”。这让Agent从一个纯粹的逻辑引擎，开始向一个更像“生命体”的方向演进。

**延伸思考**

这种记忆设计，对于构建长期陪伴型 Agent 或企业级知识 Agent 至关重要。它暗示了未来 Agent 记忆系统的方向：一个由“实时工作记忆”（上下文窗口）、“长期情景记忆”（Markdown文件库）和一个后台“记忆管理进程”（做梦Agent）构成的完整系统。

**精华收获**

- **异步记忆管理**：不要阻塞主任务来处理记忆，用专门的“分叉Agent”来做。
- **多级记忆体系**：结合实时钩子与定期后台任务，实现从“记下”到“消化”的完整闭环。

## 05. 未来狂想曲：零人公司与卡片里的经济体 `[对话 1:00:00 - 1:05:00]`

**核心观点**

Agent Harness 的终极形态将催生“零人公司”——完全由 Agent 自主运营、自我进化、自我盈利的经济实体。未来，投资一个公司可能就像购买一张运行着数百个Agent的黑色卡片。

**深度阐述**

对话最后，来新璐展开了一系列令人目眩神迷的预测，将讨论从工程实践拉升到了社会演化的层面。

**1. Agent 的“蜂群化”**

当前，我们主要研究单个 Agent 如何完成任务。下一步是研究如何让成百上千的 Agent 像蜂群一样协同作业。目前是“人”在管理和编排这些 Agent。但很快，这将变成 Agent 自己去管理和编排更多的 Agent。模型在训练时就会被注入协作与编排的能力。

**2. “零人公司”的诞生**

这是最核心的暴论。来新璐认为，未来的公司本质上将是一种“理财产品”。公司作为一个黑盒，对外提供产品和服务。客户不关心盒子内部是人还是 AI。因此，一个完全由 Agent 组成的公司是完全可能的——没有CEO，没有员工，只有运行在服务器上的 Agent。

> “我从来不觉得一人公司是贬低的事情。我认为真正 Make Sense 是零人公司。” `[对话 1:04:00]`
**3. “卡片里的公司”**

受“真格基金和十字路口发起的 Token Grant”项目启发，他想象了一个场景：未来你走在路上，从口袋里掏出一张黑色卡片，对朋友说：“这张卡里跑着五个公司，它们每年为我创造几十亿的收入。”这些 Agent 公司被部署在卡片内的芯片上，自主运行、自我维护、按需与外部世界交互。

他举了一个正在发生的雏形例子：“UU Agent”。这个 Agent 被创造出来后，就被丢进互联网，创作者不再修改一行代码，也不提供任何资金。它的目标是“有一天超越 Claude Code”。它如何生存？它在 GitHub 上开设打赏页面，画饼（画圆）来向大家募捐，用募捐来的Token购买算力，让自己进化。这已经是一个原型意义上的“零人公司”。`[对话 1:04:30]`

**个人感受**

这部分内容让整个播客的格局瞬间打开。从技术细节跳脱出来，看到一个23岁创业者眼中的未来图景，那种兴奋和憧憬极具感染力。它不再是关于“如何做一个更好的聊天机器人”，而是关于“如何创造一个全新的经济物种”。这既令人兴奋，也带有一丝对未知的敬畏。

**延伸思考**

“零人公司”的实现将对社会结构、法律、经济产生颠覆性影响。谁来为 Agent 公司的行为负责？Agent 之间如何签订和执行合同？Agent 公司的股权如何定义和交易？这些问题目前还停留在科幻层面，但技术的发展速度可能让它们比我们预想的更早成为现实议题。

**精华收获**

- **洞见未来**：Agent 不仅是工具，更是未来的“数字公民”和经济活动的基本单位。
- **改变投资逻辑**：未来的投资可能不再是投“人”和“团队”，而是投一个自我进化的“算法”或“系统”。
- **技术周期红利**：提醒所有创业者和产品经理，今天所做的一切，本质上都是在吃“技术周期变化的红利”。必须深入理解技术内核，才能吃到最大的红利。
