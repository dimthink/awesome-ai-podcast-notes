# Agent 元年第 500 天：什么在消失，什么在诞生——为什么我们不该再投资 GUI 思维的软件？

> **本期嘉宾**：真格基金投资总监 天杰Jack、归藏脏师傅（臧师傅）  
> **主持人**：Koji Yang

## 前言：Agent 元年的 500 天节点

Hi，我是 Koji，欢迎大家收看「此话当真」和「十字路口」这一期串台节目。今天我们请到两位朋友：一位是真格基金的投资总监天杰Jack，另一位是归藏脏师傅。我们也很开心真格和十字路口发起的 Token Grant 支持了归藏脏师傅的项目。

为什么要录这期播客？大概在 2024 年 1 月 1 日我们做了一期开年播客叫「Agent 元年」，到现在掐指一算，正好差不多 500 天。

这 500 天发生了什么？

- 第 100 天：ManaS 发布，因为 Claude Sonnet 3.5 提供了一个做 Agentic 的基础。
- 第 300 天：Cloud Code 发布。
- 第 400 天：Open Cloud 发布。
- 到今天第 500 天：仍然有大量和 Agent 相关的概念、技术在井喷。

最近真格基金公众号发了一篇刷屏的文章，也是天杰写的，叫《我们或许不应该再投资 GUI 思维的软件》，那篇文章引起了非常多的讨论。

## 500 天复盘：超预期还是不及预期？

**Koji**：你觉得这 500 天发展到这里是超过你的预期，还是不及你的预期？

**天杰Jack**：现在市面上市场上的产品还是在以人——以使用者的交互方式居多。就是人还是需要不停地把自己的注意力灌入某一个产品才能很好地使用它。至少市面上比如说信息消费这样的产品，我自己还没有一个特别满意的产品出现。

**臧师傅**：事实上，其实我觉得整体都是高于预期。我如果在当时是无法想象到现在这个状态的。你无法想象到现在这个工作状态和现在这个效率的提升。就是我没有办法想象说它能够让我有如此多能做的事，它极大地拓展了我的能力。

**Koji**：比如说举个例子呢？

**臧师傅**：不用说 500 天，就是 ManaS 开始，我觉得想不到我自己能维护一个近几十万行代码库，然后给这么多人提供服务。对吧，这是不可能的，对于我这种只懂一些基础的全端来说。然后第二个就是不可能认为我现在的流程的自动化会如此之高。之前我是坚决不会用 AI 写内容的。

**Koji**：什么时候开始变的？

**臧师傅**：我相信很多媒体都会有这种坚持，但是现在我发现这种坚持没有意义。当然还有很多人在吐槽很多公众号有 AI 味儿，我觉得这是一个不可避免的过程，它是会分化的。对于公众号的写作来说，它就是向你传达信息，怎么样信息传达效率最高、摩擦最低，它一定会流向那个方向。当然如果你是说在文学性的场景，我要写小说、写比较高级的东西，还是需要人去写，因为 AI 达不到。但是如果说你现在要写这些新闻、文章，想要表达你的观点，AI 有时候写得比我好。我最近看到最没有 AI 味的一篇文章是《置身钉内》。

**Koji**：所以 AI 远超你的预期。第一个是你万万没有想到今天可以维护几十万行代码的一个项目，你也想不到有一天真的把自己的自媒体可以托付给 AI 去写。还有什么呢？

**臧师傅**：不止是自媒体，其实就是我所有的工作，所有的工作都会在这个 Agent 的体系里面。就刚才说的，除了我跟线下跟人交流以外，我所有的内容都变成了一个非常自动化的事情。

**Koji**：如果要你们用一句话去总结过去 500 天 Agent 的这个行业，你们会用一句什么样的话？

**天杰Jack**：没有很多新的东西在发明，但是不变的，我觉得核心还是我们对于上下文的的管理。就是现在上下文越来越重要，不管是大模型那边的上下文，还是我们在 Agent 里的上下文管理，所有的东西都回到 context 里面。

**臧师傅**：嗯。变得不方便，就是每个东西都在变，除了上下文以外。不变的，就是始终一些新的概念在被去魅吧。之前不是一直有个笑话吗？讲的是所有那些还没有被应用、还没有被驾驭的技术，我们都可以叫它 AI，对吧。等到它真正稳定地、大规模地、高效地落到工作流里面，我们就会说那不过是软件而已。

**天杰Jack**：不变的是，我觉得模型的能力进步会依旧解锁一些我们之前没有想到的技术奇观。

## 核心讨论：我们或许不应该再投资 GUI 思维的软件

**Koji**：我们为录这期播客，尝试总结了在过去 500 天 Agent 领域发生的六个热搜关键词。我们一个一个来聊。第一个想聊的是 **Headless**。Headless 就是说软件没有头了。没有头什么意思呢？就是软件没有界面了，只剩下了自己的身子——只有数据库、只有一些工具。所以这个软件没有头的事情成为共识的背后的舆论助推手之一就是天杰。那篇文章从你写它到现在已经三个月了，你的想法有改变吗？

**天杰Jack**：恕我直说，当时写那篇文章的时候，我之所以加上“思维”两个字，是因为我们知道之前有很多软件是效率工具或者说生产性的工具，它是方便人来使用的，目的核心是让用户更好地使用这个产品完成某项任务和工作。当 Agent 的能力大量超越了部分人类时，GUI 本身对这类新的以 Agent 为本位的公司完成任务的这个工作流来说，其实就变成了一个额外的负担。所以不是说 GUI 不重要了——就像 Koji 刚刚讲到的，AutoGPT 或框架类的产品诞生在命令行里，用户接触它的门槛相对比较高；最终出圈的像 Open Cloud 或 ManaS，本质上它还是一个非常好的 GUI，用户适合使用的这么一个产品。所以 GUI 本身的价值并没有因为工作形式的变化而退潮，反而会因为它能封装更多的能力，使得用户需要接触这些能力的时候，需要一个更好的 GUI 界面。所以当时那篇文章核心，我觉得限定讨论的还是：在完成效率和生产性的任务的时候，当人类不再成为使用工具的主角的时候，GUI 变得没有那么重要了。

**Koji**：当时有句话在文章里面是说“GUI 是人类认知缺陷的补丁”，可以展开讲讲吗？

**天杰Jack**：GUI 本身都是基于这些人类注意力的机制去优化的一个产物。这个前提是，人类和信息交互的方式是某种程度上带有缺陷的，因为人类天然就是 context 有限的。人类最多只能记住七个并列的单词，要记住第八个、第九个，那个对认知的挑战就是陡增。我们看到很多效率工具，比如 Notion、Figma、包括 Slack，它们的用户体验非常好，特别符合人类对信息处理的某种习惯，让人类可以在某些工作场景长时间 focus 来完成某项任务。但不再需要人长时间聚焦工作的时候，在生产效率这个场景里，GUI 就变得没那么重要了。只要人还在，GUI 还是很重要；但 AI 用 GUI 就毫——不重要。或者说 GUI 是数字世界服务人类的一个交互界面，这也符合它原本的定义。如果某一件事物不再需要人类参与的时候，它就不需要为人类的缺陷去服务。

**Koji**：你们自己最近有用到什么所谓的 Headless 的产品吗？在工作当中？

**天杰Jack**：我主要就是一些 CLI。比如说我们常见的 FFMPEG，用它剪视频。或者我昨天给我的 skill 加了一个新功能，把它变成一个 3:4 的 layout photo 的排版，里面有很多视频。处理视频的时候，比如裁切、放大，因为给的视频不可能严丝合缝刚好能放在框上，就需要自动处理，这时候不可避免用到 FFMPEG。另外用得最多的就是那些新的 CLI。最近发现很多消费产品——不只是互联网产品，消费产品也开始推出 CLI 了。比如说飞书，飞书 CLI。

**臧师傅**：其实刚才说到 GUI 的时候，很多 GUI 是反人类的，但是我们当时没有办法。现在我们可以有另一种选择——AI 天生就理解这种复杂性，它就是为这些 CLI 之类的设计的。我们可以把这个复杂性让 AI 去承担。用户从来关注的就是——我现在就要这个功能，我现在就要知道我下周的会议，我现在就要知道我上一篇文章的总结，我就是要知道我最近这五篇文章里有什么隐含的一些关系。那么它就能帮你去拉到这个文章，去看，给出你结果。不需要你再去点这个按钮、点那个按钮，然后找“我这是在哪个文件夹、哪个分享、我给它放哪来着”——不需要找这些，这些复杂交互被消解了。

**Koji**：你刚才说消费的 CLI，说的是什么样的消费？指的是信息消费还是线下的场景消费？

**天杰Jack**：比如说最近我看到肯德基、瑞幸也都出了 SKU 或是 CLI，接入以后它可以代替你进行一些物理世界的一些操作。原来你需要去小程序这些地方点单，现在可能顺手 AI 就帮你点了。

**Koji**：你有用过吗？瑞幸或者 KFC 的？

**臧师傅**：用过。

**天杰Jack**：这种方式挺好的。他们想在 AI 时代有自己的独有的渠道，去摆脱之前原来渠道的影响。

**臧师傅**：剥削。

**Koji**：所以你认可“我们也许不应该再投资有 GUI 思维的软件”这句话吗？

**天杰Jack**：我觉得，就是他在创建软件和创建新产品的时候，是不是优先想的是“我这个界面怎么画，我的交互是什么样的”？他不应该优先想这个。他应该想的是结果——用户需要什么，我怎么样让用户更好地、让 AI 更好地帮用户解决问题和传达对应的信息。这个可以是 UI，也可以不是。UI 的话我倾向于尽量让它自动去做。Notion 的 CEO 作为非常厉害的设计师，他们在交互和体验上的打磨是非常之多的，对吧。但比如最近的 AI 试验，他们侧边栏改了五回了，我觉得这是典型的 UI 思维。

**Koji**：那如果你今天是 Notion 的产品总监、产品 VP，你要怎么做呢？

**天杰Jack**：一个是迁就老用户——不要动老用户熟悉的东西，因为相当多的老用户还是以 GUI 的思维去使用的。给它一个入口，让新用户走那个以 AI 为主的用户，和以 GUI 的用户分开。在 AI 那边去实现你这个 Headless 的想法。在原来的地方，我倾向于作为工具软件，它是不应该经常被变动的，因为那个已经很复杂了。

**Koji**：我非常同意。我是 Notion 的重度用户，我们可能订阅了快十年的时间。最早使用印象笔记，然后签到 Quip，然后又签到 Notion。可以看到在 GUI 思维下这些软件的复杂度层级是越来越高的，直到 Notion 可以按照 Block 的形式去组织信息的时候，它达到了某种意义上 GUI 思维的巅峰。在最近几个版本，我觉得它是强行往一个已经打磨得比较极致的 GUI 产品上强行加上了一些 Agent 和 AI 的概念。那些产品是在破坏老用户的价值网络，而不是在重新开辟一个新的价值。但你要开辟一个完全 disrupt 过去自己的 Notion 的全新产品，它可能是一个完全新的产品。两个产品放在一块，只会彼此冲突，让两边人都会很难受。

**臧师傅**：有没有一个正面的例子，是说它彻底摒弃掉 GUI 思维，变得非常 Agent friendly，然后做出了好产品？

**天杰Jack**：比如说飞书的 CLI。其实 Gmail 在今年，包括 Google 整个 Suite 套件今年初都是支持了这套 CLI 的交互方式。我最近在用的一个常用工作流是——你知道，因为海外有很多订阅是通过邮箱来订阅的，就导致我过去订阅了可能几百个 source，然后我的 inbox 就崩溃了。我之前花了特别多的精力去定制我的规则，比如标题里面有某一类关键词的让它在 A 分组、A tag；关键词里面涉及到 promotion 的，我把它分到一个我不用提醒的 tag，然后 skip 掉 inbox。但是你发现维护这一套标签很快就崩溃了，因为你订阅的速度永远超过你维护的成本。直到有一天我不小心设置了一个 filter，导致所有的邮件都不出现在 inbox 里，莫名其妙被归到一个很 random 的 tag 下。这在可能两年前，我每周需要花三个小时清理我的 inbox，把所有信息分门别类整理好，打上对应 tag。这是人类在使用 GUI 过程中很大摩擦的体现。

**臧师傅**：我以为你是很快乐很有心流状态，每周都花两个小时呢。

**天杰Jack**：一开始是的，但到后面就会变成负担，因为你发现“形式”和“意义”的作用大过了消费这些信息本身。我最近做了件事：通过 Codex 授权 Gmail，每天早上有一个定时任务，把你过去 24 小时的所有文件依照你自己的需求去读完，然后把你认为重要的告诉你。这个某种意义上可以抽象成一个 skill——但它非常个性化，以你自己需要的方式告诉你去昨天不能错过的信息有哪些。所以某种意义上 Gmail 的 GUI 就死了，因为它把自己内在的 mail 的能力献祭给了 Codex。对用户来说，你通过 Agent 的方式访问了你订阅邮件需求的这个信息，但你绕开了 Gmail。对 Gmail 来说，你的用户就被隔离了，就被 Agent 隔离了。所以举这个例子就是说，有些公司它实际替用户完成的任务，过去只能通过 GUI 实现，但现在通过 Agent 的方式能更好地实现。看它格局大不大。很有可能这些公司如果没有很好地转型，或者没有找到新的模式，它就会死，会被消解成最后的数据库。但假设它能跟 Agent 做这个 integration 做得特别好，它就会成为所有人默认的底层执行的工具，也很有价值，也许会有新的商业模式出现。

**臧师傅**：包括视频——生成的视频在 Chat 页面没法预览，需要点开到达本地文件夹然后按空格预览。你看 Codex，它因为体验好，很多用户过去了。Codex 有什么设计？第一，能在 Chat 页面输出视频，内嵌一个视频播放器，一点击就能播放——如果你用 FFMPEG 生成视频了，它就能吐出来。第二，它把图片的显示融合到了信息流里面，先输出图片，再说句话，再输出图片——图文混排。还有一个，文件夹——以前很多 AI 生成文件夹就给你一个地址，怎么找？没人知道。Codex 呢？本身文件夹就是个链接，点击一下你就能打开。但 GUI 还是很重要的。

**Koji**：天杰当时写那篇《我们或许不应该再投资 GUI 思维的软件》的时候，有什么触发点？

**天杰Jack**：从今年初，我们看到大量的公司在主动推出 MCP 或 CLI 这样的交互方式给到 Agent。比如 Sentry、MongoDB，这些本来主要交互界面就是文档和命令行的工具。但除此之外，像飞书、Gmail 的 Google Suite——这些原本面向人的公司也在主动开放自己的能力给到 Agent。我们从投资人角度觉得这是一个很重要的趋势。

**Koji**：那篇文章发了之后，最大的负面意见有什么？

**天杰Jack**：大家会觉得有点太极端。比如你现在刷抖音，在手机上已经被优化到了极致。但 Agent 和它解决的问题不是同一种问题，对吧，它不是为了让人类更好地刷视频。作为 PM，最重要的是定义问题。不管是创业还是做新产品，当你定义问题时，背后到底什么是对 AI 原生的交互来说是一个新问题，这是个很重要的能力——不要去解一个上个时代已经被解决得很好的问题。

**臧师傅**：现在我们有共识了——它不太需要那么重的 GUI。AI 上的 GUI 可能不太重要。但现在我觉得大部分人在一些有必要打磨的地方也打磨得不是很好。比如说最近收到很多 BP，就是 Cloud Code 和不太有品味的一些软件内容的支出——就没有用我们臧师傅的 Skill。

**天杰Jack**：Exactly，推荐臧师傅的 Skill。你的 PPT 本质上也是一种人与人之间的 GUI，对吧？如果连这个都不打磨，我觉得会有问题。就像我们录视频播客，还是要画个妆、弄个头发——那是我们的 GUI。我确实昨天看另外一个 BP，它生成了一张图，图上的中文都还是看上去半年前 AI 生成出来的。我心想，做 AI Native 的产品的创始人，居然都不用最收他的模型来做图，这好像也有问题。

**臧师傅**：或者有个标准：你用的 AI 生成的内容不要让人看出来是 AI 生成的。在这些礼仪性的场合，BP、你的官网，是你的门面，你是通过这些地方来建立信任的。上面的文案、图片、内容都在彰显你的品味、你对产品有多用心。无论你用不用 AI，你都应该把它打磨好。得投入心血，受众是能看见的。

**天杰Jack**：这就是星巴克创始人回归星巴克后写的那本书叫《将心注入》。他发现星巴克大家已经不认真做咖啡了，为了恢复士气和状态，写了这本书。

## 关键词二：CLI（命令行界面）——AI 与工具交互的新范式

**Koji**：第二个关键词叫 **CLI**（Clee）。前面已经讲到很多了。我相信听这个播客的朋友 50% 以上应该知道，但还是有一些同学可能不知道。由臧师傅给大家科普一下，CLI 到底是什么？

**臧师傅**：对于普通用户来说，理解 CLI 其实挺简单的。GUI 软件是你去点对应的按钮，它给你对应的视觉呈现。CLI 工具是这样——它的执行是通过命令来执行的，是纯文本的。你给它的命令大概类似一个伪代码。核心就是，原来你剪辑，要拖时间轴、点按钮，CLI 是你跟它说一句话它就给你剪成两节，但这句话需要固定的格式。大家用 CLI 的原因是可以给 Agent 更清晰地指令。CLI 本身是纯文本的，AI 或者大语言模型也是纯文本的。以前我们用 CLI 最大的问题：我记不下来这些命令。就算 FFMPEG 的创作者，也没办法记住他所有的命令，因为文档可能有上千页。但人类不可能记住这些，AI 能记住。所以你只要跟它说“我要把这个码率降一下”，它就会帮你翻译成对应的 CLI 命令。最大的问题消失了，你就可以非常方便地用了。

**Koji**：刚才我们在用什么 CLI？从飞书、Google 再到瑞幸、KFC 都有了自己的 CLI。那有没有什么 CLI 是你们特别想用但今天还没有推出的？

**臧师傅**：其实现在大部分我们常用的垄断性软件都没有推出自己的 CLI。比如说微信、美团、滴滴、携程、飞书、飞猪，都没有推出 CLI。因为一旦推出 CLI，用户的入口就不在我这里了，大家就变成一个“无头”的服务提供商而已。别人还可以用 CLI 去拿你的数据。所以他们可能有这方面的考虑。但对用户来说，这些 CLI 推出是好的、方便的。

**天杰Jack**：如果今天我是这些企业的决策者，我可能也不会开放 CLI。这个决定需要非常大的阻力和决心。所以我佩服那些已经是主流软件但依然开放 CLI 的公司，比如谷歌、飞书。它们有充足的理由、非常大的阻力去不推出 CLI，但依然选择推出。如果你认为你的工具可以通过 Agent 更好地找到用户，你的公司传递的价值能找到用户，你就应该开放你的 CLI。比如瑞幸咖啡，它的使命也许是更快捷地通过各种各样的方式让你获得一杯咖啡。如果你开放 CLI，还有一个潜在的优势：你会获得一定的曝光度，是很新奇的东西。比如那个“金谷雨饺子馆”——在某一个圈子里。

**Koji**：如果你们要许个愿，最希望谁开放 CLI？只说一个。

**臧师傅**：那当然是微信。太痛苦了。你在所有事情都可以 AI 自动化的时候，但你和人的沟通、和客户的沟通、和上下游的沟通，依然在用非常痛苦的 GUI 点进去的方式打字去回复。

**Koji**：要不要给张小龙喊个话？

**臧师傅**：算了算了。

**天杰Jack**：你觉得很难说服他？我觉得有任何办法可以收服微信开放 CLI 给我们吗？我可能不会期待一个传统服务去进行 CLI 转型，我可能会期待有一些新的、完全基于 Agent 或类似的服务出现。比如 Koji 你观鸟，你去找一个鸟岛，还是得通过小红书搜索。我现在用 ManaS，最近去新加坡、马来西亚、清境泽都是 ManaS 帮我找的。

**臧师傅**：我想到一个说服张小龙的可能——有没有可能有一天我们在豆瓣上互相加好友？如果已经在发生了……如果真的在豆瓣加好友，并且我们仨的聊天记录豆瓣可以用 CLI 让我们的 Agent 去看，那我可能真的会把一部分聊天沟通转移到豆瓣去。那时微信处于防御状态，有可能……我观察到一个现象：我和我的朋友、年轻人开始在抖音上聊天了，频率非常非常高。抖音聊天的 DAU 现在是一个亿。内容消费在哪，聊天就在哪——我天然要向朋友分享这些内容，这些内容是共同话题的基础。

**天杰Jack**：扎克伯格在内部分享说低估了 TikTok 的冲击。他本来以为只是一个内容平台，但没想到大家在 TikTok 上会聊天——因为很多时候我看到一个内容分享给你，就自动开启了对话。这冲击到了 WhatsApp。对微信是一样的。如果我能很好地通过微信打开分享的内容，我不会去抖音里跟朋友分享——我肯定优先分享到微信里。但抖音分享不到微信，所以会说“你关注一下我的抖音吧，我们拉个群”，然后交流就迁移了。

**臧师傅**：所以如果有一天豆瓣、微信和飞书全部打通，还是挺可怕的。

## 关键词三：Skill——Agent 能力的沉淀

**Koji**：第三个关键词是 **Skill**。臧师傅最近有一系列 Skill。你的 Skill 你最想推荐的是什么？

**臧师傅**：我肯定还是 PPT Skill。第一个是我付出心血比较多，第二个是大家确实需要。

**Koji**：如果不推荐自己的 Skill，推荐别人的 Skill，你最想推荐谁的？

**臧师傅**：完了。我不用别人的 Skill，所有的 Skill 都是我自己做的。除了我自己 Skill，AI 安装 Skill 是什么呢？飞书 CLI 是由多个 Skill 组成的，里面一连串飞书的 Skill，其他没有了。可能安装 MCP 比较多，比如 CDP 的 MCP（控制面板的），这是我们开发必须用的，涉及到测试自动化什么的。在常见的重复任务上，都是自己写的。

**Koji**：天杰有在用 Skill 吗？

**天杰Jack**：我其实把我自己的 PPT 会用臧师傅的 PPT Skill 优化一下，比如给播客起标题、找金句、写摘要，这些用得比较多。另外就是和会议相关的各种 Skill。比如开完会，我发现我的会有几种不同类型，不同类型要不同的模板去总结。所以有个 Skill 让它判断类型——是朋友之间交流、天使投资项目还是播客潜材，然后根据类型进行不同的总结，放到 Notion 固定的位置去。还有一个常用的 Skill 叫“Grill Me”，它是“榨干你”的意思。当你自己有问题想不明白时，让它来向你提问，它会用各种角度榨干式地提问，把你脑海里的想法引导出来，最后帮你做更好的决定。

**Koji**：现在大家在用 Agent 的过程中，也会讲说 Skill 的自我沉淀和自我进化。你们有感受到这样的发生吗？

**臧师傅**：其实有时它沉淀的不只是 Skill。比如很多人问我 PPT Skill 怎么做。第一个版本是在分享前，我给了它一句话，它基于我的 memory 和我以前的项目，从里面掏了一点样式代码，整合成一个符合我审美的 PPT，然后我再进行调整。第一个版本已经非常惊艳了。它就是基于上下文和你的品味，把你的品味提炼成一个结果的过程。

**天杰Jack**：在这个过程中，包括那个什么“艾玛”它有自动总结 Skill 的功能，会识别你的重复流程，帮你总结。但这种总结的 Skill 有价值，但有一个问题：缺少了你去调整的那个过程。因为 AI 总结的不一定符合你的要求，它是在猜。所以需要你再去手动调整。比如之前 Open AI 的一个 Codex 作者分享了一个提示词——因为 Codex 能看到所有聊天的记录，它也能帮你整理总结，发现重复部分，提取出来帮你总结成多个 Skill，一下给你五六个，然后问你“要不要安装”。这个流程是成立的，但还是需要人调整、参与。

**Koji**：你们认为 Skill 会长期存在在 Agent 的使用体验中吗？还是它或许只是一个过渡期的东西？

**臧师傅**：长期吧。因为最后所有东西都会被模型消化掉。但“长期”在 AI 领域半年或一年就算长期的话，从这个定义上来说，我觉得是的。模型会逐渐吃掉一部分，把它变成不用复杂就能出现的结果——我们用自然语言就行。举个例子，GPT-4 2.0，原来生成好的图片需要复杂提示词，但现在只要一句话，哪怕需求不精准也能给出好结果。最后就是这样，只是在 Skill 这块它会慢一些。而且 Skill 已经成为了共识——一旦成为共识就会被接受、被消化，存在时间会比较长。

**天杰Jack**：我觉得可能还是存在的。我始终认为最后的主体应该是一个帮你完成某一件任务的“品牌”——你依赖这个 Agent 在这件任务下持续地完成某个任务，完成得很好后你就相信它可以替你做完。至于这个 Agent 用了什么 Skill，从人类认知难度来说，要辨别两个 Skill 的不同很难——不仅要安装、测试、考虑边界条件，还要知道里面的 nuances。对人类来说很困难。把人类当“小白”的一个标准：你的奶奶会不会学会这种 Skill？很难。但奶奶想喝咖啡时，一个 Agent 帮她完成，她信赖的是那个 Agent。

## 不变的核心：上下文为王，概念不断被去魅

回顾整个对话，不变的是对上下文的重视——不管是大模型自身的上下文，还是 Agent 的上下文管理，所有东西都回到 context 里。每个东西都在变，除了上下文。不变的还有那些新概念不断被去魅。一切之前无法被驾驭的技术叫 AI，真正稳定落地到工作流后，我们只说那不过是软件而已。模型能力的进步会依旧解锁我们之前意想不到的技术奇观。

从 GUI 思维的“Headless”到 CLI 的普及，再到 Skill 的涌现，Agent 正在重塑软件的面貌。当一件事物不再需要人类参与，它就不必再为人类的缺陷服务。产品经理最重要的能力是定义问题——不要去解上一个时代已经被解决得很好的问题，而是去抓住 AI 原生交互真正的新问题。

---

## 结语

500 天前我们站在“Agent 元年”的起点，500 天后我们看到一个不断自我去魅、持续解锁奇观的领域。GUI 仍旧重要，但在效率与生产的场景中，当 AI 成为执行主体，Headless 和 CLI 成为了更自然的选择。未来，也许很多软件会像飞书、Gmail 那样开放自己的 CLI，成为 Agent 生态的一部分；也可能出现一个完全基于 Agent 的微信替代品。无论如何，每一次技术迭代都在消除摩擦，让信息以自己的方式流动。
