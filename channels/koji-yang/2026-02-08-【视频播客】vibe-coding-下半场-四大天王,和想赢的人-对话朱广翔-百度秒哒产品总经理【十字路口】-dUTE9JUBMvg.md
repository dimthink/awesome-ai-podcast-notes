## 1）从“写了 20 年代码”到“做一个让别人不写代码的产品”：[章节 1.1-1.6]

**核心观点**

朱广翔把“转向无代码/应用生成”视为计算机语言演进的必然；他对自己“放弃得太晚”的反思，本质是对趋势的服从而非背叛。

**深度阐述**

对话从一个看似玩笑、实则非常尖锐的矛盾开始：一个初中开始写代码、一路写到博士的人，工作没几年就不写了，转做产品经理，最后做出“让别人也不用写代码”的秒哒。朱广翔没有回避这种“自我否定”的冲突，反而把它上升为计算机发展史的隐喻：**每一代工具都会造出下一代工具，然后被下一代工具淘汰**。他讲了一个特别经典的例子：C 语言编译器的第一版是用汇编写的——等编译器存在之后，人们开始用 C 写更多东西，汇编的重要性被“碾压式下降”。这不是嘲讽汇编，而是说明“更接近人类意图的表达方式”会不断胜出。

他把这条曲线从机器码（0101）讲到汇编、再到 C/Java/Python，最后推到极限：**不是“用类人的语言写代码”，而是“直接用人话描述我想要什么”**。在他的话语体系里，这就是今天的 Web coding、也是秒哒要做的事：拟合这条曲线的极限。

这种叙述有两层力量。第一层是“趋势的冷酷”：如果语言抽象层级不断上升，那么“手写代码”被替代不是道德判断，而是生产力必然。第二层是“个人命运的自洽”：他把自己不再写代码描述为“被车轮碾过”，但紧接着又补了一句更狠的——他觉得自己“放弃晚了”，甚至认为“走了 20 年弯路”。这不是对过去的否定，而是对“当下最值钱能力”的重估：当工具把编码门槛打穿之后，代码能力会从“生产核心”退到“教学/思维训练”，而真正稀缺的是场景理解与产品化能力。

**重要原话**

- “**整个计算机发展史，你从最开始是机器语言……再往后高级语言……那再下一步……我就是用人类语言，说我要干嘛。其实就是现在的 Web coding，其实就是秒搭。**”`[章节 1.4]`
- “**我觉得我还是放弃晚了……我觉得走了 20 年弯路。**”`[章节 1.6]`
（原话为中文，无需翻译；其中 “Web coding” 可理解为“用自然语言/高层意图直接生成可运行应用”。）

**视觉/结构信息描述**

这一段是典型播客“故事—类比—结论”的口述结构：从个人经历切入，用编译器史做类比，最后落到“自然语言即开发”的判断。没有图表，但逻辑链条非常像一张“抽象层级上升曲线”。

**个人感受**

朱广翔的情绪不是怀旧，而是近乎兴奋的“顺势感”：他并不把过去的技术积累当成包袱，而把“被替代”当成一种荣耀——因为你曾经处在生产力跃迁的上一阶梯上。

**延伸思考**

把“被工具替代”视为进步，能让团队更敢下注；但也会带来危险：如果过度相信趋势，容易在“短期不可行”时被现实教育。所以后面他提出“15 度夹角”，其实就是给这种趋势信仰加一个现实缓冲垫。

**精华收获**

- 个人职业转换可以不靠“热爱”，而靠“抽象层级上升”这条硬趋势。
- 无代码不是“给小白的玩具”，而是语言演进的下一阶梯。
- 真正的痛点不是“会不会写”，而是“能不能把意图稳定地落成可用的东西”。

## 2）从“不信”到“信”：用户的 Aha moment 与“医生做官网”的冲击：[章节 2.1-2.7]

**核心观点**

秒哒的关键转折，不是模型突然变强，而是团队在早期用“半工程化 + 半模型”的方式先跨过可用性门槛，随后被真实用户场景反向教育：**用户先信了，产品经理才彻底信**。

**深度阐述**

朱广翔承认一个在大厂语境里不太“政治正确”的事实：最开始做 No code 时，他并不相信。原因很具体：时间点大概在“24 年下半年”，当时没有 DeepSeek（他用它代表推理/思维链能力的普及），大家对 AI 的主流想象仍然是 Chatbot，Function call 也只是浅层工具调用。Coding 模型“各家都不行”，这意味着“端到端生成应用”的难度极高——因为应用不是一段代码，而是一串任务：需求理解、页面、交互、后端、数据库、部署、分发、运维。

所以早期能跑起来靠什么？他给了一个非常真实的工程答案：**智能不够，人工来凑**——用大量“半工程化、半模型”的方式把 Demo 做出来。这里的含义是：别把“AI 产品”幻想成纯模型能力的直接投射，早期的产品竞争力往往来自工程系统（约束、路由、上下文管理、部署链路）的补齐。

真正让他从“不信”变“信”的，是两次强烈的外部反馈。

第一次是他对外演示时看到用户的“眼睛发光”。他描述那种 Aha moment 很戏剧化：台下有人本来低头玩手机，看完演示突然“瞪直眼”抬头。这种反馈对 PM 有决定性意义，因为它意味着“价值不需要教育”，用户能自发把能力映射到自己的场景里。

第二次更具冲击：他在网上搜到一个网站，同学告诉他“这是用你们秒哒生成的”。他自己“没看出来”，后来才发现作者是一位 50 多岁的医生，做的是“整个医院的官网”，而且真实上线、持续在用，百度搜索结果第一条还有官方标识。这个故事的分量在于：**严肃业务场景、非技术人群、真实上线、长期使用**——它把“AI 生成应用”从玩具推到了生产工具的那条线外。

这也解释了他后面为什么一直强调“不是 AI 玩具，而是可商业落地的应用”。很多产品死在“演示很好看”，而秒哒想跨过的是“上线后还能跑、还能迭代、还能赚钱”。

**重要原话**

- “**我觉得是我的用户比我先信了……很多人看完之后，我真的是看到了 Aha moment……大家眼睛就明显放光了。**”`[章节 2.3]`
- “**这个作者是一个 50 多岁的医生……他们整个医院的官网……真实上线的，医院一直在用。**”`[章节 2.6]`
**视觉/结构信息描述**

这一段的“画面感”来自用户反应：从低头玩手机到抬头发光，以及“你认真浏览的网站突然被告知是 AI 做的”的恍惚感。这些细节替代了视频里的视觉呈现，构成一种“产品被现实承认”的瞬间。

**个人感受**

朱广翔的语气在这里明显从“理性判断”切换到“被撞击后的确信”：不是“我们很强”，而是“我突然意识到它真的能在我没察觉的地方运行”。

**延伸思考**

一个 AI 产品的“信仰建立”，往往来自“你没参与却发生的成功案例”。因为那意味着它不是靠运营堆出来的，而是自发渗透。

**精华收获**

- 早期不要迷信模型能力，先用工程补齐可用性。
- 产品从玩具到工具的分界线：真实上线、持续使用、非技术人群也能完成闭环。
- 最强的 PMF 证据是：你在不知情时，用户已经把它用成了生产力。

## 3）不看自己的 AR，看用户的 AR：倒金字塔产业结构与“1 万个超级个体”：[章节 3.1-3.8]

**核心观点**

秒哒把商业目标定义为“用户赚到的钱（用户 AR）”，背后是一套“倒金字塔”产业观：应用层应当赚钱最多，工具/模型/硬件拿合理分成，产业才健康。

**深度阐述**

“我们不看 AR，我们只看用户的 AR。”这句话很容易被误解成鸡汤，但朱广翔给了一个非常结构化的解释：他认为 AI 产业的利润分布正在从“正金字塔”走向“倒金字塔”。过去最赚钱的是底层算力（典型是显卡），模型其次，应用层反而多赔钱——这是一种不健康结构，因为真正创造用户价值的是应用层，按理说它应当从价值中获得最大份额，再把成本分摊给工具层、模型层、硬件层。只有这样，才会形成可持续循环。

在这个框架下，秒哒的使命不是“自己先赚”，而是“扶持更多人把应用做成生意”。他把这股浪潮和 “OPC（One Person Company，一人公司）”联系起来：秒哒希望出现大量“超级个体”，用一个人完成过去需要团队完成的交付。

他给了两个非常具象的案例：

1）**超级个体“黄阿玛”**：做了三个项目，其中一个“漫剧平台”赚了 12 万；一个“门窗店官网”卖了 3 万——价格之所以能上去，是因为他抓住“用户想看装到自己家里是什么样”的核心痛点，做了“上传自家图片 + 选择门窗 + 预览效果”的功能，直接提高购买意愿；第三个是“汽车喷漆效果预览工具”。更关键的是获客方式：他把应用发在秒哒广场，第一单客户直接留言联系，形成了“作品即营销”。

2）**12 人传统交付公司**：原先做政府/行业项目，周期半年到一年；用秒哒后“一周交付一个项目”，一个多月交付两单：养老院管理系统（40 万）+ 企业内部办公平台（30 万）。这说明秒哒不是只服务“小白玩票”，也能改变“交付行业”的生产函数。

然后他把这些案例抽象成一张“倒金字塔的分账模型”：如果明年做 1 万个超级个体，每人赚十几万，总规模就是十几亿；用户拿到价值后，分 10% 给平台，平台再分一部分给硬件/底层能力——这不是空想，而是一套可以对齐团队指标、运营策略、产品路线的商业叙事。

当然，对话里也诚实地暴露了局限：他们并没有统计“赚最多的用户是谁”，目前更多通过“筑梦计划”等渠道收集到部分案例，水下可能还有大量未统计的成功者。这反而更像早期真实状态：数据体系尚未完备，但方向先立起来。

**重要原话**

- “**我们要做到秒哒用户的 AR……我们相信只要我们的用户能赚到钱，那我们早晚能赚到钱。**”`[章节 3.2-3.7]`
- “**我们的目标就是明年做 1 万个这样的超级个体。**”`[章节 3.7]`
**视觉/结构信息描述**

这一段实际上是在口述一张“产业金字塔图”：底层硬件—中层模型/平台—上层应用。以及一张“分账流水图”：用户收入→平台抽成→反哺底层。虽无图，但结构非常清晰。

**个人感受**

朱广翔的情绪像在“做一次公开承诺”：他不是在解释盈利模式，而是在解释一种更愿意“慢赚”的信念——先让用户赢，平台才有资格赢。

**延伸思考**

“用户 AR”指标一旦成立，平台就必须面对更难的事：帮助用户获客、转化、留存、交付、合规，而不仅是把生成效果做得更酷。秒哒后面强调分发、插件、后端、评估体系，其实都是在补这张“用户赚钱地图”。

**精华收获**

- AI 产业若长期“硬件赚最多、应用赔钱”，最终会反噬创新。
- 真正能让平台穿越周期的指标，是“用户是否赚到钱”。
- 超级个体的核心不是“更快写代码”，而是“更快把痛点做成可售卖产品”。

## 4）最大的风险：模型会不会把你“内化”？以及与模型保持“15 度夹角”：[章节 4.1-4.9]

**核心观点**

秒哒最大的失败风险，是误判“模型的延长线”：如果你做的东西被模型能力覆盖并内化，产品可能瞬间失去存在意义；应对方式是让产品与模型保持“15 度夹角”——既吃到红利，又不与模型正面同质化。

**深度阐述**

当主持人追问“如果有一天秒哒失败，会因为什么”，朱广翔给出的答案非常“AI 产品经理视角”：不是竞争对手太强、不是百度资源不够，而是**你做的功能可能属于模型能力的延长线**。一旦模型把这段能力“内化”，你原本提供的外部系统价值就会被吞噬。

于是他提出“15 度夹角”的经验法则：

- **不能正交**：如果你的方向与模型能力完全无关（正交），你吃不到模型进步带来的红利；
- **也不能同向重合**：如果你和模型能力完全同向，模型变强就会直接替代你；
- 最舒服的位置是“有重叠、但不被覆盖”，也就是“夹角”。
那秒哒的夹角在哪里？他给了两个层次：

1）**多模型路由**：随时调用“世界上最好的模型能力”。这意味着秒哒不把自己锁死在单一模型上，而是把产品架构当作“调度与组合能力”。

2）**云端与端到端工程能力**：从产品设计到部署分发，这些短时间内难以内化。这里的关键词是“云能力”：不仅是生成，还要承载数据库、鉴权、支付、运维、弹性扩缩容、分发渠道等。

主持人进一步把问题推到极端：中长期乃至无限长长期，Cursor、秒哒这类工具会不会都被模型吃掉？朱广翔没有回避极端结论：**无限长长期会被吃掉**，因为连“环境”也可能被内化。

他用强化学习的两派解释这个观点：Model-free（外用环境探索）与 Model-based（内化环境建模）。他用“从百度到清华怎么走”做类比：一路问路是 model-free，看地图规划是 model-based。环境本质上是一个 MDP（马尔可夫决策过程），可以被学习写进网络。一旦学进去，“外部环境”就可被 replay。

然后他把这套推理落到现实产品：IDE 的本质不是 UI，而是编译器；编译器也是代码，也可能被模型学会。于是他抛出一个“暴论”：**Cursor 会被 Claude 干掉**，因为 IDE 交互不本质，本质是后台的编译与执行能力；当模型能更直接地掌控编译/执行链路，IDE 的壳可能失去价值。进一步推，操作系统也是代码，虚拟机/OS 也可能被内化。到那时，“模型即产品”。

但他也给出了现实制动器：当生成变得重量级，幻觉、不确定性会急剧上升，可能阻碍“完全内化”发生。因此短中期的最优策略仍然是：保持夹角、快速迭代、吃到模型进步。

**重要原话**

- “**最大的风险：我们误判了模型的延长线……结果不小心做到了模型延长线上被模型内化了。**”`[章节 4.1]`
- “**所有 AI 产品最好的方式……要和模型保持一个 15 度的夹角。**”`[章节 4.1-4.2]`
- “**我很早就说：Cursor 一定会 Cloud 干掉的。因为 IDE 不本质，本质的是编译器。**”`[章节 4.6]`
**视觉/结构信息描述**

这里的“结构”像一张坐标系：模型能力方向是一条轴，产品能力方向是另一条轴；“15 度夹角”意味着你要在相近但不重合的方向上构建护城河。以及一条“内化时间线”：3-5 年内化编译器、5-10 年内化 OS（这是他的估计，属于观点而非事实）。

**个人感受**

这一段能感到朱广翔的“技术出身的冷静”：他不把产品神话化，反而公开承认“长期会被吞”。但他也不像悲观主义者，而是在寻找“在被吞之前如何最大化价值”的工程与商业窗口。

**延伸思考**

“夹角”策略本质上要求产品团队具备两种能力：一是持续跟踪模型边界变化，二是把不可内化部分做深做重（比如云、分发、数据飞轮、评估、插件生态）。这会把产品竞争从“界面/生成效果”拉到“系统工程与生态”层面。

**精华收获**

- AI 产品最致命的不是竞品，而是“模型能力边界移动”。
- 护城河不是“我比模型更聪明”，而是“我提供模型短期难以内化的系统能力”。
- 极端长期要敢承认会被替代，但短中期要抓住窗口把价值做大。

## 5）全球 AI Coding“派别”与秒哒的差异化：活的系统、强后端、场景化分发、PM 智能体与 Bench：[章节 5.1-5.10]

**核心观点**

AI Coding 工具正在分化：国外“四大天王”各自押注不同长板；秒哒的核心差异化是“活的自成长系统 + 独一档后端与云能力 + 场景深耕（小程序/分发）+ 产品经理智能体流程 + 全链路评估 Bench + 多模型多智能体路由”。

**深度阐述**

当对话进入“行业全景”，朱广翔先给出国外格局：他们内部称“四大天王”——Rasa、Replit、Botnoi、V0（字幕如此转写，具体产品名可能与现实品牌存在拼写差异，但不影响他表达的“分化逻辑”）。他用“长板”来描述它们各自的定位：

- **Rasa**：轻量化、模板与教程多、传播强，“人人都是 CMO”——更像均衡型、重市场教育。
- **Replit**：更专业、部署更灵活全面，企业场景友好——后端/部署能力更重。
- **Botnoi**：IDE 起家，开发环境与多语言支持强——面向更专业开发者。
- **V0**：前端交互设计极致，高仿真/高还原——以设计与前端为中心。
这个划分很重要，因为它揭示了一个事实：所谓“AI 写代码”不是单一赛道，而是“前端美感、后端能力、部署交付、IDE 体验、传播增长”等多维竞争的组合。

回到国内，他说直接对等的无代码应用生成并不多，但间接竞品极多：IDE 往无代码渗透（Trey 的 Solo 等），通用智能体把“做 PPT/Deep research/定时任务”扩展到“做应用”，以及传统低代码/工作流平台转向 Web coding（例如某产品改名为“编程”）。这意味着：**No code 从反共识变成了共识，越来越多人涌入**。

在激烈竞争下，秒哒的差异化被他拆成几层，每一层都对应“让用户真正上线赚钱”的链路补齐：

### （1）“秒哒是活的”：数据飞轮驱动自成长

他强调秒哒不是一次性生成，而是会“逐渐成长、自我迭代”。用户的点赞点踩、是否上线、是否卡在某一步反复修改等信号，会反馈给模型与智能体策略，让下一次生成更好。这里的关键不是“训练一次模型”，而是把产品当作持续学习系统。

### （2）插件系统：端到端服务小白

因为面向小白，必须端到端：生成、部署、上线、运维、以及地图/搜索等能力接入。插件不是附加功能，而是秒哒的一层“能力扩展框架”。

### （3）后端能力“独一档”：数据库/鉴权/支付/逻辑/弹性

他把前端比作“脸”，后端比作“大脑”。行业普遍卷前端好看，但真实业务要长期存数据、要权限、要支付、要逻辑，就必须有后端。他提到 Supabase 作为后端赛道领先者，并引用其博客：Supabase 与三家领先 AI builder 合作，其中包括秒哒海外版（字幕称 “midu”）。这被他用来证明秒哒在后端能力上获得了外部认可。

更重要的是他提出“面向 AI 的数据库”与传统数据库不同：

- 传统数据库大而重，核心是存海量数据；
- AI 时代数据库“小而多”，每个应用都可能带一个数据库，强调极致弹性与快速创建；
- 传统数据库给人（研发）用，AI 时代数据库要给 AI 用，需要 MCP/上下文/可被模型操控的接口与优化。
他给了一个非常夸张但有冲击力的数据：秒哒“一周创建的数据库”，比传统 2B 服务的数据库团队“七年累计还多”。这背后其实是在强调“云原生的规模化运维能力”是一道硬门槛。

### （4）“一句话”不是需求：产品经理智能体流程

他认为多数 Web coding 是 Query 进去直接写代码，但秒哒先产出需求/产品文档，让用户确认与修改，再进入开发执行。这是为了弥合“创意（Idea）”与“可交付需求”的鸿沟：差之毫厘，生成结果可能南辕北辙。

### （5）秒哒 Bench：端到端评估体系

他引用一句话：“AI 下半场，评估比训练更重要。”并强调：实验室榜单看看就好，真实用户场景才是真评估。因此秒哒搭建了端到端的 Benchmark（秒哒 Bench），覆盖应用生成链路的多个环节，类似产品的测试体系与护城河。

当主持人质疑“审美很主观怎么评估”，他的答案是：用脚投票 + 后验行为信号（是否上线、是否转发、停留/交互数据等），并叠加内部专业设计师/PM/运维的多渠道评审，降低“早期用户审美偏差”带来的风险。

### （6）多智能体多模型路由：像厨具把食材炒成菜

应用生成有上百个任务，秒哒用路由架构为不同任务选择最佳模型与最佳 agent 策略。他用类比解释：Benchmark 是菜谱，模型是食材，路由架构是厨具——目标是炒出用户最想吃的菜。

**重要原话**

- “**秒哒是个活的，它是会逐渐成长的，会自我迭代的。**”`[章节 5.4]`
- “**后端能力……秒哒是做的独一档的。**”`[章节 5.6]`
- “**一句话做应用……这句话不是需求，是 Idea……我们先出需求文档/产品文档再写代码。**”`[章节 5.8]`
- “**AI 下半场，评估比训练更重要。**”`[章节 5.8]`
**视觉/结构信息描述**

这一段虽然是口述，但信息密度很像一套“产品架构图”：

用户行为信号→反馈学习（模型/策略）→插件扩展→后端云能力→PM 智能体流程→Bench 评估→多模型路由。

同时还有“脸/大脑”的前后端隐喻，以及“菜谱/食材/厨具”的路由系统隐喻。

**个人感受**

朱广翔在这里的表达明显带着“想赢的人”的气质：不是守着一个点炫技，而是把整条商业闭环当作竞争单元——从生成到上线、从后端到分发、从评估到生态。

**延伸思考**

当行业都能“生成一个好看的前端”，真正分出胜负的会是：谁能把“上线后的世界”承接住（数据、权限、支付、运维、增长、合规）。这也是为什么“云”会越来越像 AI 应用的底座，而不仅是部署选项。

**精华收获**

- AI coding 不是单赛道，而是多维长板竞争。
- 秒哒的差异化不是“更会写”，而是“更会交付与运营真实业务”。
- 下半场的关键是评估体系与端到端工程能力，而不只是模型参数。

## 6）在百度做秒哒：助力、摩擦力与“我们团队没一个正常人”：[章节 6.1-6.8]

**核心观点**

百度的技术理想主义与“用两年后的技术指导当前产品”的风格，是秒哒得以启动并存活的关键助力；大厂组织的主要阻力来自对齐成本。秒哒团队用“用户是上帝指标”的文化抵消组织惯性。

**深度阐述**

主持人问得很直接：在百度做产品，外界有滤镜；秒哒遇到哪些摩擦力与助力？

朱广翔先给了一个非常明确的结论：**没有百度就没有秒哒**。理由不是资源，而是决策风格：百度愿意用“两年后的技术”去指导今天的产品投入——即使当下看起来“不行”，也敢坚定投入。他甚至用一句带笑的自嘲来形容这种投入：“我们帮老板亏了很多钱，但是我们还活下来了。”这句话的潜台词是：如果没有长期主义容忍度，No code 这种需要跨越模型能力鸿沟的产品，很可能死在早期。

阻力方面，他说得也典型：人越多想法越多，“对齐”成本高。这是大厂结构性问题，不是某个团队的错。但秒哒团队用一种“创业式结构”去对冲：产研运一体、坐在一起、OKR 全押秒哒，甚至内部有一个真实的“十字路口”工位布局（产品、运维、工程、策略四角），很多决策就在这里吵出来。

对话里还有一个很真实的舆论阻力：“如果不是百度做的，我就会用一下了。”朱广翔没有辩解，而是把它引向团队气质：秒哒团队“没一个正常人”，包括他自己“最大的优点是不听老板的话”；工程师也会“直接骂我”；运维来自数字游民、深入用户群到分不清是员工还是用户；还有用户寄椰子刻秒哒、请运维去吃年夜饭等故事。这些细节共同说明：秒哒在大厂里刻意维持一种“用户社区驱动”的气质，而不是传统大厂产品的“官样调性”。

最关键的是他提到团队的“上帝指标”：**用户说了算**。吵到不可开交，就去看用户群、看社区投票。这是一种把决策权外包给真实反馈的方式，也是一种降低内部政治消耗的方法。

对于“百度起大早赶晚集”的梗，他给的回应非常自信：“目前来看肯定是赶的大集……肯定还是一个早集。”这不是客观事实证明，而是一种战斗宣言：他们相信自己在正确的时间窗口里。

**重要原话**

- “**我觉得没有百度就没有秒哒。**”`[章节 6.2]`
- “**我们帮老板亏了很多钱，但是我们还活下来了。**”`[章节 6.2]`
- “**我们团队没一个是正常人……我最大的优点是不听老板的话。**”`[章节 6.5]`
- “**目前来看，肯定是赶的大集。**”`[章节 6.7]`
**视觉/结构信息描述**

“十字路口”工位布局是这段最强的结构信息：四个角色在四角，高频碰撞产生决策。它不仅是空间设计，更是组织设计的隐喻：减少层级、缩短反馈回路。

**个人感受**

朱广翔在谈组织时的情绪更松弛、有幽默感，但底层是强硬的：你可以骂、可以吵，但最后得对用户负责。这种文化对 AI 产品尤其重要，因为不确定性高，拍脑袋很容易死。

**延伸思考**

AI 产品的竞争越来越像“工程系统 + 运营社区 + 生态能力”的综合战。大厂的优势是资源与长期主义，劣势是对齐成本；能不能把团队做成“像创业公司一样快”，会决定窗口期能吃到多少红利。

**精华收获**

- 秒哒能启动，靠的是组织对“未来两年技术”的下注能力。
- 大厂最大成本是对齐，最好的解法是把决策锚定到用户反馈。
- 团队文化不是装出来的，它会直接影响迭代速度与产品调性。

## 7）Claude/Cloud、Scale 与上下文管理：从“改到 50 轮就崩”到“无限轮修改”：[章节 7.1-7.7]

**核心观点**

云端形态更适合服务小白与规模化多 Agent；所谓 Cloud Scale 的核心是“动态加载/渐进式披露”的上下文管理。秒哒通过隐式说明书与路由机制解决了长对话迭代崩溃问题，并计划把能力以插件形式逐步开放给用户。

**深度阐述**

当对话转向最新趋势（Claude/Cloud bot、Cloud Scale），朱广翔先把“云端 vs 本地”说清楚：秒哒选择云端不是追热点，而是因为面向小白用户，本地无限权限容易“把电脑搞崩”，大家甚至要买新机器试。云端用虚拟机隔离风险：一个应用一个虚拟机，搞坏了换一个；并且云端可以并行开 1000 个 agent、1000 个虚拟机做 1000 份工作——这就是 Cloud scale 的想象空间。

然后他抛出一个关键判断：Cloud Scale 并不神秘，本质是把“上下文管理”这类 trick 标准化并公开。问题在哪里？如果把上下文一股脑塞给模型，要么超长窗口撑不住，要么注意力分散。动态加载的方式是“需要什么叫什么、需要哪里点哪里”，让模型每次只专注一小部分。

他回忆秒哒早期的一个“惨痛记忆”：用户改到 50 轮就改不动，模型说“我改不了了”。根因就是上下文管理不行。解决方式是做一次大迭代：在后台引入类似 Scale 的“说明书/说明文档”，让 AI 每次先读说明书，再决定加载哪些工具、MCP、代码片段、上下文，再拼到系统 Prompt 里执行。这一层对用户不可见，但用户能直接感受到：**可以无限轮修改，成功率更高，模型更聚精会神**。

当主持人问“会不会开放 Scale 广场让用户导入”，他回答他们规划里本就有“隐式管理 → 开放给用户扩展”的两步走。秒哒的插件系统被他分成三类，几乎就是对“Scale/插件”形态的产品化定义：

1）API 插件：以服务形式接入应用；

2）Prompt 插件：超长 Prompt 描述复杂流程（与市面流传的各类 Scale prompt 类似）；

3）Code 插件：用代码实现格式转换、文件处理等能力，封装成可复用插件。

这段话的意义在于：它把“Scale”从营销名词落到产品模块——上下文管理是底座，插件生态是外延。

**重要原话**

- “**Scale 的本质就是一个动态加载的能力……就是上下文管理。**”`[章节 7.3]`
- “**用户改到 50 轮就改不动了……我们上线之后，他可以无限轮修改。**”`[章节 7.4]`
- “**插件三类：API 插件、Prompt 插件、Code 类插件。**”`[章节 7.6]`
**视觉/结构信息描述**

可以把这一段想象成两层结构：

- 底层：隐式上下文路由（说明书 → 选择性加载 → 拼 Prompt → 执行）；
- 上层：用户可见插件生态（API/Prompt/Code）。
它像一个“内核 + 扩展槽”的架构图。

**个人感受**

朱广翔在谈 Scale 时有一种“老工程师看新包装”的味道：不是否定，而是把热词去魅——真正重要的是工程抽象与标准化。

**延伸思考**

当上下文管理被标准化，行业门槛会被抹平一部分；下一阶段的差异化可能回到“数据飞轮、评估体系、云后端、分发渠道、生态治理”。

**精华收获**

- Cloud 的关键优势是隔离风险与并行规模，小白更需要云。
- Scale 的核心不是神 prompt，而是可复制的上下文调度机制。
- 真正让产品跨越“无限轮迭代”的，是系统性上下文工程。

## 8）反共识建议、工程师与教育：未来比“会写代码”更重要的是“懂垂直业务”：[章节 8.1-8.9]

**核心观点**

朱广翔认为自己坚持 No code 本身就是反共识；他给工程师的建议是：把计算机作为更早期的通识/教具，把大学与职业的主修资源投向垂直领域知识，因为未来“做出好应用的人”靠场景壁垒而非代码壁垒。

**深度阐述**

当被问到“反共识建议”，朱广翔说他做的事就是反共识：当大家卷 IDE、Cursor 验证程序员 PMF 时，他们坚持认为小白也能做应用。后来他用一个观察来证明趋势变化：国外 25 年 AI coding 产品在 PH 榜上“起码 200 多款”，上半年基本是 IDE，下半年 No code 比例超过 IDE 并远大于——这说明市场关注从“辅助程序员写”转向“让更多人做”。

对工程师的建议，他说得很激进：如果重读大学，他可能不学计算机，甚至希望更早（幼儿园）学 Web coding；大学去学法律、金融等垂直专业。理由是：工具会把编码能力“商品化”，真正决定应用价值的是业务理解。

他用一个震撼案例支撑：一位中石化工程师不会写代码，用秒哒做了矿井设计软件，真实运行在多个油田，还在石油大学授课，学生用它做毕业设计。对比之下，公司曾花 140 万采购程序员写的软件，因为开发者不懂矿井设计物理逻辑与展示方式，反而“用不起来”。这个对照非常残酷：**不懂业务的代码，是昂贵的废品**；懂业务的人借助工具，反而能做出真正可用的系统。

教育层面，他主张让孩子学 coding 作为教具训练思维，同时学 Web coding 作为实验更早实现想法。他还举了 Pascal 的例子：它更像教学语言，不一定实用；未来 C 也可能成为“博物馆里的代码”，主要用于教育，而真实生产依赖 Web coding/AI。

这一段的底层逻辑是：当“实现成本”下降，竞争就从“能不能做出来”转移到“做什么、为谁做、怎么卖、怎么长期跑”。这与他前面“用户 AR”的指标完全一致：**最终赢的人是懂场景且能把应用变成收入的人**。

**重要原话**

- “**如果我再重新读一遍大学，我可能不太会学计算机专业了……我真的要学一个更垂直的专业。**”`[章节 8.3]`
- “**把应用做好的的人，并不是谁代码写得好，而是谁真正懂业务，懂场景。**”`[章节 8.3]`
- “**他们公司采购 140 万的软件……程序员不懂业务……他们都不用，用不起来。**”`[章节 8.4]`
**视觉/结构信息描述**

这一段的结构是“观点—案例—对比—教育建议”。140 万采购失败 vs 零代码自建成功，是最强的对照图景。

**个人感受**

朱广翔的表达里有一种“羡慕垂直专家”的情绪：他不再把程序员视为天然主角，而把“行业里的人”视为未来应用爆发的中心。这对传统技术精英叙事是一种温柔但坚定的颠覆。

**延伸思考**

当每个行业都可能出现“用 AI 搭应用的行业专家”，平台需要思考的不只是功能，还包括模板、合规、数据安全、行业知识库、交付与售后支持——这会把平台推向更深的行业化。

**精华收获**

- No code 的对手不是 IDE，而是“旧的生产组织方式”。
- 工程师的长期壁垒可能来自“行业知识 + 产品化”，而非语法熟练度。
- 教育的重点会从“写对代码”转向“用工具表达意图并落地价值”。

## 9）2025：做对与做错；从 Workflow 到 No code；以及结尾的投资选择（Gemini）：[章节 9.1-9.8]

**核心观点**

秒哒在关键岔路口做对的是：在低代码/工作流与无代码之间选择无代码，并坚信 Web coding 会取代 workflow；做错（或过早）的部分是给模型太多自由，以及运营启动太晚。投资层面他看好 Gemini，理由是 Google 的数据与分发优势。

**深度阐述**

回顾 2025，朱广翔给出一个非常清晰的“正确选择”：在低代码与无代码之间，选无代码，并断言“Web coding 一定会取代 Workflow”。他解释这不是拍脑袋，而是来自强化学习之父 Sutton 的《The Bitter Lesson》：AI 领域的苦涩教训是——过度依赖人类经验会让系统复杂且不泛化；真正能规模化的是更好利用算力的方法，而路径是 Learning 与 Search（他把它类比为今天的 Model 与 Agent，模型内计算与模型外计算）。

在这个框架下，Workflow 像“用人类经验固化节点”，结构化但不灵活；No code 则更能发挥算力与泛化。尽管当时技术与用户需求并不支持他们的判断，他们仍用“第一性原理”押注更大市场：全球 3000 万程序员 vs 80 亿人。更关键的观察来自百度内部 Hackathon：早期是产品+设计+研发标配；大模型出现后，逐年减少程序员参与，去年很多队伍甚至没有程序员，依然能得奖，作品比拼的是创意与交付，而非代码质量。于是对话里出现了几句很有时代感的口号演进：

- “Talk is cheap, show me the code” → “Code is cheap, show me the idea” → “Idea is cheap, show me the app”。
他还用“跑步类比”解释 IDE 与 No code 的体验差别：IDE 像“两人三足”，人和 AI 绑腿协作，信息交互成本高、累；No code 像“接力”，AI 跑 99 步，人跑 1 步，只负责交接与冲刺。这种类比背后是产品哲学：**尽量把复杂过程封装掉，让用户只做关键决策**。

做错的部分，他很坦诚地说：他们“给模型太多自由”。模型像自负的年轻人，不懂也硬干，结果干错还不知道。他用歌词类比：“怎么忍心怪你犯了错，是我给你自由过了火。”这其实是 AI 产品普遍困境：越想释放模型能力，越需要约束与评估，否则错误会被放大。另一方面，他认为真正做错的是“运营太晚”：半年没有运营团队，至今没有独立运营预算，用户反问“为啥不宣传”。这解释了为什么他愿意出来上播客：开始补“传播与增长”的课。

结尾的经典提问“300 万美金投三个人”，他先说想投秒哒用户（因为能变三个亿），并提到他们确实有筑梦计划。被要求不能投用户后，他选择投 Gemini（严格说是看好 Google）：技术基因、最大量数据（尤其搜索数据）、以及全球分发渠道，这三者构成“底电”。这与他整期的产业观一致：长期胜负不只看模型参数，还看数据与分发。

**重要原话**

- “**我们做的最对的一个事情就是在低代码和无代码间，选了无代码……Web coding 一定会取代 Workflow。**”`[章节 9.1]`
- “**我们给模型太多的自由……怎么忍心怪你犯了错，是我给你自由过了火。**”`[章节 9.5]`
- “**我们运营做得太晚了……为啥你们不宣传啊？**”`[章节 9.6]`
- “**我肯定会投 Gemini……Google 有最大量的数据……还有全球最大的分发渠道。**”`[章节 9.8]`
**视觉/结构信息描述**

这一段像一份“复盘清单”：

- 战略选择（No code）
- 方法论来源（Bitter Lesson：Learning/Search）
- 体验类比（两人三足 vs 接力）
- 失误（模型自由度、运营滞后）
- 长期下注（数据 + 分发）
**个人感受**

朱广翔在复盘时既有理想主义（押注更大市场），也有现实的懊恼（运营太晚）。这种混合让整期对话不飘：他不把成功归因于“我们更聪明”，也不把失误推给外部。

**延伸思考**

当 No code 成为共识，下一阶段的竞争可能会从“能不能生成应用”变成“谁能规模化地让用户赚到钱”。这要求平台具备增长、支付、分发、合规、行业模板、生态治理等综合能力——也回到他最初的倒金字塔：应用层必须赚到钱，否则整个结构不可持续。

**精华收获**

- 选择 No code 是对“算力与泛化”趋势的押注，而不是短期变现最优解。
- 模型越强越需要约束：自由度与可靠性必须被评估系统与上下文工程平衡。
- 长期竞争看“数据 + 分发 + 云系统能力”，不只看模型本身。

# 全文总结：把这期播客带走的三条主线

1) **历史主线**：编程抽象层级持续上升，Web coding/自然语言生成应用是曲线拟合的极限方向。

2) **产业主线**：健康的 AI 产业应当让应用层赚钱最多，平台的北极星指标可以是“用户 AR”。

3) **产品主线**：下半场拼的不只是生成效果，而是端到端交付系统——后端、云、分发、评估、上下文管理、插件生态，以及与模型保持“15 度夹角”的战略自觉。
