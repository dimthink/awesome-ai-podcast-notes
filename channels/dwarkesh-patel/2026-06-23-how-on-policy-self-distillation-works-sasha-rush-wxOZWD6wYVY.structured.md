# 策略自我蒸馏（On-Policy Self-Distillation）的工作原理

## 从序列知识蒸馏到策略蒸馏

好的，我们来谈谈蒸馏。过去，人们使用一种叫做**序列知识蒸馏**的方法来训练模型。其工作方式是，你有一个**教师模型**，它能生成一个输出序列。这个教师是一个已知在某个任务上表现非常出色的智能模型。你希望有一个较小的**学生模型**，基本学得接近教师的能力。具体做法就是使用标准交叉熵损失，让学生去匹配教师的输出。这种方法在机器翻译等任务中很常见，也是早期训练语言模型完成这些任务的常用方式。

几年前，另一种方法变得相当流行，叫做**策略蒸馏（On-Policy Distillation，OPD）**。策略蒸馏有一个很好的优势：它可以在**强化学习（RL）**中使用，并且与人们常用的一些RL技术配合得非常好。

它的工作方式并非让教师生成输出，而是先让学生生成输出。学生先生成一个序列——这个序列可能是一系列工具调用，比如解决一个编码问题。这个序列可能质量不高。然后你让教师对这个序列中的每一个 Token 进行预先评分。本质上，教师在这些数据上运行一次前向传播，在每个位置上判断它更希望出现哪些 Token。这样你就有了两组对数概率：学生产生的，以及教师产生的。然后你在这两组对数概率之间计算 **KL 散度损失**，并更新学生使其更接近教师的输出。

**提问：** 我能问一个非常基础的概念性问题吗？你可以将上述一种做法理解为简单地将教师的输出作为监督学习目标来匹配。**为什么我们要训练学生去匹配教师？**

**回答：** 在序列知识蒸馏中，你确实是将教师完整的序列输出作为监督学习目标。在这种设定下，输出的**轨迹**或范围完全跟随教师的行为。而在这里（OPD），轨迹则基本跟随学生的行为。

**提问：** 我明白了。

**回答：** 直观的类比是这样的：序列知识蒸馏就像你看着**拉斐尔·纳达尔**打网球，他非常出色，你作为学生试图模仿他。而 OPD 就像你自己在打网球，纳达尔站在你身后，随时纠正你正在犯的每一个错误。

**提问：** 我明白了。

## 策略蒸馏与强化学习的结合

**提问：** 您在关于 Composer 2.5 的博客文章中似乎暗示，这种技术帮助你们解决了**信用分配问题**。您能解释一下这是如何发生的吗？

**回答：** 当然。在此之前，我们还需要再绕一个小弯，那就是指出策略蒸馏在强化学习中运作良好。当进行 RL 时，想象一下正在学习的模型：它本来就会进行 **rollout**（展开）并产生输出。因此，我们基本上已经需要为学生模型执行前向和后向步骤。我们有一个损失，即该场景下的 RL 损失。既然已经有了这个损失，你可以考虑如何移动教师，并添加第二部分损失，我们可以称之为**离策略蒸馏损失**。这个损失通过已知的、来自学生的对数概率与教师会给出的对数概率进行比较来计算得到更新。如果你有这个更新，就可以将其添加进去并获得额外的好处。

**提问：** 有趣。

## 自我蒸馏：没有教师模型时的策略

现在回到 Composer：实际上我们并没有一个教师模型——没有另一个比当前模型更好的模型。因此我们使用了最近变得流行的方法，叫做**离策略自我蒸馏（Off-Policy Self-Distillation）**。其核心思想是，我们人工**合成一个教师**来帮助自己。虽然我们没有更好的模型，但我们可以假装有一个，方法是对学生的损失进行操作并注入**文本反馈**。这个文本反馈是一种额外信息，会被注入到轨迹中，告诉你希望模型做什么。然后你可以计算学生的对数概率，以及学生加上文本反馈后的对数概率。由于条件作用在真实信息上，这两个对数概率在得到反馈消息的位置会产生差异。

**提问：** 我猜这里仍然存在一个问题：你需要不断重新运行这个轨迹，因为最初学生遵循自己的策略时虽然注意到了错误，但仍会陷入循环，一直犯同样的错误。你需要反复重试直到修复那个错误，才能真正学会正确的行为。

**回答：** 是的，如果文本反馈不够强，或者目标不准确的话。

**提问：** 或者假设在别处还有另一个错误。但如果你对学生进行在策略（on-policy）的 rollout，学生已经卡在那里了，它永远不会遇到这个额外的改进。而在序列知识蒸馏的世界里，它就会看到教师所有可能的动作，即使你第一次发球失误，也能看到教师后续更好的表现。

**回答：** 对，我们每次只能变好一点点。从理论上讲，如果你跟随整个输出，也许能做出一些神奇的事情。

**提问：** 然后这里我们要说的是：我们不想学会“在已经完成了一个纳达尔式发球之后该做什么”，因为我现在还没做出纳达尔式发球，然后我怎么回球？我想学的是“我接下来该怎么做”。所以我只是按照自己的方式展开轨迹，然后比较——如果纳达尔发现自己处于我这样糟糕的截击位置，他接下来会做什么？

**回答：** 对，就是这样。你是从一个好的起点开始（序列蒸馏），而这里是从一个很差的起点开始，然后慢慢改进。

## Composer 2.5 中的实现：文本反馈与效果

**提问：** 现在来说说你们实际训练 Composer 2.5 的过程。你们是如何确定合适的**文本反馈**的？是自动化的吗？有没有人参与标注？具体怎么做？

**回答：** 在 RL 过程中，我们有能够读取轨迹的模型。我们编写了一些**手动提示**，用于查找轨迹中样式或过程本身的特定错误。这些提示会建议在什么时候加入文本反馈是有用的，并指向要注入的特定消息。之所以需要这样做，是因为在进行 RL 时，我们不仅仅有三轮交互，可能有数百轮。RL 本身很难定位错误所在。但我们可以用模型告诉我们哪里有问题，然后通过文本反馈直接针对那个问题进行纠正。

**提问：** 效果怎么样？

**回答：** 效果相当不错。你可以看到，当某个特定错误出现时，加入文本反馈后，这些错误的消失速度比完整的 RL 过程快得多。不过目前这还不能完全取代 RL，我们仍然将其与实际系统配合使用。

**提问：** 明白了。嗯，你需要能够判断这是一个容易识别的问题，还是一个更微妙的问题。

**回答：** 是的。我们首先专注于那些容易识别的问题作为起点。不过，扩大这些反馈的范围是一个很有前景的方向。我认为**文本**是沟通信息的一种很好的方式，也是公司和人们之间存储信息的绝佳介质。现代 RL 有一个令人沮丧的方面：即使我们能读取轨迹并知道它们很差，其他模型也能读懂它们，但我们无法直接将那些信息注入进去。

**提问：** 确实有趣。

## 技术细节：重新计算对数概率而非重新 rollout

**提问：** 要执行这个协议，学生先产生一个输出轨迹；然后一个读取模型（另一个模型）查看轨迹并指出错误；接着基于读取模型的分析，构造一个新的提示；然后你**重新执行另一个轨迹**。除非你保留了之前模型的**对数概率**，否则至少要在新模型上重新运行一次前向传播。无论如何，这似乎是一个多步骤过程：必须重新生成轨迹，然后调用评估模型。能不能在**在策略**内完成？如果必须保持读取器，能否与训练循环并发进行？

**回答：** 实际过程比你想象的要简单。你生成了学生输出，它就是一个 Token 序列。把这个序列发送给读取模型，它会隔离出一个特定的位置，指出那里有问题。然后我们只做**文本操作**：直接把这个轨迹拿来，在某个位置插入一些额外的 Token。我们不需要做任何推断（inference），只是把这个带有额外信息的新轨迹拿去运行一次前向传播，这样就能计算出对数概率。然后我们将它从计算图中分离，再与 RL 损失一起反向传播。

**提问：** 我可能误解了什么。但似乎至少需要两个不同的、连续的 rollout：学生一次 rollout，评估者查看并发现问题，写出新提示，基于新提示再做一次 rollout，然后比较。但您听起来好像只需要一次 rollout。我肯定理解错了。

**回答：** 实际上更简单：只有一次 rollout。这里的 Token 和这里的 Token 是完全相同的。事实上，这里和这里的 Token 也完全相同。我们没有生成任何新内容。我们做的只是**重新计算**这些 Token 在一个略微修改过的上下文中的对数概率。也就是说，我们把这个 rollout 的文本拿过来，在某个位置插入一段新文本，然后重新计算在那个新文本下，这些原有 Token 的对数概率会发生怎样的变化。回到网球比喻：这就像我那糟糕的一挥，而纳达尔在我肩膀上对我犯的每一个小错误进行纠正。

**提问：** 所以你不需要额外再做解码（decode）。

**回答：** 不需要重新解码。你只是轻微地修改一下自己做过的一小部分。没错，不过正如你之前指出的，这样做的缺点是你不会取得巨大、惊人的进步。到了最后，你仍然在做之前做的那些笨事，只是在已经发生的事情上获得一些**本地修正**。

**提问：** 明白了。感谢解释。

**回答：** 不客气，谢谢。
