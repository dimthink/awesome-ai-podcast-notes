# 浏览器已死，Codex 刚刚取代了它们

## 引言：AI 代理与超级应用的新方向

2025 年，所有人都以为我们会创造新的浏览器，然后把 AI 注入其中。但我们开始意识到，实际情况完全不是这样。在这期视频中，我想聊聊如何为未来做好准备——未来我们将使用 AI 代理（AI agent）在像 **Claude Desktop** 和 **Codex** 这样的全新超级应用（super app）里工作。接下来我会播放 Lenny 播客中的一些片段。他邀请了一位叫 Dan Shipper 的嘉宾，我认为他对 AI 代理的走向有着极其敏锐的直觉。

## Dan Shipper 的洞察：超级应用将接管一切

Dan 是这样说的：

> 基本上，Anthropic 在某个时刻意识到，当你拥有一个能构建任何东西的编码代理——比如 **Claude Code**——它在处理任何你想做的工作时都非常有用。于是人们开始用各种方式 hack Claude Code，让它完成自己所有的工作。所以 Anthropic 就打造了 **Cowork**，它算是 Claude Code 的一个更精美的包装，但本质还是一回事。我觉得 OpenAI 下了几手不同的棋。他们推出了 **Codex** 桌面应用。在我看来，Codex 桌面应用几乎吸收了 Anthropic 的所有经验——从 Claude Code 到 Cowork，你从 Anthropic 桌面应用 UI 的标签页上就能看出这点。OpenAI 就像是说：我们看到方向了，直接跳到那里吧。现在 Codex 是我的日常工具，我几乎所有时间都泡在里面。我偶尔会切换到 Claude，但我觉得他们抓住了正确的范式。很明显，不管谁现在领先（因为我认为局面还会变化），你做的所有工作都会汇集到其中一个界面上。
## Codex 的应用内浏览器演示

这就是 Dan 的看法。他相信人们大部分的工作都会在这两类应用中的某一个里完成。这些超级应用非常擅长处理各种编程任务或知识工作（比如创建文档、电子表格），基本上你在电脑上能做的事它们都能做。

Dan 接着提到他使用的一个叫 **Proof** 的工具。他的公司 Evry 开发了 Proof，它是一个文档编辑器——类似于 Obsidian 或 Google Docs。但这个编辑器是专门为 **Agent 原生（agent-native）** 设计的，人类和代理可以同时使用。

关键是，这些超级应用（包括 Claude Codex 和 Claude Code）都在内置一个完整的浏览器。这个浏览器就放在代理聊天窗口旁边，让你可以一边使用代理一边像用 Chrome 一样浏览网页。这将彻底改变人们使用电脑的方式。

Dan 进一步解释道：

> 举个例子，当我在写文档时，Codex 内部有一个内置浏览器（in-app browser）。我每个项目都有一个单独的 Codex 会话线程。打开内置浏览器，进入文档——我通常是用自己写的在线 Markdown 编辑器 Proof——然后让 Codex 运行并“看着我”在 Proof 里的操作。Codex 能看到我在做什么，我也能看到 Codex 在做什么。所有东西都集中在同一个地方，这正是最初让 Claude Code 如此好用的延伸。我感觉自己有一个并行工作的伙伴，它不仅能在文档里回复和写作，还能去做调研，能用我的电脑做几乎所有我能做的事。这非常强大。
Riley 接着展示了实际的演示：

为了说明这一点，我现在打开了 Codex 的浏览器。我可以直接在 Google Docs 里输入。我也可以说：“嘿 Codex，请看一下你的记忆，帮我列出所有我常用的工具，然后放进这个文档。” 因为 Codex 能完全看到我的浏览器界面，它拥有浏览器上下文，并且具备**浏览器操控（browser use）**能力。我还可以说：“浏览器，完成后请用浏览器操控检查一下格式是否正确。” 我还可以让 Codex 直接控制这个浏览器来确保格式正确。这些 AI 代理工具可以完全控制你的电脑，拥有你所有集成工具的记忆和上下文。

看，它写下来了。现在它正在验证 Google Docs 的格式。现在它正在完全控制一个浏览器——那个小鼠标不是我，是 Codex 在通过浏览器操控功能控制。

**来自未来的补充：** 就在我准备上传这个视频前一个小时，Codex 刚刚更新了它的内置浏览器。现在当你使用 Codex 时，你可以一直保持登录状态。看，我在 Twitter 上已经登录了。打开一个新标签页，打开浏览器，我可以登录任何网站。访问 twitter.com，依然登录着我的账号。打开 docs.google.com，我已经登录了 Google Docs。我的 Notion 也是登录状态。如果我新建一个会话（按 Command N），打开浏览器，输入 notion.com，只需要点击登录——没错，我已经登录了。一个完整的网页浏览器正在直接嵌入 Codex 内部。

## 从浏览器标签到任务标签：范式的转移

这个变化之所以重要，是因为我们正在从“浏览器标签”的世界转向“任务标签”的世界。

之前有一些 AI 浏览器，比如 **Comet**，它在浏览器里加了一个助手侧边栏，这个侧边栏可以实际控制浏览器，从而创造了新型 AI 浏览器。还有 **Dia** 浏览器、OpenAI 的 **Atlas**，甚至 Google Chrome 自己也开始加入 AI 功能。

但新的未来是：在你的 AI 代理平台或超级应用里，不再是杂乱的一堆浏览器标签，而是左侧边栏里的一堆**任务标签**。当你点击一个任务标签，它会打开一个代理会话线程，同时你会看到你的浏览器——代理可以帮你打开浏览器，你也可以手动打开，但所有与该代理任务相关的浏览器标签都集中在一起。

对于高效工作而言，你不再需要开着几十个标签页并迷失其中，你只需要有：你的任务、你的代理，以及与这个任务相关的浏览器标签。

我们现在正从这种状态（一堆混乱标签页）切换到这种状态：不同的任务，每个任务都有不同的侧边栏。例如，我的个人登陆页面下打开着一个浏览器，这个浏览器运行的网站是这个任务相关的。当我切换到另一个任务，Google Doc 就打开了。我可以与代理交互，代理能控制文档，我也能控制文档。这个应用就是**Agent 原生应用**——你和代理共同控制它，你们可以直接编辑，互相看到对方的修改。整个过程变成了与 AI 代理协作的过程。

随着 AI 代理越来越聪明，理论上它们能够完成更多工作。再加上代理可以控制你的电脑和浏览器，意味着随着时间的推移，它甚至能预测你想做什么，然后直接替你完成。

## 对 SaaS 的影响：Agent 原生应用

这一切对 SaaS 或整个软件行业意味着什么？

Dan 之所以构建 Proof，是因为他认为直接使用 Google Docs 并非理想体验——你需要登录、认证很麻烦、界面也不适合配合 AI 代理使用。于是他构建了一个更轻量、更适合 AI 代理与人协作的文档编辑器。

他的核心论点是：所有最好的 SaaS 公司都会逐渐变成这类 **Agent 原生应用**——你可以带着你的 AI 代理一起使用它们。与在 Google Docs 里嵌入一个 Gemini 按钮不同，那个与 Google Docs 绑定的 AI 缺少 Codex 所拥有的完整上下文。Codex 连接了你所有的应用，它了解关于你的一切。

所以，SaaS 的未来可能不再是构建一个内嵌 AI 代理的应用，而是创建一个让你的现有代理（越来越多人会使用的代理）能够访问的应用。让你的用户带着他们自己的代理（bring their own agent），使用他们自己的 token。而不是像 Lovable、Replit 或 Gamma 那样，由平台提供代理和 token。

## 未来预测与建议

作为视频的总结，我想做一些预测和建议。

### 预测一：每个会话（或任务）将支持多个浏览器标签

我认为 Claude Code 和 Codex 都会添加这个功能。目前我只能在一个会话里打开一个浏览器，但我相信将来你可以在每个聊天线程里打开任意多个浏览器。

### 预测二：代理将在你启动任务时自动打开一个或多个浏览器标签

假设我想创建一些 Twitter 帖子，我通常使用 Typefully 起草，同时还会打开 Notion。未来 Codex 可能会自动学习我的习惯：每次我说要写推文，它就自动打开 Notion 和 Typefully。你在超级应用里交互的代理会根据你在浏览器里的活动学习，然后自动开始做事。也许工作流自动化的未来不再是手动构建流程，而是反复做高质量的任务，然后你的 AI 代理慢慢学会为你完成这些任务。我相信这是方向。

### 预测三：Agent 原生应用将演变为一种新类型——生成式 UI，与你自己的工具集成

在 Codex 里你可以创建插件。比如我有一个 Gmail 插件，我可以说：“读取我最近的 20 封邮件，找出三封需要回复的，并生成全部三封草稿。” 我认为未来超级应用中的 AI 会生成一个 UI（或小应用），显示出输入信息和回复。你的代理会自动创建这些**生成式 UI（generative UI）**让你完善。你可以在浏览器里打开它，检查代理的工作，然后直接在代理生成的应用里发送。你将能够创建与所有工具集成的迷你应用，并使用所有集成功能。这样我们就过渡到了“贾维斯”式的未来：你对电脑说一句话，砰，生成式 UI 出现，你像托尼·斯塔克一样做点微调，然后说“发送，发送出去”。

我认为这个未来并不遥远——我称之为“生成式迷你应用”，大概未来 3 到 6 个月就能实现。

### 建议

1. **从任务和 SOP 层面思考你的整个工作。** 把你做的所有事情整理成一个任务列表，记录完成一件事的每一步。一旦你创建了这些文档，你就能在任务级别上组织工作。当 Codex 和 Claude Code 添加完整的浏览器功能（我保证它们会）时，你将能够把工作高效地拆分成任务，完成更多事情。
1. **如果你是构建者、独立黑客，或者想通过销售应用赚钱，我强烈建议你冒一次险。** 从提交到应用商店的静态应用，转向迷你应用或 Agent 原生应用——你和代理可以共同使用的应用。一旦你深入这个领域，你会发现非常多的有趣想法。你可以创建一个“技能”，让其他人安装到 Codex 上，从而更轻松地完成特定任务。
1. **去学习 Codex 和 Claude Code。** 我认真告诉你，Codex 和 Claude Code 是所有知识工作的未来。编程任务、通用知识任务，大部分事情都会在 Codex 和 Claude Code 里完成。我完全同意 Dan Shipper 在 Lenny 播客的说法，我觉得他说得完全正确。学习如何使用 Codex 和 Claude Code 是当前商业世界里最重要的事情之一。
