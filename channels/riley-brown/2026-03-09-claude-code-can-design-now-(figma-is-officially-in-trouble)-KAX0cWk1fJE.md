### 一、挑战宣言：Figma 的危机与 Paper 的登场 `[视频开头]`

**核心观点**

Riley 开门见山地宣告了一个可能让设计圈震动的消息：设计师们正在从 Figma 转向一个名为 Paper 的新工具，因为它天生就是为 AI 智能体（如 Claude Code）设计的。这场转移的核心原因不是功能多少，而是“可被 AI 操控”的能力。

**深度阐述**

视频一开始，Riley 坐在镜头前，带着一贯的兴奋语气说：“今天，我们要用 Claude Code 在 Figma 上设计个人网站。等等，不是 Figma。”他故意制造了一个悬念，然后揭晓答案——Paper。这个工具官网是 paper.design，它在外观上跟 Figma 极其相似，有画板、图层、工具条，但底层架构完全不同：Paper 从设计之初就考虑到了 AI 智能体的接入。这意味着，AI 不仅可以读取设计文件，还可以像人类用户一样创建、修改、删除元素。

Riley 提到，“许多设计师告诉我他们正在从 Figma 切换到 Paper，因为它更容易与 AI 智能体配合使用。”这句话背后折射出一个关键趋势：随着 AI 编程工具（如 Cursor、WindSurf）的普及，设计师和开发者之间的界限正在模糊。如果一个工具能让 AI 直接操刀设计，那么“设计”本身就不再是专业软件的专利，而变成了“意图表达”的副产品。Figma 虽然强大，但其操作逻辑是为人类手指和鼠标优化的，AI 要控制它需要复杂的 API 调用和插件支持；而 Paper 则直接为 AI 提供了“原生接口”——即我们后面会讲到的 MCP 服务器。

Riley 的个人频道一直聚焦于“用 AI 智能体高效经营业务”，这个视频正是他探索前沿工具链条的又一次尝试。他的热情和急迫感很容易感染观众：仿佛错过今天，就会错过一个时代。

**个人感受**

Riley 在提到设计师们的迁移时，语气中带着一丝“早告诉过你”的得意。他显然对自己正在探索的这条路径充满信心，这种情绪在后续的演示中不断被放大——每一次 AI 的成功操作，都会引来他一句情不自禁的“This is crazy”。这种真实的情感流露，让观众能感受到技术带来的震撼，而不是冷冰冰的教程。

**延伸思考**

Figma 被挑战的根源是什么？不是功能不足，而是“交互范式”的错位。过去二十年，设计软件都在优化“人机交互”，而未来十年，核心将是“机机交互”——AI 智能体需要能够无缝地读取、理解和操作设计文件。如果 Figma 不尽快提供类似 MCP 这样的原生 AI 接入层，那么像 Paper 这样的新物种就会逐渐蚕食其市场。这不仅仅是设计工具的战争，更是整个创意软件领域的范式转移。

### 二、工具准备：Paper 下载与 Claude Code 环境搭建 `[下载 Paper - 打开 Cursor]`

**核心观点**

要实现 AI 驱动设计，需要两个核心工具：Paper 设计软件和 Claude Code（通过 Cursor 终端调用）。整个流程的起点非常简单，但背后是全新的“AI 即用户”理念。

**深度阐述**

Riley 的演示非常务实。他首先打开浏览器，访问 paper.design，下载并安装 Paper。安装过程如同普通 Mac 应用：拖动到 Applications 文件夹，然后双击打开。初次启动的 Paper 界面让 Figma 用户倍感亲切——左侧是图层列表，中间是画板，右侧是属性面板。但 Riley 提醒观众：“控制 Paper 的不是鼠标，而是 Claude Code。”

接着他打开 Cursor（一个集成了 AI 能力的代码编辑器），用快捷键 `Cmd+Shift+N` 创建新项目，新建一个名为 `control-paper` 的文件夹。在 Cursor 的终端中，他输入命令 `claude` 启动 Claude Code，并带上了 `--dangerously-skip-permissions` 和 `--high-effort` 参数。前者是为了绕过权限询问（演示环境），后者则是要求 Claude 投入更多计算资源来确保任务质量。

这一步看似平常，却蕴含深意：Claude Code 不再是一个简单的聊天机器人，而是一个能够执行终端命令、读写文件、甚至调用外部服务的“智能体”。它运行在开发者最熟悉的环境中（终端），却能跨应用操作，这得益于即将登场的 MCP 协议。

**视觉/结构信息描述**

视频中展示了 Paper 打开后的默认界面：一个空白的画板，左侧工具栏有箭头、形状、文本等图标，右侧是空白的属性面板。Cursor 的终端则显示 `claude` 命令启动后的欢迎信息，以及当前工作目录。这种并置的画面直观地展示了“双工具”协作模式。

**个人感受**

Riley 在操作时非常流畅，显然已经熟悉这套流程。但他也不忘自嘲：“我喜欢在 Cursor 里用 Claude Code，虽然不确定这是不是最佳方式。”这种坦诚让观众感觉他不是高高在上的专家，而是一个和你一起探索的同行者。

**延伸思考**

MCP（Model Context Protocol）是 Anthropic 推出的开放协议，旨在让 AI 模型能够与外部工具进行标准化交互。如果把 Claude 比作大脑，MCP 就是神经系统，连接着各种感官和肌肉（如 Paper、浏览器、文件系统）。未来，任何软件只要实现了 MCP 服务器，就能被 AI 直接操控，这为“软件即服务”带来了全新的想象空间——我们或许不再需要学习每个软件的快捷键，只需要告诉 AI“我想做什么”。

### 三、建立连接：MCP 服务器的自动配置 `[终端中首次输入指令]`

**核心观点**

Claude Code 能够自主搜索互联网，学习如何为 Paper 设置 MCP 服务器，从而获得对 Paper 的完全控制权。整个过程无需人工干预，AI 展现出了“自我工具化”的能力。

**深度阐述**

Riley 在终端中向 Claude Code 发出了第一个指令：“我希望你搜索互联网，找出如何设置 paper.design 的 MCP 服务器，这样你就能控制我的 Paper 设计，创建和编辑设计。完成整个流程。完成后不要问我，我希望你能告诉我你能否控制我当前在 Paper 中看到的画板，并添加一句简单的文字‘Hello’。”

这个指令极其关键，因为它包含了几个层次的意图：

1. **自主搜索**：Claude 需要联网查找 Paper 的 MCP 服务器配置方法。这意味着它必须具备实时获取信息的能力（Claude Code 确实支持联网）。
1. **目标导向**：最终目标是“控制画板”，而不是仅仅了解概念。
1. **验证标准**：添加“Hello”文字，以此证明控制成功。
Riley 没有手动配置任何东西，而是完全交给 Claude。几秒钟后，Claude 开始行动：它可能访问了 Paper 的官方文档或 GitHub 仓库，找到了 MCP 服务器的启动方式，然后尝试在本地运行该服务器。终端输出显示“Paper is installed. MCP is running on port 29979”，但紧接着出现“404”错误。Riley 注意到问题，并意识到 Claude 需要重启才能使用新安装的服务器。于是他清空对话，重新提问：“你能在我的 Paper 画板上添加‘Hello’吗？”

这次，Claude 成功了。它通过 MCP 服务器向 Paper 发送指令，创建了一个新的画板，并在中央添加了文本“Hello”。Riley 切换到 Paper 窗口，画板上赫然出现了预期的内容。他乘胜追击，让 Claude 把文字改为“Hello, I’m Claude”，Paper 中的文字应声而变。

**视觉/结构信息描述**

视频中，当 Claude 执行操作时，Paper 界面会实时更新：先是出现一个带有“Hello”的画板，然后文字瞬间变成“Hello, I’m Claude”。这种实时反馈让人惊叹，仿佛有一双无形的手在设计软件中操作。

**个人感受**

Riley 在文字成功修改后忍不住欢呼“Boom! There you go.”，语气中充满了孩子般的惊喜。这种兴奋感极具传染力，让观众也能体会到“命令即创造”的魔力。他随后让 Claude“清空页面，从头开始”，画板瞬间恢复空白，整个过程行云流水。

**延伸思考**

MCP 服务器的自动配置能力，标志着 AI 进入了一个新阶段：它不仅能用已有的工具，还能自己“安装”和“配置”新工具。这类似于人类程序员阅读文档、搭建环境的过程。当 AI 具备了这种能力，任何软件只要提供公开的 API 或协议，就能瞬间被 AI 生态吸纳。对于软件开发者来说，这意味着未来的竞争焦点将从“功能多寡”转向“是否容易被 AI 集成”。

### 四、设计灵感与任务描述：从一张截图开始 `[描述设计目标]`

**核心观点**

Riley 展示了他在 Twitter 上发现的一个设计灵感图，以及另一张展示信息布局的参考图，然后以极其口语化的方式向 Claude 描述了他的需求。AI 需要同时理解视觉风格和内容结构，并将其转化为 Paper 中的设计。

**深度阐述**

在清空画板后，Riley 开始阐述设计目标。他展示了两张图片：

- 第一张是一个简洁的个人页面，左侧是列表，右侧是详细信息区域。
- 第二张是一个卡片式布局，展示了某个“智能体”的详细信息，包括名称、描述、技能标签等。
他的需求是：“我想创建一个关于我的 OpenClaw 智能体团队的页面。左侧面板是智能体列表，右侧面板显示每个智能体的信息，信息格式参考第二张图。”

这里的关键在于，Riley 没有提供任何设计文件（如 Figma 链接），只是用自然语言描述，并让 Claude“使用这两张图片作为输入”。这意味着 Claude 需要具备多模态理解能力（虽然 Claude Code 本身可能不支持图像识别，但 Riley 可能通过某种方式让 Claude 能够访问这些图片，或者他假设 Claude 可以通过描述理解）。实际上，Claude Code 是基于文本的，但 Riley 可能通过上传图片到某个可访问的 URL 或使用其他多模态模型辅助。视频中没有详细说明，但可以推测他可能把图片放在了某个 Claude 能访问的地方（如 GitHub 仓库），或者他描述得足够详细。

**视觉/结构信息描述**

视频中，Riley 展示了这两张图片（可能是屏幕截图）。第一张图片是典型的“列表-详情”布局，左侧有“Agent 1, Agent 2...”等条目，右侧是空白区域；第二张图片则是一个卡片，顶部是智能体名称，下面是描述、技能标签、一个圆形图标等。这些视觉元素将成为 Claude 的设计参考。

**个人感受**

Riley 在描述时显得非常自然，就像在跟一个人设计师沟通：“左边是列表，右边是信息，每个智能体的信息就像这张图里那样。”这种轻松的语气让人感觉设计门槛被大幅降低了——你不再需要精确的尺寸、颜色代码，只需要说“像这样”就可以了。

**延伸思考**

这种“基于参考图的设计”对 AI 来说是一个巨大的挑战，因为它需要理解布局的意图，并能在新画布上重新演绎。如果 AI 能够做到这一点，那么“设计”将变成一种“对话式重构”：你给出一堆灵感图，AI 综合它们的特点，生成符合你口味的独特设计。这比单纯的“文生图”更进一步，因为它是在专业的矢量设计软件中生成可编辑的元素，而不是像素图片。

### 六、迭代与细化：字体、图标与模块调整 `[修改指令]`

**核心观点**

**深度阐述**

Riley 对初版设计基本满意，但希望尝试不同风格。他发出新指令：“创建另一个完全相同的副本，但将标题和正文的字体都换成 Instrument Sans，如果系统没有该字体，请下载安装，放在当前版本旁边。” 几秒钟后，Paper 中出现了两个并排的画板，右边的版本使用了新的字体。Riley 比较后觉得新字体更顺眼。

接着，他要求改进图标：“请为这些智能体使用更好的图标，比如社交媒体就用对应的品牌 logo。” Claude 接收到指令后，开始替换左侧列表项的图标，出现了 Twitter、YouTube 等熟悉的 logo。

然后，Riley 想增加一个新功能模块：“在现有版本右侧创建一个新画板，上面有一个叫‘Learn’的新板块，里面有一个页面叫‘What is OpenClaw’，这是一个介绍 OpenClaw 的学习页面，要让它可视化，并在右侧展示出来。” Claude 迅速响应，创建了新画板，添加了学习内容，甚至保留了原有的智能体列表。

**视觉/结构信息描述**

视频中可以看到，新画板被创建在右侧，标题为“Learn”，下面有“What is OpenClaw”的卡片，卡片内包含一些要点和简单的图示。图标也变得更精致，比如社交媒体智能体旁边出现了对应的品牌图标。

**个人感受**

Riley 在发出指令时，语气就像在跟一个效率极高的设计师同事沟通：“请把这个改成那样，再试一下那个。” 而 Claude 的每次响应都让他更兴奋：“This is so sick. So easy to use.” 他特别提到“AI generated design”并笑着摇头，仿佛不敢相信这一切真的发生了。

**延伸思考**

这种迭代方式极大地降低了设计修改的成本。传统设计中，更换字体可能需要逐个文本框修改，而 AI 可以在几秒内完成全局替换。更重要的是，AI 能够同时生成多个版本供人选择，这解放了设计师的创造力，让他们从执行中抽身，更专注于概念和决策。

**核心观点**

**深度阐述**

在基本设计成型后，Riley 希望增加一个管理功能：“添加一个设置模块，当我点击右侧的按钮时会弹出，里面有一个‘启用编辑模式’的开关，需要密码，这样管理员才能编辑网站数据并添加新智能体。” 但他意识到这个功能可能太复杂，于是改为：“我希望这是一个新页面，放在当前画板的右侧，像一个新画板。”

**视觉/结构信息描述**

视频中，新的画板显示了一个类似模态框的界面，顶部是“Settings”，下面有一个密码输入框和一个“Enable Edit Mode”的开关。齿轮图标替换了原来的“R”，显得更加专业。

**个人感受**

Riley 对这个复杂指令的执行结果非常满意，连说两次“This is awesome”、“I'm pretty blown away by this”。当看到 Claude 同时修改了两个画板时，他感慨“This is fire”，情绪达到了高潮。

**延伸思考**

这个例子展示了 AI 不仅能处理视觉设计，还能理解交互逻辑。密码保护、编辑模式这些概念对 AI 来说属于“功能需求”，而 Claude 能够将其转化为具体的 UI 元素，说明它具备一定的产品设计思维。未来，AI 或许能根据一句话需求，自动生成包含完整交互的原型。

## 精华收获

通过 Riley Brown 的这次实验，我们至少获得了以下核心洞察：

1. **范式转移：AI 即用户**
MCP 协议让 AI 能够像人类一样操作软件，这意味着未来每一个软件都需要考虑如何被 AI 调用。那些为 AI 提供原生接口的工具（如 Paper）将占据优势，而固守“人类操作逻辑”的软件可能被边缘化。

1. **设计-开发一体化**
AI 打通了从设计到代码的壁垒，设计文件不再需要手动还原，而是可以直接转化为可部署的应用。这将极大缩短产品迭代周期，让创意快速落地。

1. **自然语言即编程语言**
整个流程中，Riley 没有写一行代码，却完成了一个功能完整的网站。他使用的“语言”是描述性、对话式的，这预示着未来编程将向自然语言靠拢，更多人将获得“创造软件”的能力。

1. **迭代成本趋近于零**
在 AI 的帮助下，生成多个设计方案、尝试不同字体、修改布局，都只需几秒钟。设计师可以更专注于探索，而不是执行。

1. **对设计师的新要求**
虽然 AI 承担了大量执行工作，但人的作用并未消失——设定目标、提供灵感、做出决策、评估结果，这些仍然需要人类的审美和判断。未来的设计师可能更像“AI 训练师”或“创意导演”，需要学会如何向 AI 清晰表达意图，并引导其生成符合预期的结果。

这场由 Claude Code 和 Paper 掀起的浪潮，才刚刚开始。Figma 是否真的陷入麻烦，取决于它能否快速适应这个“AI 即用户”的新世界。但对创作者而言，一个更激动人心的时代已经到来：你只需想象，AI 便会帮你实现。
