Anthropic因npm registry中的map文件意外泄露了Claude Code的完整源代码，Riley以“好奇心驱动”下载并决定进行实验性改造，明确声明不用于商业目的。

### 二、用OpenAI的CodeEx“解剖”Claude Code `[视频 01:30-03:00]`

**核心观点**

Riley选择用OpenAI的CodeEx工具来分析Claude Code的源代码，而不是直接用Claude本身，以避免“用自己的代码分析自己”的尴尬或潜在风险。

**深度阐述**

拿到源码后，Riley面临第一个有趣的选择：用什么工具来分析它？按理说，Claude Code是Anthropic的产品，用Claude来分析似乎最自然。但Riley感到“用Claude Code编辑Claude Code的源代码会感觉很奇怪，我觉得这样做可能会被放进某个奇怪的数据库里” `[视频 01:35-01:45]`。于是，他选择了竞争对手OpenAI的CodeEx——一个专门用于代码理解和重构的AI工具。

在CodeEx中，Riley添加了源码文件夹作为新项目，然后输入了一个简单的指令：“请告诉我你在这个项目中看到了什么？总结这个agent。” `[视频 01:45-01:55]` CodeEx的返回结果让Riley大为震撼：

> “这是Anthropic的Claude Code CLI的源代码树。CLI入口点直接打印‘Claude Code’，并且有一个专用的快速通道用于远程控制、守护进程工作器、后台会话、MCP助手和其他agent运行时模式。这包含了Claude Code拥有的一切。内置的自定义插件agent、多智能体委托在这里是真实存在的。这真的很酷。” `[视频 02:00-02:30]`
这段话揭示了几个关键信息：Claude Code不仅仅是一个简单的聊天界面，而是一个复杂的多agent系统。它支持后台会话、插件机制，甚至可以委托多个子agent协作完成任务。Riley特别强调“multi-agent delegation is real here” `[视频 02:25]`，说明这个特性在AI编程助手中尚属前沿。

**个人感受**

Riley的兴奋再次爆发：“这真的很酷！” `[视频 02:28]` 他像考古学家发现了宝藏一样，逐字逐句地阅读CodeEx的分析报告。这种“偷窥”顶级AI产品内部实现的快感，是所有技术爱好者都能共鸣的。同时，他选择CodeEx而非Claude也透露出一丝幽默的“阵营意识”——用对手的工具来分析对手的代码，这本身就是一种行为艺术。

**延伸思考**

这里引申出一个重要问题：为什么AI公司如此重视“多agent委托”？传统上，一个AI助手只能串行执行任务，效率低且容易超时。而多agent系统可以让一个“主管agent”将子任务分发给多个“worker agent”并行处理，然后汇总结果。例如，如果你要求“分析这个代码库并生成文档”，主管可以派一个agent扫描文件结构，另一个agent分析函数依赖，第三个agent撰写描述。Claude Code内置了这一能力，意味着它从设计之初就是为了处理大型、复杂的编程任务。Riley的发现也暗示：未来的AI编程工具将越来越像“AI项目经理”而非“AI打字员”。

**精华收获**

- 用竞争工具分析源码是一种有趣的“技术中立”实践
- Claude Code的核心架构包含多agent委托、后台会话、MCP助手等高级特性
- 源码分析能揭示产品的真实能力，远比官方文档更可靠

### 三、让代码“跑起来”：创建Riley Code `[视频 03:00-04:30]`

**核心观点**

Riley让CodeEx帮助配置本地运行环境，将修改后的版本命名为“Riley Code”，并成功在终端中启动了一个可交互的AI编程助手。

**深度阐述**

分析只是第一步，Riley真正想做的是：“我想像运行Claude Code一样运行这段代码，但我要运行这个版本——然后我要开始做出改变，改变它的人格。” `[视频 02:40-02:50]` 他向CodeEx下达了指令：做一切必要的事情来在终端中本地运行这段代码，并且修改UI，将应用名称改为“Riley Code”，这样他就能确认修改成功。

这里有一个技术细节：运行Claude Code需要Anthropic的API Key。Riley前往Anthropic控制台创建了一个新密钥，并在视频中小心地避免泄露 `[视频 03:10-03:30]`。CodeEx处理了一段时间（Riley说“响应非常长” `[视频 03:35]`），最终给出了运行命令：在终端中输入`riley code`。

Riley打开一个新的终端窗口，输入命令。屏幕上的输出变成了：“Riley Code. Say hello. How can I help you today?” `[视频 03:55-04:05]` 这一刻，实验成功了一半——他拥有了一个自己命名的、基于Claude Code源码的AI编程助手。

**个人感受**

当终端第一次打印出“Riley Code”时，Riley的语气里有一种孩子般的成就感。他没有大张旗鼓地庆祝，而是平静地继续提问：“你是Claude Code吗？” `[视频 04:00]` 这个问题既是在测试AI的反应，也是一种微妙的身份确认：我创造的这个东西，到底还是不是你，Claude？

**延伸思考**

“重命名”看起来只是换了个字符串，但在软件工程中，这是“分叉”（fork）的第一步。许多伟大的开源项目都是从重命名开始的——比如LibreOffice从OpenOffice分叉，MariaDB从MySQL分叉。当然，Riley无意创建一个长期维护的分支，但这个动作象征着：当代码在你手中，你就拥有了定义它的权利。这种“所有权”感，是单纯使用API无法获得的。

**精华收获**

- 本地运行AI Agent需要API Key，但过程比想象中简单
- 重命名是赋予代码“自我认同”的第一步
- 成功的标志不是代码无报错，而是你看到自己的名字出现在终端中

### 四、“Status Maxing”与“Aura Farming”：修改Agent的动词库 `[视频 04:30-06:00]`

**核心观点**

Riley发现Claude Code在执行任务时会显示一些特定的“动作动词”（如“spelunking”、“symbioting”），他决定将这些动词替换为网络流行梗，让Agent的行为更有趣。

**深度阐述**

在初步运行成功后，Riley开始探索代码的个性化空间。他问CodeEx：“你能在代码库中找到那些动词吗？比如‘spelunking’（洞穴探险）、‘working’（工作中）。我想知道所有的动词，因为我想制作我自己的版本。” `[视频 04:10-04:25]` 他注意到当他说“我想构建一个应用”时，AI会回复“symbioting”（共生） `[视频 04:20-04:25]`——这些动词是Claude Code在思考或执行任务时显示的状态提示，用于让用户感知到AI正在“做某事”。

Riley觉得这些动词太正经了。他想要更幽默、更符合网络文化的版本。于是他向CodeEx提出：“请查找像‘maxing’、‘jester’、‘gooning’、‘frame mogging’这样的动词，以及其他在流行文化中有趣的动词，给我列出20个。” `[视频 04:45-04:55]`

CodeEx返回的列表包括：`status maxing`（状态最大化）、`clout maxing`（影响力最大化）、`mewing`（舌头顶上颚，一种网络健身梗）、`aura farming`（气场耕作）、`fire`（炸裂）、`gooning`（沉迷）等 `[视频 05:00-05:20]`。Riley将这些动词复制到配置文件中，然后重新运行Riley Code。

效果立竿见影。当他输入“hello make an app”时，AI回复中出现了“status maxing”和“clout maxing”；当他要求做一个待办列表时，出现了“mewing” `[视频 05:30-05:50]`。Riley忍不住笑出声：“这太搞笑了。” `[视频 05:45]`

**个人感受**

这是整个视频中最具娱乐性的部分。Riley显然非常享受这种“把正经AI带偏”的过程。他选择这些网络梗并非随意，而是经过深思熟虑——他先让CodeEx调研了“流行文化中有趣的动词”，然后再应用。这说明他追求的不仅仅是搞笑，而是一种“文化共鸣”：让AI使用用户（尤其是年轻开发者）日常交流中的语言，从而降低对话的距离感。

**延伸思考**

修改动词看似只是表面功夫，但实际上是改变AI“人格”的最直观手段。在UI设计中，“微文案”（microcopy）——按钮上的文字、加载状态提示、错误信息——对用户体验的影响远超其篇幅。一个说“正在处理，请稍候...”的应用和一个说“status maxing... gonna be lit”的应用，给人的感觉完全不同。前者是工具，后者是伙伴。Claude Code的设计者显然也意识到这一点，所以他们把动词做成了可配置的字符串，而不是硬编码。Riley只是将这种可配置性发挥到了极致。

**精华收获**

- AI的“人格”很大程度上体现在它的语言风格上，尤其是那些非核心功能的“微文案”
- 网络流行语能极大降低AI与用户的距离感，但需要谨慎使用以免过时
- 修改配置文件比修改核心代码更安全、更易于回滚

### 五、“19岁实习生”人格：改变AI的说话方式 `[视频 06:00-07:30]`

**核心观点**

Riley进一步深入，要求CodeEx修改Agent的核心人格：全小写、直接、快节奏、多消息分段、像群聊一样随意，模仿“聪明但不太懂编码的19岁实习生”。

**深度阐述**

动词改完后，Riley觉得还不够。他想要更深层次的个性改造。他提出了一组具体要求：

> “我希望它全部用小写回复，非常直接，就像一个19岁的实习生，非常聪明，有点像个通才，不太懂编码。他们回复的节奏很快，有时会发多条消息，但低调地超级聪明。你就把你当成在一个群聊里。” `[视频 06:00-06:25]`
这是一个非常具体的“人物设定”。Riley要求CodeEx“深入思考如何改变代码库中agent的人格”，然后实施这些修改 `[视频 06:25-06:35]`。

在等待处理的同时，Riley还让CodeEx将整个代码库的架构分解为12个关键部分，并生成一份文档 `[视频 06:35-06:50]`。CodeEx创建了一个`personality.txt`文件，并解释说：“我将人格作为专用的提示词文件，这样在不接触打包代码的情况下也能轻松调整。” `[视频 06:55-07:05]` 接着，“启动器现在通过支持的系统注入了Persona。” `[视频 07:05-07:10]`

Riley重新运行Riley Code，输入“Hey, I have a task for you.” AI回复道：“hey, what's the task?”（小写，无标点），然后继续：“what it is, it's a coding, not just a chatbot.” `[视频 07:15-07:30]` 效果完全符合预期——全小写、分段输出、语气随意。Riley确认：“我们确实稍微改变了agent的人格。这很酷。” `[视频 07:25-07:30]`

**个人感受**

Riley对这个结果显然非常满意。他设计的这个“19岁实习生”形象，本质上是对传统AI“一本正经”风格的反叛。在现实生活中，年轻人之间的群聊往往是碎片化、跳跃式、充满缩写的。让AI模仿这种风格，会让它感觉更像“我们中的一员”，而不是一个高高在上的专家系统。Riley的幽默感在这里体现得淋漓尽致：他不要一个更聪明的AI，他要一个更“像人”的AI——哪怕这个人是一个还没毕业的实习生。

**延伸思考**

这里触及了AI人格设计的核心问题：我们到底希望AI像谁？大多数人默认希望AI像“一个知识渊博、礼貌耐心的专家”。但Riley的实验表明，对于特定场景（比如年轻开发者之间的协作），一个“接地气、有点自嘲、不装深沉”的AI可能更受欢迎。这背后是一种“去权威化”的趋势：用户不再需要一个“老师”，而是一个“队友”。此外，Riley让CodeEx将人格分离到外部文件的做法，本身就是一种良好的软件设计模式——将“内容”与“逻辑”解耦，使得非技术人员也能调整AI的行为，而无需理解复杂的代码结构。

**精华收获**

- AI的人格可以通过独立的提示词文件进行热插拔式修改
- “19岁实习生”人格的核心特征是：小写、直接、快节奏、群聊感
- 好的架构设计让AI的“灵魂”与“躯体”分离，便于实验和迭代

### 六、构建桌面应用：从终端到图形界面 `[视频 07:30-09:00]`

**核心观点**

Riley决定为Riley Code创建一个基于Electron的桌面应用，灵感来自T3 Chat和OpenAI的CodeEx，要求界面简洁、显示工具调用、支持终端UI的交互方式。

**深度阐述**

终端界面虽然有效，但不够现代。Riley想要一个真正的桌面应用。他提供了具体的视觉参考：“我想要创建一个像OpenAI CodeEx一样 sleek 的编码应用。我甚至抓取了T3 Chat的图片。这应该是一个Electron应用，让我能像在终端UI中一样与Riley Code交互。请确保所有的工具调用等都能显示出来，同时保持极简。” `[视频 07:40-08:00]`

他给CodeEx看了T3 Chat的截图，以及一张山景图作为背景参考。CodeEx生成了8个不同的设计方案，Riley选择了其中一个基于“workspace文件”的方案，并要求“请实现这些文件，并将它们放入agent的指令中以维护这些文件。” `[视频 08:10-08:25]`

经过一段时间的处理，CodeEx成功启动了一个桌面应用窗口。Riley打开后，界面显示着之前的聊天记录。他输入“Hello there. My name is Riley.” AI回复：“jester gooning. hey riley. nice to meet you. what are we working on today?” `[视频 08:40-08:55]`

但很快遇到了问题：当AI尝试执行bash命令时，系统提示“需要你的批准”。Riley不想每次都点击确认，于是要求“请让它使用`dangerously skip permissions`” `[视频 09:05-09:15]`。CodeEx照做了，并提示“桌面应用现在以跳过权限模式启动”。

**个人感受**

### 八、内置浏览器：像Claude Artifacts一样预览内容 `[视频 10:30-11:30]`

**核心观点**

Riley要求桌面应用内置一个浏览器面板，用于预览AI生成的链接（如localhost服务）和文件路径，模仿Claude Artifacts的功能。

**深度阐述**

在修改UI的同时，Riley想到了一个更强大的功能：“我希望在这个应用中内置一个小型网页浏览器，默认关闭，可以打开在右侧。非常类似于Claude Artifacts。我点击的任何链接都应该在那个浏览器中打开。” `[视频 10:30-10:50]`

这个想法源于实际使用场景：当AI生成一个Web应用或一个本地服务器链接时，用户需要复制链接到外部浏览器查看，体验割裂。如果能在应用内部直接预览，就能实现“生成即预览”的无缝体验。Riley进一步要求：“任何时候我生成内容，消息中最后一个链接应该自动打开那个小浏览器。如果有文件路径，也应该在那个区域中打开。” `[视频 10:50-11:05]`

这个功能实现起来比较复杂，Riley自己也承认“这可能需要一段时间” `[视频 11:05]`。但CodeEx最终成功集成了一个内置浏览器面板。Riley用生成的PDF文件链接进行了测试——点击链接，文件在右侧面板中打开。他兴奋地说：“瞧，它就这样切换了。” `[视频 11:25]`

**个人感受**

这个功能让Riley的应用从“聊天界面”升级到了“开发环境”。他不再满足于只是和AI对话，而是希望AI生成的内容能够立即被消费和交互。这种“对话即产品”的愿景，正是Claude Artifacts、ChatGPT的Code Interpreter等功能的核心理念。Riley通过自己的实验，复现了这一理念，并且完全基于泄露的源码和自己的修改。

**延伸思考**

内置浏览器本质上是一个“安全沙箱”。在传统开发中，要预览一个本地服务器，你需要手动打开浏览器输入`localhost:3000`。而Riley的应用将这一步骤自动化了。更进一步，如果这个浏览器能够支持交互（比如点击按钮、提交表单），那它几乎就成了一个“内嵌的IDE预览窗口”。对于前端开发场景，这意味着AI可以在对话中直接展示UI修改的实时效果，极大缩短反馈回路。这也解释了为什么Anthropic要花大力气做Artifacts——它不仅仅是“显示内容”，而是“让内容可操作”。

**精华收获**

- 内置预览功能消除了“生成”与“消费”之间的摩擦
- Claude Artifacts的设计理念可以复现和扩展
- 自动化打开最后一个链接是一个聪明的用户体验细节
