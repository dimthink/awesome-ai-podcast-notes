# 用 OpenAI Codex 构建一个能“编写”应用的 iOS 应用

## 项目背景与目标：打造一个更好的 Vibe Coding 移动应用

你不仅可以用零编程经验来“氛围编程”（vibe code）应用，你甚至可以“氛围编程”出一个能用来“氛围编程”应用的 App。今天，我就要来做这件事。

我整个周末都在测试 Replet，但它并不是我最喜欢的工具。所以，我打算在不写一行代码的情况下，制作一个我自己的 Replet 版本。在本视频中，我将设计并构建一个 Swift 应用，用来替代 Replet。我将使用 **Codex 应用**，这是我在桌面上最喜欢的新编程工具。同时，我还会在这个 Swift 应用中集成 **Claude Code**，让它真正驱动应用的生成。

让我们别浪费时间了，直接开始构建一个**能构建应用的应用**。

## 现有问题与解决方案构想

### Replet 移动应用的痛点

这就是 Replet 移动应用。对我来说，它所有的东西都太小了，界面非常拥挤。我真的不太喜欢这个应用的体验。我觉得它的主屏幕有点令人困惑，而且很枯燥。

### 我们的解决方案：构建一个更好的替代品

所以，今天我们要构建一个更好的“氛围编程”移动应用。我们将使用 **OpenAI 的 Codex** 来完成。Codex 是一个桌面应用，我越来越频繁地使用它，并且已经爱上了它。我们将用 Codex 来构建我们自己的 Replet 版本。除了 Codex 应用，我们还将使用 **Xcode**。Xcode 允许我们用 Swift 创建应用，然后我会把它运行在我的手机上。我们现在就开始开发这个应用，这肯定会非常有趣。

### 应用的核心提示词

以下是我们为这个更好的 Replet iOS 版本设定的提示词。我们先快速过一遍提示词，然后就开始行动。

我想让你构建一个能构建**网页应用和移动应用**的 Swift 应用。我想构建一个 Replet 移动应用的克隆版，因为它确实不太好用。我这么说可能有点苛刻，但我觉得它并不是真的很差，只是它的用户界面（UI）我真的不太喜欢，用起来没那么有趣。

所以我想要一个**更简单的体验**。我希望它简单，并且像 Replet 一样，但更多地借鉴 **iMessage** 的风格。整个流程会简单得多，感觉就像在给朋友发短信。你会觉得你在和一个非常聪明的“氛围编程”高手或者说一个非常聪明的开发者**聊天，他的名字叫 Jerry**。这个应用的名字也将是 **Jerry**。

技术实现要点

- **编程语言与框架**：我们将使用 **Swift**。
- **数据存储**：所有消息都将**本地保存**到手机上。
- **AI 引擎**：我们将使用 **Claude SDK**（确切地说是 Claude Agent SDK），它允许你将 Claude Code 集成到你的应用中。我们基本上就是把 Claude Code 放进我们的应用里。
- **应用生成与托管**：Claude Code 将与全新的 **Vibe Code CLI** 进行通信，后者将处理沙盒环境，并实际创建应用，然后将链接发送回 Claude Code。
用户交互流程示例

1. 用户发送一条消息，比如：“帮我建一个健身追踪器。”
1. 用户将与由 Agent SDK 驱动的 Jerry 聊天。你们可能会来回沟通，Jerry 可能会问：“嘿，你想要应用用什么颜色？想要什么功能？”
1. 用户提供更多答案后，Jerry 就会用 Vibe Code CLI 生成并托管应用。
1. Jerry 会返回一个应用的 URL 链接，并说：“嘿，这是链接。” 然后它会**自动打开这个 URL**。
1. Vibe Code CLI 会返回应用的链接，消息中的最后一个链接将自动在应用的浏览器预览中打开。用户看到后，可以继续与 Jerry 来回沟通，更新这些应用。
到视频结束时，我认为我们会创建一个非常引人入胜的“氛围编程”平台。这就是我们要做的事情。

## 构建过程

### 第一步：在 Xcode 中创建项目

首先，我喜欢手动进入 Xcode 并创建一个新项目。我们将创建一个应用，点击“下一步”，命名为 **Jerry**，然后点击“下一步”创建这个项目。项目打开后，这就是你看到的：最基本的 iOS 应用模板，一个显示“Hello World”的白屏。你在这里看到的就是任何 Swift 应用的起始模板。

### 第二步：在 Codex 中配置项目

现在项目已经打开，我回到 Codex。在 Codex 中，我创建一个新项目，同样命名为 **Jerry**。我们需要确保选择最高级别的项目文件夹（不是 Jerry 文件夹下的子文件夹）。然后打开它，我们就有了一条新线程。这个线程将专门用于这个特定文件夹（即包含项目所有代码的目录）。我们可以在 Codex 中创建许多线程并行运行。

并行工作流程

在本视频中，我不会线性地推进。当我使用 Codex 时，我经常并行做很多事情，无论是研究、设计还是构建应用。你越熟练使用这些工具，就越能并行工作。如果我操作得有点快，那只是因为我习惯了这种方式。我会尽力解释我的过程。如果任何地方我动作太快，请在下面留言，我很乐意回答尽可能多的问题。

### 第三步：撰写计划与设计

创建计划线程

我们将在 Jerry 项目中创建一个新线程。我说这是我的应用想法，然后我从 Notion 复制出提示词并粘贴进去。我还放入了我们之前画的流程图。

我按下 **Shift+Tab** 开启“计划模式”（plan mode），让 Codex 为这个项目制定一个计划。

并行进行 UI 设计

当 Codex 在制定计划时，我去处理设计。我打开 **Paper**，这是一个专门为 AI 代理设计的、类似 Figma 的工具。我打开一个新面板，粘贴我的想法：“请用 paper.design 设计出最重要的四个屏幕。使用 MCP 工具来创建设计。”

我之前有个视频详细介绍了如何设置和使用 Paper。这就是我的工作流程。这个 MCP 工具会看到我打开的 Paper 应用（就像 Figma 一样），然后它应该会在这里创建出设计。它正在根据我的需求绘制应用结构。

我可以在这个过程中引导 Codex。例如，我可以补充说：“顺便提一下，把大部分注意力集中在聊天和预览页面上。保持简洁。” 这个反馈会作为引导信息被 Codex 理解，然后它会继续工作。你可以输入任意多条引导信息。

设计进展

它正在设计那四个屏幕：

- Jerry 收件箱
- Jerry 对话
- Jerry 预览
- 构建结果
看起来很不错。我们让设计窗口保持打开，继续工作。

计划中的技术决策与调整

在计划中，Codex 会问一些问题，比如：

- “Jerry 的代理运行时（agent runtime）应该放在哪里？” 我选择“托管桥接（hosted bridge）”，虽然我不完全确定那是什么意思。
- “第一个可发布的里程碑应该包括什么？” 我添加上下文说：“Vibe Code CLI 对待移动应用和网页应用是一样的。两者都会在网页预览中打开。所以我们应该两者都支持。”
设计迭代

计划完成后，我们同时也在对设计进行迭代。这是一个相当不错的 Jerry 设计，对吧？ “告诉我你想构建什么，我就帮你构建它。网页优先。” 然后它开始构建，并显示预览页面。

我们可以对设计做一些调整。比如，我对预览页面的要求：

- 不要显示链接。
- 不要显示“自动打开 Jerry 的最新链接”。
- 在右上角放一个 Safari 图标。
- 保留返回 Jerry 的返回按钮，以最小化空间占用。
- 让预览基本上占据整个屏幕。
对于主页面的要求：

- 顶部不要有消息。
- 放一张 Jerry 的大照片，就是一个黑色的 J。
- 顶部的文字是“Build with Jerry”。
- 聊天记录中应该显示项目名称，而不是 J 图标。用 AI 根据我的第一个提示词来命名项目，并在旁边显示一个小应用图标。
我可以把这些要求作为引导输入进去。

### 第四步：获取 API 密钥

接下来，我们需要一些 API 密钥。我打开浏览器去获取：

- **Claude API 密钥**：在 platform.claude.com 上创建。
- **Vibe Code CLI 的密钥**：在 vibecode.dev 上获取。
我会先提交计划，然后作为后续操作，把这两个密钥粘贴进去。

### 第五步：等待初始构建与首次测试

初始构建花了很长时间，大概 7 分钟还没完成。我在 Codex 上的初始提示通常耗时最长。所以我出去散了 30 分钟的步，等我回来时，构建应该已经完成了。

让我们回到 Xcode，点击“播放”按钮。应用会自动在我的 iPhone 上打开。现在我们进入了一个空白的聊天界面。

遇到第一个 Bug

我点击“新聊天”（new chat），输入“Hello there, what's up?”，结果报错：“数据无法读取，因为它丢失了”。我点击“确定”，然后在 Xcode 底部的日志里看到一堆错误信息。我复制这些日志，回到 Codex，粘贴进去并说：“我收到了这条消息，请修复这个问题。”

持续迭代修复

- **问题1：AI 无响应**：修复后再次运行，点击新聊天，输入“Hi, I'm Riley.”，结果显示“Jerry is thinking through the build.”，但没有实际回复。我把这个截图拖进 Codex 的聊天中，并解释：“我希望和它像普通代理一样聊天。Vibe Code CLI 技能应该让我们能在需要时创建应用。现在我没看到代理的任何回复。它应该只是一个聊天回复。当我添加 Vibe Code CLI 功能后，它会回复一个应用链接。所以，请先专注于让这个聊天像 Claude Agent SDK 一样工作，然后我们再添加 Vibe Code 功能，并测试应用是否真的能在手机上打开。”
- **问题2：格式错乱**：再次测试后，回复功能正常了（“Hey, I'm Jerry. What are you building today?”），但发现界面格式全乱了。我创建了一个新线程，把错乱的截图拖进去，要求修复：“我想让它看起来像 iMessage 聊天，支持多行。确保所有东西都对齐。AI 回复时，请显示像 iMessage 那样的三个点。”
- **问题3：CLI 未集成**：修复格式后，测试发现 Jerry 说自己没有 Vibe Code CLI 的访问权限。我要求确保代理能访问 Vibe Code CLI 和技能。我去 vibecode.dev 复制了安装说明，连同 API 密钥一起给到 Codex。但再次测试，代理仍然说没有安装。我发现新创建的项目里没有技能文件（skills file），于是把这个信息也反馈给了 Codex。它似乎自己弄明白了。
### 第六步：测试应用生成功能

我删除了手机上的旧应用，重新运行。

成功生成第一个应用

测试：“Do you have access to the Vibe Code CLI?” 回答：“Yes, I have access to the Vibe Code CLI.” 然后我让它：“请用 Vibe Code CLI 为 Joe and the Juice smoothies 构建一个简单的落地页。”

由于它使用了 Vibe Code CLI，我的 Vibe Code 应用（我是联合创始人之一）就介入了。这个应用（Vibe Code）可以让你直接构建应用。我觉得它的 UI 比 Replet 简单易用得多。项目正在创建中，我们应该会得到一个链接。我们希望点击链接后能停留在应用内，而不是跳转到外部浏览器。理想情况下，浏览器预览应该在 iPhone 应用内打开。这就是我们的目标。

遇到超时错误并修复

第一次尝试超时了，出现错误。我把错误信息发给 Codex，它进行了调整。修复后再次尝试，这次成功了！我下楼拿了个酸奶回来，就看到应用预览已经出现了。我们有一个完整的应用，可以点击“Get Started”、“Learn More”。完美。

### 第七步：优化链接预览体验

自动打开链接

我注意到应用预览是自动打开的，但当我点击应用内的任何链接时，它却把我带出应用，在外部 Safari 中打开了。我希望点击链接也能在应用内的预览中打开。我要求 Codex 修复这个问题。修复后，任何包含链接的消息都会自动在应用内预览中打开。

测试移动应用生成

现在，我测试让它创建一个移动应用：“请创建一个待办事项清单（todo list）应用，风格类似 Notion，要酷一点。” 结果很棒，我们得到了一个基本的笔记应用，可以添加项目，功能正常。

优化预览界面

- **全屏预览**：我希望 Web 预览能更全屏，去掉顶部的白条。我把截图和要求发给 Codex 调整。
- **主屏幕重新设计**：我另一个线程的设计完成了，我想让主屏幕看起来像那个设计。结果第一次搞错了，因为我没给图片。补上图片后，它按照要求重新设计了主页，增加了项目列表和“查看更多”的按钮。
- **增加底部输入栏**：受 Twitter 打开链接方式的启发，我要求当打开应用预览时，在屏幕底部增加一个聊天输入框，就像 Twitter 的底部栏一样。这样我就可以边预览应用边输入指令。我甚至要求加了一个语音录制按钮。
添加语音输入功能

- **集成 Whisper**：我要求语音按钮使用 OpenAI 的 Whisper 模型。当按下时，可以录制我的声音，并将其转录成文字输入到文本框中，这样我就可以在不打开键盘、同时查看应用预览的情况下发送消息。我提供了 API 密钥（如果需要的话）。
### 第八步：最终成果展示

经过一系列的质量更新，我们得到了应用的**最终版本**。

- **核心功能**：你可以创建应用。例如，我们创建了一个 Notion 风格的待办应用。
- **无缝预览**：点击链接，应用预览会立即在界面中加载。
- **语音控制**：新增的语音功能非常酷。我可以对着手机说：“请把这个应用改成浅色模式，背景用非常浅的灰色，接近白色但偏灰，彻底改变字体，然后添加一个你觉得很酷的功能。” 然后我直接发送语音指令，应用就会在后台开始修改。这个过程只需要几十秒到一分钟。
最后的小优化

我还注意到聊天界面中粗体、项目符号等格式没有正确渲染，我要求也修复这个问题。至此，所有功能都按预期工作了。

## 总结

我知道，这是我第一次构建这样的应用，所以很难用一种非常结构化的方式来呈现整个过程。但这就是我们最终创造出来的：一个**Swift 应用**，它能帮我们构建其他应用，并且我们可以直接在应用内打开它们，这真的很酷。我们甚至可以用语音来对应用进行修改。

这个体验非常有趣。你可以疯狂一点，添加 30 个功能，做一些疯狂的事情。我可以看着应用，直接指挥它修改，提交提示词，一切都发生在这个聊天式的体验中。这是一个 Swift 应用，我们甚至可以把它提交到 App Store。

希望你们学到了很多东西，也希望对 Codex 有了更多的了解。我将来会做一个关于 Codex 的更深入的指南。无论如何，谢谢大家的观看。
