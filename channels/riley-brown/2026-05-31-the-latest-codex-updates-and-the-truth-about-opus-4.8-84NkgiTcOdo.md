如果你最近在关注 AI 领域，你一定会感受到一种强烈的认知撕裂。

一方面，Anthropic 高调发布了 Claude Opus 4.8，官方宣称这是“世界上最先进的 AI 模型”，配以华丽的数据和更具“判断力”的表述。另一方面，当真正的重度用户——那些每天都在用 AI 代理构成工作流的人——坐下来测试它时，却发现了一个令人不安的事实：他们根本看不出这东西和自己用了几个月的上一代模型有什么区别。

与此同时，OpenAI 没有在 Model 领域进行针锋相对的宣传，而是悄悄地对它的超级应用 Codex 进行了一连串疯狂的升级。有些更新甚至没有通过官方渠道正式公布，但它们正在从根本上改变“人与 AI 协作”的方式。

Riley Brown 在这个视频里所揭露的，不仅仅是一周的产品动态，而是一个深刻的行业转折信号：**我们正在从“模型飞跃时代”彻底进入“超级应用整合时代”。** 一个更聪明一点的模型或许还能占据头条，但真正决定未来格局的，是谁能搭建出最好的“AI 代理操作系统”。

今天，我们将跟随 Riley 的视角，深入这场无声的革命。

## 详细内容

### 1. 高原期的到来：为什么 Opus 4.8 像极了新 iPhone `[章节 00:00 - 03:XX]`

**核心观点**

大型语言模型的迭代升级正在进入实际感知的收益递减阶段。Anthropic 的 Claude Opus 4.8 在基准测试上或许有所提升，但当经验丰富的用户在实际工作中对比使用时，几乎无法将其与上一代版本区分开来。

**深度阐述**

当 Anthropic 发布 Opus 4.8 时，官方发布文案依旧充满气势——“建立在 Opus 4.7 的基础上，拥有更敏锐的判断力，对自己能力的边界更加诚实，并且能够比前代更长时间地独立工作。” 这听起来确实很棒。在模型卡上，它在编程方面的 SWE-bench Pro 成绩低于 GPT 5.5，但在推理、控制电脑、知识工作（文档、表格、演示文稿）和其他金融任务上超过了所有其他模型，包括 Opus 4.7 和 GPT 5.5。

但在 Riley 这个级别的主播看来，这并不足以让他激动。他分享了一个极具说服力的个人实验：

> “说实话，我原本是打算做一个完整的关于 Opus 4.8 的视频的。但我在比较 Opus 4.8 和 Opus 4.7——他们之前的模型——时，花了整整 3 个小时。猜猜怎么着？我 literally 无法分辨这两个模型之间的区别。” `[章节 00:XX]`
他不是一个人。节目的常客 Greg Eisenberg 说得更加直白：

> “我没有在我的播客里报道 Claude Opus 4.8，因为我不认为它比 GPT 5.5 有任何实质性的提升。”
> “我们正在进入一个模型发布感觉像 iPhone 发布的时代。还记得每部新 iPhone 都有真正的飞跃吗？现在只是一个稍微好一点的摄像头，你根本看不出区别。这就是模型现在的发展方向。4.6 到 4.7 到 4.8。每一个都略有不同，但你根本说不清哪个才是最好的。”
Riley 进一步坦诚了自己的使用习惯：

> “事实上，我可以从个人经验告诉你。我仍然在使用 AI 代理从 iMessage 运行任务……我使用的是 Opus 4.6。我认为这是目前最适合通用代理工作的模型——至少基于我的使用方式。而我 literally 无法区分这三个模型之间的区别。” `[章节 01:XX]`
不仅仅是以上的主观感受，深层的数据同样揭示了这一点。Riley 引用了 Deep Suite 的数据。这是一家专门测量前沿编码代理在长期、原创性软件工程任务中表现的公司。Deep Suite 从三个核心维度进行衡量——**成本、时间、输出 Token**——然后将这三个维度分别与任务得分绘制成图表。

**视觉/结构信息描述：**

想象一个散点图。X 轴代表成本（越向右越贵），Y 轴代表得分（越向上越好）。

- GPT 系列的各个版本（GPT 5.5 Medium, High, Extra High 等）聚集在图表的**左上角**位置，意味着它们获得了高分，但成本却非常低。
- Opus 系列（4.7, 4.8）则分布在图表的**右下侧**，意味着它们的得分相对更低，但成本却明显更高。
在“时间 vs. 得分”的图表上，同样是 GPT 系列更高效：完成任务耗时更短，得分却更高。

在“Token 消耗 vs. 得分”的图表上，结果依然一致。

不仅如此，社区也开始涌现出一种关于“信任感”的讨论。Riley 注意到越来越多的人在讨论任务的深度与信任度：

> “我可以信任 GPT 5.5 处理那些我永远不会交给 Opus 4.8 的棘手任务。”
> “是的，Opus 4.8 感觉很好，用起来可以很上瘾，尤其是在找感觉（vibing）的时候，但那是表面层面的。”
**个人感受**

Riley 在这里展现了一种非常真实的人文色彩。他本怀着对“世界上最先进模型”的期待，花了整整一个下午的工作时间去进行对比评测。但他没有强行凑出一堆晦涩的差异化点来出一期“深度测评”视频，而是诚实地承认这根本就是一个软更新。“我放弃了制作这个专题视频，因为没什么好说的。”

这种从“模型崇拜者”到“应用务实主义者”的转变，其实代表了一种深刻的行业情绪的转变。当 AI 的热度中心从“谁的模型更聪明”转移到“谁的产品更好用”，整个行业的创新动力和商业逻辑都在迁移。

**延伸思考**

这件事引出了一个更大的行业洞察：当模型的“基础智能”水平趋于平缓，竞争的主战场就必然从“模型研发部”转移到了“产品体验部”和“生态整合部”。Riley 敏锐地点出了这个时间节点：大约 9 个月前，他还是超级关注模型更新的，因为每次更新都是一个步进式函数的大跳升。而现在，“所有激动人心的创新都在应用层”。

### 2. 超级应用的崛起：OpenAI Codex 的无声革命 `[章节 03:XX - 10:XX]`

**核心观点**

如果 Opus 4.8 是 iPhone 的“常规升级款”，那么 OpenAI 对 Codex 本周的更新就是 iOS 的一次跨代大版本更新——它正在从一个强大的编程 IDE，进化为一个能够无缝控制你整个数字工作流的“AI 代理操作系统”。

**深度阐述**

Riley 将 OpenAI 本周在 Codex 上释放的更新称之为“超级应用时代”的标志。这些更新充满了细节，但有些甚至没有被正式公告。

**第一：Windows 计算机使用 (Computer Use) 与 远程控制 (Codex Remote)** `[章节 03:XX - 04:XX]`

此前，Codex 的计算机使用功能主要面向 Mac。现在，Windows 用户正式可以使用 `@computer use` 让 Codex 内的 GPT 5.5 控制他们的整台电脑。

更具革命性的是“Codex Remote”功能。

- **场景复现**：如果你有 Windows 电脑和 iPhone。打开 Codex，左下角有一个电话图标，点击会出现一个二维码。用手机上的 ChatGPT App 扫描后，**你可以直接在手机上对 ChatGPT 说话或打字，来控制你电脑上的 Codex，然后由 Codex 操作你的电脑。**
- 这意味着，你可以躺在床上，对着手机说：“嘿，帮我检查一下我办公室的电脑桌面，看看有什么文件。”然后你的电脑就会自动响应，就像魔法一样。Riley 评价这是“一个被严重低估的酷功能”。
**第二：浏览器的永久登录与多标签页** `[章节 04:XX - 06:XX]`

这是 Riley 称“过去 72 小时内我每小时都在用的功能”。

以前，Codex 里的浏览器每次打开都要重新登录各种网站。现在，**这些浏览器会话保持登录状态。**

- **实操演示**：Riley 在 Codex 中输入指令：“请把我最新关于 Agent Native 的视频链接整理到 Notion 上，并给我一个链接。”
- 由于 Codex 已经通过 Notion 插件获得了授权，它找到文档后，将链接返回。Riley 右键点击，选择“在浏览器中打开”。
- **惊人的一幕出现了**：浏览器打开后，他不仅直接看到了那个 Notion 文档，而且是**已经自动登录好的状态**！这意味着 Codex 的浏览器已经成为了一个“永久在线的 AI 工作台”，任何 Agent 为你生成或修改的内容，都可以直接在内部编辑和应用，无需离开这个超级应用。
同时，Codex 的浏览器现在支持 **Command + 点击** 打开多个标签页。Riley 展示了同时打开 5-6 个标签页的场景，并且可以在这些标签页和主对话之间无缝切换。

**第三：Agent 繁衍 (Agent Spawning)** `[章节 06:XX - 08:XX]`

这也许是本次更新中最引人注目的一项功能。现在，你可以让一个主 Agent 自行创建多个子 Agent 会话进行协同工作。

- **实操演示**：Riley 向 Codex 发送一个“超级提示词”：“我想让你在 Codex 内部创建新的聊天会话……现在启动它，它实际上会创建 6 个新的聊天会话。”
- **视觉/结构信息描述**：画面切到 Codex 的侧边栏。随着指令的确认，左侧的聊天列表像打地鼠一样开始弹出新的会话窗口。AI 自行将对话标题逐个重新命名：“Triage……Boom. Boom. Boom.” 瞬间，6 个拥有独立、清晰任务目标的子 Agent 出现在侧边栏。
- 这些子 Agent 由 AI 自动编写 Prompt，然后由 Codex 的另一个线程触发。
- Riley 预测这解锁了全新的工作流：主 Agent 是项目经理，分配任务给子 Agent，后续甚至需要创建一个自动化脚本去验收这些子 Agent 的完成情况。
**第四：搜索与 GitHub 活动页** `[章节 08:XX - 10:XX]`

- **强化搜索**：现在可以通过 `Command + G` 进行**全文搜索**，不仅能搜标题，还能搜索所有闲聊记录里的具体关键词。
- **活动贡献图**：Codex 新增了一个类似 GitHub 仓库贡献度的页面，展示你的连续使用天数（Riley 显示 43 天）、最长任务时长（3 小时 7 分钟），以及消耗的总 Token（40 亿）。这是一个很小的细节，但赋予了产品一种“你在塑造自己使用记录”的社群感和成就感。
**个人感受**

Riley 毫不掩饰他对这些更新的狂热。他在讲解 Agent 繁衍时，言语间充满了“这太酷了”的感叹。他承认对这个功能的使用方式还处于探索阶段，计划做专题视频。他之所以如此激动，是因为这些功能正在真正逼近他一直在节目中倡导的“Agent Native”体验——让 AI 不再只是一个问答工具，而是一个可以分配任务、繁衍下一级代理的自主系统。

### 3. Vibe Coding 的终局：一切将被超级应用吞噬 `[章节 10:XX - 15:XX]`

**核心观点**

像 Lovable 和 Replit 这类专门为“氛围编程(Vibe Coding)”设计的平台，其核心价值主张正在被超级应用（特别是 Codex）及其插件生态无缝覆盖。一场从专用平台向通用平台的迁移正在发生。

**深度阐述**

Riley 观察到，很多人开始在 DM 里问他关于“Vibe Coding”平台的问题。他发现一个趋势：很多人正在从 Lovable、Replit、Bolt 这些专门的平台上迁移出来，转到 Codex 或 Claude Code。

原因很简单：Codex 正在实现这些平台的核心价值。

**视觉/结构信息描述：**

Riley 演示了一个 Prompt 示例来生成一个完整的全栈应用：

> “请为我公司构建一个追踪视频数据的内部工具……”
> - 使用 Neon 作为数据库（`@mention` Neon 插件）
> - 使用 Google 登录
> - 使用 Vercel 托管（`@mention` Vercel 插件）
> - 使用 AI Gateway 驱动所有 AI 功能
> - 使用 FAL 的 Gen Media 插件支持图片/视频模型
> - 运行大量安全检查（GPT 5.5 Extra High 非常擅长）
就这么一句 Prompt，加上提前配置好的 API Key。Codex 就创建了一个拥有认证、数据库、存储、AI 能力的完整 Web App。而这正是 Replit/Lovable 提供的核心体验：数据库、认证、托管、一键部署、安全扫描。

**延伸思考**

Riley 在这里抛出了一个对 2026 年下半年的尖锐预测：**“有人将会构建一个 AI 原生的 Replit 和 Lovable，但它是一个运行在 Codex 内部的插件。”**

他的逻辑非常清晰：

1. **模型质量劣势**：专门的 Vibe Coding 平台使用的 AI 模型往往不如 Codex 中直接使用的 GPT 5.5（尤其是 Extra High 模式）。
1. **成本劣势**：OpenAI 为了推广 Codex，在大量补贴 GPT 5.5 的使用。Vibe Coding 平台需要自己承担模型推理成本，难以在价格上与 OpenAI 竞争。
1. **商业模式创新**：未来的赢家平台将采用 **BYOT (Bring Your Own Tokens，自带 Token)** 和 **BYOA (Bring Your Own Agent，自带 Agent)** 模式。
1. **结论**：未来的场景是，一个简单的 $10/月的插件，仅仅提供一个流畅的 UI 预览、配置界面和部署服务，而 Agent 的核心智能完全调用用户自己的 Codex 资源。这意味着 Vibe Coding 将不再是某个独立平台的特权，而是任何 AI 超级 Agent 的一项基础技能。

### 4. 未来已来：Agent Mini App 与新的交互范式 `[章节 15:XX - 20:XX]`

**核心观点**

对话式 UI 不是终点。未来人机交互的核心将是 **Agent Mini Apps**——由 AI Agent 根据当前任务动态生成的、深度绑定你所有个人工具（插件）的即时交互界面。

**深度阐述**

这是 Riley 过去两个月来最大的执念，也是他判断未来行业走向的核心概念。这个概念源于 Dan Shipper 创建的“Proof”应用——一个专门为 Agent 协作优化的文档编辑器（Agent Native App）。现在 Google Docs、Notion 等都已经可以通过 Agent 完全控制，具备了“Agent 原生功能”。

但是，当前的 Codex 有一个巨大的鸿沟：**你可以通过插件授权 Agent 访问你的 Gmail、Slack、Typefully，但 Agent 无法创建一个直接利用这些授权来执行操作的“内部应用”。**

**视觉/结构信息描述：**

想象一个场景，不再是一长串的文字对话：

- 你告诉 Codex：“我需要处理今天的邮件。”
- 当前的方案：Agent 搜索你的收件箱，理解内容，然后开始写长文草稿。然后你必须一条条回复：“发送第 1 封，不要发送第 2 封，修改第 3 封的措辞，再给第 4 封加个附件……”
- **未来的方案 (Agent Mini App)：** Agent 直接在你的 Codex 侧边栏渲染出一个**“卡片式邮件处理界面”**（Riley 称之为“Tinder for Email”）。
当你在使用这个 Mini App 时，Agent 在**学习**。每一次你点击“归档”，Agent 就学会了“这类邮件不需要回复”；每一次你修改措辞，Agent 就学会了你的语气和风格。下一次它写的回复会更加精准，你几乎只需要点击“发送”。

这就是 Agent Mini App 的真正意义：**它不再是建议，而是执行。它不再是文字，而是界面。** 它让 AI 从一个需要你长篇大论下达指令的“实习生”，变成了一个可以托管一部分工作流权限、并提供最直观操作界面的“部门主管”。

**个人感受与公司动态**

说到此处，Riley 的情绪变得非常激昂。他透露自己的公司 Chorus (chorus.com) 已经从旧金山搬迁到了纽约，正在全力解决这个问题。他已经在 iMessage 中部署了一个 AI Agent，可以接受任务，然后给用户发回一个链接。点击链接，就是一个 Mini App。

> “我真心相信，所有主要的平台都将围绕着这个想法。这就是将‘Jarvis’带入现实的方式。为什么我要用别人搭建的外部平台，如果我的 AI Agent 能在我需要的时候，即时生成一个完美集成我所有工具的 UI 给我？”
**延伸思考**

这段话极具冲击力。它意味着未来的企业软件——CRM、邮件客户端、项目管理系统、社交管理工具——可能不再需要独立的 App。它们将变成由 Agent 根据你的需求动态生成的“一次性”或“持续性” Mini App。Agent 不仅仅是你的大脑助手，它正在成为你与所有数字服务之间的**操作系统的中层**。它在底层调用 API（插件），在前端生成组件（Mini App），在人机交互的最后 10% 的决策和执行。

这是对传统“SaaS 申请->等待授权->学习界面->手动操作”范式的彻底颠覆。未来的 SaaS 商业模式，或许不再是买 License，而是购买“Agent 的一次授权调用”和“Mini App 的一次生成渲染”。

## 精华收获

1. **认清模型的“高原期”**：别再被模型发布时的基准测试数字迷惑。对于 90% 的工作流，GPT 5.5 和 Claude Opus 4.X 的实际感知差异已经小到可以忽略。真正的战场不在“模型”，而在“集成”。
1. **投身于“超级应用”生态**：技能培养的重心应该从“如何 Prompt 大模型”彻底转移到“如何运用超级应用生态”。学会在 Codex 中配置插件、利用 Agent Spawning、掌握远程控制和工作流搭建，是未来 1-2 年内最值得投入的核心竞争力。
1. **警惕 Vibe Coding 中台陷阱**：如果你正在用 Lovable 或 Replit 严肃地构建产品，请高度关注 Codex/Claude Desktop 的插件生态。你可能很快就能在超级应用中以更低的成本、更强的 AI 实现完全一样的功能。未来的 Vibe Coding 平台将只是一个薄薄的 UI 插件层。
