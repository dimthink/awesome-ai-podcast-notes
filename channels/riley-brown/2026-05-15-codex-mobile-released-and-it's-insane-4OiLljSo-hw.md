### 一、Codex 是什么？为什么这次更新是“游戏规则改变者”？ `[视频 00:00-01:30]`

**核心观点**

Codex 是 OpenAI 开发的桌面端 AI 代理，能完全控制你的电脑。现在通过 ChatGPT 移动应用连接 Codex，手机就成了电脑的“远程遥控器”。

**深度阐述**

要理解这次更新的重量，首先要明白 Codex 本身的定位。Riley 把它称为“OpenAI 版本的 Claude Code”，但功能上更为激进——Codex 不是只能读取代码或给出建议，而是能够 **直接控制整个电脑**：移动鼠标、打开文件、运行命令、部署应用，就像有一个看不见的助手坐在你的电脑前执行任务。

> 原话：“Codex can control my entire computer, which means that through the ChatGPT app, I can now control my entire computer.” `[视频 00:45]`
> 翻译：Codex 可以控制我的整台电脑，这意味着通过 ChatGPT 应用，我现在可以控制我的整台电脑。
这个逻辑链条非常清晰：ChatGPT 移动应用 → Codex 桌面应用 → 你的电脑。手机发出的指令先传到 Codex，然后 Codex 在电脑上执行操作。结果再通过 Codex 传回手机。你不需要 VPN、不需要远程桌面软件、不需要复杂的端口转发——只需要两部设备连接同一个 Wi-Fi。

**视觉/结构信息描述**：

Riley 在视频左侧展示了 Codex 桌面应用的界面，右侧则是手机上的 ChatGPT 应用。当他在手机上发起请求时，桌面应用会实时显示代理正在执行的动作（例如滚动文件列表、打开浏览器、运行命令行）。整个过程有一个“连接状态”指示器，显示两个设备是否同步。

**个人感受**

Riley 的兴奋感贯穿始终。他在说“this is the biggest update to ChatGPT maybe ever”时，语气是那种“我知道你们会觉得我夸张，但你们听我说完”的认真。他多次重复“insane”、“really really cool”，仿佛一个孩子发现了新玩具。这种情绪是真实的——作为长期关注 AI 代理发展的创作者，他深知“远程控制”解决了移动端 AI 助手的最大痛点：过去你只能让 AI 生成代码，但无法运行它；现在，你可以让 AI 在家里的电脑上真正执行代码。

**延伸思考**

这实际上模糊了“本地操作”和“远程操作”的界限。传统远程控制需要你手动操作，而 AI 代理可以自主决策。那么问题来了：如果 AI 可以在你不监督的情况下控制你的电脑，安全边界在哪里？Riley 后面会提到“Full Access”模式，这确实需要用户对 AI 行为有充分信任。另一个思考：当这种能力普及后，“拥有一个‘在云端运行的电脑分身’”可能会成为标配——你只需要一个能联网的终端（手机、手表、甚至眼镜），就能指挥你的数字分身完成任务。

**精华收获**

- Codex 不是简单的代码生成器，而是一个能执行操作的系统级代理。
- 手机→Codex→电脑的链路，本质上是“移动端交互”与“桌面端算力/环境”的桥接。
- 这次更新的核心突破在于：让 AI 代理的“行动能力”脱离了物理位置的限制。

### 二、首次设置：三步搞定连接 `[视频 01:30-03:00]`

**核心观点**

设置过程非常简便：确保 Codex 桌面端和 ChatGPT 手机端都是最新版本，连接同一 Wi-Fi，授权后即可同步聊天记录。

**深度阐述**

Riley 详细演示了设置步骤，并强调了一些容易忽略的细节。整个过程大约需要 2-3 分钟。

**步骤还原**：

1. **更新 Codex 桌面应用**：打开 Codex，在界面某处（Riley 说“up here somewhere”）会看到一个“Update and Restart”选项。点击后应用会自动重启。这一步很关键——旧版本没有“Codex Mobile”连接选项。
1. **更新 ChatGPT iOS 应用**：去 App Store 确保 ChatGPT 是最新版本。Riley 特别指出“iOS app”（目前 Android 版本是否支持未提及），所以 iPhone 用户是第一批体验者。
1. **再次重启 Codex**：更新后，再手动重启一次 Codex。当界面出现“Connect to Codex Mobile”选项时，说明准备就绪。
1. **手机端连接**：打开 ChatGPT 应用，在侧边菜单（左上角或右上角？Riley 展示了右上角的按钮）中应该能看到“Codex”选项。点击后，手机会自动搜索同一 Wi-Fi 下的 Codex 桌面端。
1. **授权同步**：手机上会弹出“Setup Codex Mobile”界面，点击“Connect”→“Continue”→选择账户→“Authorize”。授权成功后，两个设备上的聊天记录会完全同步。
**视觉/结构信息描述**：

Riley 在桌面端 Codex 界面中点击了一个带有手机图标的按钮，随后手机上弹出了一个授权对话框。他特别强调：“The chats that show up here should match the chats that show up here” —— 他用手指指着桌面端和手机端的聊天列表，两者内容完全一致。

**重要原话**：

> “You want to make sure that you're connected to the same Wi-Fi.” `[视频 02:15]`
> 翻译：你要确保连接的是同一个 Wi-Fi。
**个人感受**

Riley 在演示过程中反复提醒“重启”、“确保最新版本”，语气像是一位耐心的朋友在教新手。他没有抱怨设置复杂，反而用“that's the first thing I would do”这样的建议来优化体验。这种细节感说明他真的在日常使用这个工具，而不是单纯为了拍视频做演示。

**延伸思考**

连接同一 Wi-Fi 的要求意味着当前版本不支持远程连接（比如你在咖啡馆，电脑在家）。但考虑到安全性和延迟，本地局域网连接是最稳妥的起点。未来是否会支持云端中继（类似 Tailscale）？很可能。另一个技术细节：为什么需要重启两次？可能是因为 Codex 的连接模块是在启动时加载的，更新后需要完全刷新才能激活。

**精华收获**

- 设置极为简单，无需技术背景。
- 关键是“最新版本”和“同一 Wi-Fi”。
- 聊天记录双向同步意味着你可以在手机上继续电脑上未完成的任务。

### 三、移动端界面优化：让“聊天优先于项目” `[视频 03:00-05:30]`

**核心观点**

Codex 移动端支持聊天和项目两种组织方式。对于非编码任务，建议将“聊天”列表放在“项目”上方，以提升日常使用效率。

**深度阐述**

当你成功连接后，ChatGPT 应用内会出现一个专门的 Codex 模块。Riley 展示了界面布局：

- 顶部有一个按钮（右上角）可以切换到 Codex 模式。
- 主区域显示“Chats”（聊天）和“Projects”（项目）两类对话。
- 默认情况下，Projects 可能在 Chats 之上，但 Riley 建议调整。
**如何调整顺序**：点击右上角的三个点（菜单），选择“Chats First”。这样每次打开时，你会先看到所有独立的聊天记录，而不是项目文件夹。Riley 解释说：“如果你要用 Codex 处理非技术任务（比如查资料、整理文档、生成相册），把 Chats 放在上面会更方便。”

**创建新聊天**：点击界面上的“+”按钮即可新建。Riley 特别推荐使用**语音模式**（voice mode）——他按下语音按钮，然后开始说话：“Can you please do research on the latest update for GPT's update that includes the new Codex remote?” 这个请求直接通过手机发送到桌面端的 Codex，Codex 开始搜索网络（可能会用到浏览器插件）并整理成文档。

**视觉/结构信息描述**：

Riley 在手机上展示了语音输入界面，同时桌面端的 Codex 窗口出现了三个“spinning circles”（旋转的圈圈），表示代理正在执行任务。当任务完成时，旋转圈圈会变成蓝色（与桌面端一致）。手机还会收到推送通知。

> 原话：“When they're done, they'll show up in blue just like the desktop app.” `[视频 04:10]`
> 翻译：任务完成后，它们会像桌面应用一样显示为蓝色。
**个人感受**

Riley 对语音模式的喜爱溢于言表。他建议“Go for a walk. Use the voice mode.”——这不仅仅是功能演示，更是一种生活方式建议。想象一下：你散步时随口说出一个复杂任务（“帮我研究一下最新的量子计算芯片，做成 PPT”），回家后电脑已经帮你做好了。这种“异步创造”的体验确实令人向往。

**延伸思考**

界面设计的细节反映了 Codex 的使用场景分化：Projects 适合大型代码项目（需要管理多个关联文件），Chats 适合一次性任务。Riley 的“Chats First”建议暗示普通用户可能更多使用即兴任务，而非长期项目。另一个值得注意的点：移动端的 Codex 并非独立运行，而是桌面端的“镜像”。也就是说，所有计算都发生在电脑上，手机只是一个“显示器和麦克风”。这解释了为什么处理复杂任务时手机不会发热或耗电。

**精华收获**

- 语音模式是移动端最自然的交互方式，让“边走边创造”成为可能。
- 界面可定制（Chats First）体现了对非技术用户的关怀。

### 四、插件 vs. 技能：当前移动端的限制与变通方法 `[视频 05:30-07:30]`

**核心观点**

在移动端，可以通过“@”符号使用插件（plugins），但“/”符号调用的技能（skills）暂不支持。不过，你仍然可以用自然语言描述技能名称来间接调用。

**深度阐述**

这是视频中一个非常技术性的部分，但对于理解 Codex 的架构至关重要。

Riley 区分了两个概念：

- **Plugins（插件）**：通常是官方或第三方集成的服务（如 Documents、Spreadsheets、Browser、Chrome、Vercel 等）。在移动端，按 `@` 键会弹出插件列表，你可以选择并激活。
- **Skills（技能）**：用户自定义的自动化脚本或流程（例如 Riley 自己创建的“YOLO mode”）。在桌面端，按 `/` 键可以调出技能列表；但在移动端，按 `/` 没有任何反应。
**演示过程**：Riley 在手机上按下 `@` 键，屏幕弹出了 Documents、Spreadsheets、Presentations、Browser、Chrome 等选项。他选择了“Documents”插件，然后可以要求 Codex 创建一个文档。接着他尝试按 `/` 键——没有任何菜单出现。这证实了技能的快捷方式在移动端暂不可用。

**变通方法**：虽然不能通过 `/` 快速调用技能，但你可以在自然语言中直接说出技能名称。例如：“Please use YOLO mode to deploy this app.” 因为 Codex 桌面端实际上拥有所有技能，手机端的指令最终会传递到桌面端执行。只要桌面端理解“YOLO mode”是一个技能，它就会执行。

> 原话：“You can ask it to just use a skill in natural language if you know which skills your agent has and it will still use it.” `[视频 06:15]`
> 翻译：如果你知道你的代理有哪些技能，你可以直接用自然语言要求它使用该技能，它仍然会执行。
**视觉/结构信息描述**：

Riley 在手机上展示了按下 `@` 后滚动的插件列表，并且有一个“Computer use”选项（可能是访问文件系统）。他同时也展示了 `/` 无响应的空状态界面，形成鲜明对比。

**个人感受**

Riley 的语气中带着一丝遗憾（“skills don't show up”），但他很快给出了解决方案，显示出他作为深度用户的应变能力。他不是在抱怨产品缺陷，而是在分享“目前怎么用最好”。这种务实态度值得学习。

**延伸思考**

为什么技能在移动端不可用？可能原因：技能通常涉及复杂的本地脚本或环境依赖，移动端 UI 无法动态列举所有自定义技能；或者 OpenAI 尚未完成移动端的技能选择器开发。但好消息是，自然语言调用仍然有效——这本质上是 AI 代理的核心能力：理解意图并映射到已有功能。随着时间推移，技能在移动端一定会得到原生支持。

另一个亮点：**通知功能**。当代理执行长时间任务（10-30 分钟）时，手机推送通知非常实用。你不需要一直盯着屏幕，完成后一键点击即可查看结果。

**精华收获**

- 移动端目前不支持 `/` 快捷键调出技能，但 `@` 插件完全可用。
- 自然语言调用技能是一个可靠的后备方案。
- 长任务通知是体验痛点的重要解决方案。

### 五、YOLO 模式：让移动端“边写边看”成为可能 `[视频 07:30-10:30]`

**核心观点**

YOLO 模式是一个自定义技能，每次代码更改后自动部署到互联网（如 Vercel），并返回公开链接。这样，手机端就能直接预览和测试运行中的应用——解决了本地服务器无法从手机访问的核心问题。

**深度阐述**

这是视频中最具创新性的部分。Riley 指出了一个关键问题：当你通过手机让 Codex 创建一个网页应用（比如一个披萨店的 landing page），Codex 会在电脑上启动一个本地开发服务器（通常是 `localhost:3000` 这样的地址）。这个地址只能在电脑本机访问，手机无法打开——因为手机上并没有运行那个服务器。

**YOLO 模式如何解决**：

YOLO 模式是一个预设的“规则”：当代理构建任何应用或网站时，每次修改后都自动部署到生产环境（production），并返回一个公共 URL。Riley 使用 Vercel 作为托管平台（可以是任何支持自动部署的服务）。

**设置过程（30 秒完成）**：

1. 在 Codex 桌面端进入 Plugins 设置。
1. 搜索“Vercel”并启用该插件。
1. 用自然语言定义一个技能：“Make a skill called YOLO mode. When I'm building an app, every single change is deployed to prod. This is used because I'm likely vibe coding from my phone. I need a deployed link to see it.”
就这么简单。之后，当你在手机上说“Please YOLO it”时，Codex 会理解：构建 → 自动部署 Vercel → 返回 `xxxx.vercel.app` 链接。

**演示效果**：

Riley 先要求 Codex 制作一个 landing page（普通模式）。Codex 完成后返回 `localhost` 链接，手机无法打开。接着他说：“Make a landing page for me based on your memory of me. Please YOLO it.” Codex 回复了一个 Vercel 链接。点击链接，手机浏览器中立刻显示了精美的 landing page。

接着他继续迭代：“Hey, I like this, but it kind of sucks. Can you please make it dark mode? Change the font. Be creative.” 再次使用 YOLO 模式，Codex 自动修改、重新部署，同一个链接现在变成了深色模式。手机上刷新即可看到新版本。

**视觉/结构信息描述**：

Riley 在手机上展示了部署过程的界面：桌面端 Codex 窗口中出现了截图（screenshots），显示代理正在浏览器中操作。手机上的 Vercel 链接可以直接点击，并在 ChatGPT 内置浏览器中打开，也可以跳转到 Safari。

> 原话：“This becomes a full vibe coding platform when you have something like YOLO mode.” `[视频 08:50]`
> 翻译：当你有了 YOLO 模式，这就变成了一个完整的“氛围编码”平台。
**个人感受**

Riley 在演示 YOLO 模式时，声音明显提高了兴奋度。他反复说“look at that”、“really really cool”。这确实是一个“aha moment”——当你发现自己可以在手机上反复修改一个实时在线的应用，而且每次都自动部署，那种创造的自由感很难用语言形容。他提到“I set this up in 30 seconds”，强调了这个模式的门槛之低。

**延伸思考**

“YOLO”这个名字本身就很有趣——You Only Live Once，暗示了“不求完美，先上线再说”的快速迭代哲学。这实际上是现代 DevOps 中“持续部署”（Continuous Deployment）的极端简化版。传统上，自动部署需要配置 CI/CD 流水线，而现在只需一句话。这预示着一个未来：非技术人员也能拥有“一键部署”的能力。

另一个深层含义：YOLO 模式解决了移动端 AI 编码的最大障碍——预览。没有部署链接，你只能依赖代理的文字描述（“我改了颜色”），无法真正验证。有了部署链接，移动端就变成了一个功能完整的开发测试环境。

**精华收获**

- YOLO 模式本质上是“自动部署 + 返回公网链接”的自动化流程。
- 设置仅需启用 Vercel 插件并定义一条自然语言规则。
- 它让手机变成了一个可以“边编码边预览”的真实开发平台。
