# 开篇引入

2024 年，AI Agent 的能力竞赛已经进入白热化阶段。各家大模型公司不仅比拼模型智商，更在争夺一个关键的阵地——用户的工作环境。当全球最顶尖的两家 AI 实验室——OpenAI 和 Anthropic——不约而同地在同一天，为他们分别属于两大阵营的王牌产品（Codex 与 Claude Desktop）发布了一个看似普通却极具颠覆性的更新时，整个行业的信号已经强大到无法被忽略：一个功能完备的、支持多标签页操作的应用内浏览器。

这究竟意味着什么？

如果 AI 只是大脑，那么浏览器就是它用来操控数字世界的双手。在此之前，AI 无法直接与 Web 应用交互，它只能给你一个链接，或者一段需要你手动粘贴的命令。而现在，在 Riley Brown 的现场演示中，我们看到 AI 可以同时打开你的 YouTube 后台、Notion 项目、X 通知和 Gmail 收件箱，并把它们排列在同一个应用窗口的不同标签页中。这不再是简单的“问答”，这是 AI 作为“代理”（Agent）真正开始接管和操作你的电脑。

这看起来只是产品的一个更新？不，这是一场新操作系统的加冕仪式。

# 详细内容

## 第一部分：多代理并行的魔法——从黑暗森林到一站式指挥舱 `[章节 1 / 00:00-02:00]`

### 核心观点
OpenAI 和 Anthropic 在同一天将多标签浏览器植入各自的 AI 应用，标志着 AI 代理从“信息汇总者”向“数字世界执行者”的关键跃迁。这种能力让 AI 能够同时在多个平台上执行复杂任务，结果不再是单纯的文字，而是可交互的用户界面。

### 深度阐述

在视频的开场，Riley 没有丝毫拖泥带水，直接向观众展示了这个功能最令人头皮发麻的威力。他身处于 Codex（即重构后的 ChatGPT 高级版）应用中，对着 AI 下达了一个在过去看来绝对属于痴人说梦的指令：

> “打开我的 YouTube Studio 分析页面、我的 Notion 视频数据库、我的 X 通知、我最重要的五封邮件，还有挪威足球比赛的比分。把所有这些都作为新标签页在我的应用内浏览器中打开。”

请注意，这个提示词（Prompt）的核心主语是 **“打开”**（Open）。这是一条行动指令，而非信息指令。

在屏幕上，我们可以看到 Codex 内部系统正在启动多个 **“子代理”（Sub-agents）**。这意味着 AI 并没有按部就班地顺序执行，而是把不同的任务分发给内部的虚拟助手，让它们协同并行工作。每一个子代理都有明确的职责：一个去爬取 YouTube 数据，一个去检索 Notion 数据库，另一个去读取 Gmail 收件箱。

几秒钟后，结果以一种完全不同于传统 AI 输出的方式呈现出来——不是文字，不是列表，而是一排加载完成、可以交互的浏览器标签页：

- 第一个标签页：**YouTube Studio 后台的实时数据面板**，展示了 Riley 频道的实时观看量。
- 第二个标签页：**X 通知列表**，最新互动一目了然。
- 第三个标签页：**Notion 视频数据库**，所有正在制作的视频项目表格清晰可见。
- 第五个标签页：**挪威足球比赛的比分页面**，2-1，挪威队输了。Riley 幽默地评论道：“他们输了。安息吧，挪威。”（Rest in peace, Norway.）`[01:30]`

这并非 bug，而是 AI 正确找到了实时网页数据并给出了结果。这个冷笑话恰恰展示了 Agent 带来的真实感——它确实在为你浏览真实的互联网。

### 个人感受

Riley 的兴奋是极具感染力的。他不是一个冷静的评测员，而是一个发现了超级武器的前线战士。他对这个功能的评价是 **“Massive”**（巨大的）。这种演示不仅展示了技术上的突破，更让他对自己作为“内容创业者”的日常工作效率产生了前所未有的期待。在过去，他需要打开四个不同的网站，登录四个不同的账号，切换四个不同的浏览器窗口。而现在，他所需要做的只是说一句话，AI 就为他构建好了专属的战斗指挥舱。

他在演示过程中多次发出惊叹，尤其是当子代理一个接一个启动、标签页一排接一排打开的时候。他的潜台词非常明确：这不是未来的概念，这是今天就可以使用的工作方式。

### 精华收获

此前的 AI 代理工具虽然强大，但往往完全在后台运行，用户看不到过程，导致缺乏信任感。或者，它们只能输出文本和代码。多标签浏览器的引入，创造了一个介于“完全黑箱”和“纯文本输出”之间的新状态：**过程可视化、结果可交互、决策可干预**。用户可以在 AI 完成“打开”动作后，自己去核实、阅读和操作，既保留了 AI 的高效，又保留了人类的最终控制权。这种 **“信任但验证”**（Trust but Verify）的模式，是 Agent 大规模落地的关键基石。

## 第二部分：跨越鸿沟——Claude Code 的进化与 Codex 的真正上位 `[章节 2 / 02:00-08:00]`

### 核心观点
虽然 Claude Code 也更新了网页浏览能力，但 Codex 凭借更深度的应用集成（插件/技能系统）、全局的记忆（Memory）和更灵活的 UI 设计（Command+T），在体验上已经实现了超越，将自身塑造成了一个真正的“知识工作者操作系统”。

### 深度阐述

为了全面客观，Riley 首先回顾了 Claude Code 的更新。在三天前，Claude Code 的浏览器功能还仅仅局限于打开本地主机（Localhost）的链接，方便开发者调试代码。新版本终于打破了枷锁，可以直接访问外部网站。

Riley 用它做了一个简单的测试：他告诉 Claude，“请为我打开关于使用 Claude Code 最佳实践的链接，不要想太久，直接打开。” AI 开始依次打开几个充满教程和官方文档的网页标签。`[02:30]`

但 Riley 敏锐地指出了 Claude Code 在交互上的两个明显不足：

**第一，效率问题**。Claude Code 的浏览器只能逐个打开标签，无法并行在后台一次性全开。它像早期的拨号网络——“它会一个一个地打开，而不是同时在后台加载。” `[03:00]` 在追求速度的工作流中，这种串行操作带来了明显的摩擦。

**第二，启动门槛**。Claude Code 的浏览器在新建的空白对话中无法直接调用，必须等对话开始后才能使用。这意味着你不能像打开普通浏览器那样“先开浏览器，再决定要做什么。”

### Codex 的真正优势

真正的天花板在 Codex。Riley 直接在 Codex 中还原了他的典型工作流：内容创作。而展示的第一个场景——**AI 驱动的 Google Docs 写作**——完美诠释了为什么 Codex 的浏览器体验远超对手。

#### 工作流深潜：AI 驱动的 Google Docs 写作

想象一下，你正在写一篇关于“AI 浏览器”的深度文章。在 Codex 中，你可以直接按下 **`Command + T`**，调出内置浏览器，导航到 Google Docs，新建一个文档。然后，你告诉 AI：“这是我正在写的一篇文章，标题是‘Riley 对超级应用浏览器的痴迷’，请根据你对我所有的了解，帮我写两段引言。”

这是关键的一刻。Riley 对比了一个惊人的差异：

> “Google 自己的 AI 助手虽然也在侧边栏，但它只知道我的 Google Drive 里有什么。Codex 知道我的一切——它连接了我的 Notion、我的云端硬盘、它保存着关于我所有任务的记忆，真的是一切。那是一个完全不同的层次。” `[04:30]`

因此，Codex 写出来的内容更像是“Riley”写的，具有其独特的语言风格和信息偏好。它可以引用你的项目历史、你喜欢的措辞风格、甚至你最近在 Notion 中关注的主题。而 Google 的原生 AI 只是一个陌生人。

#### 从写作到发布的全链路闭环

能力远不止于写作。Riley 随即想让 AI 将写好的内容格式化，变成推文。他给 AI 下达了一个命令：

> “帮我做三个版本的推文，然后把三个版本都在浏览器中打开成新的标签页，在 X 的发布界面里预先填好。不要真的发出去，只是让我来选择。”

`[06:00]` 44 秒后，三个 X.com 的推文编辑标签页在 Codex 的浏览器中打开，每一个都预填了不同的文案版本。Riley 迅速浏览，关闭不喜欢的，留下最满意的一个，手动点击发送。

在这个过程中，AI 不仅 **思考**（生成内容），它还 **行动**（打开编辑框、填充内容、等待审批）。这是一个从前端到后端、从思考到执行完成闭环的能力。

### 个人感受

Riley 对 Claude Code 和 Codex 的偏好非常明显。这不是简单的站队，而是基于工作性质的必然选择。作为一个以营销和内容创作为主的创业者，他重度依赖 Notion、Drive、Gmail、X 等 SaaS 工具。Codex 由于构建了深厚的“插件与技能”生态（他称之为 Skills / Plugins，在设置中可以找到“Connectors”配置），能够完美驾驭这些工具。他调侃 Claude Code 的 `/drive` 指令毫无反应，是因为 Claude 的“技能”没有从桌面端同步到 Web 端。

他半开玩笑半认真地说：“在 Codex 里，我用 `@Google Drive` 就能调用 Drive。在 Claude 里，你打 `@drive`，什么反应都没有。它太混乱了。” `[07:30]`

这揭示了一个深刻的洞察：在通用模型能力接近天花板时，**应用层的生态集成度和个性化程度**决定了 AI 对普通用户的实际价值。Codex 知道用户是谁，知道用户拥有什么，知道用户正在做什么。

### 精华收获

一个高频使用的隐喻是 **“操作系统”**（Operating System）。如果说图形界面（GUI）是 Mac / Windows 的操作系统，那么对于 AI 时代的知识工作者来说，Codex / Claude 正在成为新的 OS。而这个 OS 的“桌面”和“窗口”，就是这个多标签浏览器。

Riley 提出了一个震撼的观点：**你对电脑的一切操作，最终都可以抽象为在浏览器中打开一个页面、点击一个按钮、填写一个表单。当 AI 能够代理完成这一切时，它就不再只是一个聊天机器人，而是这个新操作系统的“内核”。** 浏览器，就是这个内核最完美的外接界面。

## 第三部分：从“告诉我做什么”到“问我该做什么”——新操作系统的定义 `[章节 3 / 08:00-12:00]`

### 核心观点
多维浏览器的力量最终导向了人机交互的终极范式转变：从“人类告诉 AI 执行具体步骤”模式（Tell），进化为“AI 自主分析全局后给出建议，人类选择执行哪条线索”模式（Ask + Pull the Thread）。这是 Jack Dorsey 提出的核心理念，也是 Riley 演示的最精彩、最震撼的部分。

### 深度阐述

Riley 引用了一条来自 Twitter 前 CEO、现任 Block CEO 的 Jack Dorsey 的推文，这成为了本段的理论基石，也是整个视频的灵魂：

> “我从告诉代理要做什么，转变为问它应该做什么，然后拉最有力的那根线。”（I’ve shifted from telling agents what to do to asking them what to do and pulling the best thread.）`[08:30]`

这是一个巨大的角色反转。在传统工作流中，你是 CEO，也是执行者。你必须在看完了所有的 Slack、邮件、日历和 Notion 面板后，才能做出“我今天该干什么”的判断。这个过程充满了认知负担和上下文切换的疲劳。

现在，Riley 向 Codex 发问：

> “根据你对我所有的了解，以及接下来重要的截止日期，告诉我我现在最应该处理什么任务？给我三个选项。” `[09:00]`

接下来的画面极具冲击力。Codex 并没有直接生成一个文字列表。我们看到这个“操作系统”开始真正 **“行动”** 了：

- 它在读取 Google 日历（`Reading Google Calendar.`）
- 它在搜索 Notion 数据库
- 它在扫描 Gmail 收件箱
- 它甚至读取了 Riley 过往的工作习惯和项目优先级

它在替人类 **“看仪表盘”**。

几秒钟后，AI 给出建议：“最重要的任务是准备周四要录制的两期播客。”

“好，就这么办。” Riley 接受建议——这就是 **“拉绳子”**（Pulling the thread）的精髓。他没有去反驳，没有去重新思考，而是信任了 AI 的全局分析，然后将注意力聚焦在下一步。他继续下达命令：

> “请打开这两期播客的 Google Doc。在旁边的标签页里，打开所有相关的外部研究资料。对于每一位嘉宾，在聊天区（不是文档里）给我准备 10 个访谈问题，并指导我如何设计结构以获得最多的观看量。” `[10:00]`

瞬间，所有相关资料被打开——嘉宾的文档、嘉宾的 LinkedIn 主页、嘉宾上一次做客的播客链接、甚至相关的行业研究文章。整个桌面环境被 AI 完全重新编排。

Riley 描述了这个过程的震撼之处：

> “在旧世界里，我需要自己打开电脑，处理信息，决定该做什么，打开所有标签页，打开 Notion。现在，是一个应用帮我把所有东西都打开。这是一种完全不同的与电脑交互的方式。” `[10:30]`

### 上下文切换的终结

这是这个概念最深层的力量。**上下文切换是知识工作者最大的敌人，而 Codex 正在消灭它。** 在传统模式下，每切换一个任务（从查邮件到写文档到看日程），你需要 10-15 分钟来重新找到状态。而在 Codex 的模式中，AI 基于当前任务，主动为你构建了全部的上下文。你所做的只是一个确认、一个指令，所有材料就已经到位。

Riley 展示了一个进一步的例子：他让 AI 在听了一位嘉宾之前做客的播客后，自动将内容整合进还未开始写的节目文档中，直接写出一段开头的引导语。AI 直接编辑了 Google Doc，将“Hook”（吸引人的引子）直接插入文档。这是一个从“搜索”到“分析”到“创作”到“输出”的全自动闭环。

### 个人感受

Riley 对这种新体验的描述是：“完全是一种与电脑交互的崭新方式。” 在旧世界，是他自己去搜索、去打开、去整理。而现在，他只是在做决策和审批。他感觉自己从“操作员”升级成了 **“指挥官”**。这种降维打击带来的快感是巨大的，他形容这是一种“流动”状态（Flow State）而不是“忙碌”状态（Busy State）。他不必再担心自己忘了什么、错过了什么，因为 Agent 已经替他看过了所有。

### 延伸思考

这背后的深远意义在于，**AI Agent 的终极形态不是一个工具，而是一个“工作伴侣”**。它了解你的长期目标和短期任务。你不需要事无巨细地分配工作，只需要设定高维的目标和价值观，它就会规划路径、收集资源、搭建环境。

多标签浏览器在此扮演的角色是 **“可视化的工作空间”**，让人类得以监督和介入。没有这个空间，AI 变成了一个无法被监督的黑箱；有了这个空间，AI 是透明的、可理解的、可干预的。这正是“人机协作”的完整体现。

### 精华收获

衡量一个 AI 工具是否进入了“OS”级别，标准不再是它能回答多少问题，而是它能否替你浏览、操作和编排你的数字工具链。Riley 对 Codex 的评价非常高，因为它正在从“提问机”进化为“执行器 + 环境调度器”。范式转移的核心在于：**人类不再是信息的收集者，而是决策者；AI 不再是回答者，而是信息世界的操作员。**

## 第四部分：四大使用模式与六大实战工作流——将 AI 浏览器编程为生产力引擎 `[章节 4 / 12:00-18:00]`

### 核心观点
将浏览器操作嵌入到具体的自动化“技能”中，实现一次配置、反复使用，是目前通往 AI 超级生产力的最短路径。通过邮件草稿、B-Roll 搜索、社交排期和并行多任务处理的深度实例，Riley 展示了 AI 浏览器如何从“玩具”变成“工具”。

### 深度阐述

在进入具体工作流之前，Riley 总结了他使用 Codex 中浏览器功能的 **四大模式**：

**模式 1：手动导航**（Manual Navigation）
在新建对话中，直接使用 `Command + T` 快捷键调用内置浏览器。它可以像普通浏览器一样导航到任何网站。Riley 说：“有时候我开一个新对话，就直接打开了浏览器，完全把它当浏览器用。” 这是最基础、也最惊人的功能——你的 AI 应用本身就是浏览器。

**模式 2：请求 AI 打开**（Ask AI to Open）
你不需要自己动手，直接告诉 AI：“嘿，我们在 Notion 里有一个关于 XX 长视频的页面，请帮我把它在浏览器中打开。” 由于 Codex 深度集成了 Notion 的 API，AI 可以快速定位并打开该页面。这是对自己应用生态的掌控。

**模式 3：AI 批量打开**（AI Opens Many Tabs）
这是力量的一次展示。Riley 让 AI 将 Notion 中全部 16 个“进行中”的视频项目全部在浏览器标签页中打开。`[09:00-10:00]` 瞬间，16 个标签页铺满了浏览栏。Riley 指着其中一个标签说：“这个，就是现在这个视频的脚本。”

这种能力对于项目管理、竞品分析、信息收集极其惊人。传统上你必须手动浏览 Notion 条目、找到页面、逐个点开。现在，AI 替你遍历整个数据库，把视觉结果全部呈现在你面前。你的大脑不再需要花时间在“调取”信息上，可以直接聚焦于“处理”信息。

**模式 4：嵌入技能**（Putting it in a Skill）
这是最强大、最成熟的形式，也是 Riley 整个视频中反复强调的核心。你可以将浏览器操作 **编程为可持续复用的技能**（Skills）。每当触发某个特定场景（如“检查支持票”、“准备推文”），AI 会自动执行一系列浏览器操作。

### 六大实战工作流

Riley 展示了四个深度工作流，每个都颠覆了传统的做事方式。

#### 工作流 #1：自动邮件回复系统 `[11:30-12:30]`

“我需要你查看当前所有的支持票据，然后为每一封草拟回复。在回复写好之后，以草稿形式在应用内浏览器中打开，让我来审核和发送。什么都不要真的发送出去。”

AI 收到了指令，它调用了邮件阅读技能，逐封分析邮件内容（包括用户遇到的问题、附上的收据截图等），然后为每一封生成精准的回复草稿，并在新的标签页中打开预览。Riley 可以逐一点击标签，阅读 AI 生成的回复。满意就手动发送，不满意就编辑一下。

```
“Sorry to hear about this. Thank you for sending the receipt. I’ve flagged the $20 charge.”
```

Riley 看着这些自动打开的标签页，强调说：“这是我用来处理所有邮件的实际工作流。” `[12:00]`

#### 工作流 #2：视频 B-Roll 图片搜索 `[12:30-14:00]`

视频创作者最大的痛点之一：写好了脚本，但找配图（B-Roll）要花很久。Riley 做了令人震撼的演示。

他告诉 AI：

> “帮我找至少 15 张在视频中做配图的图片。要求：按照脚本中的每一行顺序，在 Google Images 中打开搜索结果的标签页，按这个顺序排列。让我可以一个一个标签页看过去，快速挑选图片。”

AI 开始逐行分析 Riley 最新的视频脚本。它理解了每一句话的语义，决定该搜索什么图片。然后——这件事发生了——它以视频播放的时间顺序，在 Google Images 中逐行搜索，并将每一个搜索结果都打开成一个新的标签页：

- “OpenAI GPT 5.6 模型发布” → 打开了 OpenAI 的 Logo 图片搜索页面
- “Anthropic CEO AI 安全警告政府” → 打开了相关新闻图片搜索结果
- “新 AI 模型发布” → 科技感的背景图片搜索结果

整个过程在数十秒内完成。Riley 惊叹：“通常我需要手动搜索一个个关键词，现在它替我按时间顺序全部排列好了。我只需要一个标签页一个标签页地看过去，把合适图片拖进视频编辑器就行。”

#### 工作流 #3：自动排队推文及配图 `[14:00-15:30]`

AI 不仅仅处理文字，它还能操作本地文件与社交媒体的交互。

Riley 让 Codex 扫描他的“下载”文件夹，找到最近的截图，分析图片内容，然后自动开始排队推文。AI 做了三件事：
1.  读取本地文件夹，选择图片文件
2.  分析图片内容（如“这是我的 YouTube 视频最佳表现的截图”）
3.  在 X 的发布页面打开新的标签页，预填入图片和最佳配文

> “你们看，它选了图片，写了文案，在标签页里都准备好了，我只需要按发送。” `[15:00]`

这种方法可以扩展到 Instagram 等任何社交媒体平台。你只需要告诉 AI：“去我的‘待发布’文件夹，把所有图片都准备好，每个打开一个标签页。”

#### 工作流 #4：多线程并行任务执行（视频的最高潮）`[15:30-18:00]`

这是 Riley 整个演示的巅峰，也是真正展示 Agent 能力的“杀手锏”。他做了之前任何 AI 应用都做不到的事情——在同一会话中，创建三个独立的并行任务。

他向 Codex 下达了一个极其复杂的命令：

> “我想让你创建三个独立的 Codex 线程。每个线程都必须在应用内浏览器中打开大量标签页。
>
> 线程一：分析我下一支视频的钩子。请搜索 10 个同类热门视频的钩子，在一个 Google Docs 中创建报告，并打开每个视频的链接。
>
> 线程二：做产品调研。我需要一台升级版的 Windows 电脑，用来做多摄像头视频录制和后期制作。请做深度调研，打开所有相关的电商链接、测评页面。
>
> 线程三：寻找播客嘉宾。帮我找到 10 位理想的播客嘉宾。打开他们的主页或包含他们最佳内容的 YouTube 链接，标注为什么他们是好的选择。”

Codex 没有拒绝，没有困惑。它接收命令，开始在后台调度。三个独立的“子会话”迅速创建：

1.  **Hook 研究线程**
    打开了 Notion 文档，搜索了 YouTube，找到了 10 个热门视频的链接，并为 Riley 创建了一个汇总文档。

2.  **PC 调研线程**
    对这个线程来说，Codex 并不知道自己也在执行其他任务。它专注于寻找最佳的视频剪辑电脑配置，打开了数十个专业的电商和测评标签页。

3.  **嘉宾搜索线程**
    列出了潜在嘉宾名单，并打开了他们的最高质量内容网页。

Riley 逐一检查每个线程：“所有线程都完成了！看，它有这么多标签页。点击任何一个都能看到详细资料。它还专门为 PC 调研建了一个文档，列出了所有型号和链接。对于嘉宾搜索，它打开了相关的 YouTube 视频链接和直接的时间戳。”

以前，你是一个人在多个窗口间焦头烂额。现在，你拥有了三个并行运作的 AI 研究员，互不干扰，成果斐然。

### 个人感受

在这一环节，Riley 的兴奋达到了顶点。他反复强调“这太酷了！” 相比于聊天问答，这些具体的、自动化的、多线程的工作流让他看到了真正意义上的“工作效率提升 10 倍”的可能性。他不再是一个普通的视频博主，他正在打造一套属于自己的“AI 工作流工厂”。

他特别强调，在三个线程并行执行时，自己可以完全专注于其中一个。他不需要去切换上下文，因为系统已经自动为他划分好了。他甚至不需要担心其他任务的进度——它们会在后台静默完成。

### 精华收获

如果我们将 AI 视为一种新的劳动力，那么 **“技能”（Skills）就是为这个劳动力编写的 SOP（标准作业程序）**。以往编写 SOP 复杂且难以执行；现在，在 Codex 的架构中，你只需要用自然语言演示一遍，AI 就能学会，并以极低的成本、极高的速度进行大规模复制。浏览器，就是这个大规模自动化执行的舞台。它让 AI 从一个“头脑”长出了可以操作一切的“双手”。

Riley 最后总结说，所有这些功能都是“仅在两天前发布的”。这意味着未来几周内，随着用户和开发者发现更多使用方法，工作流的可能性会呈指数级增长。

## 第五部分：Agent Native 的未来浪潮——为什么你必须现在就适应 `[章节 5 / 18:00-结尾]`

### 核心观点
这两天的更新只是一个开始。Riley 坚信，在未来的几周内，我们的工作流将发生剧变。应用内浏览器消灭了“跳出 AI 应用去手动操作”的摩擦，是 AI 超应用最终形态的关键拼图。他呼吁大家开始思考如何成为一个“Agent Native”（以 Agent 为原生交互方式）的人。

### 深度阐述

在视频的结尾，Riley 给出了他的终极思考。

为什么浏览器功能如此重要？因为他认为，**“浏览器是所有 AI 代理最被低估的界面”**（The browser is the most underrated interface for all AI agents.）`[18:30]`

他解释了这个洞察背后的逻辑：
- 浏览器已经“坐在”几乎所有我们使用的工具之上
- 给 AI 浏览器权限，加上记忆和上下文，再加上行动能力
- 它就不再是一个你浏览互联网的地方
- **它变成了一个你完成工作的地方**

他非常反感当一个 AI Agent 需要执行任务时，突然弹出一个 Chrome 浏览器新窗口，把你从沉浸的 AI 工作区里拖走。那种摩擦是致命的，是打断心流的元凶。

> “我只需要它在我正在工作的应用里打开一个新标签页就行了。仅此而已。” `[18:45]`

这个简单的诉求，背后是宏大的产品哲学：**一个超级应用，必须能够承载你在数字世界中的绝大部分交互。** 其他工具（如日历、邮件、Docs）不再是外部的应用，而是可以通过浏览器标签页嵌入到 AI 母体中的“子页面”。

这解释了为什么 OpenAI 和 Anthropic 不约而同地在此时押注浏览器。因为这是通往下一代操作系统门票的关键战斗。谁的应用内浏览器体验更好、更集成、更能承载“技能”的自动执行，谁就掌控了未来十年的计算入口。

Riley 将自己的使命定义为 **“帮助人们成为 Agent Native”**（becoming agent native）。这不只是一个流行词，而是对新一代数字公民的期望：

- 从“操作界面”转变为“指导代理”
- 从“记忆信息”转变为“调度信息”
- 从“亲手执行”转变为“审批交付”

### 个人感受

视频最后，Riley 坐在他新布置的演播室里，难掩疲惫但眼神发光。他计划将产出视频的频率提高到每周 4-5 个，因为“有太多东西需要覆盖了”。他极其真诚地渴望观众的反馈，希望大家分享自己使用这个新功能的工作流，以及“下一期视频我应该做什么”。

这种激情不是来自流量或者算法，而是来自他作为一个重度用户，亲身感受到了技术带来的巨大冲击。他是在用他的实际行动和工作流，去验证一个未来的可能性，并迫不及待地分享给所有人。

他说：“所有我真正关心的，就是帮助人们成为 Agent Native，用 AI Agent 在自己的企业中完成工作，做更好的工作，做更有创造力的工作。” `[19:30]`

### 延伸思考

**“Agent Native”** 这个概念极有可能成为 2025 年最重要的生产力话题之一。它意味着你要学会信任 AI 去执行，学会用 AI 能理解的逻辑去构建流程。

对于企业和个人而言，关键不再是“我会用 ChatGPT 提问”，而是“我能设计一个让 ChatGPT/Claude 自动替我完成整条工作流的技能”。这不是淘汰人类，而是解放人类。

Riley 的演示也暴露了一个先决条件：要实现这样的能力，你需要一个像 Codex 这样 **深度集成你的个人数据和应用** 的 Agent 系统。那些没有连接你的 Notion、Drive、Gmail 的 AI 工具，将永远无法做到这一点。这也是为什么生态集成能力如此重要——它决定了 AI 能否真正“理解你”并“为你行动”。

### 精华收获

1.  **内嵌浏览器 = Agent 的双手**。没有这个更新，Agent 只能思考无法行动。有了它，Agent 可以操控世界上几乎所有的 Web 应用。这是 Agent 从“被动”到“主动”的关键一跃。

2.  **技能化是关键**。别每次都写 Prompt。把高频任务（邮件回复、竞品调研、社交排期）编写成可复用的“技能”，让 AI 一键启动、批量开窗、等你审批。这是从“零散使用”到“系统化赋能”的进化。

3.  **从 Tell 到 Ask 的思维转变**。不要自己先分析再告诉 AI 怎么做。反过来，让 AI 先分析所有全局信息，给你选项，你来决策。这是将 AI 效率最大化的核心心法，也是 Jack Dorsey 那条推文的精髓。

4.  **拥抱并行多线程**。不要只开一个聊天窗口。你的大脑可能只能线性思考，但你的 AI 团队可以同时调研、创作和分析。把它们部署在 Codex 的不同线程中，并行轰炸你的工作难题。一个线程查资料，一个线程写文档，一个线程找嘉宾，互不干扰。

5.  **摩擦是敌人**。任何需要使用“跳出应用”的工具都会打断心流。好的 Agent 工具应该把所有操作都内嵌在同一个界面中，让你不需要离开战场。Codex 和 Claude Desktop 正在走向这个方向。

# 结语

这场由 OpenAI 和 Anthropic 在同一天引发的浏览器革命，并非一场临时起意的功能竞赛。它是人类与 AI 交互的一座里程碑。

Riley Brown 的深度体验直接告诉我们：不管你相不相信，一个新的操作系统已经在你的屏幕里悄然启动了。这个系统的“桌面”不是壁纸和图标，而是一个 AI 驱动的、可以完成工作、打开网页、起草邮件、写文档、找素材、做调研的多标签浏览器。人类不再需要在数字世界的迷宫中手动穿梭——我们有了一个能够替我们打开所有门的代理。

正如 Riley 所说：“我将从告诉代理做什么，转变为问它该做什么。”

这，就是 Agent Native 的未来。
