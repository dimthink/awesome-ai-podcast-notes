# AI代理时代：机遇、挑战与未来

> 本文根据 a16z 播客《The Era of AI Agents》中 Aaron Levie 的对话整理，探讨了 AI 代理对企业软件、系统集成、经济模式及组织安全带来的深远影响。
## 一、AI 能力的扩散速度：硅谷的乐观与现实差距

**AI 能力的扩散速度将比硅谷人们意识到的要慢得多。**

- 认为可以靠“直觉编码”（vibe code）就打造出像 SAP 那样的系统，这种想法是荒谬的。
- 所有的领域知识，并不只是存在于某个精心编排的数据层中。
**工程计算预算的讨论，将成为未来几年最疯狂的话题之一。**

- 目前最大的问题是，每个人都在试图弄清楚这一切的经济账，但他们对这个机会的规模低估了至少一个数量级。
## 二、为代理而构建的软件：新范式的兴起

**如果你拥有的代理数量是人类的一百倍甚至一千倍，那么你的软件就必须为代理而构建。**

- 人们抽象地谈论：“你现在要向代理营销，就像你是一个 API，你有一个好的接口定义语言（IDL）。”
- 但实际上，这个观点几乎完全错了。
**我们都在思考如何为代理构建软件接口。**

- 如今，我们花在思考代理如何访问我们工具上的时间，与思考人类界面一样多。
- 原因在于我们的假设：如果代理数量远超人类，那么软件就必须为代理而构建。
- 代理将如何与你的系统交互？通过 **API**、**CLI** 或 **MCP** 等。
**一种正在兴起的成功范式：给编码代理（coding agent）访问你的 SaaS 工具、知识工作流程和上下文的权限。**

- 代理不仅能读取数据、理解信息，还能通过编码或 API 来完成它试图实现的任务。
- 这种范式正在产生复合效应，例如 Claude 和 Cloud 的协同工作现象，以及 OpenAI 正在研发的超级应用、Perplexity 计算机等。
- 这似乎是这类技术终极体现的一种合理方式。
## 三、算法思维：人类与代理协作的关键瓶颈

**从理论上看，代理范式是合理的；但在实践中，我们必须非常小心。**

- 关键在于**算法思维**（algorithmic thinking）。对于绝大多数有工作的人来说，这是非常非常困难的。
- 如果你让任何一个人为他们要做的事情画一个流程图，他们很可能会失败。
- 在任何组织中，比如一个 50 人的营销团队负责一个大型产品线，可能只有一个人能理解并记录下整个流程图。
**如果你把代理工具放在人们面前，他们解释“该做什么”的能力非常有限。**

- 那么，如果这成为与计算机交互的新方式，你只能通过循环迭代来适应吗？
- 本质上，你只是在开发下一个抽象层。历史上，每个抽象层的开发都由组织中高度熟练的特定个人完成。
- 他们构建的小部件成为人们执行特定任务的工具。有些人能把它们拼接在一起，有些人不能。这和过去的回形针、图钉一样，未来也会如此。
**不变的是：工作只是向上移动了一个台阶，你需要学习一套新技能。**

- 从这个角度看，AI 代理带来的变化并没有什么不同，只是你获得的杠杆效应是惊人的。
- 例如，有一个关于 Anthropic 增长营销人员的推文很火：一个人使用 Claude Code，自动化了原本需要 5 到 10 个人在各自孤岛中完成的工作。
- 这之所以有趣，是因为你需要成为一个**系统思考者**才能做到这一点。这个人本身已经足够懂技术。
**未来工作的想象：每个工作旁边都有一个无限的工程师池，可以自动化该人想要的任何东西。**

- 你需要把自己的工作看作一个系统，才能实现这一点。也许代理会随着时间的推移变得更好，能够引导你朝那个方向前进。
- 例如：将 Google AdWords 中有效的关键词移植到 Facebook，并确保它们被复制，然后吸收市场的新信号。
## 四、真实世界的例子：从电子表格到代理

**一个关于“代理”的旧时代例子：**

- 一位 MBA 毕业的亲戚，在计算技术刚兴起时加入第一份工作。她在研究生院没用过电子表格，当电子表格出现时，她不是“电子表格人”。
- 公司告诉她：你想招多少实习生就招多少。于是她第一年工作就监督了一整个房间的“代理”——那些大学生来做所有的电子表格工作。
- 神奇的是，几年后，她和她的同龄人都变成了“电子表格人”。
- 旧的抽象层（坐在那里用 HP 计算器手动建模 M&A 交易）消失了。新的抽象层（管理实习生做表格）出现了，然后这个抽象层本身也消失了。
**我们现在与代理的关系，正处在类似的阶段：**

- 你觉得需要 50 个代理，由一个超级聪明的人协调它们。很快，这一切都会坍缩，只剩下一种技能集合，你可以称之为“营销类代理”。
- 你可以向它询问营销事宜，并让它去执行。
- 我有点怀疑，直到 AI 的不可复现、非随机元素消失之前，“执行”会变得非常昂贵，从而引出“人机协同”的讨论。
**当前状态：你必须是一个火箭科学家加上增长营销人员，才能创建 42 个代理并让它们全部运转。**

- 但“火箭科学”这部分很快就会消失。
- 然后，话题又回到了**领域专家**身上。
## 五、代码、终端与计算机使用：代理交互的演变

**我们实际上正在走向相反的方向：**

- 一开始，我们是在 SaaS 软件中加入 AI，这是使用代码的极端版本。
- 但现在，SaaS 软件仍然是 SaaS 软件，代理把它当作计算机来使用，因为它非常擅长这一点。
- 我们从代码（code）走向终端（terminal，实际上代码更少），而今年将是**计算机使用（computer use）**的一年。
- 代理更像人类一样使用计算机，而不是生成代码。这感觉像是一个中间步骤。
**我的观点是，无论是计算机使用、API 使用还是即时编写代码，我都把它们归为一类。**

- 我们正在开发一个代理，它会自行判断：应该使用现有技能、使用 Box 的现有工具，还是编写代码来解决问题。
- 它能在任何时刻做到其中任何一项，这非常有用。因为有时你想执行某个特定操作，编写代码更快，我们无法预规划用户可能对文档做的所有事情。
- 模型足够好，可以即时编写代码，这是一个很棒的特性，尽管 90% 的情况下它应该使用现有工具。
## 六、软件与代理的长期演进：系统整合与数据层

**关于“iPhone 上只有 7 个应用”的观点：**

- 人类没有心智带宽去学习太多应用。但一个将要使用工具、API 并能编写代码的代理，没有我们这样的限制。
- 你可以说，有太多事情要做，你可以让接口足够通用。
**软件演进中一个非常有趣的点：**

- 以 SAP 为例，我整天用它做财务，生成报告。有人想要一个特定视角的报告，我得去 SAP 帮助系统里艰难寻找。
- AI 非常擅长导航这个庞大的表面积。人类一直是过去 25 年软件能力挖掘的瓶颈。
**我完全接受“消费层”的完美流畅 UI。但在后端，记录系统（systems of record）可能会收敛到某种通用数据库或一组通用 API。**

- 我在周末实现我的 nano cloud bot 时，一开始要为所有东西做集成。OpenCloud 有所有集成，NanoCloud 很少。
- 但两三天后，你就有了你需要的工具集成。
- 但当涉及到拥有全球供应链的公司，处理 75 条信息、30 个不同系统时，确实需要代理有强大的处理能力，这是我们以前任何架构都无法做到的。
**这正是过去 50 年一直在做的事情：系统集成与冗余。**

- 我有个朋友是退伍军人事务部（VA）的 CIO，他所有时间都花在将 75 个 VA 系统粘合在一起。
- 对于集成，代理是最棒的。但关键是**按需集成**：IT 团队没有预先连接的查询，现在需要在运行时完成。
## 七、CFO 与 CIO 的担忧：系统安全与控制

**我在一群 CFO 和 CIO 面前说过类似观点（虽然没那么乐观），结果有六个人事后跑来说：“你疯了，你在我这里失去了所有信誉。”**

- 他们的恐惧不是代理本身，而是**人类**利用代理进行集成，从而破坏他们的记录系统。
- 你在系统 27 和系统 38 之间创建了一个新 API，如果那个人想犯错，那是他的事，但你不能……
**我认为在很长一段时间内，我们会有只读（readonly）版本的这种操作，而且 N（集成数量）会非常大。**

- 很多情况下，消费层的人类消费者才是重点。
**实际案例：Box CLI**

- 我们推出了官方的 Box CLI。你给 Claude Code 这个 CLI，就能通过自然语言与整个 Box 系统交互。
- 例如：“把这个文件夹从我的桌面全部上传到 Box”或“处理这个文件夹中的所有文档”。
- 但想象一下，一家 5000 人的公司，每个人都能访问共享仓库，并且每个人都运行着带有 CLI 的 Claude……我们遇到了新的挑战：
- 这将成为每个 CFO 和 CIO 火烧眉毛的新问题。
**我在玩你的例子（创建营销计划目录）时，很快就陷入了一个循环，不停地创建目录。**

- 我想知道 Box 对嵌套目录的限制是多少，因为我快达到了。
## 八、代理的身份、权限与安全困境

**直觉告诉我们构建新的控制层，但实际情况相反：**

- 当我们拿起个人代理时，我们给它们 API 密钥、电子邮件地址。
- 现在大家做的是：给代理自己的**电话号码**。我甚至给我的 NanoClaw 一张自己的信用卡（希望是你在 CVS 买的 Visa 借记卡）和它自己的 Gmail 账户。
- Gmail 有各种权限系统。你可以把代理当作一个独立的**人类**来对待，而不是构建另一个抽象层。
**企业中的问题：**

- 假设一个 50 人的团队，50 个人类加 50 个代理在同一个共享空间。我完全掌控我的代理，但如果我的代理与别人的代理协作，不小心访问了我不该访问的资源呢？
- 这个有状态（stateful）的自主代理正在处理别人的信息。
**默认的端到端论证是“把它们当作人类”。但这行不通。**

- 对于真正的人类，你不能查看他们的 Slack 频道，不能以他们的身份登录，不能监督他们。他们要为自己的执行负责。
- 但代理，你**完全有监督权**，而且你可能需要这种监督。它们没有隐私权。
- 我需要能够给它们访问权限，但也需要在它们搞砸时以它们的身份登录，撤销一切。但如果我可以登录，它们又如何在现实世界中与其他代理协作并保持机密和安全？
- 所以，代理仍然是你的**延伸**，几乎不可能绕过这一点。
**我的看法：**

- 对于我的员工，我可以登录他们的账户（在诉讼等情况下）。但我不经常登录，因为他们发了一封邮件。
- 但代理的风险要大一千倍。它们会乐意在收到提示时就把你的信息泄露出去。
- 你认为最终状态是这些“邋遢的计算机”永远无法控制信息？
- 我认为，让代理不泄露上下文窗口中的秘密是一个非常难解决的问题。如果任何东西能进入上下文窗口（因为它们有资源访问权限），理论上你可以通过提示注入把它弄出来。我们目前不知道如何解决这个问题。
- 如果我新代理的电子邮件地址，我给它发邮件，我可以比人类容易 10 倍地进行社会工程攻击。
**但这不就是所有 AI 的问题吗？**

- 当我们内部使用 AI 和代理时，它们确实就是这样工作的——作为你的延伸。我们还不知道如何让它们不成为你的延伸。
- 解决这个问题的完美例子：我们已经用**开源**经历过这一切。模型是：所有代码都在那里，你挑选使用。然后很快，所有人都意识到你刚才谈论的所有问题：许可、质量等等。于是各种规范发展起来了。
- 当前关于代理的辩论，只是新技术发展的现代产物，一切都在实时发生。大家只是想快速到达终点状态，比实际能到达的速度快得多。
**我们需要的是标准。**

- 我们对终点状态有不同的直觉。一种直觉是：这些东西最终会收敛到与人类相同的可靠性水平，就像自动驾驶一样。然后我们使用与保护人类完全相同的机制：考虑内部威胁、人员可能被收买、人会犯错，并建立操作流程。
- 另一种直觉是……
- 实际上，我们可能并不分歧终点状态。战略上我们在对冲，我们会构建代理用户和常规用户。我喜欢“OpenClaw 拥有一个 Box 账户”的想法，这会带来双倍的账户量。我太喜欢了。
- 但在地面上，我们目前还不知道如何给它一个 M&A 数据室，并让它完全安全地运行。
**威胁向量会变得非常复杂。我们有一场猫鼠游戏：你不能假设代理的行为像今天的人类，因为它会是有史以来最快、最周到、最疯狂的“人类”，试图以某种被注入的方式泄露信息。**

- 企业客户会关闭一切，直到有一些理智出现。与此同时，个人（尤其是开发者）会取得巨大进步。
- 这种紧张关系最令人兴奋：企业会被这些高级个人甩在后面，然后这些个人看起来像初创公司，初创公司会比大公司跑得快得多，因为它们没有这些问题。
**在初创公司里，代理“叛变”了怎么办？**

- 初创公司里本来就有员工经常叛变。那只是《硅谷》里的一集，没什么大不了的。
- 我同意风险类似。但有一些不同：我不能像威胁普通员工那样威胁 Claude Code（比如“我要拔掉你的插头”）。至少 95% 的人不是故意做坏事的，但他们可能**无意中**做坏事。
- 我认为，让人不要错误地分享文件，比让代理遵循同样的指令要容易得多。而且你有工具可以在不同的抽象层阻止它。
## 九、AI 能力扩散的长期障碍：大企业与初创公司的鸿沟

**AI 能力的扩散速度比硅谷人们意识到的要慢，原因就在这里：**

- 我们看到初创公司可以从零开始，没有我们谈论的任何风险（因为它们没什么可炸毁的），我们把这视为轨迹。
- 但你去摩根大通问：“你们打算如何很快地用 NanoClaw 自动化业务？” 答案是：“哦，那会有点差距。”
**这开启了一个有趣的问题：大公司与小初创公司/企业之间的分裂。**

- 当前的 SaaS 供应商（在所谓的“SaaS 末日”中挣扎）实际上销售的是智能、领域专业知识和整个系统，而不是业务线数据。
- 而代理这边只想购买**数据**，并拥有无限访问权限。但这从来不是 SaaS 供应商的业务，长期以来一直是 Workday、SAP 等公司的痛点：开放多少 API 访问？
**从技术角度看：在人们想要访问数据（用于训练或日常运营）的情况下，“记录系统”（system of record）意味着什么？**

- 他们的担忧是：一个大客户，他们的供应商想要在客户的数据上构建训练层。
- 即使不涉及训练，他们也担心货币化：通过互联网发送一点数据 vs. 在你的 UI 里，这是完全不同的货币化水平。
**以 SAP 为例，他们哪儿也去不了。**

- 认为你能靠“直觉编码”达到 SAP 的水平，是荒谬的。
- 所有这些领域知识不仅仅存在于一个精心编排的数据层中。还有大量知识在 UI 中、在中间层中、在你如何使用它之中。
- 我不确定这会如何演变，因为 SAP 不会消失。这将减缓 AI 在那个特定数据源上的扩散，无论它是代理化的 AI 还是只读报告。
## 十、为代理构建：从营销到系统质量

**我喝下了“为代理想要的东西而构建”的 Kool-Aid。**

- 经过足够多次迭代，代理将在很大程度上决定它想要实现和使用哪些工具。
- 代理不能换掉一个企业系统，但经过足够多的世代，代理可能会在你的软件上遇到太多墙，然后说：“你最终必须淘汰你的遗留 HR 系统，否则我无法为你自动化这个工作流。”
- 想象一下，软件上的代理流量是人类的一百倍或一千倍。做足够多次，软件栈最终必须为代理而构建。
- 你的业务绩效将与你的代理能否获取所需信息的能力相关。因此，你的企业 IT 栈必须以支持代理的方式设置。代理某种程度上是“掌权者”。
**这对 SaaS 或软件公司意味着：**

- 游戏规则变成了：你能构建高质量的 API 吗？你有货币化的方式吗？你能处理代理的身份和访问控制吗？这是软件公司必须解决的新问题。
- 如何货币化？Workday 是按拉的每条 HR 记录收费一分钱吗？我们会搞清楚的。
- 在某些业务中可能意味着更少的收入，在其他业务中可能意味着更多的收入。每个代理都喜欢处理文件，所以未来文件会更多。我们押注这对我们的商业模式是乐观的。
**我要对“为代理构建”这个说法挑个刺。**

- 人们抽象地说：“向代理营销，最重要的是你是一个 API，你有好的 IDL。” 我认为这几乎完全错了。
- 代理非常擅长找到自己的路。最终重要的是**语义**。代理很擅长根据成本参数、持久性等选择正确的后端。它们实际上拥有我们使用这些平台的集体智慧。
- 所以，我认为我们行业过于关注接口，而实际上，我们会被推动去构建**更好的系统**，这才是代理会选择的。
**我们实际上完全一致。**

- 我不是说营销之类的东西。我的意思是，如果你的工具对代理关闭，代理最终会为公司找到一个更好的工具。
- 曾经你去找 Gartner 问“该用什么系统”。经过足够多次迭代，代理会告诉你：“你应该为此类操作使用这种数据库。” 如果你不在那里，你就出局了（DOA）。
- 我们应该庆祝这一点，因为代理在选择技术上相当聪明。过去，其他因素（如销售）才是购买决策的原因。
- 但别担心，硅谷会很快破坏这种 meritocracy。我会在推荐上花钱。Workday 的营销代理将有购买推荐的能力。
**真实的风险：代理会启动一个事实上的新记录系统。**

- 每个公司都有文件共享，里面有最好的文档、幻灯片、财务模型。人们熟悉它们，找不到就创建新的。许多组织基本上像自由市场一样运作。
- 代理会严重碎片化 IT 人员认为的中间件/终端用户 BS 区域。宏（macros）最终会运行公司。
- 他们看过这部电影：让营销人员自己去买个网站办活动，结果安全漏洞、邮件列表泄露、公司被起诉。
- 所以，现实世界中的紧张关系比我们表现出来的要多得多。
**两种对立的观点：**

1. **“端到端坍缩”观点**（类似 Elon）：提示词直接生成机器码，所有现有接口和层都会消失。
1. **“层永不消失”观点**：历史上，层只会被叠加。许多层实际上是组织边界、状态边界或兼容性要求。它们会为了兼容性而保留。代理会映射到这些层上。
- 我倾向于后者。系统会以相当相似的方式被使用，也许有更多代理在使用它们，但不会演变太多。
## 十一、从第一性原理出发的新公司

**我着迷于那些从零开始、纯粹从第一性原理出发的服务类公司。**

- 例如：营销机构、工程咨询公司、律师事务所、建筑/设计公司——任何知识工作者服务公司。
- 你可以用完全不同的方式构建你的公司：没有信息壁垒，可以给代理所有需要的上下文，可以即时编写特定软件。
- 这将在一定程度上具有破坏性，直到大型现有企业能够摆脱束缚。这将为新型公司提供先例和案例研究。
- 但随着时间的推移，它们仍然会遇到与所有其他公司完全相同的问题：地理、市场细分、分销挑战——任何你小墙之外的东西，你都会遇到物理世界。
**我喜欢新商业模式开启的想法。**

- 有大量的信息或软件被低估了 100 倍，因为没人愿意花 5 美分访问一条数据，或花 1 美元使用一次工具。
- 但如果你给代理一个预算和一个协议，突然之间，它们可以在飞行中为深度研究任务获取医学研究，并愿意支付 3 美元。这为互联网打开了全新的商业模式世界。
## 十二、经济账：机会规模被低估了一个数量级

**目前最大的问题：每个人都在试图弄清经济账，但他们把这个机会的规模低估了至少一个数量级。**

- 人们会想出全新的模型，因为每次新技术出现都会这样。
- 华尔街的人试图用旧世界的方式证明 GPU 和 token 的合理性。他们把收入世界看作线性的，试图证明所有费用的合理性。
- 这与 PC 时代的问题一样：人们认为 PC 市场有限，因为他们只看到了 MIPS 的消耗。他们没有想过把 MIPS 放到每个桌面上会发生什么。而且人们以为软件随 MIPS 而来，没人想到可以单独卖软件——有一个人想到了，结果是个好主意。
- 云时代也一样：人们说“我们要把所有服务器业务（每年 6 万台）移到别人的数据中心”，然后瓜分价格。没有人说“人们会使用一千倍的资源”。
- 这让我非常恼火：华尔街的模型是**零和思维**，认为公司的花费是一个固定饼图。
**Salesforce 的例子：**

- CRM 业务是每年 20 亿美元，你得买服务器、Oracle 许可证、多年的部署和咨询。如果你能让销售人员单独注册，没有摩擦，每个人都会注册。
- 毫无疑问，AI 也会发生同样的事情。
**我投资 10 年了，有 240 家公司。基础设施公司都在过去 6 个月里呈指数级增长。**

- 为什么？因为现在写的软件比以往任何时候都多。不是因为企业客户，只是因为基础设施层的消耗量巨大。
- 有更多的软件，更多的代理，就会有更多的计算资源消耗。
- 我们甚至还没到每个人手机都成为 AI 大消费者的地步。一旦每个人的设备上都在消耗 AI，用量会再增加十亿。
**关于微支付（micropayment）：**

- 每项技术都伴随着微支付，人们总认为你可以得到零点几分钱。但在企业端，人们会批量购买，因为更便宜、更容易，而且你希望可预测性。
- 我喜欢这个想法：这是第一次，代理不关心小额交易的摩擦。所以你可以把资源放在付费墙后面，有东西愿意为此付费。
- 世界已经建立了基础设施，将这些支付聚合为客户服务的高效方式。由于 token 目前是 COGS 的重要组成部分，这正在推动行业走向**基于使用量（usage-based）**的定价。
- 我们从永久许可转向订阅制时经历了巨大变化，现在正在经历同样的变化，走向基于使用量。
**工程计算预算的讨论将是最疯狂的：**

- 你应该把多少工程费用分配给 token？根据你读的推特，可能是 1%，也可能是 100%。
- CFO 们必须知道答案。华尔街会逼他们给出一个数字并坚持执行，然后他们会因此被解雇。
- 科技公司的研发费用占收入的 14% 到 30%。计算成本是工程团队成本的 2 倍还是多 3%，这就是全部每股收益（EPS）的差别。
- 我愿意为此牺牲几位 CFO。
**原因是我们现在不知道答案。这发生在互联网带宽、真空管、晶体管、程序员身上。**

- 但我们从未有过这样的时刻：组织中的每个最终用户都能完全弹性地为他们自己启动资源。
- 这与 2000 年代初的云非常相似：从资本支出（capex）到运营支出（opex）再到无限支出。CFO 们会说：“我们是农业公司，只懂资本支出”或“我们是运营支出公司，我们喜欢云”。
- 不要低估**本地计算引擎**作为所有这一切的泄压阀。什么时候会发生？看看历史，它走向了客户端（80 年代），然后又回到了云端（10-15 年）。最近很多人早上醒来会说：“我们要把一些关键但静态的工作流搬回本地” —— 对于 AI 来说，这是真的。
**作为工程领导者，你现在该如何管理计算预算？**

- 长期来看，这个问题会消失。初创公司会烧光可用资本，假装这不是问题。大公司会害怕得冻结，什么都不做。中间地带会有人愿意下注，并成为领导者。
- 你不希望你的工程师现在就得考虑计算预算，因为我们还在开发……哦，那句话会让你超过预算。
- 我们关于云的讨论已经持续了 15 年。2016-2018 年，有一整类公司基本上是开发者的仪表盘，因为云和 API 花费失控了。
- 但这次不同。每个工程师的每个提示，你都得决定：要一个长时间运行的提示？一个长时间运行的代理？要并行化吗？你对浪费 token 的容忍度是多少？
- 我现在认为我们应该浪费很多 token，因为这意味着我们在尝试新事物。你的工程主管应该为并行运行 10 个实验而高兴，即使浪费了 90% 的 token，因为你会选出一条成功的路径。
- 还是你想告诉团队：在你做之前，先设计完美的系统？
- 我们有一大堆开放问题。就在录这期节目的时候，人们正在因为新的 Claude Code 最大计划而抓狂，因为他们提示了三次就被阻止了。
- 这将是一个非常真实的话题，直到我们能找到建设数据中心容量的方法。
**但这是不同的问题。假设我们建设更多容量，价格会下降，因为我们现在的定价是基于有限容量。这会被解决。**

- 我为一个必须立即决定哪 17 个人这周不能再获得 token 的人感到难过。
- 但这一切都会消失。毫无疑问，在十年时间框架内会消失。最大的原因是：如果你付给一个企业销售人员一百万美元一年，你得问他们的工具值多少钱。如果你付给工程师 X 美元一年，他们的工具绝对物有所值。
- 如果是短期容量问题，那是不同的问题在驱动价格。
- **大数定律**会解决这个问题。最终你有足够多的工程师，他们使用这么多计算，但我们正处于一个过渡期。大多数人认为两年前在 AI 上的花费水平（就是一个聊天机器人）是合适的，但他们错了。
**你嘲笑真空管的例子，但曾经人们认为整个达科他州会布满真空管仓库，人们穿着旱冰鞋在过道里跑来跑去更换真空管，只是为了打二战。然后有人说：“晶体管怎么样？”**

- 我们将会有一个“晶体管时刻”。可能是更多的供应，也可能是一个根本性的算法变化，或者硬件变化。
- 现在每个人都在谈论 token，就像 IBM 和大型机时代人们谈论 MIPS 一样。有一天，IBM 每年都在以更少的美元卖出更多的 MIPS，他们自己都没意识到，仍然按 MIPS 定价，直到有人指出他们正处在一个递减曲线上，因为他们制造 MIPS 的速度超过了收费的速度。
- **这（价格下降）是肯定会发生的。**
- 保证。
- 我说得很硬核，听起来好像我知道自己在说什么。
- 我实际上可能相信它。
