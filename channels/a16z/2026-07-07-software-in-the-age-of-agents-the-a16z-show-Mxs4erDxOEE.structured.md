# 软件在代理时代：无头化、粘性企业软件与创业新机遇

在 a16z 播客中，主持人邀请到企业团队合伙人**塞玛·安布尔**（Sema Amble）和董事会合伙人**斯蒂芬·索诺斯基**（Steven Sonowski），围绕塞玛撰写的文章《软件正在失去头部吗？》（*Is Software Losing Its Head?*）展开对话。两人深入探讨了无头软件（headless software）的含义、Salesforce 宣布无头 360 的行业影响、企业软件粘性的真实来源、SAP 等巨型系统的不可替代性，以及 AI 代理（agent）时代创业公司面临的真正机会。以下为讨论精华。

## 一、无头软件：概念与现实

主持人首先请塞玛解释什么是无头软件，它带来了哪些变化。

塞玛指出，无头软件本身并非新术语，但最近成为公众讨论的热点。Salesforce 宣布推出“无头 360”，这更像是一个经典的营销动作，意在承认正在发生的转变：传统软件围绕人类访问构建，工作流程用于捕获数据；但在代理世界，用户界面（UI）可能不再必要，因为代理不通过 UI 访问软件。无头的核心在于**数据、逻辑等底层内容才是真正的价值所在**，而不是顶层的流程追踪。

主持人追问，这究竟是一个重大变化，还是对现有 API 的重新包装？几个月来有何实质改变？

塞玛认为，在 Salesforce 的语境中，可能没什么真正改变——他们的“无头 360”产品其实就是一直存在的 API 现在被重新命名。API 已经存在很久了。但更广泛的趋势是，Salesforce 以及其他公司都在思考如何为代理世界构建自身。如果一个代理需要访问 CRM 中的数据，它是通过 UI 还是使用 API？Salesforce 的声明只是在说：“我们理解变化正在发生，我们提供一个无头版本供代理交互。” 另一个例子是**Notion**，它也推出了无头产品，而且可能更合理——Notion 用户整体上更懂技术、更倾向于作为构建者去 “代理化”。Notion 只是众多正在探索“我该暴露什么 API”的公司之一。

斯蒂芬补充说，我们正处于定义的混乱期。技术浪潮的一部分就是用新词包装过去做过的事。“代理”从这个角度看，就是一个运行时间很长、可能不会结束的程序——我们以前称之为 bug，现在却成了最酷的新功能。但更重要的是区分代理和 API 到底在做什么不同的事。斯蒂芬将代理的行为分为三类：

- **查找**：这是最轻量级的，大多数系统都擅长，很多新宣布的无头代理 API 其实就是提供一个新的查找接口。
- **执行**：要真正改动系统记录，就得冒用具体人员的身份、使用他们的凭证，牵涉到付费席位、权限等一系列企业软件难题。
- **分析**：涉及查询多个系统、迭代、路由到不同模型获取答案并对比。这种场景非常适合代理，但也是**幻觉**问题最突出的地方，因为必须验证分析每一步是否正确。

因此，当人们谈论“无头”和“代理”时，往往混为一谈，必须明确所指的具体层面。

## 二、历史粘性：是什么让软件难以替代？

主持人将话题引向历史：“是什么让软件保持粘性？代理又如何开始颠覆这一点？”

塞玛认为，很多粘性源于**人类交互的方式**。UI 因为读写频率、操作频率、下游工作流、以及所有围绕软件形成的未文档化的标准操作流程（SOP）而变得粘性。例如 CRM 的粘性在于销售代表需要频繁进出、习惯使用、团队依赖，甚至财务和营销也依赖其输出。另外，**单点真相**也非常重要——比如在 ERP 或薪资系统中，法律和合规要求每个数字都准确无误。这些因素共同驱动了粘性和持久性。

斯蒂芬补充说，最粘性的做法其实是**从客户那里收钱**。一旦客户在付钱，他们很难停止付款，也很难想到停止后该怎么办。真正的粘性来自于软件在某个地方被使用。当你深入挖掘原因时，你会发现不同角色有不同的答案：合规人员关心是否符合法规，管理员关心新用户导入，终端用户关心肌肉记忆和快捷键。最好的方法是把软件卖进去，然后从市场反馈中归纳粘性因素。斯蒂芬以**Microsoft Outlook**为例：任何试图取代它的产品很快都会学到“委托访问”和“多所有人日历”这些隐藏的粘性点——这些功能从未在规划会议上被特意设计成粘性点，却成了企业迁移的最大障碍。

塞玛认同惯性是强大力量，同时也指出：当软件触角遍布整个组织、人们长期使用并付费，它就会渗入工作方式，难以拔除。

## 三、SAP 的不可替代性：逻辑比数据更重要

斯蒂芬谈到塞玛合著的一篇关于**SAP**的文章，称 SAP 是粘性软件的终极代表。唯一比 SAP 更粘的是保险公司后台的系统——有些已经运行了 50 到 75 年，永远无法替代。而 SAP 本身固化了整个公司的运营规则：如果你从大型汽车制造商那里拿掉 SAP，公司几乎就蒸发了，因为公司的业务、流程、决策全部编码在这个产品中。

塞玛强调了一个当前的误解：有人认为可以用 PostgreSQL 数据库加 API 就取代 SAP。这**完全错误**。SAP 中蕴含的业务逻辑和所有其他东西远远重要于数据恰好在某个数据库里这一事实。

斯蒂芬回忆起 1990 年代末的**拉里·埃里森**（Larry Ellison）关于企业软件定制的著名 rant。他说企业软件客户定制化很愚蠢，应该只用 80% 的方案。但现实是，以汽车行业前十大公司为例，它们都造车，真正差异在于**内部运营和流程**——所有这些都装在 SAP 里。公司是靠人坐在会议室里盯着 SAP 屏幕运行的，而福特、丰田、通用、戴姆勒之间的不同，正是它们选择了哪些屏幕、做了哪些定制。

斯蒂芬还分享了一个故事：早期推广 Excel 时，在 Goldman Sachs，对方说：“我们从 Excel 中赚的钱比你们多。”他们通过构建插件、编写代码、定义工作流，把 Excel 变成了高度差异化的工具。这引出了对“**vibe code**”（用直觉随意编码）的警示：人们严重低估了用它进入企业软件的难度。塞玛也提到，有一家上千人的公司自建 Salesforce 实例，以为字段和导入很容易，但最难的是决定捕获什么、如何匹配组织流程，以及长期维护。你当然可以“vibe code”一个 CRM，但没人维护的项目很快会过时。

## 四、代理时代：上下文、异常与长尾

塞玛指出，AI 当前的主要应用是让遗留系统更易用、更可对话。你不需要登录 SAP 界面或写 SQL，而是通过自然语言查询、自动生成自定义报告。这就是**UI 变得可选**的体现，但数据和业务逻辑仍然必须存在。

斯蒂芬进一步指出企业软件的一个普遍现实：**它几乎总能做到用户想做的事，用户只是不知道怎么让软件做到**。SAP 能生成任何报告，但用户可能没有权限或找不到路径。因此，企业软件最常用的功能反而是“导出到 Excel / CSV / PDF”。现在，语言模型让消费这些导出数据变得前所未有的容易，也为新公司创造了机会——就像当年 Excel 文档催生了 CRM 公司（先是 Siebel，后是 Salesforce）。

塞玛提到，对于代理而言，不仅需要结构化数据，还需要**上下文**——那些未在 CRM 字段中记录的边缘案例、处理规则、权限策略、各地差异等等。这些上下文往往是别人脑子里的东西。代理必须先观察足够多的交互，才能开始理解那些例外。

斯蒂芬回应：“几乎所有企业里有趣的事情都是例外。” 他举例 Amazon 的客户服务：他们通过 AI 学习如何处理退货，并直接用数据改进仓储和描述。Amazon 的哲学是“决定偏向客户”，AI 驱动了全新的异常处理方式。当自动化提升后，新的场景会不断涌现，**长尾不是变短，而是以不同方式变长**。

主持人问：“自动化长尾是否仍然是最大的困难？”

塞玛和斯蒂芬都同意，这只是部分难题。还有**权限管理**、代理间交互等问题亟待解决。斯蒂芬补充说，人们常犯的错误是线性地看待生产率：认为现有工作量减少就意味着自动化完毕，却忘了生产率会带来新需求。放射科医生欢迎 AI，结果却出现了放射科短缺；法律合约会因 AI 变得更长、更复杂，从而创造更多法律服务需求。**创新是不断再造的成长蛋糕，而非静态的饼**。

## 五、MCP 与中间件困境

主持人提到斯蒂芬去年写的一篇文章，关于 **MCP（模型上下文协议）** 以及它与早年微软的中间件诉讼的关系。斯蒂芬解释，MCP 非常符合工程师的理想架构——希望每个工具都有清晰的命令行式 API。但现实是，**没有软件愿意被上层的中间层去中介化**。任何软件的提供商都不希望自己沦为简单的底层数据存储。客户也不愿意从众多供应商那里拼凑场景，因为系统稳定性取决于最不稳定的那个环节。

塞玛以实践说明：Workday 虽然有 API，但你无法干净地提取所有数据并完全脱离 Workday 运作——它们故意让文档难以获取且不暴露全部端点。对于大型企业，即使是竞争对手“打成平手”也算是胜利，因为它们可以把自己捆绑进现有产品中。因此**中间层非常不稳定**，虽然网络架构图很好看，但实际很难持久。

## 六、创业公司的三种路径

塞玛总结了当前软件市场中面临的三种选择：

1. **使用现有厂商的代理层**：比如 Salesforce 推出 Agentforce，你可以把代理都建在它上面，但 Salesforce 并不想只做后台数据库，因此效果参差不齐。塞玛不看好现有巨头能自己建出优秀的代理。
2. **完全 DIY**：控制权最大，但极难。相当于在病人还活着的时候做开胸手术，所有业务逻辑、权限、协作都要从头重建，对于初创公司几乎不可能。
3. **作为补充层，与现有系统并行**：这是当前很多成功的 AI 创业公司走的路。它们吸收数据、在后台构建，提供增强体验，允许用户在现有数据上运行代理，而不必扔掉积累的规则。同时创建新的记录系统（如语音代理收集新数据、转录文档等），逐步观察业务运作，未来可能以系统化方式替换后端记录系统。

斯蒂芬强调，**绝对不要正面攻击一个已有的大类别**。最大的机会永远是观察现有企业软件地图，找到两个大玩家之间的空隙，用全新的方式做事。就像 HTTP 和 HTML 取代客户端-服务器那样——不是因为它做到了所有事情，而是它用了截然不同的方法。大玩家会拼命把 AI 捆在现有产品上，不会自毁产品线。如果你在它们之间创造新方式，你就不用回答 20 年积累下来的框架问题，而是可以自己定义存在理由。

## 七、网络效应与企业内部传播

主持人问：“企业软件是否可能实现网络效应？” 斯蒂芬认为，企业软件中真正的网络效应往往是**公司内部的**。就像 Excel 最初在金融界病毒式传播的例子——有人在工作现场展示新功能后，周围人立刻追问“你是怎么做到的？”2025 年的今天，Chat 正在企业内部复制同样的迭代：员工看到新工具让自己的工作更好，就会自发性地推广。塞玛补充说，**连接两个不同职能的翻译层**（如销售与财务之间的信息传递）也是全新的机会所在。软件过去只卖给单一部门，但跨部门的手工整合现在可以被 AI 填补。

## 八、总结：在指数变化中找到自己的路

在技术变革中，没有人能真正理解指数增长发生时的全貌。起步阶段容易线性格外推。但创业公司如果能够看到现有大玩家犹豫不前的领域，避开正面碰撞，并用 AI 原生方法解决两个业务功能之间的沟通断层，就可能在代理时代占得先机。正如斯蒂芬所言：“**自动化的长尾永远不会变短，但你可以用新的方式让它变得更长、更有趣。**”
